vLLM Cheat Sheet

Serve flags, CPU knobs, API calls, and metrics used across the course. Canonical: docs.vllm.ai.

Serving

# native
vllm serve Qwen/Qwen3.5-0.8B --max-model-len 4096 --dtype bfloat16

# docker (our CPU image; args after image go to `vllm serve`)
docker run -d --name vllm -p 8000:8000 \
  -v hf-cache:/root/.cache/huggingface \
  -e VLLM_CPU_KVCACHE_SPACE=4 \
  vllm-cpu:local \
  --model Qwen/Qwen3.5-0.8B --max-model-len 4096 --dtype bfloat16

Flags that matter here

FlagEffect
--modelHF repo ID; downloads to HF cache on first run
--max-model-lenContext cap → bounds per-request KV memory
--dtype bfloat16Weight/activation precision (M4-friendly)
--portAPI port (default 8000)
--api-keyRequire bearer token on all endpoints
--served-model-nameAlias returned by /v1/models
--enable-prefix-cachingReuse KV across shared prefixes (pairs with session routing)
--max-num-seqsMax concurrent sequences in a batch
--enforce-eagerSkip graph compile; faster start, slower serve — debugging aid
--tensor-parallel-sizeShard model across N GPUs (GPU world; =GPU count)

API calls (OpenAI-compatible)

# list models
curl -s localhost:8000/v1/models

# chat
curl -s localhost:8000/v1/chat/completions -H "Content-Type: application/json" -d '{
  "model": "Qwen/Qwen3.5-0.8B",
  "messages": [{"role":"user","content":"Hello"}],
  "max_tokens": 60,
  "temperature": 0.7
}'

# streaming (tokens as they generate — what UIs use)
curl -sN localhost:8000/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"Hi"}],"stream":true,"max_tokens":30}'

# health + metrics
curl -s localhost:8000/health          # 200 = engine up
curl -s localhost:8000/metrics | grep ^vllm:
# python — official openai client, one line changed
from openai import OpenAI
client = OpenAI(base_url="http://localhost:8000/v1", api_key="unused")
r = client.chat.completions.create(model="Qwen/Qwen3.5-0.8B",
    messages=[{"role":"user","content":"Hello"}], max_tokens=60)
print(r.choices[0].message.content)

Key metrics (Prometheus names)

MetricTypeMeaning
vllm:num_requests_runninggaugeIn the active batch now
vllm:num_requests_waitinggaugeQueued — growth = saturation
vllm:kv_cache_usage_percgaugeKV block usage 0–1
vllm:prompt_tokens_totalcounterPrefill tokens → wrap in rate()
vllm:generation_tokens_totalcounterDecode tokens → rate() = tok/s
vllm:time_to_first_token_secondshistogramTTFT → histogram_quantile()
vllm:time_per_output_token_secondshistogramTPOT / inter-token latency
vllm:e2e_request_latency_secondshistogramWhole request wall time
vllm:request_success_totalcounterCompleted OK (label: finished reason)

Names drift between versions — verify against your build: curl -s :8000/metrics | grep ^vllm:. Full list: metrics docs.

Glossary

Continuous batchingBatch recomposed every decode step; new requests join mid-flight
KV cacheStored attention keys/values for generated context; the memory hog
PagedAttentionBlock-based KV allocation via indirection table — no fragmentation
PrefillProcessing the prompt (parallel, compute-bound) → produces TTFT
DecodeToken-by-token generation (memory-bandwidth-bound) → produces TPOT
Prefix cachingReusing KV of identical prompt prefixes across requests
TTFT / TPOTTime to first token / time per output token
