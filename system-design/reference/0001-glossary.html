System Design Glossary

The course vocabulary. Every lesson uses these terms exactly as defined here. Grows as lessons are added.

Performance

TermDefinition
LatencyTime for one request to complete, measured from the client's perspective. Units: ms. Lower is better.
ThroughputHow many requests a system completes per unit time. Units: requests/second (RPS or QPS), or tokens/second for LLMs. Higher is better.
Percentile (pN)The value below which N% of measurements fall. p50 = median; p99 = the latency that 99% of requests beat — and 1% suffer.
Tail latencyThe slow end of the distribution (p99, p999). Dominates user experience at scale because someone always lands in the tail.
SLOService Level Objective — a target like "p99 latency < 500 ms". What you design against.
SaturationThe state where a resource (CPU, GPU, queue) is near capacity; latency explodes as load approaches saturation.

Estimation

TermDefinition
Back-of-envelope estimationFive minutes of powers-of-ten arithmetic that turns a prompt's numbers into machines. Within an order of magnitude is success; a third significant digit is theater.
DAUDaily Active Users — the top of every estimation pipeline: DAU × actions/user ÷ 10⁵ seconds/day ≈ average QPS.
Peak factorMultiplier from average to peak traffic, ×2–10. A business fact, not a math constant: social evenings ~2–5×, retail promos 10×+.
Read:write ratioReads per write — 10:1 typical, 100:1 for feeds. The single number that decides whether you buy caches or queues.
Replication factorNothing durable is stored once: multiply raw storage by ×3 for replicas (or ~×1.5 with erasure coding), and say which you assumed.
Erasure codingDurability via parity fragments instead of full copies — ~1.5× storage overhead instead of 3×, paid for with slower recovery.
Hot setThe data actually being read right now. If it fits in RAM (a few hundred GB), cache it whole and the design changes shape.
Metadata/blob splitSmall hot rows in a database, large immutable blobs in object storage behind a CDN. The shape of every photo, video, and document system.
False precisionSaying "13,889 QPS" instead of "~15k". Precision beyond one significant figure is noise wearing rigor's clothes.

Traffic & Edge

TermDefinition
DNSThe hierarchy that maps names to IPs, cached at every step. Usually free, slow to change (TTL), and itself a dependency that fails.
DNS routingWeighted (canaries), latency-based (nearest region), or geolocation (data-residency law) answers. Good for geography and coarse shaping; too slow for failover.
CDNContent Delivery Network — copies of static content on hundreds of edge servers, so the request never crosses the ocean at all.
Push vs pull CDNPush uploads content at release time (fast first hit, pays for storage everywhere); pull fetches from origin on first miss (fits long tails).
Versioned URLsShip app.9f3ab2.js, cache forever, never purge. The professional answer to CDN invalidation, which is unreliable at the worst moments.
Load balancerOne virtual IP (VIP), many real servers. Earns its keep the moment a backend dies.
L4 load balancingTransport layer: sees IPs and ports, forwards packets near line-rate. Fast and blind.
L7 load balancingApplication layer: terminates TLS, reads the HTTP request, routes by path/header/cookie. Smart, and you pay CPU for it.
Least-connectionsRoute to the backend with fewest in-flight requests. The self-correcting choice when request costs vary — and they always do.
Sticky sessionsPinning a client to one server so in-memory state survives. How small systems grow into unscalable ones — it makes servers non-interchangeable.
Stateless / shared-nothingNo request depends on one box's local state; sessions live in Redis, files in object storage. The prerequisite for horizontal scaling.
Health checkPeriodic probe (GET /health) gating rotation membership. Detection time ≈ interval × failure threshold; over-eager checks cause flapping.
Floating IPAn IP the passive balancer claims (via VRRP/keepalived) when the active dies — the standard answer to "who balances the balancer?"
Reverse proxyA server fronting your app servers: TLS termination, compression, rate limiting, a stable public face. A load balancer is just one that balances.

Consistency & Replication

TermDefinition
CAP theoremNot "pick two" — partitions are forced on you. During one, each request chooses: refuse (CP) or answer possibly stale (AP). Operations choose, not systems.
PACELCCAP's completion: if Partitioned, Availability vs Consistency; Else, Latency vs Consistency. The trade you make every day, not just on bad days.
Eventual consistencyStop writing and replicas converge, on no particular deadline. A weak promise — never say it without naming the anomaly you're accepting.
Strong consistencyLinearizable: every read sees the latest write, as if one copy existed. For payments, inventory decrement, unique usernames — bought with write latency.
Read-your-writesA session guarantee: you always see your own writes, even if strangers' replicas lag. Fixes the "my comment vanished" bug via leader pinning.
Monotonic readsTime never runs backwards across your reads. Pin a session to one replica so a user may lag but never time-travels.
Leader–follower replicationAll writes to one leader; followers replay the change stream and serve reads. The default RDBMS topology.
Replication lagThe window where followers are behind the leader. Every follower read races it; every async failover can lose it.
Multi-leader replicationSeveral nodes accept writes (multi-region, offline clients). Buys locality; pays with write conflicts — never propose it without a conflict story.
Last-write-winsConflict "resolution" by timestamp. Simple, and it silently destroys data — fine for a view counter, unforgivable for orders.
CRDTData structures mathematically guaranteed to converge regardless of delivery order. The machinery under collaborative editors.
Quorum (R + W > N)If read set and write set must overlap, a read cannot miss the latest acknowledged write. Pigeonhole arithmetic, not probability.
FailoverPromoting a replacement when the leader dies. The wounds: lost in-flight writes, premature promotion, and timeouts that are only ever tuned, never correct.
Split-brainThe "dead" leader wasn't dead, and now two leaders accept conflicting writes. Prevented by quorum election and fencing (STONITH).
RTO / RPORecovery Time Objective (how long until recovered) and Recovery Point Objective (how much data you may lose — for async replication, the lag at crash time).
NinesAvailability targets as downtime: 99.9% = 8.8 h/year, 99.99% = 53 min. Series dependencies multiply availabilities down; parallel redundancy multiplies failure probabilities down.

Databases

TermDefinition
ACIDAtomicity, Consistency, Isolation, Durability — the transaction contract. Needed when several rows must change together and a partial state is unacceptable.
BASEBasically Available, Soft state, Eventually consistent. Not "no guarantees" — a different contract: availability and scale, bought with staleness.
B-treeThe index structure: wide, shallow, sorted. Depth 4 reaches billions of rows; a point read is ~one 150 µs SSD touch.
Write amplificationEvery index is another B-tree updated on every write. Indexes are prepaid reads, billed to the write path.
Covering indexAn index containing every column the query touches — the engine never visits the table at all.
Leftmost-prefix ruleA composite index (user_id, created_at) serves equality-then-range, and is useless for a range on the second column alone. Phone books work the same way.
Scaling ladderRead replicas → caching → federation → sharding, in that order. Each rung is cheaper and more reversible than the next; say why you're stopping.
FederationSplit by function: users DB, catalog DB, orders DB. Buys independent failure domains; costs cross-database joins.
ShardingSplitting one table's rows across machines by a shard key. The only rung that scales writes — and the last resort, priced accordingly.
Shard keyThe column that maps rows to shards. Choose it by asking what the dominant query filters on — then defend it against the hot key.
Hot keyThe celebrity problem: hashing balances keys, not load, and Zipf traffic puts 100M followers on one shard. Salt it, isolate it, or route it differently.
Snowflake IDStructured ID (timestamp + worker + sequence in 64 bits), because every shard's auto-increment happily generates the same colliding 1, 2, 3…
DenormalizationCopying data so a three-way join becomes one lookup. Paying writes to buy reads — the right trade whenever reads dominate, which is nearly always.
Materialized viewDenormalization with the bookkeeping handled by the database: a precomputed query result, refreshed on schedule.
Fan-out on writePrecompute each follower's feed at post time; reads become one lookup. Twitter's hybrid: fan out for normal accounts, pull-on-read for celebrities.
Key-value storeget/put/delete and nothing else — that austerity is the point. Redis for caches and sessions, sub-millisecond, no query planner in the way.
Document storeThe whole aggregate in one JSON document, indexed and queryable. Schema evolves per document; if you constantly join, you chose wrong.
Wide-column storePartition key routes, clustering key sorts, LSM trees append. Cassandra swallows write-heavy firehoses — but the primary key is the query plan.
Graph databaseNodes and edges with direct pointers: "friends of friends of friends" is a walk, not N self-joins. Use only when traversal is the workload.
LSM treeLog-structured storage: writes append sequentially, compaction tidies later. Why wide-column stores love write-heavy time series.
Consistent hashingHash nodes and keys onto one ring; a key belongs to the first node clockwise. Adding a node moves ~1/(N+1) of keys, not the ~83% that mod N moves.
Virtual nodesGive each physical node hundreds of ring positions so arc sizes stop being a lottery — 3.2× overload smooths to ~1.1×, and a dead node's load disperses.
Polyglot persistenceOrders in Postgres, sessions in Redis, events in Cassandra, fraud in Neo4j — each store earned its place through a named pressure. Start SQL; leave when forced.

Caching

TermDefinition
Cache-asideApp checks cache; on miss reads DB and fills. On writes, delete the entry rather than update it. The default pattern; Facebook runs memcache this way.
Write-throughEvery write hits cache and DB synchronously. Never stale, always warm after writes — paid for with slower writes and caching data nobody reads.
Write-behindWrite to cache, ack, flush to DB later. Fast, batches beautifully — and loses data if the cache dies before the flush. Treat with fear near money.
Refresh-aheadRe-fetch hot entries before they expire so users never see the miss. Strongest for keys you can predict — which is exactly its limit.
TTLTime-to-live: an expiry clock on each entry. A freshness bound, not a space policy — and "prices may be 60 s stale" said aloud is what makes it engineering.
LRU / LFUEvict least-recently vs least-frequently used. LRU loves temporal locality but a scan flushes it; LFU resists scans but forgets yesterday's hot keys slowly.
Hit rateFraction of reads served from cache — simultaneously a latency number and a capacity number. Size at the knee of the curve; Zipf skew front-loads the value.
Zipf distributionThe k-th most popular key gets ~1/k of the traffic. Caching works because traffic is unfair; on uniform traffic a small cache is nearly useless.
Cache stampedeThundering herd / dogpile: a hot key expires and every in-flight request recomputes at once — a self-inflicted 200× spike on your hottest data.
SingleflightPer-key locking: one request recomputes, the rest wait for its result. Instagram's "promises", Facebook's leases — the stampede fix with a guarantee.
TTL jitterRandomize expiries so thousands of keys never expire in the same instant. The fix for cache avalanche; it decorrelates reloads, it doesn't reduce them.
Cache penetrationRequests for keys that don't exist miss forever and hammer the DB — accidentally or adversarially. No amount of capacity helps a key with no value.
Negative cachingCache the "not found" itself, with a short TTL. Pair with a Bloom filter ("definitely not present") for hostile or vast keyspaces.
Cache invalidationOne of the two hard things: a message about a write, delivered to a different system, racing the reads. Event-driven for expensive staleness, TTL as backstop.

Async & Communication

TermDefinition
Message queueBuffers data between services so producers and consumers are decoupled (Kafka, RabbitMQ, SQS). Absorbs bursts; does not create capacity.
Task queueOne level up: manages the execution of background jobs — workers, retries, schedules — on top of a broker (Celery, Sidekiq).
Queue depthThe best early-warning signal you have: depth ÷ drain rate is next-minute latency. Latency percentiles report the past; depth predicts.
BackpressurePushing overload back toward the source instead of absorbing it into an unbounded queue. Bound every queue, reject early, tell clients how to come back.
At-least-once deliveryThe honest default: the broker redelivers on ambiguity, so duplicates happen. The standard pairing is at-least-once plus an idempotent consumer.
Exactly-once deliveryMostly a lie above the transport. Real within Kafka's own boundary; the moment you touch the outside world, it's at-least-once plus idempotency again.
Idempotency keyA unique key per logical operation; the server records processed keys and duplicates return the stored result. How Stripe makes payment retries safe.
Exponential backoff with jitterWait 1 s, 2 s, 4 s… plus a random smear. The jitter is the half people forget — without it, synchronized retries re-create the outage forever.
Retry amplification3 retries × 3 layers = 27 requests from one tap, exactly when the bottom service can least serve them. Retry at one layer, cap attempts, budget retries.
Little's lawL = λW: items in system = arrival rate × time in system. Sizes queue bounds, worker pools, and wait forecasts, distribution-free.
Utilization hockey stickQueue wait scales like ρ/(1−ρ): flat until ~70–80% utilization, then 90%→95% doubles the wait. "We're at 95% and fine" is a lit fuse.
Load sheddingDropping the cheapest-to-lose work first, by pre-declared priority: telemetry before recommendations before search, never checkout.
TCP vs UDPTCP: ordered, reliable, and willing to wait for late packets. UDP: just sends. Choose UDP when late data is worse than lost data — live media, game state.
REST vs gRPCREST (HTTP + JSON) at the public edge for loose coupling and curl-ability; gRPC (HTTP/2 + protobuf) inside the datacenter for typed contracts and streaming.

Interview Process

TermDefinition
4-step frameworkClarify (5–8 min) → estimate (~5) → high-level design (~15) → deep dive (~20). You drive; silence reads as lost, an agenda reads as seniority.
Functional requirementsWhat the system does: add to cart, charge the card exactly once, email the receipt.
Non-functional requirementsHow well it must do it: p99 < 500 ms at 30k QPS, 99.99% availability, no double-charges ever. The grading contract for the hour.
Out-of-scope listWritten exclusions ("auth and fraud are existing services"). Scoping is not dodging — it is what senior engineers do to real projects every week.
Spending a numberEvery estimate must justify a decision: "50:1 reads is why this cache exists." Numbers computed and never referenced again are ritual, not engineering.
Scatter-gatherFanning a query out to every shard or service and merging the results — as slow as the slowest responder, and the tax on any query not anchored on the shard key.
SignpostingEnumerate → choose → because: "three options, I'd pick B because…". Proves you saw the design space and can still commit.
Deep diveThe largest time budget for a reason: two components dissected under load — bottleneck and failure mode named unprompted — beat ten components named.

LLM serving (preview — deep dive in Phase 3)

TermDefinition
TTFTTime To First Token — latency until the first output token arrives. What users feel as "responsiveness".
TPOTTime Per Output Token — average gap between tokens after the first. What users feel as "typing speed".
KV cachePer-session attention state kept on the GPU (~2 × layers × kv_heads × head_dim × 2 bytes per token). The memory that decides how many sessions fit per GPU.
Prefix cachingRouting requests that share a prompt prefix (consistent-hashed) to the worker already holding that prefix's KV cache. The system prompt is the ultimate hot key.
