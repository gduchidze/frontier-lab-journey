Scaling to Millions of Users — the Evolution Rep

Every rep so far designed one system at one scale. The capstone rehearses the other axis: one system, every scale — the canonical single-box-to-multi-region evolution that underlies every scaling story you will ever tell.

Mission tie. This capstone rehearses the SHAPE of every scaling narrative — an inference platform evolves through exactly the same stages with GPUs in place of app servers: one GPU box → LB + replica pool → KV-cache tier → sharded model workers → multi-region serving.
Rep protocol — this rep is different Instead of one 40-minute design, narrate the EVOLUTION aloud stage by stage before reading: start from "everything on one box" and repeatedly ask "what breaks first at 10× load?" — name the broken component, the fix, and the new problem the fix introduces, until you reach multi-region. Only then read the lesson, and log every stage where your narration diverged as deltas.

1 · The rep, and why interviewers love it

"Scale this site from zero to millions of users" is the interviewer's cheapest X-ray of your judgment. It does not test whether you know microservices, Kafka, or Kubernetes — it tests whether you scale by need, not by fashion. The candidate who deploys sixteen services and a service mesh for 200 QPS fails the same way the candidate who keeps everything on one Postgres at 50k writes/s does: both ignored the load numbers. Premature microservices is a named smell (lesson 0010); so is premature sharding, premature multi-region, and premature anything.

The winning shape is a ladder: at each stage, name (a) what breaks first at the next 10×, (b) the minimal fix, and (c) the new problem the fix introduces — because every fix introduces one. That three-beat rhythm, repeated six times, is the answer. The Primer's Scaling on AWS solution walks this exact ladder; this lesson compresses it into interview-ready form.

2 · Stage 0 — a single server, and why it is correct

Everything on one box: web server, application code, database, cron jobs, files on local disk. One vertical scaling lever — buy a bigger machine: more cores, more RAM, faster disks. This is not a naïve starting point; it is the correct one. One box means no network hops between tiers, no distributed failure modes, no consistency questions, trivial deploys, and total system comprehension in one head. YAGNI applied to infrastructure.

Its real ceiling: a dynamic page costs milliseconds of CPU, so one well-tuned box serves on the order of hundreds of QPS — and, more fatally, it is a single point of failure and vertical scaling has a hard top (the biggest rentable machine) with cost rising faster than capacity. The interview line: "I start on one box and I stay there until a measured number forces me off — the first crack is usually the DB and the web server fighting for the same CPU and memory."

3 · Stage 1 — split web and DB tiers; the load numbers

First crack, first fix: move the database to its own machine, put DNS in front of the web box. Now each tier scales and fails independently — web CPU spikes no longer evict DB pages from memory. New problem introduced: a network hop between app and DB (add ~1 ms and a new failure mode: the link itself), and now there are two single points of failure instead of one.

Estimate checkpoints — the loads at which each stage cracks. Assumptions stated: a dynamic page costs ~5–10 ms of CPU on one core, so an 8-core box serves a few hundred QPS; ~90% of traffic is reads.

Stage 0 cracks  ~200 QPS      one box does web + DB + everything
Stage 1 cracks  ~500 QPS      DB box saturates while web box idles
Stage 2 cracks  ~2–5k QPS     single DB primary is the wall behind N web servers
Stage 3 cracks  ~20–50k QPS   writes outgrow one primary despite cache + replicas
Stage 5 target  100k+ QPS     sharded writes, NoSQL hot paths
Stage 6 target  ~1M QPS       multiple regions, each a full stack

These are powers-of-ten guardrails, not facts — say your assumptions aloud and adjust. The point is that each stage has a number, and you transition when the number arrives, not before.

4 · Stage 2 — load balancer + N stateless web servers

The web tier saturates next. Fix: a load balancer (L4 or L7 — lesson 0003) in front of N identical web servers, spread across availability zones, with health checks so a dead server is ejected in seconds. But horizontal scaling has a precondition: statelessness. If user sessions live in web-server memory, the LB must pin users to servers (sticky sessions), which breaks the whole model — a server death logs users out, and load skews.

So sessions move out: externalize them to Redis (callback 0003). Any server can now serve any request, servers become cattle, and deploys become rolling restarts. New problems introduced: the LB is itself a component that can fail (run a pair), Redis is a new dependency on the hot path, and you now operate N machines instead of two — health checks, config drift, and fleet-wide deploys become real work. This stage buys you roughly 10× on the web tier and exposes the next wall: every one of those N servers hammers the same single database.

5 · Stage 3 — DB replication, cache, CDN

The single DB cracks under read load first (reads dominate ~10:1). Three fixes land together:

  • Replication — one primary takes writes, read replicas take reads (lesson 0005). The old friend that returns: replication lag (lesson 0004) — a user writes to the primary and reads their own missing comment from a replica. Read-your-writes routing or bounded-staleness rules come back with it.
  • Cache-aside layer — Redis/Memcached between web and DB (lesson 0007): check cache, on miss read DB and fill. Kills most residual read traffic; introduces invalidation, staleness windows, and thundering herds on hot-key expiry.
  • CDN — static assets (images, JS, CSS) leave your servers entirely and get served from edge locations (lesson 0003). Often the single biggest bandwidth win of the whole evolution.

New problems introduced: three kinds of staleness (lag, TTL, CDN cache) now coexist, and "which copy did this read come from?" becomes a debugging question. The write path, meanwhile, is untouched — one primary still absorbs every write. Remember that; it is the stage-5 wall.

6 · Stage 4 — autoscaling, queues, object storage

Traffic is no longer flat — daily peaks are 3–10× the trough, and paying for peak 24/7 is waste. Fix: autoscaling groups — the web fleet grows and shrinks against a utilization target. Precondition: the stateless-everything discipline from stage 2, now absolute — any server must be killable at any second without a user noticing.

Two companions arrive with it:

  • Queues decouple slow work (lesson 0008): video transcoding, email, report generation move behind a message queue with their own worker pool. The web tier answers in milliseconds and the queue absorbs spikes — with backpressure and bounded queues, or the lesson-0008 hockey stick returns.
  • Object storage for user uploads: files leave local disks for an S3-style store — infinitely scalable, replicated, and reachable from every server; local disk was quietly a statefulness violation all along.

New problems introduced: autoscaling lag (instances take a minute to boot — spikes faster than that still need headroom or shedding), queue-consumer failure semantics (at-least-once → idempotent workers), and cost that now moves — a bug can autoscale your bill.

7 · Stage 5 — shard the writes, NoSQL the hot paths

Now the wall deferred since stage 3: write volume outgrows the single primary. Replicas do not help — every replica replays every write. The fix is sharding (lesson 0005): partition data across N primaries by key (user ID, tenant ID), each shard taking 1/N of writes. Costs arrive immediately: cross-shard queries and joins get hard, transactions stop at shard boundaries, resharding is a live-surgery project, and a bad shard key creates hot shards — the celebrity problem from the Twitter rep (lesson 0020): one famous user's data melts one shard while fifteen idle.

In parallel, hot non-relational paths migrate to purpose-built NoSQL stores (lesson 0006): session-like key-value data, activity feeds, counters — workloads that never needed joins pay relational overhead for nothing. The decision table from 0006 applies: relational where transactions and joins earn their cost, NoSQL where access patterns are simple and scale is the constraint. New problems: two (or more) storage systems to operate, and data consistency between them becomes application logic.

8 · Stage 6 — multi-region

Last ladder rung, driven by two forces: latency (light takes ~70 ms one way across an ocean — no server tuning fixes physics) and survival (a whole-region outage should degrade you, not erase you). Fix: full stacks in multiple regions, with geoDNS routing users to the nearest one, and data-residency rules sometimes pinning a user's data to a legal jurisdiction regardless of latency.

The pain concentrates in cross-region replication: 70–150 ms of lag is physics, so synchronous replication across regions is off the table for the hot path — you are choosing an asynchronous, eventually-consistent world (lesson 0004) and every consistency question from Phase 1 returns at continental scale. The central choice:

  • Active-passive — one region serves, one warm-standby replicates and waits. Simpler; failover loses the async tail (RPO > 0) and the passive region is paid-for idle capacity.
  • Active-active — all regions serve. Full capacity utilization and instant failover, but writes can now conflict across regions — conflict resolution (last-write-wins, CRDTs, or per-user home-region pinning) becomes your problem.

Failover itself must be drilled, not assumed — an untested failover path is a rumor (lesson 0004).

9 · The cost curve — scale is bought with complexity

StageQPS ceiling ~Monthly cost ~Ops burden
0 · single box200$100Trivial — one pet machine
1 · web/DB split500$300Low — two boxes, one link
2 · LB + fleet2–5k$2kMedium — health checks, fleet deploys
3 · replicas + cache + CDN20k$10kHigh — lag, invalidation, 3 staleness kinds
4 · autoscale + queues50k$30kHigh — async failure semantics, moving cost
5 · shards + NoSQL100k+$100kVery high — resharding, hot shards, N stores
6 · multi-region1M$1MExtreme — conflicts, failover drills, residency

Read the table diagonally: each 10× of ceiling costs roughly 10× in dollars and a qualitative jump in operational complexity. That is the thesis — scale is bought with complexity, which is why jumping stages early is a smell, not ambition. And every jump has a prerequisite the ladder hides: monitoring and alerting (lesson 0016). You cannot autoscale on metrics you don't collect, detect a hot shard you don't graph, or fail over on an outage you notice from Twitter. Instrumentation precedes each rung.

The counter-move is always the same sentence: "at our load numbers, that stage isn't earned yet."

11 · Interview talking points

  • The one-breath evolution (30 seconds): "One box until ~hundreds of QPS; split web from DB; then LB plus stateless web servers with sessions in Redis; then read replicas, cache-aside, and a CDN as reads grow; autoscaling, queues for slow work, object storage for files; shard the primary when writes outgrow it, NoSQL for hot simple paths; finally multi-region with geoDNS — active-passive first, active-active if the business pays for conflict resolution. Each step is bought with complexity, so I take it only when a measured number forces me."
  • Opening a scaling question: "Before picking components, let me find which stage of the ladder this load actually lives at."
  • The three-beat per stage: "What breaks first, the minimal fix, and the new problem the fix introduces — every fix introduces one."
  • Anti-fashion signal: "I'd stay on the monolith here — at 300 QPS, microservices buy me network hops and distributed debugging, not capacity."
  • Monitoring as gate: "Before each jump, the prerequisite is observability — you can't autoscale on metrics you don't collect."
  • Unprompted failure mode: "The stage-3 trap is replication lag: a user posts a comment and can't see it. Read-your-writes routing closes it."
