განაწილებული ძებნა 1 მილიარდ დოკუმენტზე

ორი სისტემა ერთ ლაბადაში: offline indexing pipeline (ინდექსაციის მილსადენი) და online query plane (მოთხოვნების სიბრტყე). ამ rep-ზე ის კანდიდატები იჭრებიან, ვინც მხოლოდ ერთს აპროექტებს.

მისიასთან კავშირი. ეს Anthropic-ის რეპორტირებული პრომპტია — „distributed search over 1B documents" კითხვების ბანკში სიტყვასიტყვით ფიგურირებს, ამიტომ ეს rep-ი საბრძოლო მასალაა; იგივე retrieval ინფრასტრუქტურა უდევს საფუძვლად RAG-ს production მასშტაბზე (ფაზა 6, გაკვეთილი 41): „keyword" შეცვალე „embedding"-ით და sharding-ი, fanout-ი და კუდის ლატენტურობის მათემატიკა უცვლელი გადარჩება.
REP PROTOCOL ჯერ 40-წუთიანი სოლო rep-ი დროზე — ხმამაღლა, ქაღალდზე ან Excalidraw-ში, ჩაუხედავად. პრომპტი: „დააპროექტე ძებნის სისტემა 1 მილიარდ დოკუმენტზე." მხოლოდ ამის შემდეგ წაიკითხე გაკვეთილი, შეადარე და ყველა დელტა (გამორჩენილი კომპონენტი, მცდარი რიცხვი, დაუსახელებელი failure mode) learning record-ად ჩაწერე.

1 · პრომპტი და რას ამოწმებს სინამდვილეში

ინტერვიუერი ამბობს ერთ წინადადებას: „დააპროექტე keyword ძებნა მილიარდ დოკუმენტზე." ფარული რუბრიკა: შეამჩნევ თუ არა, რომ ეს ორი სისტემაა ერთში? Offline indexing pipeline (batch/streaming სისტემა, რომელიც ნედლ დოკუმენტებს საძიებო ინდექსად აქცევს) და online query plane (დაბალი ლატენტურობის fanout სერვისი, რომელიც ამ ინდექსზე top-k queries-ს პასუხობს). ვინც მხოლოდ query plane-ს ხატავს, კითხვაზე „და ინდექსი საიდან მოდის?" ჩერდება; ვინც მხოლოდ pipeline-ს — ლატენტურობას ვერასდროს ეხება. ორი სიბრტყის გაყოფა პირველივე წუთში თქვი — ეს ამ rep-ის ყველაზე მაღალსიგნალიანი წინადადებაა. მეორეული რუბრიკა: ინდექსის ზომის მათემატიკა, sharding-ღერძის არჩევანი (სექცია 6) და fanout-ის კუდის ლატენტურობა (სექცია 7).

Freshness-ის კითხვა ცხადად დასვი და ხმამაღლა თქვი, რომ მისი შერბილება არის გასაღები: თუ ახალი დოკუმენტის საძიებოდ გახდომას წუთები შეიძლება დასჭირდეს, indexing-ი offline batch job-ად გაეშვება, რომელიც immutable (უცვლელ) ინდექსის ფაილებს აწარმოებს და გრაფიკით გზავნის serving კვანძებზე. ორი სიბრტყე სრულად იყოფა — pipeline-ი შეიძლება ჩავარდეს, თავიდან გაეშვას და გადააშენოს ისე, რომ query ლატენტურობას არ შეეხოს. თუ ინტერვიუერი წამებში სიახლეს მოითხოვს, ბევრად რთული ამოცანა აღმოაჩინე (სექცია 10). Out of scope: crawling-ის politeness (გაკვეთილი 0024), auth, რეკლამა.

3 · Estimate (5 წთ): რიცხვები, რომლებსაც დახარჯავ

კორპუსი:  1B დოკ. × 5 KB ტექსტი          ≈ 5 TB ნედლი ტექსტი
Postings: ~500 უნიკ. ტერმინი/დოკ.        → 1B × 500 = 5 × 10¹¹ posting ჩანაწერი
ჩანაწერი: ~4 B შეკუმშული (delta docID + tf, positions ამორტიზებული)
ინდექსი:  5 × 10¹¹ × 4 B ≈ 2 TB          → ინდექსი ≈ კორპუსის 40% (ტიპურია 20–50%)
Shard-ები: 2 TB ÷ ~50 GB ინდექსი/კვანძი  = 40 shard → 50 მარაგით
Fanout:   ყოველი query ყველა shard-თან   → per-shard QPS = სრული 10k QPS
          10k QPS × 50 shard = 500k shard-query/s კლასტერზე
Replica:  კვანძი ~2k QPS-ს უძლებს        → 5 replica/shard ≈ 250 serving კვანძი

ახლა დახარჯე თითოეული. 2 TB ინდექსი → ერთი კვანძი არ კმარა; sharding იძულებითია. 50 shard-ი → fanout 50 ყოველ query-ზე — სექცია 7-ის კუდის მათემატიკა ირთვება. Per-shard QPS = კლასტერის სრული QPS (ყოველი query ყველა shard-ს სტუმრობს — doc-partitioned გადასახადი) → replica-ების რაოდენობას ჯამური QPS კარნახობს. 40% ინდექსი/კორპუსი → შეკუმშვა (სექცია 5) ფულადი გადაწყვეტილებაა. გამოუყენებელი შეფასება 0009-ის კლასიკური ანტი-პატერნია — ეს ოთხივე ქვემოთ მეორდება.

4 · High-level: ორი სიბრტყე ცალ-ცალკე დახატე

INDEXING PLANE (offline, throughput-ზე ოპტიმიზებული)
  crawl/ingest → parse + tokenize → ნაწილობრივი ინდექსები (batch-ებად)
      → merge immutable სეგმენტებად → სეგმენტების გაგზავნა shard replica-ებზე

QUERY PLANE (online, ლატენტურობაზე ოპტიმიზებული)
  client → query service (parse, cache-ის შემოწმება)
      → fanout ყველა 50 shard-ზე (თითოზე 1 replica)
      → თითო shard: ლოკალური top-k′ → gather → merge + rank → top-10 → პასუხი

დაფაზე ორ განცალკევებულ დიაგრამად დახატე. Indexing plane MapReduce-ის ფორმის batch სისტემაა: mapper-ები დოკუმენტების batch-ებს ტოკენიზებენ და ნაწილობრივ term→postings რუკებს უშვებენ; reducer-ები მათ დალაგებულ, შეკუმშულ სეგმენტის ფაილებად წყობენ; დისტრიბუციის ნაბიჯი თითო სეგმენტს თავისი shard-ის replica-ებზე გზავნის. Query plane არასდროს წერს: immutable სეგმენტებს ტვირთავს და ემსახურება. სიბრტყეებს შორის ერთადერთი კავშირი სეგმენტების გადაცემაა — ვერსიონირებული, ატომური „ჩატვირთე ნაკრები N+1, ჩამოაგდე N". ეს განცალკევება სწორედ შერბილებულმა freshness-მა გიყიდა — თქმა Step 1-ს Step 3-ს უკავშირებს ინტერვიუერის თვალწინ.

5 · Deep dive: inverted index-ის ანატომია

Inverted index არის term → postings list: ამ ტერმინის შემცველი docID-ების დალაგებული სია, თითო ჩანაწერს ახლავს term frequency (tf) და ხშირად positions (ტოკენის ოფსეტები — phrase queries-სა და snippet-ის გამოსაყოფად). სამი ფაქტი გასახმოვანებლად:

  • დალაგებული + delta-encoded + შეკუმშული. Postings docID-ით ლაგდება, ამიტომ ინახავ სხვაობებს (delta) და არა ნედლ 8-ბაიტიან ID-ებს; სხვაობები პატარაა და varint/bit-packing-ით ძლიერ იკუმშება. აქედან მოდის სექცია 3-ის 40%-იანი შეფარდება — შეუკუმშავად ინდექსი კორპუსს გაუტოლდებოდა.
  • Skip pointers. AND query postings სიებს კვეთს; skip pointer-ები ყოველ √n ჩანაწერზე კვეთას გრძელი სიის მონაკვეთების გადახტომის საშუალებას აძლევს — ღირებულება სიგრძეების ჯამიდან უმოკლესის სიგრძისკენ იწევს.
  • Immutable სეგმენტები + background merge. განახლება სეგმენტს ადგილზე არასდროს ცვლის: ახალი დოკუმენტები ახალ პატარა სეგმენტებში მიდის; წაშლა tombstone bitmap-ია; ფონური პროცესი პატარებს დიდებად წყობს. იცანი ინსტინქტი — ეს გაკვეთილი 0012-ის LSM tree-ა ძებნის ტანსაცმელში: თანმიმდევრული writes, lock-free reads, write amplification ფონში. Lucene — და მაშასადამე Elasticsearch — ზუსტად ასე მუშაობს.

6 · Deep dive: document-partitioned vs term-partitioned

ამ პრომპტის კანონიკური განშტოება: დაყო დოკუმენტით (თითო shard = სრული ინდექსი დოკუმენტების ნაჭერზე) თუ ტერმინით (თითო shard ლექსიკონის ნაჭრის მთლიან postings სიებს ფლობს)?

განზომილებაDoc-partitionedTerm-partitioned
Query fanoutყოველი query → ყველა 50 shardმხოლოდ query-ის ტერმინების მფლობელები (2–3)
მრავალტერმინიანი ANDკვეთა ლოკალურია თითო shard-შიPostings shard-ებს შორის იგზავნება — GB-ები ხშირ ტერმინებზე
ცხელი ტერმინებიდატვირთვა თანაბრად ნაწილდება„the"-ს მფლობელი shard დნება — hot-term skew
ინდექსის აგება / დოკ. დამატებაერთი shard ეხება თითო დოკუმენტსყოველი დოკუმენტი ~500 term shard-ს ეხება
Shard-ის ჩავარდნაRecall-ის ნაჭერი იკარგება; ყველა query მდოვრედ დეგრადირდებაზოგი ტერმინი ყველა შედეგიდან მთლიანად ქრება

Term-partitioning მიმზიდველად გამოიყურება (მცირე fanout!) სანამ მრავალტერმინიან სტრიქონს არ ნახავ: „distributed AND search"-ის კვეთა ერთი ტერმინის postings-ის — შესაძლოა მილიონობით ჩანაწერის — მეორის shard-ზე გაგზავნას ითხოვს, ყოველ query-ზე. Hot-term skew საქმეს ასრულებს. ამიტომ თითქმის ყველა doc-partitioned-ს აწვდის — ეს Elasticsearch/Lucene-ის მოდელია და Google-ის თავდაპირველი არქიტექტურაც. ცხრილი ხმამაღლა თქვი, აირჩიე doc-partitioned რადგან ლოკალური კვეთები და თანაბარი დატვირთვა მცირე fanout-ს სჯობს, და დაასახელე ფასი: სრული fanout, რომელსაც სექცია 7-ში იხდი.

7 · Deep dive: scatter-gather და კუდის ლატენტურობის გადასახადი

Coordinator query-ს ყველა 50 shard-ზე შლის; თითო თავის ლოკალურ top-k′-ს აბრუნებს; coordinator 50·k′ კანდიდატს წყობს და გლობალურ top-10-ს აბრუნებს. ორი დახვეწილობა სენიორის სიგნალს ატარებს:

რატომ k′ > k. ყველა shard-ს იდენტური გლობალური სტატისტიკით რომ ეფასებინა, per-shard top-10 გლობალურ top-10-ს დამტკიცებადად შეიცავდა (თითო გამარჯვებული ზუსტად ერთ shard-ში ცხოვრობს). მაგრამ shard-ები ლოკალური IDF-ით (საკუთარი document frequency-ებით) აფასებენ, early-termination მიახლოებებს იყენებენ და deletion tombstone-ებს ატარებენ — ამიტომ გლობალურად ძლიერი დოკუმენტი ლოკალურ მე-12 ადგილზე შეიძლება იჯდეს და coordinator-ამდე ვერასდროს მივიდეს. პრაქტიკული გამოსავალი: მოითხოვე k′ = 2–3×k თითო shard-ზე (რამდენიმე ასეული merge-კანდიდატი — იაფია), ან პერიოდულად დააგზავნე გლობალური ტერმინ-სტატისტიკა. ლაბი სწორედ ამ recall-vs-k′ მრუდს ზომავს.

Fanout კუდს აძლიერებს. Query ისეთივე ნელია, როგორც მისი ყველაზე ნელი shard-ი: P(ყველა n shard სწრაფია) = pn. Fanout 100-ზე და 99%-ში სწრაფ shard-ებზე 0.99100 ≈ 0.37 — queries-ის მხოლოდ 37% ცდება ყველა shard-ის ნელ 1%-ს, ანუ per-shard p99 შენი ბოლოდან-ბოლომდე მედიანა ხდება. Fanout 100-ზე სუფთა p99-სთვის თითო shard-ი 99.99-ე პერცენტილზე უნდა იყოს სწრაფი: 0.9999100 ≈ 0.99. შერბილება: hedged requests (თუ shard-მა p95-მდე არ უპასუხა, იგივე მოთხოვნა სხვა replica-ს ესროლე და პირველი პასუხი აიღე — ~5% ზედმეტი დატვირთვა, კუდის მკვეთრი მოჭრა) და replica-გათვითცნობიერებული balancing, რომელიც ნელ ან GC-ში მყოფ კვანძებს გვერდს უვლის.

8 · Deep dive: ranking — TF-IDF-დან ორ ფაზამდე

TF-IDF ინტუიცია: დოკუმენტის ქულა = Σ query-ტერმინებზე tf (რამდენჯერ ჩნდება ტერმინი აქ) × idf = log(N/df) (რამდენად იშვიათია კორპუსში). იშვიათი ტერმინები დომინირებს; „the" ნულს იძლევა. BM25 TF-IDF-ია ორი შესწორებით და მხოლოდ ფორმის ცოდნა გჭირდება: tf ჯერდება (saturation — 50-ე გამოჩენა თითქმის არაფერს ამატებს: tf/(tf+k₁) ნედლი tf-ის ნაცვლად) და ქულები სიგრძით ნორმალიზდება (ტერმინი tweet-ში წიგნში მოხვედრილს ჯობს — b·dl/avgdl წევრი). ფორმა და არა გამოყვანა — ეს არის ინტერვიუს ბარი.

ორფაზიანი ranking: ძვირი ranker მილიონობით დამთხვეულ დოკუმენტზე ლატენტურობის ბიუჯეტს დაანგრევს, ამიტომ ფაზა 1 თითო shard-ში იაფი BM25-ით რამდენიმე ასეულ კანდიდატს არჩევს, ფაზა 2 კი მხოლოდ მათ აფასებს ძვირი მოდელით (ათობით feature ან ML ranker) coordinator-ზე. შეურიე სტატიკური ხარისხის prior-ები — query-სგან დამოუკიდებელი ქულა (PageRank, წყაროს ხარისხი), ინდექსაციისას გამოთვლილი. ბოლოს query შედეგები cache-ში (გაკვეთილი 0007): პოპულარობა zipf-ურია, ამიტომ მოკრძალებული cache coordinator-ის წინ 10k QPS-ის დიდ ნაწილს fanout-ამდე შთანთქავს.

9 · Failure mode-ები და მონიტორინგი

  • ერთი ნელი shard-ი ყველა query-ს წამლავს. სრული fanout ნიშნავს: ერთი ცუდი კვანძი (GC პაუზა, კვდომადი დისკი, merge-ქარიშხალი) გლობალურ p99-ს ათრევს. შერბილება: hedged requests, მკაცრი per-shard timeout-ები და ნაწილობრივი შედეგები degraded=true დროშით query-ის ჩავარდნის ნაცვლად — 49/50 shard-ი ისევ შესანიშნავი recall-ია.
  • Segment-merge ქარიშხლები. ფონური merge I/O-მძიმეა; დაურეგულირებელი merge serving კვანძზე queries-ს ეჯიბრება და ზუსტად ზემოთხსენებულ ნელ shard-ს აწარმოებს. შერბილება: merge I/O-ს ლიმიტი, ან მძიმე merge serving გზიდან მოშორებით და მზა სეგმენტის გაგზავნა.
  • ინდექსი/serving ვერსიების აცდენა. Rollout-ის შუაში N და N+1 ნაკრებზე მყოფი shard-ები სხვადასხვანაირად ქულავს და შერწყმულ გვერდებზე დოკუმენტები ორმაგდება ან იკარგება. შერბილება: ვერსია ყოველ სეგმენტ-ნაკრებს, coordinator-ი query-ზე ერთ ვერსიას ამაგრებს, ახალი ინდექსი ჯერ canary shard-ებზე.
  • მეტრიკები, რომლებიც ადამიანს აღვიძებს: per-shard p99 და ბოლოდან-ბოლომდე p99 (მათი შუალედი შენი fanout-გადასახადია), hedge-ის სროლის სიხშირე, indexing lag (freshness SLO) და recall canary — ფიქსირებული query→მოსალოდნელი-დოკუმენტი წყვილები უწყვეტ გამეორებაზე; recall-ის ჩუმი ვარდნა ლატენტურობის დაფებზე უხილავია.

10 · ვარიანტები და twist-ები

  • Real-time indexing. „ახალი დოკუმენტი წამებში საძიებო" → თითო shard-ზე პატარა in-memory delta ინდექსი, რომელიც დიდ immutable სეგმენტებთან ერთად იკითხება და პერიოდულად სეგმენტად იწერება. ეს ზუსტად Lucene-ის near-real-time refresh-ია — ისევ LSM memtable ინსტინქტი.
  • Phrase queries. „ზუსტი ფრაზა" სექცია 5-ის position მონაცემებს ითხოვს: postings-ის კვეთა და მეზობელი პოზიციების შემოწმება; ხშირი ბიგრამების ინდექსები („new york") ყველაზე ცუდ შემთხვევებს წინასწარ იხდის.
  • მრავალენოვნება. ტოკენიზაცია ენაზეა დამოკიდებული — CJK ენებში ჰარები არ არის (სეგმენტაციის მოდელები), გერმანული კომპოზიტები იშლება, stemming/lemmatization ენის მიხედვით ინდექსაციისა და query-ის დროსაც. Analyzer-ის არჩევა ინდექსაციისდროინდელი ვალდებულებაა: შეცვლა გადაინდექსაციას ნიშნავს.

AI შენიშვნა. Embedding-ზე დაფუძნებული (vector) ძებნა term-match retrieval ფაზას მკვრივ ვექტორებზე approximate nearest-neighbor ძებნით ცვლის, მაგრამ ამ გაკვეთილის ჩონჩხს ინარჩუნებს — sharded ANN ინდექსები, scatter-gather, per-shard top-k′, merge, rerank. თანამედროვე RAG სტეკები ჰიბრიდულია: BM25 და ვექტორული retrieval პარალელურად, rerank-ზე შერწყმული. სრული განხილვა ფაზა 6-ში, გაკვეთილი 41.

11 · ინტერვიუს გასაღები ფრაზები

  • გახსნა: „ეს ორი სისტემაა — offline indexing pipeline და online query plane — და freshness-ის წუთებამდე შერბილება მაძლევს მათი განცალკევების უფლებას."
  • შეფასების დახარჯვა: „2 TB ინდექსი 50 GB-იან კვანძებზე 40–50 shard-ია — და ეს fanout რიცხვია მიზეზი, რომ დროს კუდის ლატენტურობაზე დავხარჯავ."
  • Sharding-ის განშტოება: „doc-partitioned თუ term-partitioned: სრულ fanout-ს ავიღებ ლოკალური კვეთებისა და hot-term shard-ის არქონის სანაცვლოდ — Elasticsearch-ის მოდელი."
  • დაუკითხავი კუდის მათემატიკა: „fanout 50-ზე P(ყველა სწრაფია) = per-shard p ხარისხად 50 — per-shard p99 ჩემი მედიანა ხდება, ამიტომ hedged requests მჭირდება და არა უკეთესი საშუალო."
  • სიღრმის შეთავაზება: „შემიძლია ჩავუღრმავდე BM25-პლუს-rerank-ს ან სეგმენტების merge-ს — რომელი გირჩევნია?"
  • დახურვა: „გავაღვიძებდი ბოლოდან-ბოლომდე p99-ზე, indexing lag-სა და recall canary-ზე; მეტი დროით: real-time delta-ები და მრავალენოვანი analyzer-ები."

12 · ლაბი + პროექტის ნაბიჯი

პრაქტიკა (ქვიზამდე გააკეთე) labs/lab-0026-inverted-index/-ში python3 search_lab.py აშენებს 20k-დოკუმენტიან zipf-ურ კორპუსს, inverted index-ს და TF-IDF ranking-ს; ყოფს doc-partitioned პრინციპით; შემდეგ ზომავს ორ ახლახან წაკითხულ მტკიცებას — ranking recall vs per-shard k′ (k′=k რეალურ შედეგებს კარგავს გლობალურ ჭეშმარიტებასთან შედარებით) და p99-ის გაძლიერება shard-ების რაოდენობის მიხედვით, hedging გადამრთველით. იწინასწარმეტყველე გაშვებამდე: README სავარჯიშოები E1–E4.

პროექტის ნაბიჯი. ლაბის შემდეგ გაიმეორე 40-წუთიანი rep-ი ხმამაღლა და დელტები პირველ ცდას შეადარე. შემდეგ learning record-ში ერთი აბზაცი დაწერე, რომელიც ამ rep-ს spine პროექტს უკავშირებს: შენი inference პლატფორმის RAG retrieval შრე (ფაზა 6) ზუსტად ეს არქიტექტურა იქნება, shard-ის ადგილას ANN ინდექსით — დაასახელე ორი უცვლელად გადამავალი კომპონენტი (scatter-gather hedging-ით, immutable სეგმენტების გაგზავნა).
