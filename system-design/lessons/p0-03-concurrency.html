Concurrency: Races, Locks, Event Loops, and the GIL

Why "it worked on my laptop" is the most expensive sentence in multithreaded code.

Mission tie. Anthropic's most-reported design prompt — an inference batching service where synchronous callers block while a scheduler groups their requests and routes responses back to the right waiter — is a concurrency problem before it is an ML problem: shared queue, worker pool, locking discipline, and an event loop holding thousands of open connections. Every mechanism in this lesson appears in that design. (Exponent's Anthropic guide, Inside vLLM)
Lab first (ADHD rule) Before reading further, run python3 concurrency_lab.py in labs/lab-p0-03-races-and-gil/. Watch a counter lose 70% of its updates, a deadlock fire, and threads fail to speed up CPU work. Then read on — every section below explains something you just saw.

1 · Concurrency is not parallelism

Concurrency is structure: many tasks in progress at once, interleaved on however many CPUs you have — possibly one. Parallelism is execution: many tasks literally running at the same instant on multiple cores. A single-threaded web server handling 10,000 open connections is highly concurrent and not parallel at all. The distinction decides which tool you reach for. (OSTEP: Concurrency and Threads)

Threads (lesson p0-02) share one address space — same heap, separate stacks. That sharing is the entire value proposition and the entire danger: any thread can write memory any other thread is reading.

2 · The lost update: anatomy of a race

counter += 1 is not one operation. It is read → modify → write, and the scheduler may preempt a thread between any two steps:

Thread A: read counter (=100)
Thread B: read counter (=100)      <- preempted A; reads the SAME value
Thread B: write 101
Thread A: write 101                <- B's update is silently destroyed

Two increments happened. The counter moved by one. No error was raised.

This is a race condition: the result depends on the timing of interleavings you do not control. The nasty property is silence — no exception, no crash, just wrong data. In the lab, 4 threads × 50,000 increments lose 50–75% of updates, a different amount every run. (OSTEP: Locks)

Vocabulary The code region that must not interleave is the critical section. Atomicity means the section executes as one indivisible unit — all or nothing, with no observable middle state.

3 · Locks: correctness at a price

A mutex (lock) makes a critical section atomic: acquire before, release after; only one holder at a time. In the lab, the locked counter is exact — and roughly 4× slower, because threads now queue for the section. That queue has a name you already know: contention, and it shows up in p99 first.

The senior-level knob is granularity. One coarse lock over everything is easy to reason about but serializes the world (this is literally what Python's GIL is — section 7). Fine-grained locks scale better but multiply the ways to deadlock. Real systems also use reader–writer locks (many concurrent readers, one writer) and atomic instructions (compare-and-swap) for single-word updates.

4 · Deadlock, and how to never write one

Deadlock requires four conditions simultaneously: mutual exclusion, hold-and-wait (grab one lock, wait for another), no preemption (locks can't be forcibly taken), and circular wait (T1 holds A wants B; T2 holds B wants A). Break any one and deadlock is impossible. (OSTEP: Common Concurrency Problems)

The industrial-strength fix is the cheapest one: a global lock ordering — every code path acquires locks in the same agreed order, so a cycle can never form. The lab shows the same two locks deadlocking in opposite order and completing instantly in a fixed order. Secondary defenses: acquire with timeouts, or hold at most one lock at a time.

5 · Thread pools and the sizing math

Spawning a thread per request does not survive contact with production: stacks cost megabytes, and creation/teardown costs dominate at high request rates. A thread pool pre-creates N workers pulling tasks from a shared queue (Python: concurrent.futures). The queue is itself a lock-protected structure — the pattern is called producer–consumer.

How big should N be? For I/O-bound work, the answer is Little's law applied to threads:

Gateway target: 2,000 req/s; each request waits 45 ms on a downstream
call and needs 5 ms of CPU.

In-flight requests   L = λ × W = 2,000/s × 0.050 s = 100
Threads (1/request)  ≈ 100, round up for headroom → 128
Stack memory         100 × 8 MB default stacks = 800 MB of pure bookkeeping
asyncio alternative  100 coroutines × ~few KB   < 1 MB for the same concurrency

That last comparison is why the next section exists.

6 · Async I/O and the event loop

Most server "work" is waiting — for sockets, disks, databases, downstream APIs. Waiting needs no thread. An event loop runs on one thread, asks the OS "which of my 10,000 file descriptors are ready?" (epoll/kqueue), and resumes only the tasks that can make progress. Tasks are coroutines that await at every wait-point, yielding the thread voluntarily — cooperative scheduling, versus the OS's preemptive kind. (Python: asyncio)

In the lab, 1,000 concurrent 50 ms waits finish in ~0.05 s on one asyncio thread; a 50-worker thread pool needs ~1 s (1,000/50 × 50 ms). Nginx, Node.js, and every serious API gateway are built on this model. The iron law: never block the loop — one CPU-heavy or synchronously-blocking call starves every other task (lab E5 makes you cause this).

7 · The Python GIL — one lock to rule them all

CPython's Global Interpreter Lock allows only one thread to execute Python bytecode at a time, per process — a single coarse lock protecting the interpreter's internals. (Python glossary: GIL)

Consequences, measured in the lab:

  • CPU-bound threads don't scale. Four threads of pure-Python math ≈ sequential time (sometimes worse — GIL hand-offs cost). Use processes (one GIL each) or push work into C/NumPy/GPU kernels, which release the GIL.
  • I/O-bound threads are fine. The GIL is released during blocking syscalls, so threads waiting on sockets overlap freely.
  • The GIL does not make your code thread-safe. The lab's lost-update race happens with the GIL on — it protects interpreter internals, not your read-modify-write sequences.

For LLM serving this is why vLLM and friends run the model in GPU kernels (GIL released), keep the Python layer as an async coordinator, and scale CPU-heavy steps like tokenization with separate worker processes. PEP 703 (free-threaded CPython) is removing the GIL, but the architecture lesson stands. (Inside vLLM)

8 · Choosing a model: the one table to remember

ModelParallel CPU?Best forCost per unitSharp edge
Threads + locksYes (no, in CPython)Blocking I/O, shared state~MB stack, ~10 µs switchRaces, deadlock, contention
ProcessesYesCPU-bound work, isolation~10 ms spawn, IPC copiesNo shared memory, IPC overhead
Async event loopNo (one thread)Huge numbers of waits~KB per coroutineOne blocking call starves all

Production servers compose all three: an async loop at the edge, a thread pool for blocking libraries, process workers for CPU. That composition is the standard answer to "how does your gateway handle 10k concurrent connections?"

9 · Sharp edges (failure modes)

  • Check-then-act (TOCTOU). if balance >= amount: withdraw() — the balance can change between the check and the act. The check and act must share one critical section (or one atomic DB operation).
  • The race that "never happens." Timing-dependent bugs hide for months, then fire under production load or on different hardware. Lab E2 shows the same race going quiet when the preemption interval grows — it is still there.
  • Deadlock only under load. Lock cycles need the unlucky interleaving; low traffic rarely finds it. Load tests and lock-order audits do.
  • Blocking call on the event loop. One synchronous requests.get() or heavy loop inside an async handler freezes every connection on that loop — the #1 async production bug.
  • Pool exhaustion. All N workers stuck on a slow downstream → queue grows → timeouts cascade. Fix with timeouts, bounded queues, and circuit breakers (Phase 2, lesson 0015).
  • Assuming the GIL protects you. It serializes bytecodes, not your multi-step invariants. You still need locks in threaded Python.

Interview talking points

  • "Concurrency is structure — tasks in progress; parallelism is simultaneous execution. A single-threaded event loop can be massively concurrent."
  • "counter += 1 is read-modify-write; unsynchronized threads silently lose updates — I'd protect it with a lock or an atomic, and keep the critical section minimal because contention shows up in p99."
  • "Deadlock needs mutual exclusion, hold-and-wait, no preemption, and circular wait — I prevent it by imposing a global lock acquisition order."
  • "For I/O-bound thread pools I size with Little's law: threads ≈ arrival rate × wait time, plus headroom."
  • "The GIL serializes Python bytecode, so CPU-bound work needs processes or C/GPU kernels that release it; I/O threads are fine — and the GIL still doesn't make compound operations thread-safe."
  • "My default server shape: async loop at the edge, thread pool for blocking libraries, process workers for CPU — never block the event loop."
