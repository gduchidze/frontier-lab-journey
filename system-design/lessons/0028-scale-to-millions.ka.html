გაკვეთილი 0028 · ფაზა 3 — კლასიკური დიზაინის Rep-ები · Capstone · English version

მასშტაბირება მილიონობით მომხმარებლამდე — ევოლუციის Rep-ი

აქამდე ყველა rep-ი ერთ სისტემას ერთ მასშტაბზე აპროექტებდა. Capstone-ი მეორე ღერძს ავარჯიშებს: ერთი სისტემა, ყველა მასშტაბი — კანონიკური ევოლუცია ერთი ყუთიდან multi-region-ამდე, რომელიც ყველა მასშტაბირების ისტორიის საფუძველია.

მისიასთან კავშირი. ეს capstone-ი ავარჯიშებს ყველა მასშტაბირების ნარატივის ფორმას — inference პლატფორმა ზუსტად იმავე ეტაპებს გადის, ოღონდ app სერვერების ადგილას GPU-ებით: ერთი GPU ყუთი → LB + replica pool → KV-cache ფენა → დაშარდული model worker-ები → multi-region მიწოდება.
Rep-პროტოკოლი — ეს rep-ი განსხვავებულია ერთი 40-წუთიანი დიზაინის ნაცვლად, კითხვამდე ხმამაღლა თხრობით გაიარე ევოლუცია ეტაპობრივად: დაიწყე „ყველაფერი ერთ ყუთზეა"-დან და განმეორებით იკითხე „რა ტყდება პირველი 10× დატვირთვაზე?" — დაასახელე გატეხილი კომპონენტი, გამოსავალი და ახალი პრობლემა, რომელიც გამოსავალს შემოაქვს, სანამ multi-region-ს მიაღწევ. მხოლოდ ამის შემდეგ წაიკითხე გაკვეთილი და ჩაწერე დელტები ყველგან, სადაც შენი თხრობა განსხვავდებოდა.

1 · Rep-ი, და რატომ უყვართ ის ინტერვიუერებს

„გაზარდე ეს საიტი ნულიდან მილიონობით მომხმარებლამდე" ინტერვიუერის ყველაზე იაფი რენტგენია შენი განსჯისთვის. ის არ ამოწმებს, იცი თუ არა microservices, Kafka ან Kubernetes — ის ამოწმებს, აწყობ თუ არა საჭიროებით და არა მოდით. კანდიდატი, რომელიც 200 QPS-ზე თექვსმეტ სერვისს და service mesh-ს შლის, ისევე იჭრება, როგორც ის, ვინც 50k ჩაწერა/წმ-ზე ყველაფერს ერთ Postgres-ზე ტოვებს: ორივემ დატვირთვის რიცხვები დააიგნორა. ნაადრევი microservices დასახელებული სუნია (გაკვეთილი 0010); ასევე ნაადრევი sharding-ი, ნაადრევი multi-region და ნაადრევი ნებისმიერი რამ.

გამარჯვებული ფორმა კიბეა: ყოველ ეტაპზე დაასახელე (ა) რა ტყდება პირველი შემდეგ 10×-ზე, (ბ) მინიმალური გამოსავალი, და (გ) ახალი პრობლემა, რომელიც გამოსავალს შემოაქვს — რადგან ყველა გამოსავალს ერთი შემოაქვს. ეს სამტაქტიანი რიტმი, ექვსჯერ გამეორებული, არის პასუხი. Primer-ის Scaling on AWS ამოხსნა ზუსტად ამ კიბეს გადის.

2 · ეტაპი 0 — ერთი სერვერი, და რატომ არის ის სწორი

ყველაფერი ერთ ყუთზე: web სერვერი, აპლიკაცია, მონაცემთა ბაზა, cron-ები, ფაილები ლოკალურ დისკზე. ერთი ბერკეტი — ვერტიკალური მასშტაბირება (vertical scaling): იყიდე უფრო დიდი მანქანა. ეს გულუბრყვილო საწყისი კი არა, სწორი საწყისია: ერთი ყუთი ნიშნავს ნულ ქსელურ ჰოპს ფენებს შორის, ნულ განაწილებულ ჩავარდნის რეჟიმს, ნულ consistency-კითხვას, ტრივიალურ deploy-ს და მთელი სისტემის გაგებას ერთ თავში. YAGNI ინფრასტრუქტურაზე გადატანილი.

რეალური ჭერი: დინამიკური გვერდი მილიწამების CPU ღირს, ამიტომ ერთი კარგად აწყობილი ყუთი ასეულობით QPS-ს ემსახურება — და, უფრო ფატალურად, ის single point of failure-ია, ხოლო ვერტიკალურ მასშტაბირებას მყარი ჭერი აქვს (ყველაზე დიდი ხელმისაწვდომი მანქანა) და ფასი სიმძლავრეზე სწრაფად იზრდება. ინტერვიუს ფრაზა: „ერთ ყუთზე ვიწყებ და ვრჩები, სანამ გაზომილი რიცხვი არ ჩამომაგდებს — პირველი ბზარი, ჩვეულებრივ, DB-სა და web სერვერის ბრძოლაა ერთსა და იმავე CPU-სა და მეხსიერებაზე."

3 · ეტაპი 1 — web და DB ფენების გაყოფა; დატვირთვის რიცხვები

პირველი ბზარი, პირველი გამოსავალი: მონაცემთა ბაზა საკუთარ მანქანაზე გადადის, web ყუთის წინ DNS დგება. ახლა თითო ფენა დამოუკიდებლად მასშტაბირდება და დამოუკიდებლად ეცემა — web-ის CPU-პიკები DB-ის გვერდებს მეხსიერებიდან აღარ დევნიან. შემოტანილი ახალი პრობლემა: ქსელური ჰოპი app-სა და DB-ს შორის (~1 ms და ახალი ჩავარდნის წერტილი — თავად ბმული), და ახლა ორი single point of failure გაქვს ერთის ნაცვლად.

შეფასების საკონტროლო წერტილები — დატვირთვები, რომლებზეც თითო ეტაპი ბზარდება. დაშვებები: დინამიკური გვერდი ~5–10 ms CPU ერთ ბირთვზე, ანუ 8-ბირთვიანი ყუთი რამდენიმე ასეულ QPS-ს ემსახურება; ტრაფიკის ~90% წაკითხვაა.

ეტაპი 0 ბზარდება  ~200 QPS      ერთი ყუთი აკეთებს web + DB + ყველაფერს
ეტაპი 1 ბზარდება  ~500 QPS      DB ყუთი ივსება, web ყუთი უქმად დგას
ეტაპი 2 ბზარდება  ~2–5k QPS     ერთი DB primary კედელია N web სერვერის უკან
ეტაპი 3 ბზარდება  ~20–50k QPS   ჩაწერები ერთ primary-ს აჭარბებს cache + replica-ების მიუხედავად
ეტაპი 5 მიზანი    100k+ QPS     დაშარდული ჩაწერები, NoSQL ცხელი გზებისთვის
ეტაპი 6 მიზანი    ~1M QPS       რამდენიმე რეგიონი, თითო სრული stack-ით

ეს ათის ხარისხების მოაჯირებია და არა ფაქტები — დაშვებები ხმამაღლა თქვი და შეასწორე. მთავარია, რომ თითო ეტაპს რიცხვი აქვს, და გადადიხარ მაშინ, როცა რიცხვი მოვა — არა ადრე.

4 · ეტაპი 2 — load balancer + N stateless web სერვერი

შემდეგი ივსება web ფენა. გამოსავალი: load balancer-ი (L4 ან L7 — გაკვეთილი 0003) N იდენტური web სერვერის წინ, გაშლილი availability zone-ებზე, health check-ებით, რომ მკვდარი სერვერი წამებში გამოირიცხოს. მაგრამ ჰორიზონტალურ მასშტაბირებას წინაპირობა აქვს: statelessness (უმდგომარეობა). თუ სესიები web სერვერის მეხსიერებაში ცხოვრობს, LB-მ მომხმარებლები სერვერებზე უნდა მიაკრას (sticky sessions), რაც მთელ მოდელს ტეხავს — სერვერის სიკვდილი მომხმარებლებს ალოგინებს და დატვირთვა იხრება.

ამიტომ სესიები გარეთ გადის: Redis-ში ექსტერნალიზდება (callback 0003). ახლა ნებისმიერი სერვერი ნებისმიერ მოთხოვნას ემსახურება, სერვერები cattle ხდებიან, deploy-ები — მოძრავი (rolling) რესტარტები. შემოტანილი ახალი პრობლემები: LB თავად ჩავარდნადი კომპონენტია (წყვილად გაუშვი), Redis ახალი დამოკიდებულებაა ცხელ გზაზე, და ახლა N მანქანას აოპერირებ ორის ნაცვლად. ეს ეტაპი web ფენაზე ~10×-ს გაძლევს და შემდეგ კედელს ააშკარავებს: ყველა ეს სერვერი ერთსა და იმავე ერთადერთ ბაზას ურტყამს.

5 · ეტაპი 3 — DB replication, cache, CDN

ერთადერთი DB ჯერ წაკითხვის დატვირთვის ქვეშ ბზარდება (წაკითხვები ~10:1 დომინირებს). სამი გამოსავალი ერთად მოდის:

  • Replication (რეპლიკაცია) — ერთი primary ჩაწერებს იღებს, read replica-ები — წაკითხვებს (გაკვეთილი 0005). ბრუნდება ძველი მეგობარი: replication lag (გაკვეთილი 0004) — მომხმარებელი primary-ში წერს და საკუთარი კომენტარის არარსებობას replica-დან კითხულობს. მასთან ერთად ბრუნდება read-your-writes routing-ი და bounded-staleness წესები.
  • Cache-aside ფენა — Redis/Memcached web-სა და DB-ს შორის (გაკვეთილი 0007): ჯერ cache, აცდენაზე DB და შევსება. კლავს ნარჩენი წაკითხვების უმეტესობას; შემოაქვს invalidation-ი, staleness-ფანჯრები და thundering herd-ი ცხელი გასაღების ვადის გასვლაზე.
  • CDN — სტატიკური ფაილები (სურათები, JS, CSS) შენს სერვერებს სულ ტოვებენ და edge ლოკაციებიდან მიეწოდება (გაკვეთილი 0003). ხშირად მთელი ევოლუციის ყველაზე დიდი bandwidth-მოგება.

ახალი პრობლემები: staleness-ის სამი სახე (lag, TTL, CDN cache) ერთად თანაარსებობს და „რომელი ასლიდან წამოვიდა ეს წაკითხვა?" debugging-კითხვა ხდება. ჩაწერის გზა კი ხელუხლებელია — ერთი primary ისევ ყველა ჩაწერას შთანთქავს. დაიმახსოვრე — ეს ეტაპ 5-ის კედელია.

6 · ეტაპი 4 — autoscaling, queue-ები, object storage

ტრაფიკი ბრტყელი აღარაა — დღიური პიკები მინიმუმის 3–10×-ია, და პიკზე 24/7 გადახდა ფლანგვაა. გამოსავალი: autoscaling group-ები — web ფლოტი utilization-სამიზნის მიხედვით იზრდება და იკუმშება. წინაპირობა: ეტაპ 2-ის stateless-everything დისციპლინა, ახლა აბსოლუტური — ნებისმიერი სერვერი ნებისმიერ წამს უნდა იყოს მოკვლადი ისე, რომ მომხმარებელმა ვერ შეამჩნიოს.

  • Queue-ები ნელ სამუშაოს თიშავენ (გაკვეთილი 0008): ვიდეო-ტრანსკოდირება, ელფოსტა, რეპორტები message queue-ს უკან გადადიან საკუთარი worker pool-ით. Web ფენა მილიწამებში პასუხობს და queue პიკებს შთანთქავს — backpressure-ითა და შემოსაზღვრული queue-ებით, თორემ 0008-ის ჰოკეის ჯოხი დაბრუნდება.
  • Object storage ატვირთვებისთვის: ფაილები ლოკალური დისკებიდან S3-სტილის საცავში გადადის — უსასრულოდ მასშტაბირებადი, რეპლიცირებული, ყველა სერვერიდან მიწვდომადი; ლოკალური დისკი მთელი ამ ხნის განმავლობაში ჩუმად statefulness-დარღვევა იყო.

ახალი პრობლემები: autoscaling-ის დაყოვნება (ინსტანსი წუთში იტვირთება — უფრო სწრაფ პიკებს headroom ან shedding სჭირდება), queue-consumer-ის ჩავარდნის სემანტიკა (at-least-once → idempotent worker-ები), და ღირებულება, რომელიც ახლა მოძრაობს — bug-მა შეიძლება შენი ინვოისი დააავტოსქეილოს.

7 · ეტაპი 5 — დაშარდე ჩაწერები, NoSQL ცხელი გზებისთვის

ახლა ეტაპ 3-იდან გადადებული კედელი: ჩაწერის მოცულობა ერთ primary-ს აჭარბებს. Replica-ები ვერ შველიან — ყველა replica ყველა ჩაწერას იმეორებს. გამოსავალი sharding-ია (გაკვეთილი 0005): მონაცემები N primary-ზე ნაწილდება გასაღებით (user ID, tenant ID), თითო shard-ი ჩაწერების 1/N-ს იღებს. ფასები მაშინვე მოდის: cross-shard კითხვები და join-ები რთულდება, ტრანზაქციები shard-ის საზღვართან ჩერდება, resharding-ი ცოცხალი ქირურგიაა, ცუდი shard-გასაღები კი ცხელ shard-ებს ქმნის — Twitter rep-ის celebrity პრობლემა (გაკვეთილი 0020): ერთი ცნობილი მომხმარებელი ერთ shard-ს ადნობს, სანამ თხუთმეტი უქმად დგას.

პარალელურად, ცხელი არარელაციური გზები სპეციალიზებულ NoSQL საცავებში მიგრირებს (გაკვეთილი 0006): სესიის-მაგვარი key-value მონაცემები, activity feed-ები, მთვლელები — დატვირთვები, რომლებსაც join-ები არასდროს სჭირდებოდათ. 0006-ის გადაწყვეტილების ცხრილი მოქმედებს: რელაციური იქ, სადაც ტრანზაქციები და join-ები ფასს იმსახურებენ; NoSQL იქ, სადაც წვდომის პატერნები მარტივია და შეზღუდვა მასშტაბია. ახალი პრობლემები: ორი (ან მეტი) საოპერაციო საცავი, და consistency მათ შორის აპლიკაციის ლოგიკა ხდება.

8 · ეტაპი 6 — Multi-region

კიბის ბოლო საფეხური, ორი ძალით ნაკარნახევი: დაყოვნება (სინათლეს ოკეანის გადაკვეთა ~70 ms ერთ მხარეს სჭირდება — ფიზიკას სერვერის tuning-ი ვერ ასწორებს) და გადარჩენა (მთელი რეგიონის ჩავარდნამ უნდა დაგაზიანოს და არა წაგშალოს). გამოსავალი: სრული stack-ები რამდენიმე რეგიონში, geoDNS-ით მომხმარებლები უახლოესისკენ მიემართებიან, data-residency წესები კი ზოგჯერ მომხმარებლის მონაცემებს იურისდიქციაზე აკრავენ დაყოვნების მიუხედავად.

ტკივილი cross-region რეპლიკაციაში გროვდება: 70–150 ms lag ფიზიკაა, ამიტომ სინქრონული რეპლიკაცია რეგიონებს შორის ცხელი გზისთვის გამორიცხულია — ირჩევ ასინქრონულ, eventually-consistent სამყაროს (გაკვეთილი 0004) და ფაზა 1-ის ყველა consistency-კითხვა კონტინენტურ მასშტაბზე ბრუნდება. ცენტრალური არჩევანი:

  • Active-passive — ერთი რეგიონი ემსახურება, მეორე თბილი standby-ა და ელოდება. მარტივია; failover ასინქრონულ კუდს კარგავს (RPO > 0), პასიური რეგიონი კი გადახდილი უქმი სიმძლავრეა.
  • Active-active — ყველა რეგიონი ემსახურება. სრული სიმძლავრის გამოყენება და მყისიერი failover, მაგრამ ჩაწერები ახლა რეგიონებს შორის შეიძლება დაუპირისპირდნენ — კონფლიქტების მოგვარება (last-write-wins, CRDT-ები, ან per-user home-region მიმაგრება) შენი პრობლემა ხდება.

თავად failover-ი ვარჯიშით უნდა შემოწმდეს, არა დაშვებით — შეუმოწმებელი failover-გზა ჭორია (გაკვეთილი 0004).
