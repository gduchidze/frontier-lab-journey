Design Rep: API Rate Limiter

კლასიკური გამახურებელი ამოცანა, შესრულებული როგორც სრული 4-ნაბიჯიანი rep-ი — ხუთი ალგორითმი, ერთი Redis race condition და სიზუსტე-vs-latency trade-off-ი, რომელიც სენიორ შეთავაზებებს წყვეტს.

მისიასთან კავშირი. „დააპროექტე rate limiter" Anthropic-ის დაფიქსირებული კითხვების ბანკში კლასიკურ გამახურებლად ჩანს — ხშირად უფრო რთული ამოცანის წინ. ეს ნიშნავს, რომ ფასდება შესრულების სისწრაფე და სიმკვეთრე, არა სიახლე. სენიორი კანდიდატი მთელ ციკლს 30–35 წუთში ასრულებს: ალგორითმების trade-off-ები ბაიტებში დათვლილი და backing store-ის ჩავარდნის პოლიტიკა დასახელებული. ეს გაკვეთილი ზუსტად ამ შესრულების reference-ია.
Rep-პროტოკოლი (ჯერ ეს წაიკითხე) ფაზა 3-ის გაკვეთილები REFERENCE SOLUTION-ებია და არა პირველი წასაკითხი. სანამ ამ ყუთს ჩაუვლი: გააკეთე 40-წუთიანი სოლო, დროში გაზომილი დიზაინი, ხმამაღლა, გაკვეთილი 0009-ის 4-ნაბიჯიანი ჩარჩოთი, სექცია 1-ის ამოცანაზე. ШЕМდეგ წაიკითხე ეს გაკვეთილი და ჩაწერე ყველა დელტა — გამორჩენილი კომპონენტი, დაუხარჯავი რიცხვი, failure mode, რომელიც reference-მა დაასახელა და შენ ვერა. სწავლა დელტებშია; მხოლოდ კითხვა სწავლა არ არის.

1 · ამოცანა და რას ამოწმებენ სინამდვილეში

ამოცანა ერთი წინადადებაა: „დააპროექტე rate limiter საჯარო API პლატფორმისთვის." დანარჩენის სტრუქტურირება შენზეა. მოჩვენებითი სიმარტივის მიღმა ინტერვიუერები ოთხ რამეს ამოწმებენ:

  • ალგორითმების ცოდნა. შეგიძლია ერთზე მეტი ალგორითმის დასახელება და — ნამდვილი ტესტი — მათი trade-off-ების დათვლა მეხსიერებაში, burst-ქცევასა და სიზუსტეში, სახელების ჩამოთვლის ნაცვლად?
  • განაწილებული state-ის ინსტინქტები. ერთმანქანიანი ვერსია ტრივიალურია. N gateway კვანძის მომენტში თვლა ხდება განაწილებული სისტემების პრობლემა — race-ებით, ცხელი გასაღებებით (hot keys) და consistency-ის სახელურით. Rep-ი აქ იგება.
  • ჩავარდნის პოლიტიკა. რა ხდება, როცა counter store კვდება? „Fail open vs fail closed" ერთსტრიქონიანი კითხვაა, რომელიც ჰყოფს კანდიდატებს, ვინც სისტემები აუმუშავებია, მათგან, ვინც მათზე მხოლოდ წაუკითხავს.
  • API-ის მანერები. უარყოფილი გამომძახებლები იღებენ პასუხს, რომელზეც მოქმედება შეუძლიათ (429, Retry-After), თუ შემთხვევით thundering herd-ს აპროექტებ?

რადგან გამახურებელია, ტემპი მნიშვნელოვანია: ~5 წუთი clarify, 4 estimate, 8 high-level დიზაინი, დანარჩენი — ალგორითმებსა და განაწილებულ თვლაში სიღრმისეულად. სწრაფად და ხმამაღლა მართვა თავად არის შედეგი.

2 · ნაბიჯი 1 — Clarify (5 წთ)

დაასკოუპე კონკრეტულ ბიზნესად: საჯარო e-commerce API პლატფორმა (კატალოგი, შეკვეთები, ძიება), რომელიც წვდომას მესამე მხარის ინტეგრატორებზე ყიდის. ჩაწერე კონტრაქტი:

ტიპიმოთხოვნა
ფუნქციურილიმიტი API key-ზე (მაგ. 600 req/min); სხვადასხვა ლიმიტი გეგმის (plan tier) და endpoint-ის მიხედვით; უარყოფა 429-ით + ინფორმაციული header-ებით; ლიმიტების შეცვლა redeploy-ის გარეშე
ფუნქციურიგანზომილებები: per-user (API key) — მთავარი; per-IP — არაავთენტიფიცირებული ტრაფიკისთვის; per-endpoint — ძვირი მარშრუტებისთვის (ძიება vs პროდუქტის წაკითხვა)
არაფუნქციურიLimiter-ის overhead p99 < 5 ms — ის ყველა მოთხოვნაზე ზის; დამცავი თავად ვერ გახდება bottleneck
არაფუნქციურიAvailability ≥ თავად API-ისა; limiter-ის ჩავარდნამ API არასდროს უნდა წააქციოს
არაფუნქციურისიზუსტე: მცირე დროებითი ზედმეტი დაშვება (over-admission) ასატანია (~1–5%); ჩუმი under-admission (გადამხდელი მომხმარებლის გეგმაზე დაბლა დათრგუნვა) — არა

Out of scope, ხმამაღლა: billing/კვოტები (თვიური ჭერი სხვა, DB-ზე დაფუძნებული სისტემაა), DDoS scrubbing L3/L4-ზე (edge-ის საზრუნავი) და თავად ავთენტიფიკაცია — უკვე გადამოწმებულ API key-ს ვიღებთ. სიზუსტის ის სტრიქონი შეგნებულად სენიორული სვლაა: აცხადებს, შეცდომის რომელი მიმართულებაა მისაღები, და ყველა გვიანდელი trade-off მას დაეყრდნობა.

3 · ნაბიჯი 2 — Estimate (4 წთ), ყველა რიცხვი დახარჯული

პლატფორმა: 10M რეგისტრირებული API key, ~2M აქტიური პიკის საათში
ტრაფიკი:  500M გამოძახება/დღეში ÷ ~100k წმ/დღეში ≈ 5k QPS საშ., ×4 პიკი ≈ 20k QPS
  → ერთი limiter-შემოწმება თითო მოთხოვნაზე = 20k შემოწმება/წმ
  → ერთი Redis კვანძი ≈ 100k მარტივი op/წმ → დღეს ეტევა; 5× ზრდა შეჭამს → sharding გასაღებით

Latency:  in-DC Redis RTT ≈ 0.5–1 ms; overhead-ის ბიუჯეტი p99 < 5 ms
  → მაქსიმუმ ЕРთი round trip თითო შემოწმებაზე → check+update ერთ ატომურ ძახილში (Lua)

მეხსიერება, token bucket: key ~40 B + state 16 B ≈ 60 B overhead-ით
  → 10M key × 60 B ≈ 600 MB → МთЕЛი key space ერთ Redis კვანძში ეტევა

მეხსიერება, sliding window log @ 600 req/min: worst 600 timestamp × 8 B ≈ 5 KB/key
  → 2M აქტიური key × 5 KB ≈ 10 GB → ~16-ჯერ ძვირი → log ამ მასშტაბზე უარყოფილია

ყველა რიცხვი მაშინვე იხარჯება: 20k შემოწმება/წმ ყიდულობს sharding-გეგმას და გამორიცხავს ყველაფერს, რაც state-ს თითო შემოწმებაზე სკანირებს; ერთი-RTT ბიუჯეტი ყიდულობს Lua/ატომური-ძახილის გადაწყვეტილებას სექცია 6-ში; 600 MB vs 10 GB ის არითმეტიკაა, რომელიც სექცია 5-ში ალგორითმს აირჩევს — იქ მისვლისას ეს წინადადება ხმამაღლა თქვი, და estimate-ნაბიჯი უკუქმედებით დაამტკიცებს, რომ რიტუალი არ იყო.

4 · ნაბიჯი 3 — High-level დიზაინი (8 წთ)

მოთხოვნის გზა. Client → load balancer → API gateway → rate-limit middleware → backend სერვისები. Limiter-ი მიმართავს counter store-ს (Redis) გასაღებით {key_id}:{endpoint_class}:{window}; დაშვებულები გადიან, უარყოფილები მაშინვე ბრუნდებიან — უარყოფა მიკროწამები უნდა ღირდეს და არა backend-ის ძახილი.

429-ის კონტრაქტი. უარყოფას მოჰყვება მანქანურად წაკითხვადი ტემპის ინფორმაცია:

HTTP/1.1 429 Too Many Requests
Retry-After: 23
X-RateLimit-Limit: 600
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1721741460

წარმატებულ პასუხებსაც მიჰყვება X-RateLimit-Remaining, რომ კარგი კლიენტები კედელზე ასვლამდე შენელდნენ. ეს ერთი header ჩუმი thundering-herd-პრევენციაა (სექცია 9).

სად ზის limiter-ი — სამი ვარიანტი, signpost-ით:

  • Gateway middleware — ერთი აღსრულების წერტილი, ერთი config-ზედაპირი; gateway ისედაც ყველა მოთხოვნის გზაზეა. Default არჩევანი.
  • ბიბლიოთეკა თითო სერვისში — ზედმეტი hop არ არის, მაგრამ N ენა × M სერვისი დუბლირებული ლოგიკა და config drift. აქ უარყოფილია.
  • Sidecar — ლოკალური გადაწყვეტილებები pod-ზე, უკან გაზიარებული store; ყიდულობს სექცია 7-ის ლოკალური-bucket-ის მოდელს ფლოტის deployment-სირთულის ფასად.

აირჩიე gateway middleware, რადგან პლატფორმა-პროდუქტს უნიფორმული აღსრულება და წესების შეცვლის ერთი ადგილი უნდა — და ხმამაღლა შენიშნე, რომ sidecar მიმზიდველი ზუსტად მაშინ ხდება, როცა სექცია 7-ის სიზუსტე-vs-latency სახელური ტრიალდება.

5 · Deep dive A — ხუთი ალგორითმი, დათვლილი

ალგორითმიState თითო key-ზეBurst-ქცევასაზღვრის spike?სიზუსტე
Token bucket (ტოკენების ვედრო)~16 B (tokens, last-refill)უშვებს burst-ებს ვედროს ზომამდე — კონფიგურირებული ფუნქციააარა — მაგრამ rolling-window დაშვება capacity + refill ≈ 2× rate-ს აღწევსზუსტი საკუთარი მოდელისთვის
Leaky bucket (გამჟონავი ვედრო, meter)~16 B (level, last-leak)ყველაფერს მუდმივ გადინებამდე ასწორებს; burst-ები რიგში დგებიან ან ვარდებიანარაზუსტი; სჯის ლეგიტიმურ burst-იანობას
Fixed window counter (ფიქსირებული ფანჯარა)~16 B (window id, count)სრული ლიმიტი მაშინვე ხელმისაწვდომია ყოველ ფანჯარაშიდიახ — 2× ლიმიტამდე საზღვრის გადაღმამიახლოებითი
Sliding window log (მოცურავე ფანჯრის ლოგი)8 B × limit (~5 KB @ 600/min)ზუსტი მოძრავი ლიმიტი, spike-ების გარეშეარასრულყოფილი — ~300× მეხსიერების ფასად
Sliding window counter (მოცურავე ფანჯრის მთვლელი)~24 B (ორი counter + id)თითქმის გლუვი; მცირე მიახლოების ცდომილებაარა (გაგლუვებულია)მიახლოებითი — წინა ფანჯარაში თანაბარ განაწილებას უშვებს

საზღვრის spike-ი ინტერვიუში ერთ კონკრეტულ წინადადებას იმსახურებს: 600/min fixed window-ზე კლიენტს შეუძლია 600 მოთხოვნა 11:59:59-ზე და კიდევ 600 — 12:00:01-ზე: 1,200 მოთხოვნა ~2 წამში, ორმაგი ტემპი, ყველა „კანონიერი." Sliding window counter ამას 8 დამატებითი ბაიტით ასწორებს წინა ფანჯრის აწონვით: est = cur + prev × (1 − elapsed_fraction). აქ დახარჯე სექცია 3-ის არითმეტიკა: log-ის 10 GB vs counter-ის ათეულობით MB — ამიტომ აწვდიან რეალური პლატფორმები (მათ შორის Cloudflare) მიახლოებას და იღებენ ~რამდენიმე-პროცენტიან ცდომილებას.

6 · Deep dive B — განაწილებული counter-ები Redis-ში

ბევრი gateway კვანძით counter-ი გაზიარებული უნდა იყოს — და გულუბრყვილო იმპლემენტაციაში race ჩაშენებულია:

count = INCR key          # მოთხოვნა 1 აქ კვდება →
IF count == 1: EXPIRE key window   # ...key ახლა Სამუდამოდ ცოცხლობს
IF count > limit: reject

ორი განსხვავებული პრობლემა. პირველი: INCR-შემდეგ-EXPIRE ატომური არ არის — crash ან timeout ორ ბრძანებას შორის ტოვებს უკვდავ counter-ს, რომელიც key-ს ახრჩობს, სანამ ვინმე შეამჩნევს. მეორე: check-then-act ორ round trip-ზე კონკურენტულ მოთხოვნებს აძლევს საშუალებას, ყველამ წაიკითხოს count < limit და ყველა გავიდეს. ორივეს წამალი: გადაწყვეტილება + განახლება ერთ ატომურ ერთეულად — Lua სკრიპტი (Redis სკრიპტებს ატომურად ასრულებს, არაფერი ერევა შუაში), რომელიც ავსებს ვედროს, ამოწმებს, აკლებს და TTL-ს აყენებს ერთ round trip-ში — რაც სექცია 3-ის ერთი-RTT ბიუჯეტსაც აკმაყოფილებს. Redis-ის INCR დოკუმენტაცია ზუსტად ამ rate-limiter პატერნსა და მის race-ს განიხილავს.

Hot keys (ცხელი გასაღებები). Sharding API key-ით დატვირთვას ფანტავს — სანამ ერთი უზარმაზარი ინტეგრატორი მარტო 5k RPS-ს აკეთებს და ყველაფერი ერთ shard-ზე ვარდება. საშუალებები, ზრდადობით: ცალკე shard „ვეშაპი" key-ებისთვის; ერთი ლოგიკური ლიმიტის გაყოფა K ქვე-გასაღებად limit/K-თი (იაფი, ოდნავ უსწორმასწორო); ან per-node ლოკალური bucket-ები — რაც მთლიანად სექცია 7-ის თემაა.

7 · Deep dive C — სიზუსტის სახელური: sync vs eventual

ყველა განაწილებული limiter-ი ერთ სახელურზე ზის:

  • სინქრონული გლობალური თვლა — ყოველი შემოწმება Redis round trip-ია. სიზუსტე: თითქმის ზუსტი. ფასი: +1 ms ყველა მოთხოვნაზე, და Redis მთელი პლატფორმის კრიტიკულ გზაზეა.
  • ლოკალური bucket-ები + ასინქრონული sync — თითო gateway კვანძი ლოკალური ვედროდან უშვებს და გაზიარებულ store-თან ყოველ T ms-ში სწორდება. შემოწმებები in-memory ხდება (მიკროწამები). ფასი: sync-ებს შორის „ბრმა" ფანჯარაში N კვანძი მოძველებული გლობალური რიცხვის მიმართ უშვებს — დროებითი over-admission, შემოსაზღვრული დაახლ. N × ჩამოსვლის-ტემპი × T-თი.

აქ იხარჯება ნაბიჯ 1-ის სიზუსტის მოთხოვნა, სახელით: 1–5% დროებითი over-admission მისაღებად გამოვაცხადეთ და under-admission — მიუღებლად; ამიტომ ლოკალური bucket-ები მოკლე sync-ინტერვალით hack-ი კი არა, მოთხოვნის პირდაპირი შედეგია. T ამოირჩიე ზღვრიდან: 4 კვანძზე და 400 req/s-ით ამფეთქებელ key-ზე T = 100 ms ≈ 40 ზედმეტი დაშვების რისკია 100-იან ლიმიტზე — ~40% worst case იმ key-სთვის burst-ის დროს; ამ რიცხვს შენი ლაბი პირდაპირ ზომავს. მგრძნობიარე endpoint-ებზე T დაამჭიდროვე, იაფებზე — მოუშვი.

Stripe-ის გამოქვეყნებული limiter-დიზაინი იმავეზე ჯდება — Redis-ზე დაფუძნებული token bucket-ები, შეგნებულად შემწყნარებელი მცირე უზუსტობის მიმართ, სანაცვლოდ კრიტიკული გზის ყველაზე ცუდ percentile-ებს შორდება.

8 · Deep dive D — წესები, config და განზომილებები

ლიმიტები პოლიტიკაა, პოლიტიკა კი ყოველკვირა იცვლება; აღსრულების გზას redeploy არ უნდა სჭირდებოდეს. სტანდარტული ფორმა: წესები ცხოვრობს პატარა config store-ში (ბაზა ან config-სერვისი) — {plan_tier, endpoint_class} → {limit, window, burst} — და gateway კვანძები მათ მეხსიერებაში აქეშავენ პერიოდული განახლებით (ათეულობით წამი საკმარისია; წესის ცვლილება latency-კრიტიკული არ არის). Rules cache-ის miss-ი რომელიმე წესზე უნდა დავარდეს და არასდროს — შეცდომაზე.

განზომილებები ერთმანეთს ედება. ერთი მოთხოვნა შეიძლება რამდენიმე counter-ზე შემოწმდეს:

  • Per API key — მთავარი, გეგმაზე დაფუძნებული ლიმიტი.
  • Per IP — არაავთენტიფიცირებული endpoint-ებისთვის (login, signup), სადაც key ჯერ არ არსებობს; ესვე abuse-კონტროლის განზომილებაა.
  • Per endpoint class — ძიება პროდუქტის წაკითხვაზე 20-ჯერ ძვირია, ამიტომ იმავე key-ის ქვეშ საკუთარი, უფრო მჭიდრო counter-ი აქვს.

ჯერ ყველაზე იაფი უარყოფა შეაფასე, და ფასი პატიოსნად დათვალე: სამი განზომილება = სამი counter-განახლება მოთხოვნაზე. Lua-თი გააერთიანე ერთ სკრიპტ-ძახილში — ერთი round trip განზომილებების რაოდენობის მიუხედავად — თორემ 5 ms ბიუჯეტი ათასი ჭრილობით მოკვდება. ხმამაღლა ახსენე ფენებადობა: gateway limiter-ი პლატფორმის სამართლიანობისთვის + უხეში per-IP ფენა edge-ზე მის წინ; ორი იაფი ფენა ერთ ჭკვიანურს ჯობს.

9 · Failure mode-ები და მონიტორინგი

ჩავარდნარა ხდებადიზაინის პასუხი
Redis დაეცაყოველი შემოწმება ცდება — limiter-ს შეუძლია წააქციოს API, რომელსაც იცავსFail open commerce API-სთვის (მთელი ტრაფიკის დაგდება დამცავი მოაჯირის დასაცავად უკუღმართობაა), ლოკალური fallback limiter-ით კონსერვატიულ ტემპზე + ხმამაღალი alert-ით. Fail closed მხოლოდ იქ, სადაც limiter-ი ТАვად security-საზღვარია (auth-მცდელობები). არჩევანი უკითხავად თქვი.
Retry-ების thundering herdფანჯარა განულდა, ან outage დასრულდა — და ყველა დათრგუნული კლიენტი ერთსა და იმავე წამში retry-ობს, spike-ს სამუდამოდ ხელახლა ქმნისRetry-After jitter-იანი მნიშვნელობებით, რომ retry-ები გაიფანტოს; კლიენტებს — exponential backoff + jitter (შენი lab-0008-ის შედეგი, ახლა სერვერის მხარეს); token bucket-ის refill ხელახლა შემოსვლას აგლუვებს fixed window-ის კლდესთან შედარებით
Clock skew (საათების აცდენა)Gateway კვანძები აცდენილი საათებით ფანჯრის საზღვრებზე ვერ თანხმდებიან → ლიმიტები არათანაბრად სრულდებაგამოიყენე store-ის საათი (Redis-ის TIME Lua სკრიპტის შიგნით) დროის ერთადერთ ავტორიტეტად; N gateway-საათის შეთანხმებას არასდროს ენდო

მონიტორინგი: ადამიანის გამაღვიძებელი მეტრიკები — limiter-შემოწმების p99 latency და store-ის შეცდომების ტემპი; dashboard-ები — 429-ის ტემპი key-ზე/tier-ზე (ნახტომი = ერთი მომხმარებლის გაფუჭებული retry-ციკლი) და over-admission drift-ი (დაშვებული ტემპის შერჩევითი შედარება კონფიგურირებულ ლიმიტთან). Alert დააყენე 429-ის ნულამდე დაცემაზეც — ჩუმად fail-open-ში გადასული limiter-ი ზუსტად ჯანმრთელ მშვიდს ჰგავს.

10 · ინტერვიუერის twist-ები და ვარიანტები

  • „აქციე security-კონტროლად." სახელური ტრიალდება: per-IP და per-account ლიმიტები login/OTP endpoint-ებზე credential stuffing-ის შესანელებლად არსებობს — ამიტომ იქ fail closed, ლიმიტები დაბალი, ფანჯრები გრძელი, უარყოფები abuse-გუნდისთვის ილოგება. იგივე მექანიზმი, საპირისპირო ჩავარდნის პოლიტიკა — ამ წინადადების თქმა twist-ს იგებს.
  • „გლობალური ლიმიტი datacenter-ებს შორის." სინქრონული პლანეტარული counter-ი ყოველ მოთხოვნაზე 80+ ms cross-region RTT-ს დააწვებოდა — მიუღებელია. რეალური პასუხი: per-region კვოტები (გლობალური ლიმიტის გაყოფა ტრაფიკის დაკვირვებული წილით) + ასინქრონული შეჯერება, რომელიც წილს ანაცვლებს, დროებითი გლობალური over-admission-ის მიღებით. ეს სექცია 7-ის სახელურია პლანეტარულ მასშტაბზე.
  • „რა ღირს სიზუსტე?" შემაჯამებელი კიბე: fixed window (16 B, 2× spike) → sliding counter (24 B, ~რამდენიმე % ცდომილება) → sliding log (5 KB, ზუსტი) → სინქრონული გლობალური (ზუსტი, +1 ms ყველაფერზე). სიზუსტე ჯერ მეხსიერებით ისყიდება, მერე latency-ით. დაასახელე კიბე, დააყენე ბიზნესი საფეხურზე, დაასაბუთე მოთხოვნით.

AI-არომატი (ერთი აბზაცი). LLM API-ები ზომავენ ტოკენებს წუთში და არა მხოლოდ მოთხოვნებს — ერთმა მოთხოვნამ შეიძლება 50 ან 50,000 ტოკენი მოიხმაროს. მექანიკურად იგივე token bucket-ია, ტოკენის ერთეულებში შევსებადი, ერთი ნიუანსით: გამოსავლის ფასი დაშვების მომენტში უცნობია, ამიტომ პროვაიდერები მოთხოვნის დასაწყისში შეფასებას აკლებენ და დასრულებისას რეალურ ხარჯთან აჯერებენ — ისევ სიზუსტის სახელური, ამჯერად ფასზე.

11 · Interview talking points

  • გახსნის scope-ი: „Rate limiting იცავს ტევადობასა და სამართლიანობას წამების-მასშტაბის ფანჯრებში; თვიური კვოტები და billing სხვა სისტემაა — limiter-ზე ვსკოუპავ."
  • მეხსიერების რიცხვის დახარჯვა: „Sliding log ამ მასშტაბზე ~10 GB-ია, bucket-ები — ~600 MB; ეს არითმეტიკა და არა გემოვნება არის მიზეზი, რომ token bucket-ს ვირჩევ sliding-window-counter fallback-ით."
  • ატომურობის სტრიქონი: „გულუბრყვილო INCR-შემდეგ-EXPIRE-ს ორი failure mode აქვს — უკვდავი key crash-ზე და check-then-act race-ები — ამიტომ შემოწმება ერთი ატომური Lua ძახილი უნდა იყოს. ესვე ერთ round trip-ზე გვაჩერებს."
  • სახელური: „ზუსტი გლობალური თვლა და დაბალი latency პირდაპირ დაძაბულობაშია; ჩვენი მოთხოვნები დროებით over-admission-ს იტანს, ამიტომ ვიღებ ლოკალურ bucket-ებს ~50–100 ms sync-ით და ცდომილების შემოსაზღვრაც შემიძლია: N × rate × interval."
  • უკითხავი ჩავარდნის პოლიტიკა: „სანამ მკითხავ: Redis-ის დაცემა = fail open კონსერვატიული ლოკალური fallback-ით — commerce API-სთვის limiter-ი outage არასდროს უნდა გახდეს. Auth endpoint-ებზე ამას შევაბრუნებდი."
  • დახურვა: „Page — შემოწმების latency-სა და store-შეცდომებზე; dashboard — 429-ები tier-ზე; alert — 429-ების ნულზე გაქრობაზეც: მკვდარი limiter-ი მშვიდად გამოიყურება. მეტი დროით: multi-region კვოტების გაყოფა და per-key ადაპტიური ლიმიტები."

12 · ლაბი + პროექტის ნაბიჯი

პრაქტიკა (ქვიზამდე გაუშვი) labs/lab-0018-rate-limiter/-ში python3 rate_limiter_lab.py ერთსა და იმავე burst-იან trace-ს ატარებს token bucket-ში, fixed window-ში, sliding window log-სა და sliding window counter-ში — და ბეჭდავს თითო ალგორითმის accepted/rejected რიცხვებს, worst rolling-window დაშვებას (საზღვრის-spike-ის გაჟონვა, გაზომილი) და მეხსიერების proxy-ს. შემდეგ python3 rate_limiter_lab.py --distributed ასიმულირებს N limiter-კვანძს, რომლებიც გაზიარებულ counter-ს ყოველ T ms-ში ასინქრონებენ, და ზომავს over-admission-ს sync-ინტერვალის მიხედვით — სექცია 7-ის ზღვარი, საკუთარი თვალით. ყოველ გაშვებამდე იწინასწარმეტყველე; README-ის E1–E4 გაგატარებს.

პროექტის ნაბიჯი. თუ თავში rep-პროტოკოლის ყუთი გამოტოვე, ახლა გააკეთე: გაკვეთილი 0009-ის design_rep.py-ში PROBLEM შეცვალე „დააპროექტე rate limiter საჯარო API პლატფორმისთვის"-ით და გაუშვი სრული 40-წუთიანი დროში გაზომილი rep-ი, ხმამაღლა, ამ გაკვეთილის ხელახლა წაკითხვამდე. შემდეგ rep-notes.txt-ში ჩაწერე დელტები: რომელი ალგორითმის trade-off-ი ვერ დათვალე, დაიხარჯა თუ არა ნაბიჯ 2-ის რიცხვები, fail-open/fail-closed შენგან უკითხავად წამოვიდა თუ არა. დელტების სია — და არა ეს გაკვეთილი — არის ის, რასაც მასწავლებელი შემდეგ სესიაზე განიხილავს.

ძირითადი წყარო

წაიკითხე Stripe-ის „Scaling your API with rate limiters" — production token-bucket-on-Redis დიზაინი load shedder-ებით და კანონიკური ფორმულირება: „ჩავარდნილმა limiter-მა API თან არ უნდა წაიღოს." თანამგზავრები: Cloudflare-ის „Counting things, a lot of different things" (sliding-window-counter მიახლოება გლობალურ მასშტაბზე, გაზომილი ცდომილებით), Redis-ის INCR დოკუმენტაცია (rate-limiter პატერნი და მისი race, პირველწყაროდან) და System Design Primer-ის rate-limiter სავარჯიშო კლასიკური ჩარჩოსთვის.
