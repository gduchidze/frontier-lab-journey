გაკვეთილი 0021 · ფაზა 3 — კლასიკური დიზაინის Rep-ები · English version

ჩატის სისტემა — WhatsApp-ის მასშტაბის მესიჯინგი

აქამდე ყველა დიზაინი stateless იყო load balancer-ის უკან. ჩატი ამას არღვევს: მილიონობით ხანგრძლივი socket-ი, თითო კონკრეტულ მანქანაზე მიბმული, და შეტყობინებები, რომლებიც ერთხელ და თანმიმდევრობით უნდა ჩავიდეს — გამორთულ ტელეფონზეც კი.

მისიასთან კავშირი. Stateful connection-gateway ფენა, რომელსაც აქ ააგებ — ხანგრძლივი socket-ები და routing ცხრილი კითხვაზე „რომელი მანქანა უჭირავს user X-ს" — ზუსტად ის პატერნია, რომელსაც გაკვეთილი 37 იყენებს inference კლასტერიდან ტოკენების 100k კლიენტთან სტრიმინგისთვის.
Rep-ის პროტოკოლი ეს rep-ია და არა საკითხავი. ჯერ: 40 წუთი, მარტო, ხმამაღლა — გაუშვი 0009 გაკვეთილის სრული ჩარჩო 1-ლი სექციის ამოცანაზე, ჩაუხედავად. მერე წაიკითხე ეს ამოხსნა და ყველა დელტა (გამორჩენილი მოთხოვნა, გამოუყენებელი რიცხვი, დაუსახელებელი failure mode) ჩაწერე learning record-ად.

1 · ამოცანა და რას ამოწმებს სინამდვილეში

„დააპროექტე WhatsApp-ის მსგავსი მესინჯერი: 1:1 და ჯგუფური ჩატები, მიწოდების receipt-ები, presence, მუშაობს როცა მიმღები offline-შია."

ის, რაც ამ rep-ს ყველა წინაზე განასხვავებს: statefulness (მდგომარეობის ქონა). Feed-ი ან URL shortener-ი request/response-ია — ნებისმიერი სერვერი უპასუხებს ნებისმიერ მოთხოვნას, ამიტომ სკალირება = „დაამატე replica-ები LB-ის უკან". ჩატი ამას აბრუნებს: შენთვის გამოგზავნილმა შეტყობინებამ უნდა იპოვოს ის კონკრეტული მანქანა, რომელსაც ახლა შენი socket-ი უჭირავს — ერთ წამში, ასობით gateway-ს შორის. ინტერვიუერი ამოწმებს, შეამჩნევ თუ არა ამ ინვერსიას და დააპროექტებ თუ არა ცხადად: კავშირების დამჭერი ფენა, routing ცხრილი და პასუხი კითხვაზე, რა ხდება როცა socket-ების მანქანა კვდება.

ამოცანაში დამალული მეორეული ტესტები: ordering (დალაგება) — რატომ გიღალატებს timestamp-ები, მიწოდების გარანტიები — at-least-once + dedup და არა მითიური exactly-once, და fanout-ის დისციპლინა — ჯგუფებიც და presence-იც ფეთქდება, თუ გულუბრყვილოდ გააკეთე. თითოეულს თავისი deep dive აქვს ქვემოთ.

2 · Clarify — დაზუსტება (5–8 წთ)

მოთხოვნაჩასაწერი გადაწყვეტილება
1:1 ჩატიბირთვი. ტექსტი, მაქს. ~1 KB.
ჯგუფური ჩატიკი, ლიმიტი ~500 წევრი (თავად WhatsApp-ს წლობით 256 ჰქონდა, მერე 1024 — ლიმიტი დიზაინის გადაწყვეტილებაა: fanout-ს საზღვრავს).
Delivery receipt-ებიSent / delivered / read — მე-6 სექციის სამმდგომარეობიანი მანქანა.
PresenceOnline + last_seen კონტაქტებისთვის.
Offline მიწოდებააუცილებელი. შეტყობინებები ელოდება მოწყობილობას და თანმიმდევრობით იცლება.
Multi-device syncკი — ტელეფონი და ლეპტოპი ერთსა და იმავე ისტორიას ხედავს.
მედიაშესთავაზე scope-იდან ამოღება: „blob storage + CDN, შეტყობინებას URL მიაქვს" — ერთი წინადადება და უკან მესიჯინგზე.

არაფუნქციური კონტრაქტი: მიწოდების p95 < ~500 ms როცა ორივე online-შია; არავითარი დაკარგვა, არავითარი ხილული დუბლიკატი, დაცული per-conversation თანმიმდევრობა; 99.99% availability. ხმამაღლა სათქმელი დეტალი: ჩატს მხოლოდ საუბრის შიგნით დალაგება სჭირდება — ორი სხვადასხვა საუბრის ურთიერთგადაწყობას ვერავინ დააკვირდება, და სწორედ ეს შვება ხდის სისტემას shardable-ს.

3 · Estimate — შეფასება (~5 წთ), ყველა რიცხვი დახარჯე

შეტყობინებები: 500M DAU × 40 msg/დღეში = 20B msg/დღეში
               ÷ ~100k წმ/დღეში ≈ 200k msg/s საშუალო, ×3 პიკი ≈ 600k msg/s შემომავალი
               თითოს ≥1 მიმღები ჰყავს → gateway-ს გამავალი ≥ 1.2M msg/s პიკზე

Socket-ები:    უმოქმედო აპებიც socket-ს ღიად ინახავენ; ~20% DAU ერთდროულად → ~100M კავშირი
               WhatsApp-ის ეპოქის ფოლკლორი: 1–2M კავშირი ერთ დატუნინგებულ Erlang/FreeBSD
               მანქანაზე (მხოლოდ რიგითობის შეფასება — ინჟინრული ლორეა და არა სპეცი)
               → ~100 მანქანა; კონსერვატიული 100k/მანქანა → ~1,000. ორივე შემთხვევაში: ფენა.

Storage:       20B msg × ~100 B ≈ 2 TB/დღეში ≈ 730 TB/წელიწადში, მხოლოდ ტექსტი.

ახლა დახარჯე: 600k msg/s → სერვისები partitioned queue-თი საუბრობენ და არა RPC fan-in-ით. 100M socket-ი → gateway ფენას მეხსიერება და file descriptor-ები ზომავს და არა CPU — და ცალკე ფენა უნდა იყოს, რომ stateless ჩატის ლოგიკა მილიონი კავშირისგან დამოუკიდებლად deploy-დებოდეს. 2 TB/დღეში → wide-column საცავი, conversation-ით დაყოფილი (§8) — 730 TB/წელს ერთი Postgres ვერ შეჭამს.

4 · High-level დიზაინი (~15 წთ)

ტრანსპორტი. WebSocket და არა long polling: ჩატი ორმხრივია (სერვერი აწვდის შეტყობინებებსაც, receipt-ებსაც და presence-საც), და ერთი მუდმივი socket-ი 100M კლიენტზე ბევრად იაფია, ვიდრე თავიდან დამყარებადი poll-ციკლები. Long polling — fallback ქსელებისთვის, სადაც WebSocket ტყდება.

Stateful ნაწილი. Chat gateway ფენა socket-ებს ამთავრებს (terminate). Gateway-ები კავშირებს იჭერენ და თითქმის სხვას არაფერს აკეთებენ. როცა user X gateway 42-ს უკავშირდება, gateway წერს X → gw42 session/routing ცხრილში (Redis: ცხელი, პატარა მნიშვნელობები, TTL-მეგობრული). ეს ცხრილი მთელი statefulness-ის პასუხია: „რომელ მანქანას უჭირავს X" = ერთი lookup-ი.

შეტყობინების ნაკადი — ბოლომდე გაჰყევი ხმამაღლა:

გამგზავნი ──ws──▶ gateway A ──▶ chat service ──▶ message store (persist)
                                     │              + seq-ის მინიჭება (§5)
                                     ▼
                          queue (conversation-ით დაყოფილი)
                                     ▼
                     routing lookup: მიმღები → gateway B?
                       online  → push gateway B-თი ──ws──▶ მიმღები
                       offline → per-device offline queue (§6)

Stateless ნაწილი. ისტორია, ძებნა, კონტაქტები, login: ჩვეულებრივი HTTP API-ები (GET /conversations/{id}/messages?before_seq=N) იმავე საცავზე — cache-ვადი, replicable, განზრახ მოსაწყენი. მოსაწყენი HTTP სიბრტყის socket-ის სიბრტყისგან გაყოფა ამ დიზაინის მთავარი სტრუქტურული სვლაა.

5 · Deep dive: შეტყობინებების ordering

რატომ ტყდება timestamp-ები. კლიენტის საათები იხრება (clock skew) — NTP კარგ მოწყობილობებს ათეულ მილიწამში აკავებს, ტელეფონები კი წამებით სცდება. თუ ბობის საათი 150 ms ჩამორჩება და ალისას 100 ms-ში პასუხობს, client timestamp-ით დალაგება პასუხს კითხვაზე წინ აჩენს. სერვერის მიღების timestamp-ები უკეთესია, მაგრამ მრავალკვანძა სერვისში მაინც რბოლაშია. შენი ლაბი ზუსტად ამ ბაგს აწარმოებს.

გამოსავალი: ერთი writer-ი თითო საუბარზე. C საუბრის ყველა შეტყობინება ერთ partition-ში გაუშვი (hash conversation_id), სადაც ერთი writer-ი ანიჭებს მონოტონურად მზარდ sequence ნომერს. თანმიმდევრობა ახლა ამ ერთ writer-თან ჩასვლით განისაზღვრება — საათი აღარ მონაწილეობს. ეს happens-before-ის პრაქტიკული სახეა: ფიზიკური დრო სხვადასხვა მანქანის ერთდროულ მოვლენებს ვერ ალაგებს; დანიშნული sequencer-ი — ალაგებს. კლიენტები seq-ით ხატავენ, ხარვეზები კი დაკარგვის დეტექტორია („მაქვს 41 და 43 — 42 თავიდან მომეცი").

Dedup-ი თან მოჰყვება (0011 გაკვეთილის idempotency-ის გამოძახილი): კლიენტი გაგზავნამდე შეტყობინებას UUID-ს აწებებს. Retry-ები იმავე ID-ს იყენებენ; სერვერი და მიმღები ნანახს ყრიან. Seq აძლევს თანმიმდევრობას, კლიენტის message ID — exactly-once ეფექტს: ორივე გჭირდება.

6 · Deep dive: მიწოდების გარანტიები

Receipt-ის „ჭკები" მდგომარეობათა მანქანაა, ყოველ გადასვლას ack მართავს:

მდგომარეობაგადასვლის ტრიგერი
sent (ერთი ჭკა)Chat service-მა შეტყობინება შეინახა და გამგზავნს ack-ი დაუბრუნა.
delivered (ორი ჭკა)მიმღების მოწყობილობამ მიღება ack-ით დაადასტურა თავისი gateway-ს გავლით.
read (ლურჯი)მიმღების კლიენტმა შეატყობინა, რომ საუბარი გაიხსნა.

გარანტია არის at-least-once + dedup. ქსელი კარგავს პაკეტებსაც და ack-ებსაც; ერთადერთი პატიოსანი პოლიტიკაა: retry სანამ ack არ მოვა, და redelivery უვნებელი გახადე message-ID dedup-ით (§5). Exactly-once მიწოდება დანაკარგიან ქსელში შეუძლებელია; exactly-once გამოჩენა — საინჟინროდ მიღწევადი.

Offline გზა. თუ routing lookup-მა ცოცხალი socket-ი ვერ იპოვა, შეტყობინება ჯდება per-device offline queue-ში, seq-ით დალაგებული. Reconnect-ზე მოწყობილობა თავის ბოლო cursor-ს აგზავნის და queue თანმიმდევრობით იცლება. Ack-პროტოკოლი კუმულაციურია — „ყველაფერი მაქვს seq 47-ის ჩათვლით" — ერთი დაკარგული ack იაფ redelivery-დ ჯდება და არა დაკარგულ შეტყობინებად. მობილური ნიუანსი ერთ წინადადებაში: OS-ის push შეტყობინება (APNs/FCM) მხოლოდ გაღვიძების სიგნალია; ნამდვილი შეტყობინებები queue-დან მოდის, როცა socket-ი დაბრუნდება.

7 · Deep dive: presence

Presence heartbeat-პროტოკოლია: კლიენტი socket-ზე ყოველ ~10 წამში ping-ავს; რამდენიმეს გამოტოვება = offline + last_seen-ის ჩაწერა. Disconnect-ის მოვლენა (gateway-მ socket-ის სიკვდილი შეამჩნია) იმავეს მყისიერად აკეთებს.

რატომ ფეთქდება გულუბრყვილო fanout-ი. ყოველი online↔offline გადასვლის ყველა კონტაქტთან push-ი = fanout-ი შენს ყველაზე მოციმციმე მოვლენების წყაროზე. მეტროში მგზავრობამ ტელეფონი საათში ათეულჯერ შეიძლება „აახამხამოს"; გაამრავლე ~200 კონტაქტზე და ასობით მილიონ მომხმარებელზე — presence-ის ტრაფიკი შეტყობინებებისას აჭარბებს, ისეთი განახლებებისთვის, რომლებსაც არავინ უყურებს.

სამი სტანდარტული ჩამხშობი:

  • Debounce. გადასვლა მხოლოდ მაშინ გამოაქვეყნე, როცა ათეულობით წამი სტაბილურია; 5 წამში მორჩენილი flap-ი სერვერს არ ტოვებს.
  • Pull გახსნაზე. აპის გახსნისას ჩატების სიისთვის presence წამოიღე. მუდმივი გამოწერა მათზე, ვისაც არ უყურებ — არა.
  • ვიწრო subscribe. ცოცხალი განახლებები მხოლოდ ეკრანზე გახსნილი საუბრისთვის; დახურვისას unsubscribe.

სენიორის წინადადება: „presence best-effort, eventually-consistent მონაცემია — სიამოვნებით ვაჩვენებ 30 წამით ძველ last_seen-ს, ოღონდ fanout ფენა არ დამიდნეს." მცდარი receipt-ი ბაგია; ოდნავ ძველი presence-წერტილი — სამშაბათი.

8 · Deep dive: ჯგუფური ჩატი, საცავი, multi-device

ჯგუფის fanout-ი. ერთი შეტყობინება 500-კაციან ჯგუფში = 1 ჩაწერა + 500-მდე მიწოდება: routing lookup + push ყოველ online წევრზე, offline-queue-ში ჩამატება ყოველ offline მოწყობილობაზე. ამიტომ აქვს Clarify-ს ლიმიტს მნიშვნელობა — ის ყველაზე ცუდ fanout-ს 500×-ზე კეტავს, feed-გაკვეთილის უსაზღვრო celebrity პრობლემისგან განსხვავებით. Chat service fanout-ს ერთი persist + seq-ის მინიჭების შემდეგ აკეთებს, ასე რომ ჯგუფსაც ერთი writer-ი და ერთი საერთო თანმიმდევრობა აქვს.

Message store. ჩაწერა-მძიმე, append-only, საუბრით საკითხავი — wide-column-ის ზუსტი ფორმა. Discord-ის messages-store ბლოგი ზუსტად ამას აღწერს Cassandra-ში: partition არხით (+ დროის bucket-ი, რომ ცხელმა არხებმა უსაზღვრო partition-ები არ გაზარდონ), clustering დროში დალაგებული message ID-ით. ჩვენი სქემა:

messages(
  PARTITION KEY (conversation_id, bucket),   -- bucket ≈ 10 დღე, ზომავს partition-ს
  CLUSTERING KEY seq DESC,                   -- ახლები წინ, დისკზევე დალაგებული
  sender_id, msg_id, body, sent_at)

„ჩატვირთე ეს საუბარი" = ერთი partition, უკვე დალაგებული — წაკითხვის პატერნი თავად სქემაა.

Multi-device sync = per-device cursor-ები: ყოველი მოწყობილობა ინახავს ბოლო seq-ს თითო საუბარზე და დამოუკიდებლად ეწევა იმავე offline-queue გზით. სერვერი queue-ს ჩანაწერს შლის, როცა ყველა რეგისტრირებული მოწყობილობის cursor-ი გასცდა (ან TTL ამოიწურა).

9 · Failure mode-ები + მონიტორინგი

ჩავარდნადარტყმის რადიუსიშერბილება
Gateway კვდება (1M socket-ი)1M კლიენტი ერთდროულად უკავშირდება — TLS handshake-ების, auth-ისა და routing-ჩანაწერების thundering herd-ი გადარჩენილ ფენაზეReconnect ჯიტერიანი exponential backoff-ით; სათადარიგო სიმძლავრე (N+2); gateway-ზე მდგრადი არაფერი ცხოვრობს — სიკვდილი კარგავს მხოლოდ socket-ებს, ნულ შეტყობინებას
Queue-ს lag-იშეტყობინებები აგვიანებს; ორივე online-შია, „delivered" კი ხოხავსPage-ი end-to-end მიწოდების p99-ზე და partition-ის consumer lag-ზე — ეს მეტრიკა თავად პროდუქტია
Routing ცხრილის split brainძველი ჩანაწერი ამბობს X gw42-ზეა; gw42 მკვდარია → push-ები შავ ხვრელში ქრებაTTL ჩანაწერებზე, gateway-ს heartbeat-ით განახლებადი; push ჩავარდა → მიმღები offline-დ ჩათვალე და offline queue-ზე გადადი (queue ყოველი ეჭვის safety net-ია)
Offline queue-ს გროვადაბრუნებულ მოწყობილობას 10k შეტყობინება ხვდებაგვერდებით დაცლა, ჯერ უახლესი საუბრები; retention-ის TTL, ძველი ისტორია HTTP გზით

Dashboard: მიწოდების p50/p99, socket-ების რაოდენობა gateway-ზე, reconnect-ის ტემპი (herd-დეტექტორი), queue lag-ი partition-ზე, receipt-გადასვლების ტემპები.

10 · ვარიანტები და twist-ები

  • E2E შიფრაცია. Signal-პროტოკოლის E2EE-თი (WhatsApp-ის რეალური deployment) სერვერი გადასცემს ciphertext-ს, რომელსაც ვერ კითხულობს: სერვერული ძებნა, სპამ-სკანირება და კონტენტის მოდერაცია ქრება; მიწოდება, ordering, receipt-ები და presence ცოცხლობს — მეტადატაა. მზად იყავი თქვა, რომელი ფუნქციები კვდება.
  • Read receipt-ების პრივატულობა. Receipt-ები ქცევას ამჟღავნებს; კლასიკური პოლიტიკა ორმხრივობაა — შენსას თუ გამორთავ, სხვისას ვეღარ ხედავ. პროდუქტის შეზღუდვა = ერთი სერვერული შემოწმება.
  • ფედერაცია. Matrix/XMPP-სტილის სერვერთაშორისი ჩატი შენს routing ცხრილს domain-თაშორისი routing-ითა და ნდობით ცვლის — trade-ი: წვდომა და ღიაობა ordering-ისა და abuse-კონტროლის ერთი ოპერატორის ხელიდან გაშვების ფასად.

AI twist-ი (ერთი აბზაცი, მხოლოდ თხოვნით). LLM ასისტენტი ჩატში უბრალოდ საუბრის კიდევ ერთი წევრია, რომლის პასუხებიც token-token-ად მოედინება — gateway იმავე socket-ზე partial-message ფრეიმებს აწვდის, საბოლოო შეტყობინებას კი per-conversation seq ალაგებს. იგივე ფენა, ახალი producer-ი; გაკვეთილი 37 ამას აშენებს.

11 · ინტერვიუს სასაუბრო პუნქტები

  • Twist-ის დასახელება: „Gateway-მდე ყველაფერი stateless-ია; gateway ფენა განზრახ ერთადერთი stateful ნაწილია, და Redis-ის routing ცხრილი ის გზაა, რომლითაც stateless სამყარო მას პოულობს."
  • რიცხვის დახარჯვა: „100M ერთდროული socket-ი — აი რატომ არის gateway ცალკე ფენა: ის მეხსიერებაზე და descriptor-ებზე სკალირდება, ჩატის ლოგიკა კი CPU-ზე."
  • Ordering: „Timestamp-ებით არ დავალაგებ — clock skew სწრაფ პასუხებს გადაალაგებს. საუბრის partition-ის ერთი writer-ი ანიჭებს seq-ს; საათები ხმას ვერ აძლევენ."
  • მიწოდება: „At-least-once + კლიენტის message-ID dedup. Exactly-once მიწოდება შეუძლებელია; exactly-once გამოჩენა — საინჟინრო."
  • დაუკითხავი failure mode: „მომაკვდავი gateway მილიონ socket-ს ყრის — საფრთხე დაკარგული შეტყობინებები კი არა (gateway-ს არცერთი უჭირავს), reconnect-ის შტორმია. ჯიტერიანი backoff, გზაში მყოფს კი offline queue იჭერს."
  • Presence-ის დისციპლინა: „Presence განზრახ best-effort-ია — debounce flap-ებზე, pull გახსნაზე, subscribe მხოლოდ ხილულ ჩატზე."

12 · ლაბი + პროექტის ნაბიჯი

პრაქტიკა (ქვიზამდე გააკეთე) labs/lab-0021-chat-ordering/-ში python3 chat_lab.py შენს მანქანაზე აწარმოებს §5–§6-ის ოთხივე ბაგს: clock skew-იანი timestamp-ები საუბარს ურევს, სერვერის seq ასწორებს, retry-დუბლიკატები ჩნდება და message-ID dedup-ით კვდება, offline queue კი თანმიმდევრობით იცლება. გაშვებამდე იწინასწარმეტყველე ყოველი ტრანსკრიპტი — README-ს E1–E4 გაგატარებს.

პროექტის ნაბიჯი. ერთი გვერდი პროექტის ჩანაწერებში: შენი inference პლატფორმის streaming ფენა როგორც ჩატის სისტემა — კლიენტებს socket-ები gateway ფენასთან უჭირავთ, „რომელი gateway-ა session X-ზე" იმავე routing ცხრილში ცხოვრობს, დაგენერირებული ტოკენები კი შეტყობინებებია per-request seq-ით. ჩაწერე ერთი ასიმეტრია (ტოკენები ცალმხრივად მიედინება; receipt-ები არ არსებობს) — ეს გვერდი 37-ე გაკვეთილის ჩონჩხი გახდება.
