Storage Engine Internals

B-tree vs LSM is the read-vs-write-amplification decision — the one database-internals topic interviews actually reward. Know why MySQL and Cassandra disagree about disks, and you can defend any datastore pick with mechanisms instead of brand names.

Mission tie. When an Anthropic prompt asks you to store something — usage events at 100k writes/s, order history, an inbox — "I'll use Postgres" is a brand, not a design. The follow-up is always "why that store, and what breaks under this write rate?" Candidates who can say "this is append-heavy, so an LSM engine trades read amplification for sequential-write throughput, and here's the compaction risk" are reasoning from internals — exactly the skill Chapter 3 of Designing Data-Intensive Applications exists to teach, and this lesson compresses.

1 · The log is the database

The simplest possible database is an append-only file: every write is append(key, value), and it is nearly unbeatable at writing — because appending is the one thing disks do at full speed. The Estimation Kit numbers: a spinning disk pays ~10 ms per random seek (~100 random IOPS) but streams sequentially at 100–200 MB/s — a gap of several thousand×. SSDs shrink the gap but keep it: random 4 KB I/O still costs a controller round trip per op, and small random writes trigger internal erase-block churn (ACM Queue, "The Pathologies of Big Data"). Sequential beats random on every storage medium you will ever be offered.

That is why every serious engine writes twice. The first write goes to a write-ahead log (WAL) — a pure append, forced to disk before the client is acknowledged; the second updates the engine's real data structure, possibly much later. Crash mid-update? Recovery replays the log from the last checkpoint and reconstructs the structure — the log is the truth, the structure is a cache of it (Postgres WAL docs; DDIA ch. 3). Hold onto this: both engine families below are just different answers to "the log is fast but unsearchable — how do we add search?"

2 · Indexes: pay at write time to save at read time

A raw log gives O(1) writes and O(N) reads — a full scan per lookup. An index is an additional structure, derived from the data, that makes some reads fast. DDIA ch. 3 states the law of the land: well-chosen indexes speed up reads, but every index slows down writes — each write must now also maintain every index. There is no free index; there is only a chosen trade. Storage engines differ exactly in which structure they maintain and when they pay the maintenance bill: B-trees pay at write time, in place; LSM-trees defer the bill and pay it in the background, in bulk. Everything else in this lesson is a consequence of that one sentence.

3 · B-trees: read-optimized order

The B-tree — standard since the 1970s and the engine inside Postgres, MySQL/InnoDB, Oracle, SQL Server — breaks the database into fixed-size pages (traditionally 4 KB; InnoDB uses 16 KB) and arranges them as a tree: each internal page holds several hundred sorted keys with child-page references, leaves hold the rows (Postgres B-tree internals; DDIA ch. 3).

Worked depth example. With a branching factor of 500, depth grows as log500(N): one page addresses 500 children, two levels 250,000, three levels 125 million, and a four-level tree of 4 KB pages addresses ~5004 ≈ 62 billion leaf references — roughly 256 TB of data. Four page reads for any key in a quarter-petabyte, and the top levels are always cached. That is why B-tree read latency is famously flat and predictable.

Writes go in place: find the leaf, overwrite it. A full leaf splits into two, updating the parent — and because a crash between those multi-page steps would corrupt the tree, B-trees keep their own WAL (§1) and coordinate concurrent access with latches (short-lived page locks). Where B-trees win: point reads, range scans (leaves are in key order — "orders for customer 4711, newest 50" walks sibling pages), read-heavy mixes, and decades-mature transaction machinery built on top. E-commerce order history is the archetype: read far more than written, queried by ranges, wrapped in transactions.

4 · LSM-trees: write-optimized order

The log-structured merge-tree (O'Neil et al., 1996) refuses to do random writes at all. The path: writes go to an in-memory sorted memtable (plus WAL for crash safety); when it reaches a threshold, it is flushed — one sequential write — as an immutable sorted file, an SSTable; background compaction merges SSTables into fewer, bigger, sorted ones and drops overwritten values. Nothing is ever updated in place; the disk sees only appends and streaming merges.

Reads pay for this: a lookup checks the memtable, then SSTables newest-to-oldest, until the key appears. What rescues point reads is a Bloom filter per table — a tiny probabilistic set that answers "definitely not here" without touching the table, so absent tables cost one probe instead of a search (RocksDB wiki: Bloom Filter). The lineage runs from the 1996 paper through Google's Bigtable (OSDI '06) to HBase, Cassandra, LevelDB, and RocksDB (RocksDB wiki). Where LSMs win: sustained write-heavy ingest. A messaging inbox is the archetype — every message is a write, reads mostly touch the recent end, and Cassandra's LSM engine soaks the firehose that would grind a B-tree's random-I/O write path to dust.

5 · The amplification triangle

The engines are compared on three amplifications — and the RUM conjecture (Harvard DASlab) says you can optimize any two of Read, Update, and Memory/space cost only by sacrificing the third:

  • Read amplification — pieces of storage consulted per logical read (SSTables probed, pages descended).
  • Write amplification — bytes physically written per byte logically written.
  • Space amplification — bytes stored per byte of live data.

The arithmetic interviewers want to hear. A B-tree updating a 100-byte row must rewrite its whole 4 KB page — 40× write amplification before counting the WAL copy and any splits. An LSM with leveled compaction (fanout 10) rewrites each entry roughly 10× per level it descends; with 3–4 levels that is ~30–40× write amplification — but every one of those bytes moves in large sequential merges, which is why the LSM still wins the ingest race on real disks (RocksDB wiki: Compaction). In the lab's size-tiered mini-LSM, 100k inserts cost 292k entry-writes — write amp 2.92×, one rewrite per tier merge.

Compaction strategyWrite ampRead ampSpace ampHome
Leveled — one sorted run per level, each ~10× largerHigh (~10 per level)Low (≤ 1 table per level)Low (~1.1×)LevelDB, RocksDB default
Size-tiered — merge similar-sized tables when a tier fillsLow (~1 per tier merge)High (many overlapping tables)High (up to ~2× until merges)Cassandra STCS default

Compaction is a background tax that shares disk and CPU with foreground traffic. If ingest bursts faster than compaction can merge, tables pile up, reads slow, and engines eventually throttle or stall writes to let compaction catch up (RocksDB wiki: Write Stalls). An LSM's write throughput number is only honest if quoted with its compaction debt at steady state.

6 · Head-to-head: which engine when

The decision table to reproduce on a whiteboard (DDIA ch. 3, "Comparing B-Trees and LSM-Trees"):

DimensionB-tree (Postgres, MySQL/InnoDB)LSM (Cassandra, RocksDB, LevelDB)
Point readsFlat, predictable — one short descentGood with Bloom filters; degrades as tables accumulate
Range scansExcellent — leaves in key order, walk siblingsWeaker — must merge every table; Blooms can't help ranges
Write throughputLimited by random in-place page I/OExcellent — sequential flushes and merges only
SpaceFragmentation: pages run ~⅔ fullTighter — compaction rewrites densely, compresses runs well
Latency predictabilityHigh — no background rewrites of noteCompaction bursts cause p99 spikes and write stalls
Transactions maturityDecades of MVCC, locking, constraint machineryThinner — typically per-key atomicity, app-level everything else

Rule of thumb: read-mostly, relational, transactional → B-tree (orders, accounts, catalog). Write-heavy, append-mostly, huge ingest → LSM (events, messages, time series, logs). Say the workload shape first; the engine follows.

7 · Object vs block vs file storage

Engines need somewhere to put bytes; interviews increasingly probe the three storage models beneath them:

ModelUnit & APIPartial update?Classic home
BlockFixed-size blocks; read/write by offset (EBS, SAN)Yes — the whole pointUnder databases and filesystems
FileHierarchical paths, POSIX open/seek/write (NFS, EFS)Yes — in-place rangesShared app directories, legacy software
ObjectFlat keyspace of immutable blobs + metadata over HTTP (S3, GCS)No — replace the whole objectMedia, backups, data lakes, SSTables

The S3 model in one breath: objects are written whole and replaced whole — appending 200 bytes to a 1 GB object means uploading a new 1 GB object (or multipart-composing a new one); each object carries user metadata; LIST and reads are strongly consistent after every write since December 2020 (AWS S3 consistency); and durability is designed for eleven nines — 99.999999999%, i.e. store 10 million objects and expect to lose one every ~10,000 years, achieved by erasure-coding across devices and zones (AWS S3 durability docs). Object storage replaces a filesystem exactly when data is write-once, read-many, large — which is why modern LSM engines increasingly tier their immutable SSTables straight into S3: the storage model and the file format finally agree (DDIA ch. 3 covers the immutable-segment idea these designs build on).

8 · The failure modes that decide interviews

  • Compaction debt spiral. A sustained write burst outruns compaction; SSTables accumulate; every read now consults more tables, so read latency climbs while compaction competes for the same disk; the engine finally imposes write stalls — your "write-optimized" store rate-limits writes (RocksDB wiki: Write Stalls). Mitigations: provision compaction headroom (ingest ≤ ~½ of merge throughput), rate-limit ingest upstream, watch pending-compaction bytes as a first-class alert.
  • Unbounded tombstones. In an LSM, a delete is a write — a tombstone marker that must persist until compaction has provably merged it past every older copy. Delete-heavy workloads (queues-in-Cassandra, purged inboxes) build tombstone fields that reads must scan through before finding live data: Cassandra aborts queries past 100k tombstones by default, and slicing through tens of thousands of them is a classic production fire (The Last Pickle: About Deletes and Tombstones; Discord hit exactly this reading messages in heavily-deleted channels — How Discord Stores Billions of Messages). Fix the data model: don't build queues on deletes; bucket by time and drop whole partitions/tables instead.
  • Random keys vs B-tree pages. Insert with random UUIDv4 primary keys and every insert lands on a random leaf page: the buffer pool thrashes, pages split constantly and sit half-full, and write amplification explodes — Percona measured InnoDB falling off a cliff once the index outgrows RAM. Sequential-ish keys (auto-increment, ULID, UUIDv7) keep all inserts on the rightmost leaf — which is why Shopify's payments team uses ULIDs for idempotency keys and cut an INSERT's latency substantially by doing so (Shopify: Building Resilient Payment Systems). In the lab: switching insert order from sequential to random takes the mini-B-tree's write amp from ~2× to ~44×.

9 · One AI note, then back to fundamentals

The KV-cache manager in our LLM inference platform is a storage engine wearing a GPU costume: token generation is an append-heavy workload; PagedAttention (vLLM) allocates the cache in fixed-size blocks with an indirection table — literally the page-and-pointer trick from §3, applied to attention state to kill fragmentation; eviction and defragmentation of finished sequences are compaction's cousins, background work that reclaims space while racing foreground traffic; and prefix sharing is deduplication. Same trade triangle, same vocabulary, pricier bytes. That is the whole note — §1–§8 are what the interview is grading.

10 · Interview talking points

How to deploy this lesson live, in order:

  1. Classify the workload before naming a store: read/write ratio, point vs range reads, delete rate, transaction needs. The engine falls out of the shape.
  2. Anchor on the disk: "sequential I/O is thousands of times cheaper than random — B-trees pay random writes for cheap reads, LSMs pay deferred merges for sequential writes."
  3. Quote the depth math (§3): "branching 500, four levels, ~256 TB — B-tree point reads are flat," and the write-amp math (§5): "4 KB page per 100 B row is 40×."
  4. Name the three amplifications and say which one your choice sacrifices — that single sentence signals you know RUM is a trade, not a defect.
  5. Raise the failure mode before the interviewer does (§8): compaction debt for LSM picks, random-key page splits for B-tree picks, tombstones for delete-heavy designs.
  6. Push immutable blobs to object storage (§7) and say the constraint out loud: "no partial updates — whole-object replace — so it only fits write-once data."
  7. Close with the operational tell: "I'd alert on pending compaction bytes / buffer-pool miss rate" — engine choices come with engine-specific pager duty.

12 · Project step

Two decisions for the LLM inference platform: pick the storage engine for (a) the request/usage log (every request appended, billed, rarely re-read) and (b) the metadata store (users, API keys, quotas, model configs), and defend each in two sentences. A defensible pick: (a) an LSM engine (RocksDB-backed, or Kafka-then-object-storage) because the log is pure append-heavy ingest that never updates in place, accepting read amplification we rarely pay; (b) Postgres, because metadata is small, read-mostly, and needs transactions and constraints — B-tree territory, and its write volume will never stress the page-update path. Write your own two-sentence pair — workload shape first, engine second — and bring it to the next session for critique.
