API დიზაინი, სიღრმისეულად

API დაპირებაა, რომელსაც წლობით ასრულებ. სემანტიკა, versioning (ვერსიონირება), pagination (გვერდებად დაყოფა) და idempotency (იდემპოტენტურობა) — სწორედ აქ განსხვავდებიან senior კანდიდატები, რადგან ზარმაცი არჩევანი აქ ორმაგ ჩამოჭრად ან ორჯერ გაგზავნილ შეკვეთად იქცევა.

მისიასთან კავშირი. Anthropic-ის ყველა დიზაინ-ამოცანა ადრე თუ გვიან API-მდე მიდის: „დააპროექტე ამ სერვისის საჯარო ინტერფეისი" — და დამატებითი კითხვები ყოველთვის ერთი და იგივეა: „რა ხდება, როცა კლიენტს timeout მოუვა და ის POST-ს გაიმეორებს?" და „როგორ გაუშვებ breaking change-ს 10,000 ინტეგრაციაზე?" კანდიდატები, რომლებიც ასახელებენ idempotency key-ებს, cursor pagination-ს და deprecation (მოძველების) პოლიტიკას, senior სიგნალს იძლევიან. მასალა კლასიკური სისტემებია: Stripe-ის გადახდები, GitHub-ის API, e-commerce checkout. (Exponent-ის 2026 Anthropic გზამკვლევი)

1 · REST-ის სემანტიკა სწორად

REST-ის ნამდვილი შინაარსი „ლამაზი URL-ები" კი არა, HTTP მეთოდების ერთიანი კონტრაქტია (RFC 9110). ორი თვისება აკეთებს მთელ საქმეს: safe (უსაფრთხო — სერვერის state არ იცვლება, ე.ი. თავისუფლად ქეშირდება) და idempotent (იდემპოტენტური — N იდენტური მოთხოვნა ≡ 1, ე.ი. თავისუფლად მეორდება). ინტერნეტის ყველა retry-პოლიტიკა და proxy ამ ცხრილზეა აშენებული:

მეთოდიSafeIdempotentკონტრაქტი
GETკიკიწაკითხვა; GET-ზე არასდროს შეცვალო state
PUTარაკისრული ჩანაცვლება ცნობილ URI-ზე; გამეორება იმავე state-ს ტოვებს

POSTარაარაშექმნა/დამუშავება; გამეორებამ შეიძლება ორჯერ შექმნას — §3 ამ სტრიქონის გამო არსებობს

Status code-ებიც კონტრაქტია: 200/201/202/204 (OK / შეიქმნა / ასინქრონულად მიღებულია / პასუხის გარეშე), და წყვილები, რომლებსაც ინტერვიუერები ამოწმებენ: 400 vs 422 (გაფუჭებული სინტაქსი vs სწორი ფორმის, მაგრამ სემანტიკურად უვარგისი), 401 vs 403 (ვინ ხარ? vs არ გაქვს უფლება), 409 (კონფლიქტი — optimistic locking), 429 (ბევრი მოთხოვნა) და 503 + Retry-After (გადატვირთულია და კლიენტს ეუბნები, როდის დაბრუნდეს). რესურსების მოდელი: არსებითი სახელები (POST /orders და არა POST /createOrder), მაქსიმუმ 1–2 დონის nesting, ფილტრები query პარამეტრებით, PATCH ნაწილობრივი განახლებისთვის (Stripe: payment API design, Microsoft REST Guidelines).

2 · Pagination, სიღრმისეულად: offset vs cursor

არითმეტიკა, რომლის მოსმენაც ინტერვიუერს უნდა. LIMIT 20 OFFSET 1000000 database-ს აიძულებს, ჯერ მილიონი სტრიქონი გაიაროს და გადააგდოს — ღრმა გვერდის ფასი offset-ის პროპორციულია: ~1 წამი DB-დრო თითო მოთხოვნაზე. Cursor-ული query — WHERE (created, id) < (:cursor) … LIMIT 20 — B-tree-ში seek-ია: 3–4 გვერდის წაკითხვა, მილიწამზე ნაკლები, სიღრმის მიუხედავად (Use The Index, Luke: No Offset). და უარესი: offset არასტაბილურია პარალელური ჩაწერებისას — Slack-მა API-ის pagination სწორედ ამიტომ გადააშენა cursor-ებზე (Evolving API Pagination at Slack), Stripe-ის ყველა list endpoint cursor-ულია (Stripe pagination).

3 · Idempotency key-ები: ორმაგი ჩამოჭრის პრობლემა

სცენარი, რომელიც გადახდების ინტერვიუებს წყვეტს: კლიენტი იძახებს POST /payments-ს; სერვერი ბარათს ჩამოაჭრის; პასუხი იკარგება. კლიენტი ვერ არჩევს „არ მომხდარა"-ს „მოხდა, დადასტურება დაიკარგა"-სგან — ამიტომ იმეორებს, და კლიენტს ორჯერ ჩამოეჭრება. გამეორება სწორი ქცევაა; უბრალოდ POST იდემპოტენტური არ არის.

Stripe-ის გამოსავალი (idempotent requests docs, idempotency ბლოგი): კლიენტი თითო ლოგიკურ ოპერაციაზე აგენერირებს უნიკალურ Idempotency-Key header-ს და ყოველ გამეორებაზე ერთსა და იმავეს იყენებს. სერვერს აქვს key store: key პირველად მოვიდა → შეასრულე და პასუხი შეინახე; იგივე key მეორედ → შენახული პასუხის replay ხელახლა შესრულების გარეშე. Key-ები TTL-ით იწმინდება (Stripe — 24 საათი): TTL ყველაზე გრძელ სავარაუდო retry-ფანჯარაზე მეტი უნდა იყოს, თორემ დაგვიანებული გამეორება key-ს ვეღარ იპოვის და ისევ შეასრულებს (ლაბის E2-ში ამას შენ თვითონ გატეხავ). რიცხვები: 30% პასუხის დაკარგვა + 3 retry → მოსალოდნელი შესრულებები = 1 + 0.3 + 0.3² + 0.3³ ≈ 1.42 თითო შეკვეთაზე — გულუბრყვილო endpoint შეკვეთების ~42%-ს აორმაგებს. ლაბში: ~90 დუბლი 200 შეკვეთაზე key-ის გარეშე, 0 — key-ით.

4 · Versioning და webhook-ები

Versioning: დანამატი ცვლილებები (ველის დამატება) ვერსიას არ საჭიროებს, თუ კლიენტები უცნობ ველებს უგულებელყოფენ. Breaking ცვლილებებს სტრატეგია სჭირდება: URL-ვერსია (/v1/ — თვალსაჩინო, მაგრამ უხეში), header-ვერსია (Stripe-Version: 2024-06-20 — წვრილმარცვლოვანი) ან additive-only (არასდროს გატეხო). ოქროს სტანდარტი Stripe-ის თარიღებზე დაფუძნებული მოდელია (Stripe: APIs as infrastructure): ყოველი ანგარიში თავის თარიღიან ვერსიაზეა მიმაგრებული (pinned), breaking ცვლილება ახალი თარიღით გამოდის, თავსებადობის ფენა კი ძველ მოთხოვნებს წინ თარგმნის — 2016 წლის ინტეგრაცია დღემდე უცვლელად მუშაობს. აუცილებელია გამოქვეყნებული deprecation პოლიტიკა: ანონსი, Sunset header-ები, რეალური ფანჯარა.

Webhook-ები — push, polling-ის ნაცვლად (Stripe webhooks, GitHub webhook validation): (1) ხელმოწერის შემოწმება — HMAC-SHA256 header-ში, თორემ URL-ის მპოვნელი ნებისმიერი „payment succeeded"-ს გაგიყალბებს; (2) at-least-once მიწოდება backoff-იანი გამეორებებით — ე.ი. დუბლიკატები გარანტირებულია და მომხმარებელმა event id-ზე dedup უნდა გააკეთოს; (3) თანმიმდევრობა გარანტირებული არ არის — event სიგნალია მიმდინარე state-ის გადამოწმებისთვის და არა log; (4) უპასუხე 2xx-ით სწრაფად, რეალური სამუშაო რიგში ჩააგდე.

6 · Failure mode-ები, რომლებიც ინტერვიუებს წყვეტენ

  • ჩუმად გაშვებული breaking change. პარასკევს „უწყინრად" გადარქმეული ველი — ორშაბათს 10,000 ინტეგრაცია null-ს კითხულობს. გამოსავალი: ვერსიის pinning + deprecation ფანჯარა + კონტრაქტ-ტესტები CI-ში.
  • Retry storm არაიდემპოტენტურ endpoint-ზე. Latency-პიკი → timeout-ები → ყველა იმეორებს POST-ს → ყოველი გამეორება საქმეს ხელახლა აკეთებს და დატვირთვას ამატებს. Idempotency key-ები გამეორებას უსაფრთხოს ხდიან; backoff + jitter — ზრდილობიანს. ორივე გჭირდება.
  • Webhook-მომხმარებელი dedup-ის გარეშე. At-least-once მიწოდება + handler, რომელიც თითო event-ზე შეკვეთას აგზავნის = ერთი შეკვეთის ორი გზავნილი. Stripe-ის დოკებში წერია, უბრალოდ არავინ კითხულობს.
  • Pagination drift-ით ორჯერ დამუშავებული შეკვეთები. ღამის batch /orders?offset=…-ს დადის, checkout კი ჩაწერას აგრძელებს: სტრიქონები ინაცვლებენ, გვერდი N წინა გვერდის ბოლოს ხელახლა კითხულობს — ზოგი შეკვეთა ორჯერ იგზავნება, უძველესებამდე კი ვერასდროს აღწევს. ლაბში: 120-დან 44 დუბლირებული და 44 გამოტოვებული სტრიქონი.
LLM შტრიხი Inference API REST-ს სამ ადგილას ღუნავს: streaming — token-token პასუხი ნიშნავს SSE-ს, სადაც status code გენერაციის დასრულებამდე ფიქსირდება და შუა-stream შეცდომები in-band event-ებად მოდის; idempotency key-ები გენერაციის მოთხოვნებზე — გამეორებული POST აქ GPU-წუთებს აორმაგებს ბარათის ნაცვლად, იგივე §3-ის წამალი, უფრო ძვირი შეცდომა; token-ებით გაზომილი rate limit-ები — კვოტა tokens/min-ია და არა requests/min. იგივე ფუნდამენტები, ახალი ერთეულები.

7 · ლაბი — ჩამოაჭერი კლიენტს ორჯერ, მერე შეაჩერე საკუთარი თავი

პრაქტიკა (ქვიზამდე გააკეთე) ყველაფერი ზემოთქმული ახლა შემოწმებადია: labs/lab-0011-idempotency-api/-ში python3 idempotency_lab.py უშვებს ნამდვილ პატარა payments API-ს (stdlib http.server, ცალკე thread-ში) და სამ ექსპერიმენტს — გულუბრყვილო retry 30% პასუხის დაკარგვისას (დათვალე ორმაგი ჩამოჭრები ledger-ში), იგივე ტრაფიკი Idempotency-Key header-ითა და სერვერის key store-ით (ნული დუბლი), და offset vs cursor pagination ცოცხალი ჩაწერების ფონზე (offset დუბლირებს და კარგავს, cursor სუფთა რჩება). შემდეგ README-ს E1–E4: იწინასწარმეტყველე, შეცვალე კონფიგი, გაუშვი თავიდან, შეადარე. მცდარი პროგნოზები დავალების შედეგია — მოიტანე შემდეგ სესიაზე.

8 · პროექტის ნაბიჯი

გადაწყვეტილების დროა LLM inference gateway-სთვის: დახაზე მისი საჯარო API — სამი endpoint, idempotency-ის ამბავი და pagination-ის არჩევანი — და ორი წინადადებით დაიცავი. დასაცავი მაგალითი: POST /v1/generations (SSE streaming, სავალდებულო Idempotency-Key 24-საათიანი store-ით, რომ კლიენტის გამეორებებმა GPU-დრო არასდროს დააორმაგონ), GET /v1/generations/{id} (stream-ის ჩავარდნის შემდეგ გადამოწმება) და GET /v1/generations?starting_after=… (opaque cursor, რადგან დაფები ღრმად კითხულობენ და გენერაციები მუდმივად ემატება). თარიღზე მიმაგრებული ვერსიები პირველივე დღიდან, token-ებით გაზომილი 429-ები Retry-After-ით. დაწერე შენი ესკიზი და მოიტანე შემდეგ სესიაზე გასარჩევად.

ძირითადი წყარო
