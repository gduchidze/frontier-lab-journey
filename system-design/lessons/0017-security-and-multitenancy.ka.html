უსაფრთხოება & Multi-Tenancy

ვინ ხარ, რის უფლება გაქვს, სად ცხოვრობენ secret-ები და როგორ ატარებ tenant A-ს tenant B-ს მონაცემებიდან და latency-დანაც — თავდაცვითი ლექსიკა, რომელიც „დაამატე auth"-ს დიზაინად აქცევს.

მისიასთან კავშირი. Anthropic-ის დიზაინის ამოცანებს — „დააპროექტე multi-tenant SaaS პლატფორმის API" — ყოველთვის ამოსდის უსაფრთხოების კიდური: „როგორ არ მოქმედებს ერთი კლიენტის ტრაფიკი მეორეზე? საიდან იცი, რომ ამ მოთხოვნას ამ row-ის წაკითხვის უფლება აქვს?" ვინც „JWT-ებს გამოვიყენებთ"-ით ისვენებს, ჩერდება; ვინც ასახელებს authN vs authZ-ს, token-ის audience შემოწმებას, tenant-isolation მოდელს და rate-limit-ის განზომილებას, აგრძელებს. (Exponent-ის 2026 Anthropic გზამკვლევი)

1 · პერიმეტრი მკვდარია — zero trust

ძველი მოდელი ციხე იყო: მაგარი firewall გარეთ, რბილი შიგნით — და როცა „ქსელის შიგნით" იყავი, გენდობოდნენ. ეს მოკვდა, როცა ერთმა phishing-ლეპტოპმა თუ ერთმა კომპრომეტირებულმა სერვისმა მთელი შიდა ქსელი შეძლო შემოევლო. ჩამნაცვლებელი — zero trust (ნულოვანი ნდობა): შიგნით არაფერია. ყოველი hop თავს იმოწმებს (authenticate), ყოველი hop დაშიფრულია გზაში (encrypt in transit), ყოველი identity იღებს least privilege-ს (მინიმალურ პრივილეგიას) (Google, BeyondCorp).

სამი პრინციპი: authenticate every hop („შიგნიდან მოვიდა" credential არაა); encrypt in transit ყოველთვის, edge-ზეც და შიდა სერვისებს შორისაც; least privilege — გაჟონილმა credential-მა ერთი ოთახი უნდა გააღოს და არა შენობა. მათ ერტყმის defense in depth (თავდაცვა სიღრმეში) — შრეებრივი კონტროლი, სადაც ერთი ჩავარდნა კატასტროფა არაა. სია, რასაც ეს შრეები უნდა უძლებდნენ, არის OWASP Top 10.

2 · AuthN vs AuthZ

ინტერვიუზე ყველაზე ხშირი შეცდომა — authentication-ისა და authorization-ის ერთ რამედ აღრევა. ეს ცალკე სისტემებია, რომლებიც ცალკე ვარდებიან.

Authentication (authN)Authorization (authZ)
კითხვავინ ხარ?რის უფლება გაქვს?
აწარმოებსდადასტურებულ identity-სallow / deny გადაწყვეტილებას
როდისერთხელ edge-ზე, სესიაზეყოველ resource-ზე, ყოველ ქმედებაზე
ტიპური ჩავარდნამოპარული პაროლი, გაყალბებული tokenვალიდური user კითხულობს სხვა tenant-ის row-ს
მოდელებიPasswords, OAuth, mTLS, passkeysRBAC, ABAC, row-level security, scopes

ხაფანგი: მოთხოვნა შეიძლება იდეალურად authenticated იყოს და მაინც unauthorized. tenant A-ს user უნაკლო token-ითაც უნდა უარეყოს tenant B-ს მონაცემი — და ეს authZ-ის საქმეა, შემოწმებული resource-ზე. Session cookie (სერვერი ინახავს state-ს, cookie მხოლოდ ID-ს; revocation = row-ის წაშლა) ერთ web-აპს უხდება; token (bearer, stateless verification) — API-ებსა და service mesh-ს (OWASP Session Management).

3 · OAuth2 & OIDC

OAuth2 ერთ კითხვას პასუხობს: როგორ იმოქმედოს აპმა user-ის სახელით მისი პაროლის ფლობის გარეშე? ეს delegated authorization-ის ფრეიმვორკია, არა login-პროტოკოლი. ოთხი როლი (RFC 6749, oauth.net):

როლივინაა
Resource owneruser, ვისი მონაცემიცაა
Clientაპი, რომელსაც წვდომა უნდა
Authorization servertoken-ებს გასცემს consent-ის შემდეგ
Resource serverმონაცემს ინახავს, token-ს ამოწმებს

Authorization-code flow PKCE-ით (იცოდე ზეპირად): client user-ს authorization server-ზე გადაამისამართებს code_challenge-ით (ახლახან დაგენერირებული საიდუმლოს hash). user შედის და თანხმდება. server უკან აბრუნებს მოკლევადიან authorization code-ს. client მერე token endpoint-ს ურეკავს code-ითა და ორიგინალი code_verifier-ით; server verifier-ს hash-ავს და amოწმებს challenge-ს. PKCE (RFC 7636) ხურავს თავდასხმას, სადაც მავნე აპი გადამისამართებულ code-ს იპარავს — მოპარული code უსარგებლოა verifier-ის გარეშე. Client-credentials flow კი user-ის გარეშეა: სერვისი თავის თავად imოწმებს machine-to-machine ზარისთვის.

OIDC არის identity-შრე, რომელიც OAuth2-ს აკლია: OAuth2 წვდომას გცემს, არასდროს გეუბნება ვინ შევიდა. OIDC ამატებს ხელმოწერილ ID token-ს (JWT identity claim-ებით) (OIDC Core). დასამახსოვრებელი: access token = რის უფლება გაქვს; ID token = ვინ ხარ; OIDC = OAuth2 + identity.

4 · JWT, გულახდილად

JSON Web Token სამი base64url ნაწილია წერტილებით — header.payload.signature (RFC 7519). payload ინახავს claim-ებს (sub, iss, aud, exp). მოგება — stateless verification: ნებისმიერ სერვისს, key-ის მფლობელს, ლოკალურად შეუძლია გადამოწმება, auth-database-ზე გასვლის გარეშე. მაგრამ სწორედ ესაა ხაფანგი: JWT-ს ვერ გააუქმებ. მიტიგაცია: მოკლე TTL + refresh token-ები (access token წუთებია, revocable refresh token ახლებს ჭრის) და denylist (jti-ების პატარა სია „ახლავე გააუქმე"-სთვის, რაც state-ს აბრუნებს).

ხაფანგები: alg=none — ძველი ბიბლიოთეკები „ალგორითმის გარეშე"-ს პატივს სცემდნენ და signature-ს ტოვებდნენ; მკაცრმა verifier-მა უნდა დააფიქსიროს მოსალოდნელი ალგორითმი (Auth0: JWT vulnerabilities — ამას ლაბში გატეხავ). Key confusion (RS256→HS256) — public key-ს HMAC secret-ად იყენებენ. Audience unchecked — X-ისთვის ჭრილი token-ს Y იღებს; შეამოწმე aud, iss, exp, არა მხოლოდ signature. Secret-ები claim-ებში — payload ხელმოწერილია, არა დაშიფრული; base64-ია, ყველასთვის წაკითხვადი.

JWT (თვითაღწერადი)Opaque token + introspection
Verifyლოკალური, offlineქსელური ზარი auth server-ზე
Revocationრთული — TTL ან denylistმყისიერი — row-ის წაშლა
N სერვისზეიაფადauth server ცხელი გზა ხდება
საუკეთესოდიდი fan-out შიდა APIმგრძნობიარე, revocation-კრიტიკული

Opaque token = შემთხვევითი სტრიქონი, რომელსაც resource server introspection endpoint-ს ეკითხება (RFC 7662): revocation მყისიერი, მაგრამ ქსელური hop ყოველ გადამოწმებაზე (Okta).

5 · Service-to-service — mTLS

მექანიზმიროგორ ამტკიცებსსისუსტე
Shared API keyერთი მხარე secret-ს აგზავნისცალმხრივი; გაჟონვა უხმაუროა
Signed tokenხელმოწერილი identity claimBearer — მოპარული სრულად გამოსადეგია
mTLSორივე მხარე certificate-ს წარადგენსCert-ის lifecycle რეალური სამუშაოა

Mutual TLS — zero-trust default შიდა ტრაფიკისთვის: ჩვეულებრივი TLS სერვერის identity-ს ამტკიცებს, mTLS კი client-საც აიძულებს certificate წარადგინოს, ორივე ბოლო კრიპტოგრაფიულად იდენტიფიცირებულია. ცერტიფიკატების გაცემა/როტაცია სწორედ იმიტომ არსებობს service mesh. SPIFFE/SPIRE ყოველ workload-ს verifiable identity-ს (SVID) აძლევს და ავტომატურად ანახლებს (SPIFFE); Istio mesh-ს mTLS-ს ჩართავს აპის ცვლილების გარეშე (Istio security).

6 · Secret-ების მართვა

პირველი ჩავარდნა ალგორითმზე მაღლაა: secret კოდში, ან .env git-ში. git-ის ისტორიაში მოხვედრილი credential სამუდამოდ კომპრომეტირებულია — როტაცია, არა უბრალოდ წაშლა. სწორი მოდელი — ცენტრალური secrets manager (HashiCorp Vault): ცენტრალური სტორი (არა გაფანტული ფაილები); short-lived dynamic credentials (Vault-ის დაგენერირებული პაროლი, წუთებში გაუქმებული); audit log (ვინ, რა, როდის); rotation როგორც რუტინა.

Envelope encryption data-at-rest-ისთვის: KMS ინახავს root key-encryption key-ს (KEK), რომელიც არასდროს ტოვებს hardware-ს; ყოველ dataset-ს აქვს თავისი data-encryption key (DEK), რომელიც მონაცემს შიფრავს, თვითონ კი DEK ინახება მონაცემთან, KEK-ით დაშიფრული. root-ის როტაცია = DEK-ების ხელახლა შეფუთვა, არა პეტაბაიტების ხელახლა დაშიფვრა (AWS KMS). ასე ხდება per-tenant key-ები პრაქტიკული — DEK ყოველ tenant-ზე, ერთ KEK-ით შეფუთული.

7 · Rate limiting როგორც უსაფრთხოების კონტროლი

Rate limiting-ის ღირებულების ნახევარი უსაფრთხოებაა: brute force (5 მცდელობა/წუთი/account მილიონ-გამოცნობიან თავდასხმას საუკუნეებად აქცევს), credential stuffing (გატეხილი წყვილების გადათამაშება — გაფანტულია account-ებზე, per-account limit ვერ იჭერს, per-IP და global გჭირდება — OWASP), scraping, და abuse economics — შენი საქმეა თავდასხმა უფრო ძვირი გახადო, ვიდრე მოგება.

განზომილებაიჭერსბრმა წერტილი
Per-API-keyერთი კლიენტის გაქცეული integrationkey-ების როტაცია
Per-IPერთი მავნე hostNAT (მთელი ოფისი = 1 IP); botnet
Per-tenantერთი tenant, პლატფორმის მშიერიcross-tenant განაწილებული abuse

ორი დიზაინის შენიშვნა: დააბრუნე 429 Too Many Requests Retry-After-ით; და დაიცავი თვითონ limiter — იაფი უარყოფა ძვირ სამუშაომდე. token-bucket-ის შემოწმებას database-read არ უნდა სჭირდებოდეს, თორემ თვითონ check ხდება DoS. უიაფესი reject რაც შეიძლება ადრე გზაზე, სადაც endpoint უშვებს, auth-ამდე. გაკვეთილი 18 overload-სა და load-shedding-ს გააგრძელებს (Google SRE: Handling Overload).

8 · Multi-tenancy-ის isolation მოდელები

ერთი პლატფორმა, ბევრი tenant, ვისი მონაცემიც და წარმადობაც არ უნდა გადაერიოს. ცენტრალური გადაწყვეტილება (დაიმახსოვრე — SaaS პასუხის ხერხემალი, AWS SaaS Lens):

მოდელიგანლაგებაIsolationღირებულებაOps
SiloDB ყოველ tenant-ზეყველაზე ძლიერი — ფიზიკურიყველაზე ცუდი — ფასი per tenantmigration/backup ×N
Bridgeსაერთო DB, schema per tenantსაშუალო — ლოგიკურიუკეთესი — ერთი კლასტერიmigration ×N schema
Poolსაერთო tables, tenant_idსუსტი — filter ერთადერთი კედელიასაუკეთესო — სრული სიმჭიდროვედაბალი ops, დიდი blast radius

მათემატიკა, რასაც ინტერვიუერი ელის. 500 tenant. Silo: პატარა managed Postgres ~$50/თვე, ანუ 500 × $50 = $25,000/თვე მხოლოდ database-ის იატაკზე, პლუს 500 backup და 500 migration ყოველ რელიზზე. Pool: ერთი 3-node კლასტერი ~$2,000/თვე ≈ $6,000/თვე ყველა 500-ზე, ერთი migration რელიზზე. ~4-ჯერადი ინფრასტრუქტურის სხვაობა და N-ჯერ vs მუდმივი ops — ეს იცვლება, როცა ერთი tenant ბანკია, silo-ში გადაიხდის და საერთო table-ს გაექცევა. საუკეთესო მოდელი არაა — საუკეთესო მოდელი tenant-ის tier-ზეა (pool გრძელ კუდზე, silo „ვეშაპებზე").

Pool-ის გამაგრება RLS-ით. Pool-ის მთელი რისკი ისაა, რომ isolation ეყრდნობა ყოველ query-ს, დაიმახსოვროს WHERE tenant_id = ?. Postgres RLS ამ კედელს database-ში გადააქვს: policy USING (tenant_id = current_setting('app.tenant_id')) და engine ავტომატურად ამატებს filter-ს ყოველ query-ს — helper, რომელიც ივიწყებს, მაინც ვერ ხედავს სხვის row-ს (PostgreSQL RLS).

Noisy neighbor = performance isolation. Pool-ში tenant-ები worker-ებს, connection-ებს და queue-ებს იზიარებენ, ერთი tenant-ის batch job ყველას latency-ს ახრჩობს. გამოსწორება — RLS-ის წარმადობის ტყუპი: per-tenant quotas (token bucket per tenant), fair queuing (round-robin/weighted dequeue), bulkheads (რესურს-პულების იზოლაცია, Release It!-იდან). Isolation ორი ინვარიანტია: A ვერ კითხულობს B-ს მონაცემს, და A ვერ ჭამს B-ს სიმძლავრეს.

9 · Failure mode-ები, რომლებიც ინტერვიუებს წყვეტენ

  • დაკარგული tenant_id ერთ query-ში. ხელმოწერა-SaaS-ბაგი: ხუთი helper-იდან ოთხი filter-ავს, მეხუთე ივიწყებს — tenant A ხედავს tenant B-ს შეკვეთებს. ერთ-tenant-იან ტესტს გაივლის. დაცვა — RLS ან default-scoped data layer, filter სტრუქტურული, არა per-query დისციპლინა.
  • JWT verified, audience unchecked. signature ვალიდურია, exp კარგია, სერვისი იღებს სხვა სერვისისთვის ჭრილ token-ს. ყოველთვის შეამოწმე aud, iss.
  • დიდვადიანი გაჟონილი API key, როტაციის გარეშე. ორი წლის წინ public repo-ში ჩადებული key ისევ მუშაობს. short-lived credentials ავტომატურად დახურავდა ფანჯარას.
  • Rate limiter IP-ზე NAT-ის უკან. 300-კაციანი ოფისი ერთ IP-ს იზიარებს; per-IP login limit მთელ კომპანიას კეტავს. Key სენსიტიური limit-ები account-ზე.
  • ერთი tenant-ის batch job pool-ს ახრჩობს. fair queuing-ის გარეშე ერთი tenant-ის bulk export queue-ს ავსებს და ყველა სხვას p99 ასკდება. per-tenant quotas + fair dequeue აღადგენს.

10 · ერთი AI შტრიხი, მერე უკან საფუძვლებთან

ყველაფერი ზემოთქმული უცვლელად გადადის multi-tenant inference პლატფორმაზე. Per-tenant token quotas + fair scheduling GPU queue-ზე ზუსტად §8-ის noisy-neighbor-ია — ერთი tenant-ის გრძელი-კონტექსტის ნაკადი მეორის TTFT-ს არ უნდა აფეთქებდეს, ამიტომ batching scheduler სამართლიანად (tier-ის წონით) dequeue-ს აკეთებს და არა FIFO. Prompt/output log-ები მგრძნობიარე მონაცემია: PII-ს მოეპყარი — per-tenant DEK-ით დაშიფრე (§6), retention policy დააწესე, JWT claim-ებში არასდროს ჩადო. Model access = authZ scope: „tenant A-ს 70B მოდელის ზარის უფლება აქვს?" scope-ის შემოწმებაა (§4). იგივე pattern-ები, GPU-ფასიანი blast radius.

11 · ინტერვიუს სასაუბრო წერტილები

  1. authN-ი authZ-ისგან პირველივე წინადადებაში გამიჯნე. „gateway authenticate-ს კალერს; ყოველი სერვისი authorize-ს ქმედებას tenant-ის მიხედვით."
  2. დაასახელე token-ის ფორმა და revocation. „short-TTL JWT შიდა fan-out-ისთვის, denylist ახლავე-გაუქმებისთვის, opaque token სენსიტიურ გზაზე."
  3. mTLS east-west ტრაფიკზე, cert-როტაცია mesh-ს ეკუთვნის (SPIFFE identity).
  4. Secret-ები vault-ში ხმამაღლა: „dynamic short-lived DB creds Vault-იდან, envelope encryption per-tenant DEK-ით."
  5. Rate limiting უსაფრთხოებად ჩარჩოში: „per-account brute force-ისთვის, per-IP და global credential stuffing-ისთვის, იაფი reject auth-ამდე."
  6. აირჩიე tenancy მოდელი და ნაზავი დაიცავი: „pool RLS-ით გრძელ კუდზე, silo დარეგულირებულ ვეშაპებზე — აი 4× ღირებულების მათემატიკა." მერე performance isolation დაასახელე.
  7. Failure mode-ები ინტერვიუერამდე ახსენე: „რისკი query-ია, რომელიც tenant_id-ს ივიწყებს — ამიტომ isolation RLS-შია, არა code review-ში."

12 · ლაბი — გატეხე tenant isolation შენი ხელით

პრაქტიკა (ქვიზამდე გააკეთე) labs/lab-0017-tenant-isolation/-ში python3 tenant_lab.py უშვებს სამ ექსპერიმენტს — JWT-ის გაყალბების დრილი (ვალიდური გადის; შეცვლილი და ვადაგასული ვარდება; alg=none token-ს გულუბრყვილო verifier იღებს, მკაცრი კი უარყოფს, forged-admit რიცხვი ნულამდე), noisy-neighbor (tenant B 10× ტბორავს; FIFO queue tenant A-ს p99-ს აფეთქებს, per-tenant token bucket + round-robin fair dequeue აღადგენს და B-ს დათვლილი 429-ებით ახრჩობს), და pool-model leak drill (ხუთი helper, ერთი tenant_id filter-ს ივიწყებს; invariant checker result-ებს სკანირებს და დამნაშავე helper-ს ასახელებს). მერე README-ს E1–E4: იწინასწარმეტყველე, შეცვალე კონფიგი, გაუშვი, შეადარე. მცდარი პროგნოზები დავალების შედეგია.

13 · პროექტის ნაბიჯი

გადაწყვეტილების დროა LLM inference პლატფორმისთვის: აირჩიე tenancy მოდელი და authentication სქემა, ორივე ორი წინადადებით დაიცავი. დასაცავი მაგალითი: pool მოდელი Postgres RLS-ითა და per-tenant token-bucket quota-ებით პლუს fair GPU-queue scheduling (სრული სიმჭიდროვე გრძელ კუდზე, მაგარი data და performance კედლები), წინ short-TTL OIDC access token-ებით audience-შემოწმებით და mesh-managed mTLS gateway-ს, router-სა და worker-ებს შორის. დაწერე შენი ორი წინადადება და მოიტანე შემდეგ სესიაზე.
