გაკვეთილი 0007 · ფაზა 1 — განაწილებული სისტემების საფუძვლები · English version

ქეშირება, სიღრმისეულად

რატომ ჯობნის მეხსიერების ყუთი database-ს — და სამი failure mode, რომლებიც წყვეტენ, გადარჩება თუ არა შენი პასუხი ინტერვიუზე.

მისიასთან კავშირი. Anthropic-ის დიზაინის ამოცანები — „დააპროექტე token-generation სერვისი 100,000 RPS-ზე" — ქეშირების გარეშე შეუძლებელია, და დამატებითი კითხვა ყოველთვის ერთი და იგივეა: „რა ხდება, როცა ის cache ჩანაწერი დატვირთვის ქვეშ ვადას გაუვა?" კანდიდატები, რომლებიც ამბობენ „დავამატოთ Redis", ჩერდებიან; კანდიდატები, რომლებიც ასახელებენ update pattern-ს, eviction (განდევნის) პოლიტიკას და stampede-ის გამოსწორებას, აგრძელებენ. ეს გაკვეთილი გაძლევს ამ სახელებს. (Exponent-ის 2026 Anthropic გზამკვლევი)

1 · რატომ მუშაობს ქეშირება: ფიზიკა × გადახრილობა

ორი ფაქტი მრავლდება ერთმანეთზე. პირველი — ფიზიკა: მიმართვა ოპერატიულ მეხსიერებას ~100 ns-ია; round trip დატაცენტრის შიგნით — ~500 µs. ეს არის 5,000-ჯერადი უფსკრული Estimation Kit-იდან. მეხსიერებიდან პასუხის გაცემა კიდევ ერთი ქსელური hop-ისა და დისკის ნაცვლად ოპტიმიზაცია კი არა — სხვა რეჟიმია.

მეორე — გადახრილობა: რეალური ტრაფიკი Zipf-განაწილებულია — k-ური ყველაზე პოპულარული key იღებს ტრაფიკის დაახლოებით 1/ks-ს, ანუ key-ების მცირე ნაწილი იღებს მოთხოვნების უმეტესობას. სწორედ ამიტომ შეუძლია cache-ს, რომელიც keyspace-ის მხოლოდ 0.2%-ს იტევს, წაკითხვების თითქმის ნახევრის შთანთქმა (ზუსტად ამას გაზომავ ლაბში). ქეშირება მუშაობს, რადგან ტრაფიკი უსამართლოა; თანაბარ ტრაფიკზე პატარა cache თითქმის უსარგებლოა.

2 · სად ცხოვრობენ cache-ები, ზემოდან ქვემოთ

სტეკის ყველა ფენა ქეშირავს; ინტერვიუზე ჯერ კიბე ჩამოიარე და მერე აირჩიე შენი ფენა (Primer §Cache):

ფენარას ქეშირავსრით მართავ
Browserსტატიკური რესურსები, API პასუხებიHTTP cache header-ები
CDNსტატიკური + ქეშირებადი დინამიკური კონტენტიTTL-ები, cache key-ები, purge API
Reverse proxyმთლიანი პასუხები შენს edge-ზეNginx/Varnish კონფიგი
App-layer cacheობიექტები, query-ის შედეგები, სესიებიRedis / Memcached — შენი დიზაინის სივრცე
Database-ის შიდაგვერდები (pages), query plan-ებიBuffer pool-ის ზომა — ძირითადად ავტომატური

დიზაინის კითხვები თითქმის მთლიანად app-layer-ზე ცხოვრობს: რაზე აწყობ key-ს, როგორ იგებს cache ჩაწერების შესახებ და რა განიდევნება.

3 · ოთხი update pattern

Patternწაკითხვის/ჩაწერის გზაიგებსიხდის
Cache-asideაპი ამოწმებს cache-ს; miss-ზე კითხულობს DB-დან და ავსებსზარმაცია — მხოლოდ მოთხოვნილი მონაცემი ქეშირდებაMiss 3 მოგზაურობა ჯდება; ჩაწერის შემდეგ staleness-ის (მოძველების) ფანჯარა
Write-throughყოველი ჩაწერა მიდის cache-შიც და DB-შიც სინქრონულადCache არასდროსაა მოძველებული DB-სთან შედარებითნელი ჩაწერები; ცივი cache, სანამ არ ჩაიწერება
Write-behindჩაწერა cache-ში; DB-ში ასინქრონულად იცლებასწრაფი ჩაწერები, დაჯგუფებული flush-ებიმონაცემების დაკარგვა, თუ cache flush-მდე მოკვდება
Refresh-aheadCache ცხელ ჩანაწერებს ვადის გასვლამდე წინასწარ ანახლებსცხელი key-ები მომხმარებლის თვალწინ არასდროს ვადადებიანუსარგებლოა — ზედმეტი დატვირთვაც კი — თუ პროგნოზი მცდარია

Cache-aside ნაგულისხმევი პასუხია; მისი „აწევა" გააზრებულად გააკეთე და გაცვლა ხმამაღლა დაასახელე. (Primer §When to update the cache)

Invalidation, გულახდილად „კომპიუტერულ მეცნიერებაში მხოლოდ ორი რთული რამ არსებობს: cache invalidation (ქეშის გაუქმება) და სახელების დარქმევა." ზუსტად ორი გულახდილი სტრატეგია გაქვს: event-driven invalidation (ჩაწერები აქვეყნებენ „key X შეიცვალა"; ზუსტია, მაგრამ ახლა invalidation-ის მილსადენი თავად არის განაწილებული სისტემა, რომელსაც შეუძლია ჩამორჩეს ან დაკარგოს) ან TTL-and-tolerate (ჩანაწერები საათით ვადადებიან; მარტივია, და staleness-ის ზღვარს ხმამაღლა ამბობ: „ფასები შეიძლება 60 წამამდე მოძველებული იყოს"). სწორედ ზღვრის დასახელება აქცევს მეორეს ინჟინერიად და არა იმედად.

4 · Eviction და ზომის შერჩევა

სავსე cache-მა რაღაც უნდა გადააგდოს. LRU (least-recently-used — ყველაზე დიდი ხნის გამოუყენებლის განდევნა) ნაგულისხმევია — შესანიშნავია დროითი ლოკალურობისთვის, მაგრამ ერთ მასიურ scan-ს შეუძლია მთელი შენი working set გამორეცხოს. LFU (least-frequently-used — ყველაზე იშვიათად გამოყენებულის განდევნა) scan-ებს უძლებს, მაგრამ გუშინდელი ცხელი key-ების დავიწყება უჭირს. TTL იმდენად eviction არაა, რამდენადაც სიახლე — ის ზღუდავს staleness-ს პოპულარობის მიუხედავად და ორივე პოლიტიკას ერწყმის.

ზომა hit-rate მრუდს მიჰყვება: Zipf-გადახრილობის გამო პირველი სლოტები უზარმაზარ მოგებას ყიდულობენ, ყოველი შემდეგი გაორმაგება კი — უფრო ცოტას. ლაბში ნახავ: ~47% hit rate 100 სლოტზე, მაგრამ ~86%-ის გადასალახად 100-ჯერ მეტი მეხსიერება გჭირდება. ზომა მრუდის მუხლზე აირჩიე და 5,000-ჯერადი უფსკრულით დაასაბუთე: hit rate-ის ყოველი პუნქტი ამდენივე მოთხოვნაა უფსკრულზე გადატანილი.

5 · Failure mode-ები, რომლებიც ინტერვიუებს წყვეტენ

  • Cache stampede (thundering herd — „მეხის გუნდი"). ცხელი key ვადადება; 500 პარალელური მოთხოვნა ერთდროულად ვერ პოულობს და ყველა DB-დან ითვლის თავიდან — საკუთარი ხელით მოწყობილი 500-ჯერადი პიკი, პიკურ ტრაფიკზე, შენს ყველაზე ცხელ მონაცემზე. სამი გამოსწორება, ზრდადი დახვეწილობით: per-key locking (ერთი მოთხოვნა ითვლის, დანარჩენები ელოდებიან — „singleflight"), TTL jitter (ვადის გასვლის დროების შემთხვევითი გაფანტვა, რომ ცხელი key-ები ერთსა და იმავე წამს არასდროს ვადადებოდნენ) და refresh-ahead (ჩანაწერი განახლდება მანამ, სანამ მომხმარებლები მის ვადადებას საერთოდ დაინახავენ).
  • Cache penetration (გახვრეტა). მოთხოვნები key-ებზე, რომლებიც არ არსებობენ, სამუდამოდ miss-დებიან და ყოველ ჯერზე DB-ს ურტყამენ — შემთხვევით ან განზრახ. გამოსწორება: negative caching — თვითონ „not found"-ის დაქეშვა, მოკლე TTL-ით.
  • Hot-key გადატვირთვა. ერთი „ცნობილი" key აჭარბებს იმას, რასაც ერთი cache node ემსახურება. გამოსწორება: key-ის replication (გამრავლება) node-ებზე — მაგ., მიამატე შემთხვევითი სუფიქსი მცირე ნაკრებიდან და წაკითხვები ასლებზე გაფანტე.
LLM შტრიხი LLM serving ამ გაკვეთილის ყველა დონეზე ქეშირავს: prefix/KV cache ინახავს საერთო prompt-პრეფიქსების attention-მდგომარეობას (system prompt-ები საბოლოო ცხელი key-ებია), ხოლო semantic cache აბრუნებს შენახულ სრულ პასუხს იმ prompt-ებზე, რომლებიც ერთსა და იმავეს ნიშნავენ. მათემატიკა იგივეა: როცა პოპულარული prompt-ის დაქეშილი პასუხი ვადადება, ყოველი პარალელური მომხმარებელი GPU-ს ზომის stampede ხდება — singleflight ინფერენსის წინ არჩევანი არაა. ფაზა 4 ამათ gateway-ში ჩააშენებს.

6 · ლაბი — გატეხე cache შენი ხელით

პრაქტიკა (ქვიზამდე გააკეთე) ყველაფერი ზემოთქმული ახლა შემოწმებადია: labs/lab-0007-cache-lab/-ში python3 cache_lab.py უშვებს სამ ექსპერიმენტს — LRU-ს hit-rate მრუდი zipf(1.1) დატვირთვაზე (უყურე, როგორ ჩნდება ცხრილში შემცირებადი უკუგება), cache-aside vs write-through 90/10 წაკითხვა/ჩაწერის ნარევზე და ნამდვილი 50-thread-იანი stampede: დაუცველი vs per-key lock vs TTL jitter. შემდეგ README-ს E1–E4 სავარჯიშოები: იწინასწარმეტყველე, შეცვალე კონფიგი, გაუშვი თავიდან, შეადარე. მცდარი პროგნოზები არის დავალების შედეგი — მოიტანე ისინი შემდეგ სესიაზე.

7 · პროექტის ნაბიჯი

გადაწყვეტილების დროა LLM inference gateway-სთვის: აირჩიე მისი response-cache დიზაინი და ორი წინადადებით დაიცავი. დასაცავი მაგალითი: cache-aside TTL-and-tolerate-ით (პასუხები ნორმალიზებული prompt-ის hash-ზეა აწყობილი, 5-წუთიანი TTL, staleness ხმამაღლა ნათქვამი) პლუს per-key singleflight და TTL jitter, რომ ვადადებადი პოპულარული prompt GPU-ების ფლოტს არასდროს დაეხვიოს. დაწერე შენი ორი წინადადება — pattern + stampede-ისგან დაცვა — და მოიტანე შემდეგ სესიაზე გასარჩევად.
