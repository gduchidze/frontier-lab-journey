Notification სისტემა + Distributed Scheduler — ორი დიზაინი საერთო ხერხემლით

„გააგზავნე 100M push/SMS/email დღეში, საიმედოდ" და „გაუშვი cron მთელ ფლოტზე, ზუსტად ერთხელ" სხვადასხვა ამოცანებად გამოიყურება. სიღრმეში ერთი და იგივე მანქანაა: მდგრადი (durable) მომლოდინე სამუშაო, worker-ები რომლებმაც ორჯერ არ უნდა „გაისროლონ", და მილის ბოლოში — მუდამ მერყევი (flaky) დამოკიდებულება.

მისიასთან კავშირი. Eval-pipeline-ის ორკესტრაცია და batch-job scheduling AI ლაბებში ზუსტად ეს დიზაინია, უბრალოდ სხვა ტანსაცმლით — cron-ის ფენა ისვრის eval batch-ებს judge worker-ების fanout-ში, ხოლო rate-cap-იანი model API თამაშობს flaky provider-ის როლს.
Rep-პროტოკოლი — ჯერ ეს გააკეთე სანამ წაიკითხავ: გაუშვი 40-წუთიანი სოლო, დროში გაზომილი rep-ი ორი ამოცანიდან ერთზე — „დააპროექტე notification სისტემა (push, SMS, email) 100M შეტყობინება/დღეზე" ან „დააპროექტე distributed job scheduler-ი (cron მასშტაბზე)." ხმამაღლა, მარტო, 0009 გაკვეთილის 4-ნაბიჯიანი ჩარჩოთი. მერე წაიკითხე გაკვეთილი და ყველა დელტა (გამორჩენილი კომპონენტი, დაუხარჯავი რიცხვი, დაუსახელებელი failure mode) ჩაწერე learning record-ად.

1 · ორი ამოცანა, ერთი გაკვეთილი

ორივე ამოცანა ერთ წინადადებამდე დადის: საიმედო async fanout დროის სემანტიკით. Notification სისტემა = „მიაწოდე ეს სამუშაო N ბოლო წერტილს ახლავე, გადაურჩი retry-ებსა და flaky provider-ებს." Scheduler-ი = „მიაწოდე ეს სამუშაო worker-ს T დროს, გადაურჩი crash-ებსა და საათის დრიფტს (clock drift)." ლექსიკა მოაშორე და ორგანოები იდენტურია:

ამიტომ იზიარებენ ერთ გაკვეთილს: აითვისე ერთი ხერხემალი, მოჰყევი ორჯერ. ინტერვიუერი ორივეგან ერთსა და იმავე კუნთებს აფასებს — durable state, ownership, retries, idempotency.

3 · Estimate — რიცხვები, რომლებსაც დახარჯავ

100M შეტყობინება/დღეში ÷ ~100k წმ/დღეში ≈ 1.2k/წმ საშუალო
Flash-მოვლენა (breaking news, დიდი sale): ×20 პიკი ≈ 25k/წმ წუთებით
არხების გაყოფა: 70M push · 25M email · 5M SMS დღეში

Provider-ის rate cap-ები (საკონტრაქტო, შენ ვერ ცვლი):
  push ~10k/წმ   email ~2k/წმ   SMS ~200/წმ
SMS საშუალო = 5M/86,400 ≈ 60/წმ  → წესრიგშია
SMS flash-პიკზე = ×20 ≈ 1.2k/წმ vs 200/წმ cap
  → რიგი იზრდება ~1k/წმ; 10-წუთიანი პიკი ≈ 600k backlog
  → დაცლის დრო 600k ÷ 200/წმ = 50 წუთი

Storage: 100M/დღეში × ~0.5 KB ლოგ-სტრიქონი ≈ 50 GB/დღეში → 30 დღე ≈ 1.5 TB

ყველა რიცხვი დახარჯე: 25k/წმ პიკი ნიშნავს stateless-ჰორიზონტალურ ingestion-ფენას LB-ის უკან. SMS-ის cap-მათემატიკა ამტკიცებს, რომ გჭირდება თითო-არხის რიგები (50-წუთიანმა SMS backlog-მა push OTP-ები არ უნდა დააგვიანოს) და პრიორიტეტული ზოლები SMS-ის შიგნით (OTP კამპანიის backlog-ს გვერდს უვლის). 1.5 TB ლოგი ერთ დაპარტიციებულ საცავში ეტევა — აუდიტი იაფია, შეინახე.

4 · Notification-დიზაინი მაღალ დონეზე

კლიენტები / სერვისები
   → Ingestion API  (ვალიდაცია, auth, event_id-ის მინიჭება)
   → Preference + opt-out ფილტრი  (ჩამოაგდე რიგამდე)
   → Template სერვისი  (თითო-არხის ტექსტის აწყობა)
   → თითო-არხის რიგები  [push] [email] [sms — პრიორიტეტული ზოლები]
   → არხის worker-ები  (provider-ის cap-ებზე შეზღუდული)
   → Provider-ადაპტერები  (APNs, FCM, Twilio, SES)
   → Notification log  (ყოველი მცდელობა, ყოველი შედეგი)

ერთი არააშკარა თანმიმდევრობა ახსენი: ფილტრი რიგამდე — opt-out-იანი შეტყობინების ჩაყენება რიგში კარგავს ტევადობას და კომპლაენს-ინციდენტს რისკავს, თუ ფილტრი dequeue-ის დროს წაქცეულია. და თქვი წინადადება, რომელიც მთელ deep dive-ს კარკასს აძლევს: provider-ი ყოველთვის bottleneck-ია და ყოველთვის flaky. APNs-ს ვერ აკონტროლებ, Twilio-ს cap-ები და outage-ები აქვს, email კი greylist-ში გაგდებს. რიგის ქვემოთ ყველაფერი ამ ფაქტის გარშემოა დაპროექტებული.

5 · Deep dive: საიმედოობა — at-least-once, გულწრფელად

კონტრაქტი: at-least-once მიწოდება + idempotency; ვინც მესამე მხარის HTTP API-ზე exactly-once-ს გპირდება, ბლეფავს. ნაკრები:

  • Idempotency / dedup გასაღები. ყოველ შეტყობინებას ingestion-ზე მინიჭებული event_id მიჰყვება. Worker-ები და idempotency-key-ს მიმღები provider-ები ამაზე დედუპლიცირებენ; შენი retry-გზა — ყოველთვის.
  • Retries: exponential backoff + jitter — 0008 გაკვეთილის retry storm-ი, სიტყვასიტყვით. Provider-ი, რომელმაც ახლახან 10k შეტყობინება ჩააგდო, ერთსა და იმავე მომენტში მიიღებს 10k retry-ს, თუ jitter-მა არ გაფანტა.
  • DLQ. k მცდელობის შემდეგ — გვერდზე გადადება შეცდომების ისტორიით. Poison message-მა (გაფუჭებული token, მკვდარი მოწყობილობა) retry-ბიუჯეტი სამუდამოდ არ უნდა ჭამოს.
  • Circuit breaker თითო provider-ზე. Twilio-ს შეცდომების პიკზე — შეწყვიტე ბრახუნი: სწრაფად ჩავარდი რიგში, გაზონდე მცირე ტრაფიკით, თანდათან გახსენი.
  • Notification log. Append-only ჩანაწერი ყოველი მცდელობისა და შედეგისა. პასუხობს „მიიღო X-მა OTP?"-ს support-ში და კვებავს delivery-rate მონიტორინგს.

6 · Deep dive: dedup და collapsing

რბილი no-duplicate გარანტია რეალიზდება dedup ფანჯარით: dequeue-ზე SETNX dedup:{event_id} 1 EX 3600 Redis-ში — თუ გასაღები უკვე იყო, ეს event ფანჯარაში სხვამ მიიტანა (ან მიაქვს); ჩამოაგდე. ეს ფანჯარაა და არა დავთარი: TTL-ის შემდეგ გამეორებული event შეიძლება გაძვრეს — ზუსტად ამას ნიშნავს „soft" და ამას ხმამაღლა ამბობ.

იმავე ვიწრო წერტილში კიდევ ორი თითო-მომხმარებლის კონტროლი ზის:

  • თითო-მომხმარებლის rate limit (0018 გაკვეთილის token bucket-ი, თითო user-ზე თითო არხზე): ვერავინ მიიღებს 40 push-ს წუთში, upstream-ი რასაც უნდა შვრებოდეს. ეს პროდუქტის გარანტიაცაა და ბაგების ცეცხლგამძლე კედელიც.
  • Digest / collapse. „17-მა მოიწონა შენი პოსტი" 17 push-ის ნაცვლად. Collapse-გასაღები (user:post:like) + მოკლე ბუფერული დაყოვნება: დაიჭირე, დათვალე, ერთი გაუშვი. მარკეტინგს უყვარს; ბატარეებს — მეტად.

7 · Distributed scheduler — მაღალი დონე

მეორე ამოცანა: cron მასშტაბზე — ათასობით განმეორებადი და ერთჯერადი job-ი, გასროლილი განრიგიდან წამებში, ფლოტზე სადაც ნებისმიერი კვანძი შეიძლება მოკვდეს. ანატომია: job store (განსაზღვრებები + next_fire_at, მდგრადი DB), due-job ინდექსი fire-დროით დალაგებული, და scheduler worker-ები, რომლებიც due job-ებს იჭერენ და ისვრიან. ინდექსის სამი კლასიკური ფორმა:

მასშტაბზე — partition დროის ბაკეტით (მაგ. თითო წუთი-საათში, ან job_id-ის hash): თითო partition-ს ერთდროულად ერთი scheduler-ი ფლობს, ამიტომ არცერთი კვანძი მთელ განრიგს არ სკანირებს და „ყველაფერი 00:00-ზე ისვრის" thundering herd-ი მფლობელებზე ნაწილდება.

8 · Deep dive: scheduler-ის კორექტულობა

კითხვა კითხვის შიგნით: რა აჩერებს ორ scheduler-ს, ერთი და იგივე job-ი არ გაისროლონ?

  • Leader-ი თითო partition-ზე. თითო partition-ს ჰყავს ერთი არჩეული მფლობელი — lease-ით საკოორდინაციო საცავში (etcd/ZooKeeper, ან DB-სტრიქონი ვადით). Lease ამოიწურა → სხვა იბარებს.
  • Fencing token-ები (ფაზა 2-ის იდეა, ადგილზე შეხსენება): ყოველ lease-ს მოჰყვება მონოტონურად მზარდი token-ი. Downstream მდგომარეობა — job-run ჩანაწერი, გვერდითი ეფექტების საცავი — უარყოფს ჩანაწერს, რომელსაც უკვე ნანახზე ძველი token-ი მოაქვს. რატომ არ ყოფნის მარტო lease: leader-ი შეიძლება გაიჭედოს (GC-პაუზა, VM-გაყინვა), lease ამოიწუროს, ახალმა leader-მა job-ი გაისროლოს — მერე ზომბი გაიღვიძებს და ისიც ისვრის. ზომბის token-ი მოძველებულია, ამიტომ მისი downstream ჩანაწერი ბრუნდება. Lease = ალბათ ერთი მფლობელი; fencing = მკვდარი მფლობელის ჩანაწერები უვნებელია.
  • Missed-fire catch-up პოლიტიკა. Scheduler-ი 10 წუთი წაქცეული იყო; job-ს 10-ჯერ უნდა ესროლა. თითო-job-ის პოლიტიკა: fire-all (ბილინგის ტიკები), fire-once-latest (რეპორტის გენერაცია), ან skip (cache warmer-ები). ამის კონფიგურირებადობა სენიორის სიგნალია — უნივერსალურად სწორი პასუხი არ არსებობს.
  • გულწრფელი კონტრაქტი: at-least-once სროლა + იდემპოტენტური job handler-ები. Fencing + idempotency-გასაღებები იძლევა exactly-once ეფექტს; exactly-once სროლა crash-ების გავლით დაუპირებელია.
  • საათები. Worker-ები დრიფტავენ; NTP ზღუდავს, მაგრამ არ სპობს. „Due"-ს ერთი ავტორიტეტი წყვეტს: job store-ის საათი და არასდროს თითო worker-ის მაჯის საათი — თორემ 1.5 წმ-ით აჩქარებული worker-ი lease-ებს მოიპარავს და ადრე ისვრის (შენი ლაბი ზუსტად ამას აჩვენებს).

9 · Failure mode-ები და მონიტორინგი

10 · ვარიანტები და twist-ები

  • Webhook-ები მესამე მხარეებთან — სარკისებური სურათი: notification სისტემა, სადაც flaky provider-ი შენ ხარ. ამიტომ კარგივით მოიქეცი: მოაწერე payload-ებს ხელი, ჩადე event id, რომ მომხმარებლებმა დედუპლიცირება შეძლონ, გამოაქვეყნე შენი retry-with-backoff პოლიტიკა, დანებდი „failed webhooks" დაფაში, საიდანაც კლიენტი replay-ს გააკეთებს.
  • თანმიმდევრობა თითო user-ზე. „პაროლი შეიცვალა" არ უნდა მოვიდეს „პაროლის შეცვლა მოთხოვნილია"-ზე ადრე. დააპარტიციე რიგები user_id-ით, რომ ერთი user-ის შეტყობინებები ერთ ზოლზე სერიალიზდეს — გლობალური თანმიმდევრობა არასდროსაა მოთხოვნა; თითო-გასაღების — თითქმის ყოველთვის.
  • Quiet hours / დროის სარტყლები. „3 საათზე ღამით ნუ push-ავ ადრესატის ლოკალურით" ყოველ შეტყობინებას დაგეგმილ მიწოდებად აქცევს შემდეგ დაშვებულ ლოკალურ დროზე — ამ გაკვეთილის ორი დიზაინი პირდაპირ ერთ სისტემად ერწყმის. ეს წინადადება ინტერვიუზე თქვი.

AI twist (მოკლედ). Eval pipeline ეს ხერხემალი სიტყვასიტყვით არის: scheduler-ი ღამის eval batch-ებს ისვრის; fanout-ფენა სამუშაოს judge worker-ებს აწვდის; model API — rate-cap-იანი flaky provider-ია; fencing გაჭedილ ორკესტრატორს eval-ის ორჯერ გაშვებას უშლის; DLQ-ში judge-ის ჩამგდები prompt-ები წვანან.

12 · ლაბი + პროექტის ნაბიჯი

პრაქტიკა (ქვიზამდე გააკეთე) labs/lab-0023-scheduler/-ში python3 scheduler_lab.py უშვებს scheduler worker-ების სიმულირებულ ფლოტს delayed-job store-ის წინააღმდეგ: worker-ი სროლის შუაში კვდება, მეორეს საათი დრიფტავს, lease-ები რბოლაში ეჯახებიან — და ხედავ დუბლირებულ სროლებს fencing-ის გარეშე, მერე მათ გაქრობას fencing + იდემპოტენტური handler-ებით. ექსპერიმენტი 3 შეტყობინებებს 30%-ით flaky provider-ში ატარებს backoff + jitter-ითა და DLQ-ით. ყოველ გაშვებამდე იწინასწარმეტყველე — README-ში E1–E4-ია.

პროექტის ნაბიჯი. ჩაწერე rep-დელტები (სხვაობა შენს 40-წუთიან დიზაინსა და ამ გაკვეთილს შორის) learning record-ად. მერე spine-პროექტის ჩანაწერებს დაამატე ერთი აბზაცი, რომელიც ამ გაკვეთილს inference-პლატფორმაზე გადაიტანს: რომელი კომპონენტია scheduler-ი (eval cron), რომელია არხის რიგები (judge fanout) და რომელი provider-cap-ი თამაშობს Twilio-ს როლს (model API-ის rate limit-ები).
