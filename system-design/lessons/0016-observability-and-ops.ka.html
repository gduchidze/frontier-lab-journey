გაკვეთილი 0016 · ფაზა 2 — პროგრამული არქიტექტურა და საიმედოობა · English version

Observability და Ops

ვერ გაასწორებ იმას, რასაც ვერ ხედავ. Logs, metrics და traces; SLO-ები და error budget-ები; და alerting, რომელიც ადამიანს მხოლოდ ადამიანის ღირს პრობლემაზე აღვიძებს.

მისიასთან კავშირი. Anthropic-ის ყველა დიზაინის ამოცანა ერთნაირად მთავრდება: „საიდან იცი, რომ მუშაობს — და საიდან იცი, როცა აღარ მუშაობს?" ვინც პასუხობს „დავამატოთ monitoring", ჩერდება; ვინც ასახელებს სამ SLI-ს, burn-rate page-ს და რას აკეთებს on-call ღამის 3 საათზე, ისე ჟღერს, თითქოს პეიჯერი უტარებია. ეს გაკვეთილი Google-ის SRE წიგნის — ინდუსტრიის კანონიკური პასუხის — შეკუმშვაა, e-commerce checkout-ისა და payments API-ის მაგალითებზე.

1 · სამი სვეტი, გულახდილად

სამი ტელემეტრიის სიგნალი, სამი სხვადასხვა კითხვა, სამი ძალიან განსხვავებული ქვითარი (SRE წიგნი, თ. 6, OpenTelemetry signals):

სიგნალირა არისრა კითხვას პასუხობსღირებულების პროფილი

Metricsწინასწარ აგრეგირებული რიცხვები დროში„checkout ჯანმრთელია ახლა? საით მიდის?"იზრდება cardinality-ზე (კარდინალობა — განსხვავებული label-კომბინაციები), არა ტრაფიკზე

Cardinality-ის ხაფანგი წინასწარ: მეტრიკის ღირებულება განსხვავებული label-კომბინაციების რაოდენობაა, რადგან თითოეული საკუთარი time series-ია. დაამატე user_id label და თითო მომხმარებელზე თითო series მოიჭრება — მილიონი მომხმარებელი მილიონი series-ია. წესი: label-ები შემოსაზღვრული განზომილებებისთვისაა (service, status, region); შემოუსაზღვრელი იდენტობა (user, order) logs-სა და traces-ს ეკუთვნის (Prometheus: do not overuse labels). ამ ბომბს ლაბში უსაფრთხოდ ააფეთქებ.

2 · სტრუქტურირებული logging

Log-ები ჯერ მანქანებისთვისაა, მერე ადამიანებისთვის — ანუ JSON და არა პროზა. სამი დისციპლინა (12-Factor: logs): correlation ID ყველგან — gateway ჭრის request ID-ს და ყველა hop მას ატარებს; ერთი grep თორმეტ სერვისში ერთი მოთხოვნის ისტორიას აღადგენს. დონეები გულახდილად — ERROR ნიშნავს „ადამიანმა უნდა ნახოს"; სადაც ყველაფერი ERROR-ია, ERROR დონე აღარ არსებობს. Sampling მოცულობისთვის — ყველა error log შეინახე, happy path დაასემპლე.

ნამუშევარი არითმეტიკა. Checkout 1,000 RPS-ზე, ~2 KB log თითო მოთხოვნაზე = 2 MB/s = 173 GB/დღეში ≈ 5.2 TB თვეში, დაუმუშავებელი. ინდექსი და 3× replication → ~15 TB; vendor-ის ingest ფასით (~$0.50/GB) — ~$2,600/თვეში, სანამ რამეს გამოითხოვ. INFO-ს 1:10 sampling (ყველა WARN/ERROR რჩება) ქვითარს ~80%-ით ამცირებს. Logging პროდუქტის გადაწყვეტილებაა ერთეულის ფასით და არა უფასო გამონაბოლქვი.

ორი checklist თითქმის ყველა dashboard-ს ფარავს: RED სერვისებისთვის, USE რესურსებისთვის (Tom Wilkie: RED მეთოდი, Brendan Gregg: USE მეთოდი):

აუდიტორიაOn-call ღამის 3 საათზე — სიმპტომებიCapacity planning და დიაგნოზი — მიზეზები

პროცენტილები და არა საშუალო — 30 ms საშუალო checkout latency თავსებადია 2 წამიან p99-თან. აღრიცხე p50/p95/p99, და რაკი პროცენტილების საშუალო host-ებს შორის არ იანგარიშება, production სისტემები histogram-ებს წერენ — მთვლელები latency-ის bucket-ებად, რომლებიც სწორად აგრეგირდება (Prometheus: histograms). Bucket-ის ერთი ზღვარი ზუსტად SLO-ის ზღურბლზე დადე (მაგ. 400 ms), რომ „SLO-ში ჩამჯდარი მოთხოვნების წილი" ზუსტი იყოს.

4 · განაწილებული tracing

Trace ერთი მოთხოვნის ხეა: ყოველი hop წერს span-ს (სახელი, დასაწყისი, ხანგრძლივობა, მშობელი) და მთელი ხე ერთ trace ID-ს იზიარებს. კონტექსტი traceparent HTTP header-ით მოგზაურობს — W3C Trace Context სტანდარტი. დიზაინი Google-ის Dapper-ის ნაშრომიდან მოდის; OpenTelemetry მისი ღია სტანდარტიზაციაა.

რას პოულობს მხოლოდ trace: მეტრიკები გეტყვიან, checkout-ის p99 900 ms-მდე ავიდაო; მხოლოდ trace აჩვენებს 12-სერვისიან fan-out-ში, რომ იმ მილიწამებიდან 835 payments hop-შია. Sampling: head (გადაწყვეტილება trace-ის დასაწყისში — იაფი, მაგრამ ბრმა) vs tail (გადაწყვეტილება შედეგის გაგების შემდეგ — ყველა error და ნელი trace რჩება, დანარჩენის 1%) (OpenTelemetry: sampling). კლასიკური ჩავარდნა: 1:1000 head sampling-მა ის ერთი trace გადააგდო, რომელიც ინციდენტზე გჭირდებოდა — ლაბის E4 ამას შეგაგრძნობინებს.

5 · SLI, SLO, error budget

SRE წიგნის ცენტრალური მექანიზმი (Service Level Objectives): SLI — ინდიკატორი, საუკეთესოდ როგორც შეფარდება: კარგი მოვლენები / ყველა მოვლენა (checkout-ის availability SLI = არა-5xx / ყველა პასუხი). SLO — სამიზნე ამ შეფარდებაზე ფანჯარაში: „მოთხოვნების 99.9% წარმატებულია 30 დღეში". Error budget (შეცდომების ბიუჯეტი) — დანამატი: 1 − SLO; საიმედოობა ხდება რესურსი, რომელსაც შეგნებულად ხარჯავ deploy-ებსა და რისკზე (Embracing Risk).

ნამუშევარი მათემატიკა. 30-დღიანი თვე 43,200 წუთია. 99.9% SLO ტოვებს 0.1%-ს → 43.2 წუთი სრული წყვეტა თვეში. 99.95%-ზე ბიუჯეტი ნახევრდება (21.6 წთ); 99.99%-ზე 4.3 წუთია — ერთ ცუდ deploy-ზე ნაკლები, ამიტომ ყოველი დამატებითი ცხრიანი ღირებულებას ამრავლებს. მოთხოვნებში: 201,600 მოთხოვნა/თვეში → ~202 ცუდი მოთხოვნის ბიუჯეტი — ლაბში ნახავ, როგორ ხარჯავს ერთი 11-წუთიანი error burst ამის 72%-ს.

ბიუჯეტის პოლიტიკა რიცხვს კბილებს აძლევს: სანამ ბიუჯეტი რჩება — სწრაფად ushiპ-ავ; როცა დაიხარჯა — ფიჩერების გამოშვება იყინება და გუნდი საიმედოობაზე გადადის. და ტერმინები: SLO შიდა სამიზნეა, რომელიც ინჟინრებს აღვიძებს; SLA გარე კონტრაქტია, რომელიც იურისტებს აღვიძებს — SLO SLA-ზე მკაცრი უნდა იყოს.

6 · Alerting-ის ფილოსოფია: page burn rate-ზე

წესი პირველი (SRE წიგნი, თ. 6): page სიმპტომებზე და არა მიზეზებზე. „CPU 95%-ზეა" მიზეზია — ticket ხდება. „მომხმარებლები checkout-ზე ისე სწრაფად მტყუნდებიან, რომ SLO საფრთხეშია" სიმპტომია — ის page-ს იმსახურებს. ყოველი page ქმედითი უნდა იყოს: თუ პასუხისმგებლის პირველი ფიქრია „მერე რა?", alert-ი ადამიანებს page-ების იგნორირებას ასწავლის.

თანამედროვე მექანიზმი burn rate-ია (წვის ტემპი): რამდენად სწრაფად ხარჯავ ბიუჯეტს, მდგრადი ტემპის ჯერადად. ნამუშევარი მაგალითი (SRE workbook: Alerting on SLOs): 99.9% SLO-ზე ერთი საათი 1.44% შეცდომების ტემპით არის 14.4× burn და ხარჯავს 14.4/720 = თვიური ბიუჯეტის 2%-ს ერთ საათში — page-ის ღირსია. რეკომენდებული ფორმაა multi-window multi-burn-rate: page, როცა burn ≥ 14.4× ორივეზე — 1 სთ და 5 წთ ფანჯრებზე (მოკლე ფანჯარა alert-ს სწრაფად „აქრობს"); ticket, როცა burn ≥ 6× 6 სთ + 30 წთ-ზე. შეადარე ნაივურ ალტერნატივას — „page, თუ წუთში error rate > 2%" — ის თვითგანკურნებად წუთიან blip-ებზეც აღვიძებს და latency-ის ინციდენტს საერთოდ ვერ ხედავს, რადგან ნედლ შეცდომებს უყურებს და არა SLI-ს. ლაბი ზუსტად ამ შეჯიბრს ითვლის.

7 · ინციდენტზე რეაგირება

8 · Ops-ის სიმწიფე: მოსაწყენი ზეძალები

  • Runbook-ები. ყოველი page ბმულია დოკუმენტზე: რას ნიშნავს, პირველი სამი შემოწმება, შერბილების ნაბიჯები. იწერება მშვიდობისას, სრულდება პანიკისას.
  • ერთი dashboard თითო სერვისზე, RED ზედა რიგში. On-call სწორ გრაფიკამდე ორი დაწკაპუნებით უნდა მივიდეს.
  • Deploy-ის observability. ინციდენტების უმეტესობას deploy-ები იწვევენ: canary-ს RED მეტრიკები baseline-ს შეადარე დაწინაურებამდე; ავტომატური rollback, თუ დელტა ზღვარს გადააჭარბებს.
  • Toil-ის ბიუჯეტი. ხელით, განმეორებადი ops-სამუშაო შეზღუდე (~50% Google-ის პრაქტიკაში) და დანარჩენი მის ინჟინერიულ მოშორებაზე დახარჯე (Eliminating Toil).
  • Error-budget-ით მართული release gating. CI/CD ბიუჯეტს კითხულობს: ბიუჯეტი ჯანმრთელია → deploy-ები მიდის; დახარჯულია → მხოლოდ საიმედოობის fix-ები.

9 · Failure mode-ები, რომლებიც ინტერვიუებს წყვეტენ

  • Alert fatigue (განგაშის დაღლილობა). კვირაში ათობით უქმედო page on-call-ს გადაფურცვლას ასწავლის — და ნამდვილი SEV-1 მათთან ერთად გადაიფურცლება. გამოსწორება: წაშალე ან ჩამოაქვეითე ყველა alert, რომელსაც ქმედება არ მოჰყოლია; ზომე page-ები ცვლაზე.
  • Dashboard-ების გაფანტულობა. ორმოცი dashboard, რომელსაც არავინ ხსნის, უარყოფითი ღირებულებაა — ინციდენტისას არჩევანის პარალიჩი. ერთი სერვისზე, RED ზევით.
  • შემოუსაზღვრელი cardinality. §1-ის user_id label-ი — ქვითარი მოდის ექვსნიშნა ინვოისად ან OOM-ად. Label-ების lint code review-ში.
  • Tracing უსარგებლობამდე დასემპლილი. 1:1000 head sampling მთელი წელი ფულს ზოგავს, მერე ის ერთი trace, რომელიც ავარიას ხსნიდა, გადაგდებული აღმოჩნდება. გამოსწორება: tail-based sampling — ყველა error და ნელი trace რჩება.
  • Postmortem-ები შედეგის გარეშე. ლამაზი დოკუმენტი, ნული შესრულებული action item, იგივე ავარია მომდევნო კვარტალში. Action item-ები sprint board-ზე, მფლობელებით.
  • Monitoring-ს არავინ აკვირდება. Alert-ების მილსადენი ჩუმად კვდება და სიჩუმე ჯანმრთელობად იკითხება. გამოსწორება: dead-man's switch — alert, რომელიც განზრახ მუდმივად ვარდება (Alertmanager-ის Watchdog); ქვედა დინების სისტემა მაშინ აღვიძებს, როცა ის შეწყდება (Prometheus alerting).

10 · ერთი AI შტრიხი, და ისევ საფუძვლები

Model serving-ი SLI-ების მენიუს ცვლის და არა მექანიზმს. Inference პლატფორმისთვის: TTFT p99 (დრო პირველ token-მდე), tokens/s თითო მოთხოვნაზე და queue depth თითო tenant-ზე — RED-მხარის SLI-ებია; GPU utilization და KV-cache-ის შევსება USE-ის ტერიტორიაა — dashboard-ები და capacity, არა page-ები. ერთი დისციპლინა: eval-ხარისხის მეტრიკები ops-მეტრიკები არ არის — „კარგია თუ არა მოდელის პასუხები" offline eval-მილსადენებით იზომება საკუთარი კადენციით; შერევა ღამის 3 საათზე benchmark-რეგრესიაზე გაღვიძებას ნიშნავს (OpenTelemetry GenAI conventions). იგივე SLO მათემატიკა, ახალი ინდიკატორების სახელები.

11 · ინტერვიუს სალაპარაკო პუნქტები

  1. დაიწყე SLI-ებით, კითხვის გარეშე: „წარმატების კრიტერიუმები: availability SLI = არა-5xx/ყველა, latency SLI = 400ms-ში/ყველა, ორივე 99.9% 30 დღეზე."
  2. თქვი ბიუჯეტის რიცხვი: „99.9% თვეში 43.2 წუთის ბიუჯეტია — ეს ზღუდავს, რამდენად აგრესიულია ჩვენი deploy-ები."
  3. დაასახელე სამი სიგნალი საქმეებით (§1): მეტრიკები აღმოჩენისთვის, trace-ები ლოკალიზაციისთვის, log-ები მოთხოვნის ისტორიისთვის — correlation ID-ით შეერთებული.
  4. RED სერვისზე, USE რესურსზე (§3), histogram-ები და არა საშუალოები.
  5. Page burn rate-ზე და არა ზღურბლებზე (§6): „14.4× 1სთ+5წთ-ზე — page; 6× 6სთ-ზე — ticket; CPU-ის alert-ები ticket-ებია, არასდროს page-ები."
  6. დახატე ინციდენტის გზა (§7): page → IC → შერბილება → blameless postmortem მფლობელიანი action item-ებით.
  7. თავად შესთავაზე failure mode (§9): „ჩემი დიზაინის რისკი cardinality-ია — label-ებს code review-ში გავალინტავ." პირველად წამოწევა strong-hire სიგნალია.

12 · ლაბი — გამოიძახე საკუთარი თავი, სწორად

პრაქტიკა (ქვიზამდე გააკეთე) ყველაფერი ზემოთქმული ახლა შემოწმებადია: labs/lab-0016-observability-lab/-ში python3 observability_lab.py უშვებს შეკუმშულ „თვეს" gateway → orders → payments ჯაჭვში ორი ჩადგმული fault-ით და სამ ექსპერიმენტს — RED ცხრილები, რომლებიც ორივე fault-ს რიცხვებში ამხელენ; SLO მათემატიკა, სადაც ერთი 6-წუთიანი ფანჯარა თვიური latency-ბიუჯეტის ~78%-ს ჭამს; და burn-rate paging, რომელიც ნაივურ ზღურბლს ცრუ page-ებსა და დაფარვაში ჯობნის — პლუს span-ხის რეკონსტრუქცია, რომელიც დამნაშავე hop-ს ასახელებს. შემდეგ README-ს E1–E4: იწინასწარმეტყველე, შეცვალე კონფიგი, გაუშვი თავიდან, შეადარე. მცდარი პროგნოზები დავალების შედეგია.

13 · პროექტის ნაბიჯი

გადაწყვეტილების დროა LLM inference gateway-სთვის: განსაზღვრე მისი სამი SLI + SLO და ერთი burn-rate page, და ორი წინადადებით დაიცავი. დასაცავი მაგალითი: availability (არა-5xx/ყველა, 99.9%/30დ), TTFT p99 < 1.5 წმ (როგორც კარგი მოვლენების შეფარდება, 99%/30დ) და per-tenant queue depth როგორც saturation-სიგნალი dashboard-ზე — ერთი page-ით: burn ≥ 14.4× 1სთ+5წთ-ზე availability SLI-ზე, დანარჩენი ticket-ია. ორი წინადადება: მომხმარებლები მტყუნებებსა და პირველი token-ის დაგვიანებას გრძნობენ, ამიტომ ეს ორი აღვიძებს; queue depth მათ წინასწარმეტყველებს, მაგრამ არასრულყოფილად, ამიტომ capacity-ს აწვდის ინფორმაციას და არავის აღვიძებს. დაწერე შენი სამი SLI + ერთი page და მოიტანე შემდეგ სესიაზე გასარჩევად.
