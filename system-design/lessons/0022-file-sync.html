File Sync — Dropbox

Chunking, content addressing, delta sync, and honest conflict resolution — the rep where the client is as much of the system as the servers are.

Mission tie. Content-addressing and dedup thinking transfers directly to model-weight distribution across GPU fleets and to KV-cache prefix reuse — same trick, different bytes.
REP PROTOCOL Do a 40-minute solo timed rep FIRST — out loud, whiteboard or paper, prompt: "Design a file-sync service like Dropbox." Only then read this lesson, compare against your design, and log the deltas as a learning record. Reading first turns a rep into a recital.

1 · The prompt, and what it actually tests

"Design Dropbox: users install a client, drop files in a folder, and the files appear on all their devices — even after offline edits." Every previous rep let you treat the client as a dumb browser. This is the first design where the client is part of the system: it runs a file watcher, a chunker, a local index database, and an offline operation queue, and the interesting protocol lives between client and server. Interviewers use this prompt to test three things: whether you split metadata from bulk data (the design's load-bearing decision), whether you know why files are chunked and hashed rather than uploaded whole, and whether you handle two offline devices editing the same file without hand-waving. Candidates who draw only server boxes have already missed half the problem.

3 · Estimate (~5 min) — and spend every number

Users:    500M registered, ~100M daily-active syncers
Storage:  500M × 10 GB avg = 5 EB raw
Dedup:    assume 2:1 (installers, media, shared files) → ~2.5 EB stored
Upload:   100M DAU × ~10 MB changed blocks/day = 1 PB/day
          ÷ ~100k s/day ≈ 10 GB/s ≈ 80 Gbit/s avg, ×3 peak ≈ 240 Gbit/s
Metadata: 10 GB ÷ 4 MB blocks = 2,500 blocks/user
          500M × 2,500 = 1.25×10¹² block rows × ~100 B ≈ 125 TB

Now spend them: 2.5 EB of blocks can never live in a database → S3-style blob store. 125 TB of metadata is ~0.005% of the data — small enough for a sharded relational DB with real transactions. 240 Gbit/s peak means clients must upload blocks directly to block storage, never through the metadata service. And the 2:1 dedup assumption is itself an architecture argument: dedup only falls out for free if blocks are named by their content — section 5.

4 · High-level design: split the brain

The load-bearing decision is splitting the system along the metadata/data seam:

ComponentHoldsProperties
Metadata serviceNamespaces, file→blocklist records, version journal, ACLsSmall (TB), consistent, transactional — sharded SQL
Block storageThe chunks themselves, keyed by content hashHuge (EB), immutable, append-only — S3-style
Notification serviceMillions of idle long-poll connectionsStateless "something changed" pings — no payloads

And the client mirrors the split with three components: a watcher (OS file-system events → detect changes), a chunker (split files into blocks, hash each), and an indexer (local SQLite mapping files → block hashes → sync state, so the client knows what the server has without asking). Draw both halves; narrate one file's journey through them before going deep.

5 · Deep dive: chunking and content addressing

Split every file into blocks — Dropbox uses fixed 4 MB blocks — and name each block by its SHA-256 hash. That one choice buys three properties at once: dedup for free (same bytes → same hash → server says "already have it", upload skipped — across files and across users), immutability (a block never changes; edits create new blocks, which is exactly what file history needs), and integrity (re-hash after download; mismatch means corruption).

Fixed chunking has one famous weakness: the insert-shift problem. Insert one byte at the front of a 1 GB file and every 4 MB boundary shifts by one — every downstream block re-hashes as "new" and the whole file re-uploads. The fix is content-defined chunking (CDC): run a rolling hash (Rabin fingerprint) over a small sliding window and cut a boundary wherever the hash matches a pattern (e.g. lowest 13 bits all ones → ~8 KB average). Boundaries become functions of local content, not position, so they re-align right after an edit. Trade-off: CDC costs CPU per byte and variable block sizes; fixed is simple and fine for append-heavy or whole-file-rewrite workloads — which is why Dropbox shipped fixed and rsync/restic/borg ship CDC. Your lab makes both behave in front of you.

6 · Deep dive: delta sync

A file, to the server, is just an ordered list of block hashes — the blocklist. Edit a file and the client re-chunks it, diffs the new blocklist against the old, and uploads only the blocks whose hashes the server lacks. A 4 MB edit in a 10 GB file costs one or two blocks, not 10 GB. Each committed change appends a file-version record to a per-namespace journal: (namespace_id, journal_id, path, blocklist, size, mtime). The journal is append-only — which gives you file history and restore for free (old versions are just old journal rows whose blocks still exist), and gives clients a cursor to sync from (section 7).

7 · Deep dive: the sync protocol

Downstream: every client holds a long-poll connection to the notification service. On any change in a namespace the client subscribes to, the poll returns — carrying no data, just "wake up". The client then calls the metadata service: list_changes(namespace, cursor) — give me all journal entries after my cursor — then fetches whichever blocks it's missing from block storage, assembles files, advances the cursor. Cursor-based pull means a client offline for a week catches up with the same code path as one that blinked for a second.

Upstream, the ordering is the crash-safety argument interviewers listen for: upload blocks first, commit metadata second. The client pushes new blocks to block storage, and only when all of them are durable does it call commit(path, new_blocklist, parent_version) on the metadata service. Why that order? A crash mid-upload leaves orphan blocks — invisible garbage, cleaned by a GC job later. The reverse order would leave a committed version pointing at blocks that don't exist: a visibly corrupted file on every other device. Metadata commit is the atomic switch; blocks are just inert bytes until a journal row points at them.

8 · Deep dive: conflict resolution

Two devices, both offline, edit the same file. Locks are impossible — you cannot lock a laptop on a plane. So the commit carries the version the edit was based on: commit(..., parent_version). First committer wins normally. The second commit's parent_version is now stale — the server detects the conflict. Two policies:

PolicyBehaviorCost
Last-writer-winsNewest commit silently replacesSilently destroys a user's work
Conflicted copyKeep both; rename the loser "report (conflicted copy — Ann's laptop, 2026-07-23)"User must merge by hand

Dropbox makes the conflicted copy — and that is the honest choice: a sync engine cannot merge arbitrary binary formats, so pretending to (LWW) means losing data quietly, while surfacing both versions loses nothing and tells the user the truth. Mention version vectors (per-device counters) as the general mechanism for distinguishing "concurrent edits" from "stale copy" — one sentence is enough. Offline edits queue in the client's local journal and replay through the exact same commit path on reconnect; conflicts fall out of parent_version checks with no special offline logic.

9 · Failure modes and monitoring

FailureHandling
Crash mid-upload (half-uploaded file)Blocks-then-commit ordering → only orphan blocks; GC sweeps unreferenced blocks; no version ever points at missing data
Hash collision (two blocks, one SHA-256)Address honestly: probability ~2⁻¹²⁸ — orders of magnitude below disk/RAM corruption rates; adding byte-compare "just in case" adds cost, not safety
Notification storm (shared folder, 10k members, one change)Fan-out amplification: 1 commit → 10k wakeups → 10k journal pulls. Coalesce notifications per namespace, add jitter to client pulls, and let pings carry no payload so storms are cheap
Sync loops (client A and B endlessly re-writing — e.g. antivirus touching mtime)Compare content hashes, not timestamps, before committing; detect commit cycles per file and back off with alerting

Monitoring: the metric that pages a human is p99 sync convergence time (change on device A → visible on device B), plus journal-pull lag, block-store error rate, and conflicted-copy rate (a spike means a client bug, not user behavior).

10 · Variants and twists

  • Sharing & permissions. A shared folder is a namespace mounted into multiple users' trees; ACL checks happen on journal pull and commit. Re-sharing = re-mounting; unsharing = unmounting plus cursor invalidation.
  • Bandwidth throttling. The client rate-limits itself and backs off on 429s — sync is a background citizen; never saturate the user's uplink.
  • LAN sync. Devices on one network discover each other (UDP broadcast) and fetch blocks peer-to-peer — the content hash makes any source trustworthy: verify by re-hashing. This is the twist interviewers love, and it only works because blocks are content-addressed.

AI corner. The same content-addressed dedup runs modern ML infra: distributing a 100 GB model to 1,000 GPU nodes is a Dropbox problem (chunk the weights, hash the chunks, nodes fetch missing blocks from peers — fine-tunes dedup against the base model's unchanged layers), and KV-cache prefix reuse is hash-the-prefix-tokens, reuse-the-cached-blocks — content addressing over attention states instead of file bytes.

11 · Interview talking points

  • Opening scope: "The client is half this system — I'll design the watcher/chunker/indexer as first-class components, not just the servers."
  • The split: "Metadata is 125 TB and needs transactions; blocks are 2.5 EB and immutable — those numbers are why they're different systems."
  • Chunking: "Fixed 4 MB is what Dropbox ships; content-defined chunking fixes the insert-shift problem at CPU cost — I'd start fixed and justify CDC only for edit-heavy workloads."
  • Unprompted crash safety: "Before you ask: upload blocks first, commit metadata second — a crash leaves invisible garbage instead of a visible corrupt version."
  • Conflicts: "No locks — offline devices exist. Conflicted copy over last-writer-wins, because silently destroying a user's edit is worse than asking them to merge."
  • Closing: "I'd page on p99 convergence time and conflicted-copy rate; with more time: LAN sync and shared-folder notification coalescing."
