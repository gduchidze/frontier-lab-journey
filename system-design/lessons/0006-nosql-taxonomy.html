Lesson 0006 · Phase 1 — Distributed Systems Core

NoSQL Taxonomy & Consistent Hashing

Four data-store families, one decision table, and the hashing trick that makes distributed KV stores possible at all.

Mission tie. A reported Anthropic warm-up prompt is "design a distributed key-value store." The very first fork in that answer is: which family of store is this, and how do keys find their node? If you say hash(key) % N, the follow-up question — "now add a node" — sinks you. This lesson gives you the taxonomy, the decision framework, and the ring. (Exponent's 2026 Anthropic guide)

1 · ACID vs BASE: two contracts, not good vs bad

Relational databases sell ACID: a transaction is Atomic (all-or-nothing), Consistent (invariants hold — a transfer never leaves money in zero or two accounts), Isolated (concurrent transactions don't see each other's half-done work), Durable (committed means survived a crash). NoSQL systems typically relax that to BASE: Basically Available (the system answers even during partial failure), Soft state (replicas may briefly disagree), Eventually consistent (if writes stop, replicas converge to the same value). (Primer §12: NoSQL)

BASE is not "no guarantees" — it is a different contract. You give up read-your-writes certainty on every replica and buy availability plus horizontal scale. The classic illustration is Amazon's shopping cart: it is built on a BASE store (Dynamo) because an unavailable cart loses a sale right now, while a cart that briefly shows a stale item merely annoys — and add/remove operations can be merged when replicas reconcile. Payments at the same company run ACID, because "eventually the charge appears" is not a contract anyone signs. (Dynamo paper, SOSP 2007)

The interview skill is naming which anomalies your application tolerates — stale reads? lost update? out-of-order timeline entries? — not reciting acronyms. DDIA's framing: "eventual consistency" is a very weak promise; always ask how eventual and what happens meanwhile. (DDIA)

2 · The four NoSQL families

Every "NoSQL" product is one of four shapes. Learn one canonical system and one killer use case per family — that is what interviewers listen for.

FamilyCanonical systemKiller use caseWhy it wins there
Key-valueRedisCache & session storeO(1) get/put by key, sub-millisecond, no query planner in the way
DocumentMongoDB / DynamoDBFlexible entities (user profiles, product catalogs)Whole aggregate in one document; schema evolves per document, no migrations
Wide-columnCassandra / BigtableWrite-heavy time series at scale (metrics, IoT, messages)Append-friendly LSM storage; partition key spreads writes, clustering key sorts within
GraphNeo4jMany-to-many traversals (social graphs, fraud rings)"Friends of friends of friends" is a walk, not N self-joins

Key-value is the simplest possible API — get(k), put(k, v), delete(k) — and that austerity is the point: nothing between you and the hash table means microsecond-to-millisecond latencies. Twitter caches timelines in Redis; virtually every large web property keeps sessions in Redis or Memcached rather than paying a SQL round trip on every authenticated request. The value is opaque to the store: no secondary indexes, no queries by value. (Primer: key-value store)

Document stores keep the KV access pattern but make the value a structured document (JSON/BSON) you can index and query into. The unit of design is the aggregate: an e-commerce order with its line items, shipping address, and payment status lives in one document, so the common read ("show order 12345") is one fetch with no joins. The cost: cross-document queries and multi-document transactions are weak or bolted on — if you constantly join, you chose wrong. (Primer: document store)

Wide-column stores (Bigtable lineage) are physically sorted maps: a partition key routes a row to a node, a clustering key sorts rows within the partition. Writes append to an LSM tree — sequential I/O — which is why Cassandra swallows write-heavy firehoses that would melt a B-tree: Discord stores trillions of messages keyed by (channel_id, bucket) with messages clustered by time; Netflix pushes viewing history through Cassandra; Google built Bigtable for crawl and indexing data. The catch: you must know your query patterns up front, because the primary key is the query plan. (Primer: wide-column store, Discord engineering)

Graph databases store nodes and edges as first-class citizens with direct pointers, so traversal cost is proportional to the subgraph you touch, not to the total table size. Fraud detection is the killer demo: "does this new account share a device, card, or address with any known-fraud account within 3 hops?" is milliseconds in Neo4j and a self-join catastrophe in SQL. Use only when relationship traversal is the workload — for everything else a graph DB is an exotic liability. (Primer: graph database)

3 · SQL vs NoSQL: the decision table

Default answer: start SQL, leave when a specific pressure forces you — and name the pressure out loud. Postgres with read replicas (lesson 0005) carries you astonishingly far: it gives you ACID, ad-hoc queries, decades of tooling, and one system to operate. NoSQL is a targeted response to a named pressure, not a lifestyle. (Primer: SQL or NoSQL)

PressureStay SQL when…Move when……to
Structuredata is relational, normalizedeach entity is a self-contained blobDocument
Joinsad-hoc joins across tables matteraccess is always by primary keyKey-value / Document
Transactionsmulti-row ACID is required (money!)single-key atomicity is enoughKV / Wide-column
Scaleone big box + replicas sufficeswrites must shard across many nodesWide-column / Dynamo-style KV
Schema velocityschema is stable, shared by teamsfields change weekly per featureDocument

Two traps to call out in an interview. First, read scale is not a reason to leave SQL — replicas and caches solve reads; it is write scale that forces sharding. Second, polyglot persistence is normal: a real e-commerce platform keeps orders and payments in Postgres, sessions and carts in Redis, the product catalog in a document store, clickstream events in Cassandra, and a fraud graph in Neo4j. Each store earned its place through a named pressure.

4 · Worked example: splitting an e-commerce monolith

Concrete numbers make this real. Say your marketplace has grown to 50M users, 2M orders/day, 500M clickstream events/day, all in one Postgres:

  • Sessions: 50M users × ~1 auth check per page view. At 20k page views/sec, that is 20k point reads/sec of a 200-byte blob — pure key lookup, zero joins. Pressure: latency and connection pressure on Postgres. Move to Redis: sub-ms, and the SQL box breathes again.
  • Clickstream: 500M events/day ≈ 5,800 writes/sec sustained, 3–5× at peak, append-only, queried by (user, time-range). Pressure: write throughput and table bloat. Move to Cassandra, partition key user_id, clustering key event_time.
  • Product catalog: 10M products, each category with different attributes (shoes have sizes, laptops have RAM). Pressure: schema velocity — a new attribute per merchant onboarding. Move to a document store, one product per document.
  • Orders & payments: 2M orders/day ≈ 23 writes/sec. No scale pressure at all — and they need multi-row ACID (order + inventory decrement + payment record, atomically). Stay in Postgres.

Notice the shape of the reasoning: throughput estimate → named pressure → family. The order table staying put is as important an answer as anything that moved.

5 · The routing problem: why hash(key) % N fails

The moment you shard a KV store (or a cache tier) across machines, every key needs an owner, and every client must compute the same owner. The naive rule node = hash(key) % N is correct, fast, coordination-free — and catastrophic the day N changes. Going from 5 nodes to 6, a key keeps its owner only if hash % 5 == hash % 6, which holds for roughly 1 key in 6. About 83% of all keys change owner because you added one machine. Going 5 → 4 is just as bad: ~80% move. (DDIA ch. 6, partitioning)

Work the blast radius for a cache tier: 100M sessions cached at a 95% hit rate, backing database serving the 5% of misses. Add a sixth cache node under mod-N: 83% of keys now hash to a node that doesn't have them, so the hit rate collapses toward ~17% until the cache re-warms. Misses jump from 5% to ~83% of traffic — a ~17× read surge hits the database with zero warning. That is a self-inflicted outage caused by adding capacity. For a storage tier it's worse still: rebalancing physically moves data, and mod-N moves nearly the whole dataset across the network.

6 · The consistent hashing ring

Consistent hashing (Karger et al., 1997 — built for early web caches, immortalized by Dynamo) fixes the coupling between "where keys live" and "how many nodes exist." Hash both nodes and keys into the same circular space (0 … 264, wrapping around). A key is owned by the first node clockwise from its position. (Karger et al., consistent hashing)

Now membership changes are local:

  • Add a node: it lands somewhere on the circle and takes over only the arc between itself and its counter-clockwise neighbor — about 1/(N+1) of keys (~17% going 5 → 6), and every moved key moves to the new node. Nothing else shifts.
  • Remove a node (deliberately, or by crashing): only its arc moves, to its clockwise successor — about 1/N of keys (~20% at N=5). The other 80% of keys never notice. Since node failure is the common case at scale, this property is not optional.

Lookup is cheap and coordination-free: keep the node positions in a sorted array, binary-search the key's hash, take the next entry (O(log M)). No central router required — every client with the member list computes the same answer, which is exactly how Memcached client libraries do it.

7 · Virtual nodes: smoothing the luck

One point per physical node makes arc sizes a lottery. With 5 nodes, the unlucky draw gives one machine an arc several times its fair share — the lab measures a 3.2× overload with 1 vnode. The fix: give each physical node not one position but 100–500 positions ("virtual nodes" / "tokens"). Each physical node then owns many small scattered arcs, and the law of large numbers averages its total share toward 1/N — the same lab run drops to 1.08× of fair share at 100 vnodes.

Virtual nodes buy three things beyond smoothing. Heterogeneity: a machine with 2× the RAM takes 2× the vnodes and thus 2× the keyspace. Failure dispersion: a dead node's many small arcs are inherited by many different successors, so its load spreads across the cluster instead of doubling one neighbor's — which is what turns "one failure" into "cascading failure" in the 1-vnode design. Gradual rebalancing: moving one vnode at a time throttles data movement. Costs: the ring's memory and lookup table grow M-fold, and range scans fragment — which is why Cassandra ships with num_tokens (256 historically, 16 in newer versions as tradeoffs shifted). (Cassandra architecture docs)

8 · Who uses it, and what the alternatives are

SystemWhere the ring livesDetail
Amazon Dynamo / DynamoDB lineageServer sideRing + vnodes + replication to the next R distinct nodes clockwise (Dynamo paper)
CassandraServer sideToken ring, gossip membership, every node can route any request
Memcached clients (Ketama, mcrouter)Client sideThe servers are dumb; clients hash consistently so a pool resize doesn't flush the cache fleet
CDNs & load balancers (e.g. Envoy)Edge/proxyRing hash / Maglev policies pin a user or object to a backend for cache affinity

Two honest alternatives to name before the interviewer does. Range partitioning (Bigtable, HBase, MongoDB, TiDB): split the keyspace into contiguous sorted ranges managed by a coordinator; you gain efficient range scans, you risk hot ranges from sequential keys (timestamps!) and you now operate a coordinator. Rendezvous (HRW) hashing: for each key, score every node with hash(key, node) and pick the max — same minimal-movement property, no ring to store, O(N) per lookup; great for small N like proxy pools. Consistent hashing wins when N is large and lookups are hot. (DDIA ch. 6 covers both hash and range rebalancing.)

9 · Failure modes: what the ring does NOT fix

  • Hot keys. Consistent hashing balances key count, not key load. Real access is zipf-distributed: one flash-sale SKU, one celebrity's profile, one viral post can be 10%+ of all traffic, and it lands on exactly one node no matter how many vnodes you add. Mitigations: replicate the hot key to R nodes and spread reads; put a small local cache in front of the ring (a few hundred hottest keys in each client absorbs the head of the zipf curve); or salt the key (key#1…key#R) and fan reads across the copies. The lab's experiment 4 makes this vivid.
  • Rebalance stampedes. Even 17% key movement re-warms caches and copies data. Production systems throttle vnode handoff and warm new nodes before serving.
  • Membership disagreement. If two clients briefly disagree about the member list (gossip lag), the same key maps to two owners — reads miss, writes fork. This is why Dynamo pairs the ring with versioning (vector clocks) and read-repair; the ring solves placement, not consistency.
  • Fan-out amplification still applies. A request touching 50 shards inherits the worst shard's tail (lesson 0001). Partitioning multiplies your exposure to p99s.
One AI note The same machinery reappears in AI infrastructure twice: vector databases (Pinecone, pgvector, Qdrant) are best understood as a fifth, specialized NoSQL family — key → embedding, queried by nearest-neighbor instead of equality (Phase 5) — and LLM serving gateways consistent-hash prompt-prefix hashes so requests sharing a prefix land on the worker already holding that prefix's KV-cache (Phase 4). Same ring, new payloads.

11 · Project step

Back to the spine project, the LLM inference gateway. Assume today it keeps everything in one Postgres: user accounts & billing, API-key auth lookups, rate-limit counters, and per-request usage logs (tokens in/out, latency, model — millions of rows/day). In a scratch note, decide: which store would you move off SQL first, and to which family? A defensible answer, using §4's method: rate-limit counters and API-key lookups → Redis (key-value; read on every single request with a sub-ms budget); usage logs → wide-column (append-only time series, no joins at write time); accounts & billing → stay in Postgres (multi-row ACID, money). Name the pressure for each move, estimate the write rate that forced it, and bring the note to the next session for critique.

12 · Interview talking points

  • Open with the family, justified by access pattern. "Reads are always by key with a sub-ms budget — that's a key-value store" beats naming a product first.
  • Say the default out loud. "I'd start with Postgres and move the sessions/events off it when this pressure appears" signals judgment; reflexive NoSQL signals fashion.
  • Own the mod-N number. When sharding comes up, volunteer: "naive mod-N remaps ~83% of keys going 5→6 nodes; a consistent-hash ring moves ~17%, all onto the new node." Interviewers reward the quantified version.
  • Bring up vnodes before imbalance is asked. "With one token per node someone owns 3× their share; a few hundred vnodes flattens that to ~1.1×, and it also disperses a dead node's load."
  • Name the ring's limit yourself. "This balances key count, not load — a zipf-hot key still needs replication or a front cache." Pre-empting the hot-key follow-up is a senior move.
  • Treat node removal as the design case, not the edge case. Failures are routine at scale; a placement scheme is judged by its behavior on the worst day.
  • Know the who's-who. Dynamo and Cassandra (server-side ring), Memcached clients / Ketama (client-side ring), range partitioning as the deliberate alternative when scans matter.
