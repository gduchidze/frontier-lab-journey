Chat System — WhatsApp-scale Messaging

Every design so far was stateless behind a load balancer. Chat breaks that: millions of long-lived sockets, each one pinned to a specific box, and messages that must arrive once, in order, even to phones that are off.

Mission tie. The stateful connection-gateway tier you build here — long-lived sockets, a routing table answering "which box holds user X" — is exactly the pattern lesson 37 reuses to stream tokens from an inference cluster to 100k concurrent clients.
Rep protocol This is a rep, not a read. FIRST: 40 minutes, alone, out loud — run the full lesson-0009 framework on the prompt in section 1 with no peeking. THEN read this reference solution and log every delta (missed requirement, unspent number, unnamed failure mode) as a learning record.

1 · The prompt, and what it actually tests

"Design a messaging app like WhatsApp: 1:1 and group chats, delivery receipts, presence, works when the recipient is offline."

The twist that separates this rep from every previous one: statefulness. A feed or a URL shortener is request/response — any server can answer any request, so scaling is "add replicas behind the LB." Chat inverts that. A message to you must find the specific machine currently holding your socket, within a second, among hundreds of gateways. The interviewer is testing whether you notice that inversion and design for it explicitly: a connection-holding tier, a routing table, and a story for what happens when a socket-holding box dies.

Secondary tests hiding in the prompt: message ordering (why timestamps betray you), delivery guarantees (at-least-once plus dedup, not mythical exactly-once), and fanout discipline (groups and presence both explode if done naively). Each gets its own deep dive below.

Non-functional contract: delivery latency p95 < ~500 ms when both parties are online; no message loss, no visible duplicates, per-conversation order preserved; 99.99% availability. Consistency note worth saying aloud: chat needs per-conversation ordering only — nobody can observe whether two different conversations interleave "correctly," and that relaxation is what makes the system shardable.

3 · Estimate (~5 min) — and spend every number

Messages:  500M DAU × 40 msgs/day = 20B msgs/day
           ÷ ~100k s/day ≈ 200k msg/s average, ×3 peak ≈ 600k msg/s inbound
           each has ≥1 recipient → gateway egress ≥ 1.2M msg/s peak (groups push it higher)

Sockets:   idle apps keep sockets open; assume ~20% of DAU concurrent → ~100M connections
           WhatsApp-era folklore: 1–2M conns per tuned Erlang/FreeBSD box
           (order-of-magnitude only — engineering lore, not a spec) → ~100 boxes;
           a conservative 100k conns/box → ~1,000 boxes. Either way: a TIER, not a server.

Storage:   20B msgs × ~100 B ≈ 2 TB/day ≈ 730 TB/year, text only, before media.

Now spend them: 600k msg/s → services talk through a partitioned queue, not RPC fan-in. 100M sockets → the gateway tier is sized by memory and file descriptors, not CPU — and it must be a separate tier so the stateless chat logic can deploy independently of a million held connections. 2 TB/day → a wide-column store partitioned by conversation (section 8), because no single Postgres eats 730 TB/year.

4 · High-level design (~15 min)

Transport. WebSocket, not long polling: chat is bidirectional (server pushes messages and receipts and presence), and one persistent socket costs far less than reconnecting poll loops at 100M-client scale. Long polling is the fallback for networks that break WebSockets.

The stateful part. A chat gateway tier terminates sockets. Gateways hold connections and do almost nothing else. When user X connects to gateway 42, the gateway writes X → gw42 into a session/routing table (Redis: hot, small values, TTL-friendly). That table is the answer to the whole statefulness twist: "which box holds user X" becomes one lookup.

Message flow — narrate it end to end:

sender ──ws──▶ gateway A ──▶ chat service ──▶ message store (persist)
                                   │              + assign seq (§5)
                                   ▼
                            queue (partitioned by conversation)
                                   ▼
                     routing lookup: recipient → gateway B?
                       online  → push via gateway B ──ws──▶ recipient
                       offline → per-device offline queue (§6)

The stateless part. History, search, contacts, login: plain HTTP APIs (GET /conversations/{id}/messages?before_seq=N) over the same message store — cacheable, replicable, boring on purpose. Splitting the boring HTTP plane from the socket plane is the design's main structural move.

5 · Deep dive: message ordering

Why timestamps fail. Client clocks skew — NTP keeps well-behaved devices within tens of milliseconds, but phones drift by seconds. If Bob's clock runs 150 ms behind and he replies to Alice within 100 ms, sorting by client timestamp shows the answer before the question. Server receive-timestamps are better but still race across a multi-node service. Your lab manufactures exactly this bug.

The fix: a single writer per conversation. Route every message in conversation C to one partition (hash conversation_id), where one writer assigns a monotonically increasing sequence number. Order is now defined by arrival at that single writer — no clock involved. This is the happens-before insight in practice: physical time cannot order concurrent events on different machines; a designated sequencer can. Clients render by seq, and gaps double as loss detection ("I have 41 and 43 — refetch 42").

Dedup rides along (the lesson-0011 idempotency callback): the client attaches a UUID to each message before sending. Retries reuse the ID; server and recipients drop anything they've seen. Sequence numbers give order, client message IDs give exactly-once effect — you need both.

6 · Deep dive: delivery guarantees

The receipt ticks are a state machine, each transition driven by an ack:

StateTransition trigger
sent (one tick)Chat service persisted the message and acked the sender.
delivered (two ticks)Recipient's device acked receipt back through its gateway.
read (blue)Recipient's client reported the conversation opened.

The guarantee is at-least-once + dedup. Networks lose packets and lose acks; the only honest policy is: retry until acked, and make redelivery harmless via message-ID dedup (§5). Exactly-once delivery is impossible over a lossy network; exactly-once appearance is what you engineer.

Offline path. If the routing lookup finds no live socket, the message lands in a per-device offline queue, ordered by seq. On reconnect the device sends its last-seen cursor and the queue drains in order. The ack protocol is cumulative — "I have everything through seq 47" — so one lost ack costs a cheap redelivery, not a lost message. Mobile wrinkle worth one sentence: the OS push notification (APNs/FCM) is only a wake-up signal; the actual messages come from your queue when the socket returns.

7 · Deep dive: presence

Presence is a heartbeat protocol: the client pings over its socket every ~10 s; miss a few and the server marks the user offline, recording last_seen. Disconnect events (gateway noticed the socket die) do the same immediately.

Why naive fanout explodes. Every online↔offline transition, pushed to every contact, is fanout applied to your flakiest event source. A subway ride can flap a phone dozens of times an hour; multiply by ~200 contacts and by hundreds of millions of users, and presence traffic dwarfs message traffic — for updates nobody is looking at.

Three standard dampers:

  • Debounce. Publish a transition only after it has been stable for tens of seconds; a flap that heals in 5 s never leaves the server.
  • Pull on open. When the app opens, fetch presence for the chat list. No standing subscription for people you aren't looking at.
  • Subscribe narrowly. Live updates only for the conversation currently on screen; unsubscribe on close.

The senior sentence: "presence is best-effort, eventually-consistent data — I'll happily show a 30-second-stale last_seen to avoid melting the fanout tier." A wrong receipt is a bug; a slightly stale presence dot is Tuesday.

8 · Deep dive: group chat, storage, multi-device

Group fanout. One message to a 500-member group = 1 write + up to 500 deliveries: one routing lookup and push per online member, offline-queue append per offline device. This is why the member cap from Clarify matters — it bounds worst-case fanout to 500×, unlike the feed lesson's unbounded celebrity problem. The chat service fans out after the single persist + seq assignment, so the group still has one writer and one order everyone agrees on.

Message store. Write-heavy, append-only, read-by-conversation — a wide-column fit. Discord's messages-store writeup describes exactly this shape in Cassandra: partition by channel (plus a time bucket so hot channels don't grow unbounded partitions), cluster by time-ordered message ID descending. Our schema:

messages(
  PARTITION KEY (conversation_id, bucket),   -- bucket ≈ 10 days, caps partition size
  CLUSTERING KEY seq DESC,                   -- newest first, already sorted on disk
  sender_id, msg_id, body, sent_at)

"Load this conversation" = one partition, pre-sorted — the read pattern is the schema.

Multi-device sync is per-device cursors: each device tracks the last seq it has per conversation and catches up independently through the same offline-queue path. The server prunes a message's queue entry once every registered device's cursor passes it (or a TTL expires).

10 · Variants and twists

  • E2E encryption. With Signal-protocol E2EE (WhatsApp's actual deployment) the server relays ciphertext it cannot read: server-side search, spam scanning, and abuse review of content disappear; delivery, ordering, receipts, and presence all survive because they're metadata. Be ready to say which features die.
  • Read-receipt privacy. Receipts leak behavior; the classic policy is reciprocity — disable sending yours and you stop seeing others'. A product constraint implemented as one server-side check.
  • Federation. Matrix/XMPP-style server-to-server chat replaces your routing table with inter-domain routing and trust — name the trade: reach and openness versus losing single-operator control of ordering and abuse handling.

AI twist (one paragraph, on request only). An LLM assistant in a chat is just another conversation member whose replies stream token-by-token — the gateway pushes partial-message frames over the same socket, and per-conversation seq still orders the final message. Same tier, new producer; lesson 37 builds this out.

11 · Interview talking points

  • Naming the twist: "Everything before the gateway is stateless; the gateway tier is deliberately the only stateful part, and the Redis routing table is how the stateless world finds it."
  • Spending a number: "100M concurrent sockets is why the gateway is its own tier — it scales on memory and descriptors while chat logic scales on CPU."
  • Ordering: "I won't order by timestamps — clock skew reorders fast replies. One writer per conversation partition assigns seq; clocks never vote."
  • Delivery: "At-least-once plus client-message-ID dedup. Exactly-once delivery is impossible; exactly-once appearance is engineerable."
  • Unprompted failure mode: "A dying gateway drops a million sockets — the danger isn't lost messages (gateways hold none), it's the reconnect storm. Jittered backoff, and the offline queue catches anything in flight."
  • Presence discipline: "Presence is best-effort by design — debounce flaps, pull on open, subscribe only to the visible chat."
