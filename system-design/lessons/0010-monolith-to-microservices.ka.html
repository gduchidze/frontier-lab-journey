მონოლითი → მიკროსერვისები

Decomposition (დაშლა) გარიგებაა, რომელშიც იხდი latency-ით, ჩავარდნის მათემატიკით და ოპერაციებით. ეს გაკვეთილი ამ გარიგებას ფასს ადებს — რომ იცოდე როდის გაჭრა, როგორ გაჭრა და, რაც მთავარია, როდის არა.

მისიასთან კავშირი. Anthropic-ის დიზაინის ამოცანებზე სუსტი კანდიდატები რეფლექსით ხატავენ თორმეტ მიკროსერვისს. ძლიერები ერთი პროცესით იწყებენ, ყოველი გაჭრისთვის კონკრეტულ ძალას ასახელებენ (მასშტაბირება, გუნდის ავტონომია, fault isolation) და შემდეგ გაჭრას ფასს ადებენ: „ოცი სამ-ცხრიანი სერვისი სერიულ ჯაჭვში ≈ 98%-იანი სისტემაა". ინტერვიუერები ზუსტად ამ არითმეტიკაზე აღრმავებენ კითხვებს. (Exponent-ის 2026 Anthropic გზამკვლევი)

1 · მონოლითი სწორი ნაგულისხმევი არჩევანია

ერთი კოდბაზა, ერთი პროცესი, ერთი deploy. სანამ დასცინებ, ჩამოთვალე რას იღებ უფასოდ: ფუნქციის გამოძახება მოდულებს შორის ~50 ნანოწამია; ქსელური hop სერვისებს შორის — ~1 მილიწამი. ეს 20,000-ჯერადი უფსკრულია, რომელსაც ყოველ დაშლილ გამოძახებაზე თავიდან კვეთ. მოდულებს შორის ტრანზაქცია ერთი ACID COMMIT-ია; debugging — ერთი stack trace. DHH ამას „majestic monolith"-ს უწოდებს, Fowler-ის რჩევა კი პირდაპირია: MonolithFirst — თითქმის ყველა წარმატებული მიკროსერვისების ისტორია მონოლითით დაიწყო, რადგან სერვისების სწორ საზღვრებს მხოლოდ დომენის ოპერირებით სწავლობ.

ყველაზე ძლიერი კონტრმაგალითი თეზისისა „დიდი = მიკროსერვისები" არის Shopify: მსოფლიოში ერთ-ერთი უდიდესი Rails კოდბაზა, განზრახ შენარჩუნებული როგორც modular monolith (მოდულური მონოლითი) — კოდი კომპონენტებადაა დაყოფილი იძულებითი შიდა საზღვრებით (packwerk უკანონო cross-component გამოძახებაზე build-ს აგდებს), deploy კი ერთი ერთეული რჩება. საზღვრები ქსელის გარეშე — ეს არის ინტერვიუზე დასაცავი საწყისი პოზიცია.

2 · რა აიძულებს რეალურად დაშლას

  • დამოუკიდებელი მასშტაბირება. Checkout-ს პიკზე 50 მანქანა სჭირდება, invoicing-ს — 2. მონოლითში მთელ ბინარს ყველაზე ცხელი მოდულის საჭიროებაზე ამასშტაბირებ და სხვაობას კარგავ.
  • გუნდის ავტონომია (Conway-ის კანონი). „ორგანიზაციები ისეთ სისტემებს აპროექტებენ, რომლებიც მათი კომუნიკაციის სტრუქტურას ირეკლავენ." Amazon-მა ეს კედელი ~2001-ში მოისინჯა: Werner Vogels აღწერს retail მონოლითის სერვისებად დაჭრას, თითოეული two-pizza team-ის სრული საკუთრებით — „you build it, you run it" — რომ გუნდებს ნებართვის თხოვნის გარეშე შეძლებოდათ deploy (ACM Queue: საუბარი Werner Vogels-თან).
  • Fault isolation (ჩავარდნის იზოლაცია). Recommendations-მოდულის memory leak checkout-საც აგდებს, თუ პროცესს იზიარებენ; ცალკე სერვისებად recommendations კვდება, checkout კი ცოცხლობს.
  • დამოუკიდებელი deploy რიტმი. Pricing-გუნდი საათში ერთხელ უშვებს, ledger-გუნდი — კვირაში ერთხელ აუდიტის შემდეგ; ერთი pipeline ორივეს უფრო ნელ რიტმზე აჯაჭვებს.

ინტერვიუს ხერხი: არასდროს დაასაბუთო გაჭრა აბსტრაქტული „scalability"-ით — დაასახელე ამ ოთხიდან რომელი ძალა მოქმედებს ამ სისტემაზე, თორემ მოდული პროცესში დატოვე.

3 · სად გაჭრა: bounded context-ები და არა ფენები

ჭრი ბიზნეს-შესაძლებლობის მიხედვით და არასდროს ტექნიკური ფენის მიხედვით: „UI სერვისი + ლოგიკის სერვისი + database სერვისი" ნიშნავს, რომ ყოველი მოქმედება სამივეს კვეთს — ფენები დაანაწილე და არა დომენი. ჭრი DDD-ის bounded context-ების გასწვრივ: Orders, Payments, Inventory — ერთეულები, სადაც სიტყვას „customer" ერთი მნიშვნელობა აქვს და რომლებიც ერთი მიზეზით იცვლებიან (Fowler & Lewis, Microservices).

მიგრაცია კი ინკრემენტულია, არასდროს big-bang: strangler fig (მახრჩობელა ლეღვი) — მონოლითის წინ დგამ routing facade-ს, ერთ შესაძლებლობას თითო ჯერზე გამოაქვს მის უკან, ტრაფიკს მარშრუტ-მარშრუტ გადაანაცვლებ და მონოლითი იკუმშება, სანამ ნარჩენი წაშლადი არ გახდება. ძველი სისტემა მთელი გზა production-ს ემსახურება — ყოველი ნაბიჯი პატარაა, შექცევადი და ჩაშვებადი.

4 · Microservice tax — არითმეტიკა, რომელიც ინტერვიუერებს სმენა უნდათ

Latency: ლაბში იგივე 20 ms სამუშაო სერიულ ჯაჭვში: p99 20 ms-დან (მონოლითი) ~29 ms-მდე იზრდება N=2-ზე და ~58 ms-მდე N=20-ზე — mean ჰოპზე ~1 ms-ით იზრდება, p99 კი თითქმის ორჯერ სწრაფად, რადგან კუდები სერიულ ჰოპებზე იკრიბება. Availability: სერიული ჯაჭვი ჩავარდნას ამრავლებს: 0.99920 ≈ 98.0% — დღეში 10 მლნ მოთხოვნაზე ~200,000 ჩავარდნა, მონოლითის ~10,000-ის ნაცვლად. Retry-ები ცხრიანებს იბრუნებენ, მაგრამ დატვირთვას ამრავლებენ — ნაწილობრივი ავარიისას retry-ქარიშხალი ზუსტად ავადმყოფ სერვისს ურტყამს (AWS Builders' Library). Fan-out-ის კუდი: N სერვისს პარალელურად რომ ეძახი, ყველაზე ნელს ელოდები — ალბათობა, რომ მოთხოვნა ერთ p99 straggler-ს (ჩამორჩენილს) მაინც მოხვდება, არის 1 − 0.99N: 18% N=20-ზე, 63% N=100-ზე (Dean & Barroso, „The Tail at Scale"). მათი წამალი — hedged requests (სათადარიგო ასლი p95 დაყოვნების შემდეგ) — კუდს ~5% დამატებითი დატვირთვის ფასად ჭრის.

5 · Service discovery — როგორ პოულობენ სერვისები ერთმანეთს

ინსტანციები რეგისტრირდებიან registry-ში — Consul, etcd, ZooKeeper — health check-ებით (heartbeat/TTL), რომ მკვდარი ინსტანციები აღარ ამოიხსნას (resolve). ორი ტოპოლოგია (Primer §Service discovery):

მოდელივინ ეკითხება registry-სიგებსიხდის
Client-side (პატერნი)თავად გამომძახებელი სერვისი, ინსტანციებზე თვითონ აბალანსებსერთით ნაკლები hop; ჭკვიანი per-request არჩევანიDiscovery-ლოგიკა ყველა კლიენტში, ყველა ენაზე
Server-side (პატერნი)Load balancer / router; კლიენტები სტაბილურ სახელს ეძახიანკლიენტები „ბრიყვები" რჩებიან; discovery ერთგანააზედმეტი hop; LB საოპერაციო ინფრასტრუქტურაა

ტრანსპორტიც მნიშვნელოვანია: DNS-ზე დაფუძნებული discovery უნივერსალურია, მაგრამ ქეშირდება — TTL-ები failover-ს აყოვნებენ; API-ზე დაფუძნებული ახალია, მაგრამ registry-ს კლიენტზე გაბმული. Kubernetes-ის Service სახელი სწორედ server-side discovery-ია სტაბილური DNS-ის უკან.

6 · API gateway vs BFF

კლიენტებმა შენი შიდა ტოპოლოგია არ უნდა იცოდნენ. API gateway (პატერნი) ერთი შესასვლელი კარია, რომელიც edge-საკითხებს ერთხელ ხსნის: auth, rate limiting, TLS, routing, აგრეგაცია. ფასი სიმეტრიულია: ის შეიძლება single point of failure გახდეს (ამიტომ რეპლიცირებული და „თხელი" უნდა იყოს — routing და პოლიტიკა, არა ბიზნეს-ლოგიკა). როცა ერთი საერთო gateway per-client აგრეგაციას იგროვებს — მობილურს მჭლე პასუხი უნდა, ვებს მდიდარი — Sam Newman-ის Backend-for-Frontend edge-ს ყოფს: თითო პატარა backend თითო frontend-ზე, იმ frontend-გუნდის საკუთრებაში. დუბლირებას ავტონომიაში ცვლი — მთელი მიკროსერვისების გარიგება მინიატურაში.

7 · მონაცემების საკუთრება — დაკარგული join

წესი, რომელიც სერვისებს ნამდვილს ხდის: database-per-service (პატერნი) — თითო სერვისი თავის მონაცემებს ფლობს, სხვისი ცხრილები ხელშეუხებელია. გაზიარებული ცხრილი გაზიარებული სქემაა, ეს კი deploy-დროის ბმა, რომელიც მონოლითს ჩუმად აწყობს თავიდან. ფასი კონკრეტულია: join გაქრა. „Orders + customer-ის სახელები" ერთი SQL ხაზი იყო; ახლა ან API-კომპოზიციაა (ორივეს ეძახი, მეხსიერებაში აერთებ — fan-out-ის გადასახადით), ან materialized ლოკალური ასლი — Orders ინახავს customer-ველების read-only პროექციას, event-ებით განახლებულს. ეს ასლი eventually consistent-ია (საბოლოოდ თანმიმდევრული): რაღაც ფანჯარაში Orders ძველ სახელს აჩვენებს. Cross-service ტრანზაქციებიც გაქრა — sagas და event-driven თანმიმდევრულობა 0013–0014 გაკვეთილების თემაა (DDIA, თავები 7 და 9).

8 · Failure mode-ები, რომლებიც ინტერვიუებს წყვეტენ

  • Distributed monolith (განაწილებული მონოლითი) — ორივეს უარესი. სერვისები, რომლებიც მხოლოდ ერთად deploy-დებიან, ერთ database-ს იზიარებენ და ყველაფერზე სინქრონულად ეძახიან ერთმანეთს: იხდი მიკროსერვისების ყველა ქსელურ და საოპერაციო გადასახადს, ინარჩუნებ მონოლითის ყველა ბმას. დიაგნოსტიკა: შეგიძლია ერთი სერვისის ცალკე deploy? ერთი სქემის ცვლილება ხუთ repo-ზე ვრცელდება?
  • კასკადური ჩავარდნა სინქრონული ჯაჭვებით. A → B → C: C ნელდება, B-ის thread-ები ლოდინში იბლოკებიან, B-ის pool იწურება, A მიჰყვება — ერთი ნელი ფოთოლი მთელ ჯაჭვს აგდებს, retry-ები კი ტალღას აძლიერებენ. დაცვა: timeout-ები, circuit breaker-ები, შემოსაზღვრული რიგები, load shedding (Google SRE წიგნი: Addressing Cascading Failures).
  • ვერსიების აცდენა (version skew). N დამოუკიდებლად deploy-ადი სერვისით rollout-ის შუაში ძველი და ახალი API ვერსიები ყოველთვის ერთდროულად ცოცხლობს სადღაც. კონტრაქტები backward-compatible უნდა იყოს (ველებს ამატებ, არასდროს გადააკეთებ), contract-ტესტები კი იმ კომპილატორს ანაცვლებენ, რომელიც გამოძახებამ პროცესის დატოვებისას დაკარგა.
  • ორგანიზაცია არქიტექტურაზე პატარაა. Fowler-ის MicroservicePrerequisites — „ამ სიმაღლის უნდა იყო": სწრაფი provisioning, მონიტორინგი, სწრაფი deploy, DevOps კულტურა. ხუთი ინჟინერი ორმოცი სერვისით ამ ზღვარს ვერ ცდება. Segment-მა ცნობილად უკან დაიხია: ~140 სერვისი მონოლითში დააკონსოლიდირა, როცა საოპერაციო ტვირთმა პატარა გუნდი წალეკა — უკან შერწყმა მარცხი კი არა, გარიგების ხელახალი შეფასებაა.
LLM შტრიხი Spine-პროექტი ამ გაკვეთილზე ზუსტად ჯდება: LLM inference პლატფორმა იშლება როგორც gateway → router → batching scheduler → workers → KV-cache manager, რადგან §2-ის ძალები ნამდვილად მოქმედებს — GPU worker-ები სხვა რკინაზე მასშტაბირდებიან, ვიდრე CPU-ზე მიბმული gateway, scheduler კი ბევრად ხშირად deploy-დება, ვიდრე მოდელის წონებიანი worker-ები. მაგრამ გადასახადი მაინც წყვეტს: tokenizer პროცესშივე რჩება gateway-სთან ან worker-თან, რადგან ტოკენიზაცია სუბ-მილიწამიანი CPU საქმეა და „tokenizer სერვისი" ქსელურ hop-ზე მეტს დახარჯავდა, ვიდრე თავად სამუშაოზე — microservice tax-ის ურყევესი დარღვევა.

9 · ლაბი — გადაიხადე გადასახადი შენი ხელით

პრაქტიკა (ქვიზამდე გააკეთე) ყველაფერი ზემოთქმული ახლა გაზომვადია: labs/lab-0010-microservice-tax/-ში python3 microservice_tax_lab.py უშვებს სამ ექსპერიმენტს — იგივე 20 ms სამუშაო N=1/2/5/10/20 სერვისზე დაშლილი lognormal ჰოპებით (უყურე, როგორ უსწრებს p99 საშუალოს), სერიული ჯაჭვის availability 100k მოთხოვნაზე vs თეორიული 0.999N, და fan-out-ის კუდის გაძლიერება hedged request-ებით p95-ზე. შემდეგ README-ს E1–E4 სავარჯიშოები: იწინასწარმეტყველე, შეცვალე კონფიგი, გაუშვი თავიდან, შეადარე. მცდარი პროგნოზები არის დავალების შედეგი — მოიტანე ისინი შემდეგ სესიაზე.

10 · პროექტის ნაბიჯი

გადაწყვეტილების დროა LLM inference პლატფორმისთვის: რომელი კომპონენტებია ცალკე სერვისები და რომლები ცხოვრობენ ერთ პროცესში? ორი წინადადებით დაიცავი. დასაცავი მაგალითი: gateway, router/scheduler და GPU worker-ები ცალკე სერვისებია, რადგან სხვადასხვა რკინაზე მასშტაბირდებიან და სხვადასხვა რიტმით deploy-დებიან; tokenizer-ი და per-worker KV-cache აღრიცხვა worker-ის პროცესშივე რჩება, რადგან ~0.1 ms-იანმა CPU საქმემ ყოველ მოთხოვნაზე ~1 ms-იანი ქსელური hop არ უნდა გადაიხადოს. დაწერე შენი ორი წინადადება — ყოველ გაჭრას ძალა დაუსახელე, ყოველ შერწყმას გადასახადი — და მოიტანე შემდეგ სესიაზე გასარჩევად.
