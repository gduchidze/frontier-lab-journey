OS Essentials — Processes, Virtual Memory, Scheduling

Your server is a guest of the operating system. Here is the rent it pays — and the paging trick that vLLM later steals wholesale.

Mission tie. Anthropic's design rounds assume you can reason about what a "worker" costs, why memory limits kill pods, and why a box at 98% RAM is one request from death. Better: PagedAttention — the core idea of vLLM — is OS demand paging transplanted into the KV cache, so this lesson is a direct prerequisite for the flagship inference-batching prompt. Canon here is OSTEP (Operating Systems: Three Easy Pieces); the payoff is Inside vLLM.

1 · What an OS actually does: two illusions

An OS sells two illusions. CPU virtualization: every process believes it has the CPU to itself; the scheduler time-slices reality behind its back. Memory virtualization: every process believes it owns a huge private address space starting at zero; page tables map that fiction onto whatever physical RAM is free. (OSTEP: The Abstraction — The Process)

System design cares because both illusions leak under load — as scheduling delay and page faults — and the leaks land precisely on your p99.

2 · Processes vs threads: the ledger

A process = address space + page table + file descriptors + at least one thread. A thread = a stack and registers inside a process, sharing everything else. (OSTEP: Concurrency intro)

ThreadProcess
Address spaceShared with siblingsPrivate (isolated)
Creation cost (lab-measured)~50–100 µs~0.5 ms fork · ~10 ms spawn
CommunicationShared memory (races possible)Pipes/sockets (copy, but safe)
One crashes / leaksTakes the whole process downBlast radius contained
Typical serving useI/O concurrency in a workerWorkers, isolation, multi-GPU ranks

The design rule falls out of the ledger: processes for isolation, threads for sharing — and because creation is never free, real servers pre-fork and pool instead of creating either per request. (Compare gunicorn workers, nginx worker processes, vLLM's one-process-per-GPU ranks.)

3 · Virtual memory: pages, frames, TLB

Virtual address spaces are cut into pages (typically 4 KiB); physical RAM into same-sized frames. The page table maps page → frame; the TLB caches recent translations so the mapping doesn't cost a memory walk on every access. (OSTEP: Paging, OSTEP: TLBs)

The indirection buys three superpowers: processes can't touch each other's memory (isolation), physical RAM needn't be contiguous (no external fragmentation), and pages can live on disk and be faulted in on demand. Hold that third one — it becomes PagedAttention in §5.

Vocabulary Page: fixed-size chunk of virtual memory. Frame: its physical slot. Page fault: access to a page with no frame mapped — the OS pauses your thread, fixes the mapping (maybe reading from disk), resumes. Working set: the pages a process actually touches over a window of time.

4 · Demand paging: allocation is a promise

malloc/bytearray mostly just reserves address space. Physical frames arrive later, one page fault at a time, on first touch. A "soft" fault (frame available, just needs mapping) costs on the order of a microsecond; a "hard" fault (page must come from disk) costs whatever the disk costs — milliseconds on HDD, ~100 µs on NVMe. (OSTEP: Beyond Physical Memory)

1 GiB buffer = 1 GiB ÷ 4 KiB = 262,144 pages
first-touch cost  ≈ 262,144 soft faults × ~1 µs ≈ 0.3 s   (one-time, at startup… or at p99)

effective access time under paging pressure (hit 100 ns, hard fault 1 ms):
  0.1% faults:  0.999 × 100 ns + 0.001 × 1 ms ≈ 1.1 µs   → 11× slower than RAM
  1%   faults:  ≈ 10.1 µs                                → 100× slower than RAM

Read that twice: a 1% fault rate makes memory 100× slower. This is the same "tiny tail owns the mean" arithmetic as lesson 0001 — one level down the stack. It is why latency-critical services touch (pre-fault) their buffers at startup and pin pages (mlock) when they can.

5 · Working sets and the thrashing cliff

While the sum of working sets fits in RAM, paging is invisible. The moment it doesn't, every fault evicts a page someone needs soon, which faults again: the machine does I/O full-time and useful work collapses. That mode is thrashing, and the lab's experiment 3 shows it is a cliff, not a slope — a working set 12% past RAM ran ~1,000× slower. Practical consequences: serving boxes disable swap, and the kernel's OOM killer exists because a fast kill beats a thrashing host.

LLM connection — PagedAttention A KV cache grows one token at a time to an unpredictable length — exactly the workload that fragmented early LLM servers, which reserved contiguous max-length buffers (internal fragmentation, wasted VRAM). vLLM's fix is literally this lesson: cut the KV cache into fixed-size blocks (pages), keep a block table (page table) per request, allocate on demand, and even share blocks across requests like copy-on-write forks. Result: near-zero waste and much bigger batches. (Inside vLLM)

6 · Scheduling and the price of a context switch

More runnable threads than cores means the scheduler multiplexes: each thread runs a timeslice, then is preempted. A context switch saves registers, switches page tables, and — the expensive part — hands the next thread a cold cache and cold TLB. Direct cost is ~1–10 µs; the cache-refill shadow can be much larger. (OSTEP: Scheduling)

Two design-relevant corollaries. Timeslice trade-off: short slices give responsiveness but burn throughput on switch overhead; long slices do the reverse — the same latency/throughput axis from lesson 0001. Run-queue delay: when CPU saturates, requests wait to be scheduled before they wait for anything else; a saturated run queue is the invisible term in many p99 mysteries.

7 · cgroups and containers: a process wearing fences

A container is not a VM. It is an ordinary Linux process wearing two fences: namespaces limit what it can see (its own PIDs, network, filesystem — namespaces(7)), and cgroups limit what it can use (CPU quota, memory ceiling, I/O — cgroups(7)).

So Kubernetes limits are just cgroup knobs: a CPU limit is a quota per period — exceed it and your threads are throttled (paused mid-request, a classic tail-latency source); a memory limit is a ceiling — exceed it and the OOM killer terminates the container (OOMKilled, exit 137). Everything in §§2–6 applies unchanged inside a pod.

8 · Sharp edges: how the OS kills your p99

  • Swap on a serving box. One cold page in the request path adds milliseconds; §4's math says 1% faulting = 100× slowdown. Disable swap; size for the working set.
  • OOM killer roulette. Under memory pressure the kernel kills by heuristic score — often your biggest process, i.e. the model server itself. Set limits so the right thing dies predictably.
  • CPU-limit throttling. A pod with limit 1.0 CPU that bursts to 4 threads gets frozen ~75% of each quota period. Looks exactly like random 100 ms stalls in traces.
  • Lazy allocation surprise. "Preallocated" buffers still fault on first touch; overcommit means the allocation that "succeeded" can still OOM later, on a write.
  • fork() in a threaded process. The child inherits only the calling thread; any lock held by another thread stays locked forever in the child. (This ambushes Python multiprocessing users; more in p0-03.)

Interview talking points

  • "Processes buy isolation, threads buy cheap sharing — so I pre-fork workers for blast-radius control and use threads or async inside each for I/O."
  • "Allocation is lazy: first touch pays a page fault. For latency-critical paths I pre-fault and pin memory at startup."
  • "A 1% page-fault rate makes memory effectively ~100× slower — which is why I run serving nodes with swap off and size RAM to the working set."
  • "Thrashing is a cliff, not a slope: the moment working sets exceed RAM, the machine does I/O instead of work. The OOM killer is the kernel choosing a fast death over that."
  • "Containers are processes wearing namespaces and cgroups — so CPU limits mean quota throttling in the tail, and memory limits mean OOMKilled, not graceful degradation."
  • "vLLM's PagedAttention is OS paging applied to the KV cache: fixed-size blocks, a per-request block table, on-demand allocation, copy-on-write sharing."
