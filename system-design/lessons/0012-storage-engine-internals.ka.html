Storage Engine-ების შიდა სამყარო

B-tree vs LSM — ეს არის read vs write amplification (გამრავლების) გადაწყვეტილება: database-ის შიდა მოწყობის ის ერთი თემა, რომელსაც ინტერვიუები ნამდვილად აჯილდოებენ.

მისიასთან კავშირი. როცა Anthropic-ის ამოცანა რაღაცის შენახვას გთხოვს — usage-ივენთები 100k ჩაწერა/წმ-ზე, შეკვეთების ისტორია, inbox — „Postgres-ს გამოვიყენებ" ბრენდია და არა დიზაინი. დამატებითი კითხვა ყოველთვის ერთია: „რატომ ეს store და რა გატყდება ამ ჩაწერის ტემპზე?" ვინც ამბობს „ეს append-heavy დატვირთვაა, ამიტომ LSM engine read amplification-ს თმობს sequential ჩაწერის გამტარუნარიანობის სანაცვლოდ", ის შიდა მექანიზმებიდან მსჯელობს — ზუსტად ის უნარი, რომელსაც DDIA-ის მე-3 თავი ასწავლის და ეს გაკვეთილი კუმშავს.

1 · Log-ი თვითონ არის database

უმარტივესი database append-only ფაილია — და ჩაწერაში თითქმის დაუმარცხებელია, რადგან append (მიწერა ბოლოში) ერთადერთია, რასაც დისკი სრული სიჩქარით აკეთებს. Estimation Kit-ის რიცხვები: მბრუნავი დისკი ~10 ms იხდის ყოველ შემთხვევით seek-ზე (~100 random IOPS), sequential (თანმიმდევრული) კითხვისას კი 100–200 MB/s-ს აწვდის — რამდენიმე ათასჯერადი უფსკრული. SSD-ები უფსკრულს ამცირებენ, მაგრამ ინარჩუნებენ (ACM Queue, "The Pathologies of Big Data").

ამიტომ ყველა სერიოზული engine ორჯერ წერს: ჯერ write-ahead log-ში (WAL) — სუფთა append, დისკზე ჩასული კლიენტის დადასტურებამდე; მერე ნამდვილ სტრუქტურაში. Crash-ის შემდეგ recovery log-ს თავიდან უკრავს ბოლო checkpoint-იდან — log-ი სიმართლეა, სტრუქტურა მისი cache-ია (Postgres WAL docs; DDIA ch. 3). ორივე ოჯახი ქვემოთ ერთი კითხვის სხვადასხვა პასუხია: „log-ი სწრაფია, მაგრამ მოუძებნელი — როგორ დავამატოთ ძებნა?"

2 · Index-ები: იხდი ჩაწერისას, რომ დაზოგო კითხვისას

შიშველი log-ი O(1) ჩაწერას და O(N) კითხვას იძლევა. Index დამატებითი სტრუქტურაა, რომელიც ზოგ კითხვას აჩქარებს — მაგრამ DDIA-ის კანონი ურყევია: ყოველი index ჩაწერას ანელებს, რადგან ყოველმა ჩაწერამ ისიც უნდა განაახლოს. უფასო index არ არსებობს — არსებობს მხოლოდ არჩეული გაცვლა. B-tree-ები ანგარიშს ჩაწერისას იხდიან, ადგილზე; LSM-ები ანგარიშს გადაავადებენ და background-ში, ბითუმად იხდიან.

3 · B-tree-ები: კითხვაზე ოპტიმიზებული წესრიგი

B-tree — Postgres-ის, MySQL/InnoDB-ის, Oracle-ის ძრავა — database-ს ფიქსირებული ზომის page-ებად (გვერდებად, ~4 KB) ჭრის და ხედ აწყობს: შიდა page რამდენიმე ასეულ დალაგებულ key-ს ინახავს შვილებზე მიმთითებლებით (Postgres B-tree internals).

სიღრმის ნამუშევარი მაგალითი: branching factor 500-ით სიღრმე log500(N)-ით იზრდება — ოთხდონიანი ხე 4 KB page-ებით ~5004 ≈ 62 მილიარდ leaf-მითითებას ≈ 256 TB მონაცემს წვდება. ოთხი page-ის წაკითხვა ნებისმიერ key-მდე მეოთხედ პეტაბაიტში — ამიტომაა B-tree-ის read latency ასეთი ბრტყელი და პროგნოზირებადი.

ჩაწერები ადგილზე (in-place) ხდება: იპოვე leaf, გადააწერე. სავსე page split-დება (იყოფა) ორად — და რადგან crash-ი მრავალ-page-იან ნაბიჯებს შორის ხეს დააზიანებდა, B-tree-ებს საკუთარი WAL აქვთ და პარალელურ წვდომას latch-ებით (მოკლევადიანი page-ლოქებით) მართავენ. სად იგებენ: point read-ები, range scan-ები (leaf-ები key-ის რიგითაა — „მომხმარებელ 4711-ის შეკვეთები, ბოლო 50" მეზობელ page-ებს მიუყვება), read-heavy ნარევები, მომწიფებული ტრანზაქციები. E-commerce შეკვეთების ისტორია არქეტიპია.

4 · LSM-tree-ები: ჩაწერაზე ოპტიმიზებული წესრიგი

Log-structured merge-tree (O'Neil et al., 1996) შემთხვევით ჩაწერებზე საერთოდ ამბობს უარს. გზა: ჩაწერები მიდის მეხსიერების დალაგებულ memtable-ში (პლუს WAL); ზღვარზე ის იცლება (flush) — ერთი sequential ჩაწერით — უცვლელ დალაგებულ ფაილად, SSTable-ად; background compaction (შემჭიდროება) SSTable-ებს უფრო მსხვილებად merge-ავს და გადაწერილ მნიშვნელობებს ყრის.

კითხვა ამის ფასს იხდის: lookup ამოწმებს memtable-ს, მერე SSTable-ებს ახლიდან ძველისკენ. Point read-ებს Bloom filter-ები იხსნიან — პაწია ალბათური სტრუქტურა თითო table-ზე, რომელიც „აქ ნამდვილად არ არის"-ს ერთი probe-ით პასუხობს (RocksDB wiki: Bloom Filter). წარმომავლობა: 1996-ის ნაშრომი → Google-ის Bigtable (OSDI '06) → HBase, Cassandra, LevelDB, RocksDB. სად იგებენ: შეუჩერებელი write-heavy ingest. Messaging inbox არქეტიპია — ყოველი მესიჯი ჩაწერაა და Cassandra-ს LSM ძრავა ისრუტავს იმ ნაკადს, რომელიც B-tree-ის random-I/O ჩაწერის გზას გაანადგურებდა.

5 · გამრავლების (amplification) სამკუთხედი

ძრავები სამ გამრავლებაზე დარდება — და RUM-ჰიპოთეზა ამბობს: Read-იდან, Update-იდან და Memory/სივრციდან ნებისმიერ ორს მესამის ხარჯზე თუ გააუმჯობესებ. Read amplification — რამდენ ნაჭერს კითხულობ ერთ ლოგიკურ read-ზე; write amplification — რამდენი ბაიტი იწერება ფიზიკურად ერთ ლოგიკურ ბაიტზე; space amplification — რამდენი ბაიტი ინახება ერთ ცოცხალ ბაიტზე.

არითმეტიკა, რომლის მოსმენაც ინტერვიუერს უნდა: B-tree 100-ბაიტიანი სტრიქონის განახლებაზე მთელ 4 KB page-ს გადაწერს — 40× write amplification WAL-ის ასლამდე. LSM leveled compaction-ით (fanout 10) ყოველ ჩანაწერს ~10-ჯერ გადაწერს თითო დონეზე; 3–4 დონე = ~30–40× — მაგრამ ყველა ეს ბაიტი მსხვილ sequential merge-ებში მოძრაობს, ამიტომ ingest-ის რბოლას LSM მაინც იგებს (RocksDB wiki: Compaction). ლაბის size-tiered მინი-LSM-ში 100k ჩაწერა 292k ჩანაწერ-ჩაწერა დაჯდა — write amp 2.92×.

7 · Object vs block vs file storage

მოდელიერთეული და APIნაწილობრივი განახლება?კლასიკური სახლი
Blockფიქსირებული ბლოკები; offset-ით წერა/კითხვა (EBS, SAN)კი — ეს არის მთელი აზრიdatabase-ებისა და ფაილსისტემების ქვეშ
Fileიერარქიული გზები, POSIX (NFS, EFS)კი — in-place დიაპაზონებისაზიარო დირექტორიები, legacy პროგრამები
Objectბრტყელი keyspace, უცვლელი blob-ები + metadata HTTP-ზე (S3, GCS)არა — მთელი ობიექტი იცვლებამედია, backup-ები, data lake-ები, SSTable-ები

S3-ის მოდელი ერთ ამოსუნთქვაზე: ობიექტები მთლიანად იწერება და მთლიანად იცვლება — 1 GB ობიექტს 200 ბაიტის მიწერა ახალი ობიექტის ატვირთვას ნიშნავს; read-ები და LIST-ები 2020 წლის დეკემბრიდან strongly consistent-ია (AWS S3 consistency); durability (მდგრადობა) კი თერთმეტი ცხრიანია — 99.999999999%: შეინახე 10 მილიონი ობიექტი და ერთის დაკარგვას ~10,000 წელიწადში ერთხელ ელოდე (AWS S3 durability docs). Object storage ფაილსისტემას ცვლის ზუსტად მაშინ, როცა მონაცემი write-once, read-many და დიდია — ამიტომ თანამედროვე LSM ძრავები უცვლელ SSTable-ებს პირდაპირ S3-ში აწყობენ.

8 · Failure mode-ები, რომლებიც ინტერვიუებს წყვეტენ

  • Compaction-ის ვალის სპირალი. ხანგრძლივი ჩაწერის ტალღა compaction-ს უსწრებს; SSTable-ები გროვდება; ყოველი read მეტ table-ს ეკითხება, latency იზრდება, სანამ compaction იმავე დისკზე იბრძვის; ბოლოს engine write stall-ებს აწესებს — შენი „ჩაწერაზე ოპტიმიზებული" store ჩაწერებს ზღუდავს (RocksDB: Write Stalls). შველა: compaction-ის მარაგი (ingest ≤ ~½ merge-ის გამტარობის), ingest-ის შეზღუდვა ზემოდან, pending-compaction bytes როგორც პირველი რიგის alert.
  • უსაზღვრო tombstone-ები. LSM-ში delete ჩაწერაა — tombstone (საფლავის ქვა) მარკერი, რომელიც compaction-მდე უნდა იცოცხლოს. Delete-heavy დატვირთვები (რიგები Cassandra-ში, გასუფთავებული inbox-ები) tombstone-ების მინდვრებს აშენებენ, რომლებშიც read ცოცხალ მონაცემამდე უნდა გაიკვლიოს — Cassandra 100k tombstone-ზე query-ს default-ად კლავს (The Last Pickle: About Deletes and Tombstones; Discord-მა ზუსტად ეს იწვნია მასიურად წაშლილი არხების კითხვისას — How Discord Stores Billions of Messages). გამოსავალი მონაცემთა მოდელშია: delete-ებზე რიგებს ნუ ააშენებ; დროით დაანაწილე და მთელი partition-ები გადაყარე.
  • შემთხვევითი key-ები B-tree-ის page-ების წინააღმდეგ. Random UUIDv4 primary key-ებით ყოველი insert შემთხვევით leaf page-ზე ჯდება: buffer pool-ი იფლითება, page-ები გამუდმებით split-დებიან და ნახევრად ცარიელები რჩებიან, write amplification ფეთქდება — Percona-მ გაზომა InnoDB-ის ჩავარდნა, როგორც კი index RAM-ს გადააჭარბებს. თანმიმდევრული key-ები (auto-increment, ULID, UUIDv7) insert-ებს მარჯვენა კიდის leaf-ზე ინახავენ — ამიტომ იყენებს Shopify-ის payments გუნდი ULID-ებს idempotency key-ებად (Shopify: Building Resilient Payment Systems). ლაბში: sequential→random გადართვა მინი-B-tree-ის write amp-ს ~2×-დან ~44×-მდე ზრდის.

9 · ერთი AI შენიშვნა, მერე ისევ საფუძვლები

ჩვენი LLM inference პლატფორმის KV-cache მენეჯერი თვითონ არის storage engine GPU-ს კოსტიუმში: ტოკენების გენერაცია append-heavy დატვირთვაა; PagedAttention (vLLM) cache-ს ფიქსირებული ზომის ბლოკებად ანაწილებს მიმართვის ცხრილით — §3-ის page-და-მიმთითებლის ხრიკი, attention-მდგომარეობაზე გადატანილი; დასრულებული sequence-ების eviction (განდევნა) და defragmentation compaction-ის ბიძაშვილებია; prefix-გაზიარება კი — deduplication. იგივე სამკუთხედი, ძვირი ბაიტები. ეს მთელი შენიშვნაა — ინტერვიუ §1–§8-ს აფასებს.

10 · ინტერვიუს სალაპარაკო პუნქტები

  1. ჯერ დატვირთვა დაახარისხე, მერე store დაასახელე: read/write თანაფარდობა, point vs range, delete-ების ტემპი, ტრანზაქციები.
  2. დისკზე დაამაგრე: „sequential I/O ათასობითჯერ იაფია random-ზე — B-tree random ჩაწერებს იხდის იაფი კითხვისთვის, LSM გადავადებულ merge-ებს sequential ჩაწერისთვის."
  3. სიღრმის მათემატიკა თქვი (§3): „branching 500, ოთხი დონე, ~256 TB", და write-amp-ისა (§5): „4 KB page 100-ბაიტიან სტრიქონზე — 40×."
  4. სამივე amplification დაასახელე და თქვი, რომელს სწირავ — ერთი წინადადება, რომელიც RUM-ის ცოდნას ასხივებს.
  5. Failure mode ინტერვიუერზე ადრე ახსენე (§8): compaction-ის ვალი LSM-არჩევანზე, random-key split-ები B-tree-ზე, tombstone-ები delete-heavy დიზაინზე.
  6. უცვლელი blob-ები object storage-ში გაუშვი (§7) და შეზღუდვა ხმამაღლა თქვი: „ნაწილობრივი განახლება არ არსებობს."
  7. ოპერაციული სიგნალით დახურე: „alert-ს დავაყენებდი pending compaction bytes-ზე / buffer-pool miss rate-ზე."

11 · ლაბი — შეაჯიბრე LSM და B-tree

პრაქტიკა (ქვიზამდე გააკეთე) ორივე engine ამ გაკვეთილიდან ნულიდან აწყობილია labs/lab-0012-lsm-vs-btree/-ში: python3 lsm_btree_lab.py უშვებს სამ ექსპერიმენტს — ჩაწერის ღირებულება და write amplification 100k insert-ზე (size-tiered compaction vs buffer pool-ის page-ების flush-ები), point read-ები ცხელ / ცივ / არარსებულ key-ებზე Bloom filter-ებით და მათ გარეშე, და range scan-ები, რომლებიც LSM-ს ყველა table-ის merge-ს აიძულებენ. შემდეგ README-ს E1–E4: იწინასწარმეტყველე, შეცვალე CONFIG, გაუშვი თავიდან, შეადარე — გაანახევრე memtable, გამორთე bloom-ები, გაარანდომე key-ები, გათიშე compaction. მცდარი პროგნოზები დავალების შედეგია.

12 · პროექტის ნაბიჯი

ორი გადაწყვეტილება LLM inference პლატფორმისთვის: აირჩიე storage engine (ა) request/usage log-ისთვის (ყოველი მოთხოვნა ემატება, ბილინგდება, იშვიათად იკითხება ხელახლა) და (ბ) metadata store-ისთვის (მომხმარებლები, API key-ები, კვოტები), თითოეული ორი წინადადებით დაიცავი. დასაცავი მაგალითი: (ა) LSM ძრავა (RocksDB-ზე დაფუძნებული, ან Kafka-მერე-object-storage), რადგან log-ი სუფთა append-heavy ingest-ია, რომელიც ადგილზე არასდროს ახლდება — read amplification-ს ვთმობთ, რომელსაც იშვიათად ვიხდით; (ბ) Postgres, რადგან metadata პატარაა, read-mostly და ტრანზაქციები და constraint-ები სჭირდება — B-tree-ის ტერიტორია. დაწერე შენი წყვილი — ჯერ დატვირთვის ფორმა, მერე engine — და მოიტანე შემდეგ სესიაზე.
