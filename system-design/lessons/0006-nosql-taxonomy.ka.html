NoSQL ტაქსონომია და Consistent Hashing

მონაცემთა საცავების ოთხი ოჯახი, ერთი გადაწყვეტილების ცხრილი და ჰეშირების ხრიკი, რომლის გარეშეც განაწილებული KV საცავები საერთოდ ვერ იარსებებდა.

მისიასთან კავშირი. Anthropic-ის ინტერვიუების ცნობილი გასახურებელი ამოცანაა: „დააპროექტე განაწილებული key-value საცავი." ამ პასუხის პირველივე გზაგასაყარია: საცავის რომელი ოჯახია ეს და როგორ პოულობს გასაღები (key) თავის კვანძს (node)? თუ იტყვი hash(key) % N, შემდეგი კითხვა — „ახლა დაამატე კვანძი" — ჩაგძირავს. ეს გაკვეთილი გაძლევს ტაქსონომიას და რგოლს (ring). (Exponent-ის 2026 Anthropic გზამკვლევი)

1 · ACID vs BASE ერთ აბზაცში

რელაციური ბაზები გვთავაზობენ ACID-ს: ტრანზაქცია არის Atomic (ან სრულად, ან საერთოდ არა), Consistent (ინვარიანტები დაცულია), Isolated (პარალელური ტრანზაქციები ერთმანეთის ნახევრად შესრულებულ სამუშაოს ვერ ხედავენ), Durable (დაკომიტებული ნიშნავს გადარჩენილს). NoSQL სისტემები ამას, როგორც წესი, ამსუბუქებენ BASE-მდე: Basically Available (პასუხობს ნაწილობრივი ავარიის დროსაც), Soft state (რეპლიკები შეიძლება მცირე ხნით არ თანხმდებოდნენ), Eventually consistent (თუ ჩაწერები შეწყდება, რეპლიკები ერთ მნიშვნელობაზე თანხმდებიან). BASE არ ნიშნავს „გარანტიების არარსებობას" — ეს სხვა კონტრაქტია: თმობ read-your-writes სიზუსტეს ყველა რეპლიკაზე და სანაცვლოდ იღებ availability-ს და ჰორიზონტალურ მასშტაბს. საინტერვიუო უნარი ისაა, დაასახელო რომელი ანომალიების ატანა შეუძლია შენს აპლიკაციას, და არა აკრონიმების ზეპირად თქმა. (Primer §12: NoSQL)

4 · Consistent hashing: შესაძლებლობის მომცემი ხრიკი

როგორც კი KV საცავს დაშარდავ, ყველა გასაღებს მფლობელი სჭირდება. გულუბრყვილო წესი node = hash(key) % N მუშაობს, სანამ N არ შეიცვლება. 5 კვანძიდან 6-ზე გადადი და გასაღები ადგილზე რჩება მხოლოდ თუ hash % 5 == hash % 6 — რაც დაახლოებით 6-დან 1 გასაღებისთვისაა მართალი. ყველა გასაღების ~83% იცვლის მფლობელს ერთი მანქანის დამატებით: ქეშები ყველგან ცივდება, მათ უკან მდგარ ბაზას სტამპედი (stampede) ატყდება, და rebalance თითქმის მთელ მონაცემთა ნაკრებს გადაათრევს. (DDIA თავი 6, partitioning)

რგოლი (ring). დააჰეშე კვანძებიც და გასაღებებიც ერთსა და იმავე წრიულ სივრცეში (0 … 264). გასაღებს ფლობს მისგან საათის ისრის მიმართულებით პირველი კვანძი. ახლა კვანძის დამატება მხოლოდ ერთ რკალს (arc) იპარავს — გასაღებების დაახლოებით 1/(N+1) (~17% 5→6 გადასვლისას) გადაადგილდება, ისიც მხოლოდ ახალ კვანძზე. კვანძის მოხსნა მხოლოდ მისი რკალის გასაღებებს გადასცემს მის მემკვიდრეს.

ვირტუალური კვანძები (virtual nodes). ერთი წერტილი თითო ფიზიკურ კვანძზე იძლევა უსწორმასწორო, ჰეშის-იღბალზე დამოკიდებულ რკალებს (ერთმა კვანძმა შეიძლება თავისი წილის 3-ჯერ მეტი დაიკავოს). ამიტომ თითო ფიზიკური კვანძი რგოლზე 100–500 წერტილს („vnode"-ს) იღებს; დიდი რიცხვების კანონი რკალებს ასწორებს, ხოლო არაერთგვაროვან მანქანებს შეუძლიათ თავიანთი სიმძლავრის პროპორციული vnode-ების აღება. ვინ დგას ამაზე: Amazon-ის Dynamo (და DynamoDB-ის წარმომავლობა), Cassandra-ს token ring და Memcached-ის კლიენტის ბიბლიოთეკები (Ketama) — მარშრუტიზაცია კლიენტში ხდება, კოორდინატორი არ სჭირდება.

ერთი პატიოსანი დათქმა, რომელსაც ლაბი თვალსაჩინოს გახდის: consistent hashing აბალანსებს გასაღებების რაოდენობას და არა დატვირთვას. ერთი zipf-ცხელი გასაღები მაინც ადნობს ერთ კვანძს — გამოსავალი რეპლიკაცია ან წინ დაყენებული ქეშია, და არა მეტი vnode.

5 · ლაბი — გატეხე mod-N შენი ხელით

პრაქტიკა (ქვიზამდე გააკეთე) labs/lab-0006-consistent-hashing/-ში: python3 hash_ring_lab.py 100k გასაღებს ანაწილებს mod-N-ითაც და ნამდვილი vnode რგოლითაც, შემდეგ ამატებს მე-6 კვანძს — უყურე, როგორ გადაადგილდება გასაღებების ~83% vs ~17%. ის ასევე ხსნის კვანძს, დაატარებს vnode-ებს ∈ {1, 10, 100, 500} დატვირთვის დაგლუვების საჩვენებლად და ამთავრებს zipf ცხელი-გასაღების ექსპერიმენტით, რომელიც ზემოთა დათქმას ამტკიცებს. შემდეგ README-ს E1–E4 სავარჯიშოები: იწინასწარმეტყველე, შეცვალე კონფიგი, გაუშვი თავიდან, შეადარე.

6 · პროექტის ნაბიჯი

ისევ ხერხემალი-პროექტთან, LLM inference gateway-სთან. დავუშვათ, დღეს ის ყველაფერს ერთ Postgres-ში ინახავს: user ანგარიშები და billing, API-key ავთენტიფიკაციის ძიებები, rate-limit მთვლელები, თითო-მოთხოვნის usage ლოგები (tokens in/out, latency, model — მილიონობით მწკრივი დღეში). ჩანაწერში გადაწყვიტე: რომელ საცავს გადაიტანდი SQL-იდან პირველად და რომელ ოჯახში? დასაცავი პასუხი: rate-limit მთვლელები და API-key ძიებები → Redis (key-value, იკითხება ყოველ ერთეულ მოთხოვნაზე, მილიწამზე ნაკლები ბიუჯეტით); usage ლოგები → wide-column (append-only time series, ჩაწერისას join-ები არ სჭირდება); ანგარიშები და billing → რჩება Postgres-ში (მრავალმწკრივიანი ACID, ფული). თითო გადატანაზე წნეხი დაასახელე. მოიტანე ჩანაწერი შემდეგ სესიაზე გასარჩევად.
