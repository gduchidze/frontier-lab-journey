გაკვეთილი 0013 · ფაზა 2 — პროგრამული არქიტექტურა და საიმედოობა · English version

ნაკადები და მოვლენები

Log-ი როგორც system-of-record — Kafka-ს მოდელი, event-driven არქიტექტურა, event sourcing/CQRS, და რას ნიშნავს სინამდვილეში "exactly-once", როცა გადახდის email ზუსტად ერთხელ უნდა გავიდეს.

მისიასთან კავშირი. ყველა მასშტაბურ დიზაინის ამოცანაში ადრე თუ გვიან ჩნდება ისარი წარწერით "events" — „როგორ იგებენ downstream სერვისები, რომ შეკვეთა გაიგზავნა?" — და დამატებითი კითხვები ყოველთვის ერთი და იგივეა: „რა ordering-ს (მიმდევრობას) იღებ სინამდვილეში?" და „რა ხდება, როცა consumer-ი batch-ის შუაში კვდება?" ვინც ამბობს „Kafka-ზე დავდოთ", ჩერდება; ვინც ასახელებს partition key-ს, delivery semantics-ს და idempotency-ის გეგმას, აგრძელებს. ხერხემალი: DDIA თავი 11 და Jay Kreps-ის "The Log".

1 · Message queue vs event log

კლასიკური broker-ები (RabbitMQ, SQS) შეტყობინებას დავალებად თვლიან: ერთი consumer-ი იღებს, ack-ს აბრუნებს, broker-ი შლის (RabbitMQ: acknowledgements). Kafka შეტყობინებას ფაქტად თვლის: უცვლელი ჩანაწერი შენახულ log-ში; consumer-ები უბრალოდ იმახსოვრებენ, სად მიაღწიეს (offset), და კითხვა არაფერს ანადგურებს (Kafka design docs).

ეს ერთი განსხვავება — delete-on-ack vs retain-and-track — ყველაფერს ცვლის: შეკვეთების ნაკადს მეორე consumer-ს (ანალიტიკა) აბამ პირველის (fulfillment) ხელშეუხებლად; გასულ კვირას გამოსწორებულ bug-ში replay (თავიდან გატარება) უკეთებ; ახალ სერვისს ისტორიიდან ტვირთავ. Kreps-ის აზრი: LinkedIn-ის ათობით point-to-point მილსადენი (activity events → warehouse, → search, → recommendations) ფარულად ერთი და იგივე იყო — log shipping — და ერთ გამძლე, მოწესრიგებულ, replay-ად log-ს შეეძლო O(N²) ინტეგრაციების ჩანაცვლება. წესი ინტერვიუსთვის: queue, როცა სამუშაო უნდა მოხდეს ერთხელ, მალე, ვინც თავისუფალია; log, როცა ფაქტს მრავალი აუდიტორია ჰყავს დროში.

2 · Kafka-ს მოდელი, ზუსტად

ცნებარა არისრას გაძლევს
Topicმოვლენების დასახელებული კატეგორია (orders, clicks)სახელთა სივრცე; per-topic retention და სქემა
Partitionappend-only, სრულად მოწესრიგებული log; topic არის P ცალი მათგანიordering-ისა და პარალელიზმის ერთეული
Offsetconsumer-ის პოზიცია ერთ partition-შიიაფი გაგრძელება, გადახვევა, replay
Consumer groupconsumer-ების ჯგუფი; თითო partition-ს ზუსტად ერთი წევრი ფლობსჰორიზონტალური სკალირება per-partition ordering-ის შენარჩუნებით

Partition-ში მოვლენები სრულად მოწესრიგებულია; partition-ებს შორის — საერთოდ არა. Consumer-ი ან კვდება თუ ემატება — ჯგუფი rebalance-ს აკეთებს (partition-ების გადანაწილება). აქედან: ჯგუფში სასარგებლო consumer-ების მაქსიმუმი = partition-ების რაოდენობა. მე-9 consumer-ი 8-partition-იან topic-ზე არაფერს ფლობს და არაფერს აკეთებს (ლაბი ზუსტად ამას აჩვენებს).

ნამუშევარი მაგალითი — არითმეტიკა, რომლის მოსმენაც ინტერვიუერს უნდა. Checkout clickstream: 200,000 event/s × ~1 KB = 200 MB/s. Partition-ი ~10 MB/s ჩაწერას უძლებს → broker-ის მხარეს ≥ 20 partition-ი სჭირდება. მაგრამ consumer-ი event-ს ამდიდრებს და store-ში ~2,500 event/s სიჩქარით წერს → 200,000 / 2,500 = 80 consumer-ი → და რაკი consumers ≤ partitions, საჭიროა ≥ 80 partition-ი. Partition-ების რაოდენობას, ჩვეულებრივ, consumer-ის მხარე კარნახობს და არა broker-ის. აიღე ~96 მარაგით და ხმამაღლა თქვი: მოგვიანებით repartitioning key-ების განლაგებას არევს — ამიტომ ახლავე იღებ მეტს.

3 · Partition key წყვეტს, რას ნიშნავს „მოწესრიგებული"

Producer-ი partition-ს key-ის hash-ით ირჩევს: ერთი key-ის ყველა event ერთ partition-ში ხვდება, გაგზავნის თანმიმდევრობით. ანუ Kafka-ს რეალური გარანტია per key ordering-ია, და key-ის არჩევა ordering-კონტრაქტის არჩევაა:

  • order_id: ერთი შეკვეთის ყველა მოვლენა (created → paid → shipped) მოწესრიგებულია — ზუსტად ის, რაც შეკვეთების მილსადენს სჭირდება.
  • user_id: ერთი მომხმარებლის ქმედებები მოწესრიგებულია (fraud-წესებისთვის საჭირო) — მაგრამ ბოტი ან ცნობილი ანგარიში hot key ხდება და მისი partition-ი ჯგუფის ყველაზე ნელი ზოლი. იგივე celebrity პრობლემა, რაც sharding-სა და cache-ში — მესამედ ამ კურსში.
  • შემთხვევითი / უkey-ო: იდეალურად თანაბარი partition-ები, მაქსიმალური throughput, ნულოვანი ordering. კარგია დამოუკიდებელი ფაქტებისთვის; კატასტროფაა ყველაფრისთვის, რასაც state აქვს.

ლაბში zipf(1.1) key-ებზე ყველაზე ცხელი key ტრაფიკის 15.9%-ს ატარებს და ერთი partition-ი მოვლენების 23%-ზეა (1.87× სამართლიანი წილისა) — და მარტო ეს partition-ი ზღუდავს მთელი ჯგუფის დასრულების დროს.

4 · Delivery semantics: საიდან მოდიან დუბლიკატები

გარანტიამექანიზმიჩავარდნის ფასიროდის
At-most-onceoffset-ის commit დამუშავებამდეCrash batch-ის შუაში → მოვლენები ჩუმად იკარგებამეტრიკები, ტელემეტრია
At-least-onceoffset-ის commit დამუშავების შემდეგCrash commit-მდე → batch-ი თავიდან მოდის, ორჯერ მუშავდებანაგულისხმევი + idempotent consumer
Exactly-onceidempotent producer + ტრანზაქციები (Kafka-ს შიგნით)სირთულე; არ ფარავს გარე ეფექტებსKafka Streams მილსადენები

დუბლიკატები ორი კარიდან შემოდიან. Producer: გაგზავნა timeout-დება, producer-ი იმეორებს, მაგრამ პირველი ცდა რეალურად ჩაჯდა — log-ში event ორჯერაა. Kafka-ს idempotent producer ამ კარს sequence-ნომრებით კეტავს (Kafka docs §4.6). Consumer: consumer-ი ამუშავებს, კვდება offset-ის commit-მდე, rebalance — და ახალი მფლობელი ბოლო commit-იდან კითხულობს: ყველაფერი მას შემდეგ თავიდან მოდის. ლაბში ეს სცენარი ~148 დუბლიკატს იძლევა commit-after-batch-ით, ან ~258 დაკარგულ event-ს commit-before-თი.

„Exactly-once"-ის გულახდილი ვერსია: Kafka-ს ტრანზაქციები ატომურს ხდის read-process-write-ს Kafka-ს შიგნით (Confluent: exactly-once semantics). ისინი ვერ ფარავენ email-ს, Postgres-ის სტრიქონს, ბარათის ჩამოჭრას. გარე ეფექტებისთვის რეალური მექანიზმია at-least-once + idempotent დამუშავება: უნიკალური event ID და consumer-ის ეფექტი, რომელიც მეორე მიწოდებაზე no-op-ია (INSERT … ON CONFLICT DO NOTHING event ID-ზე). Exactly-once მიწოდების გარანტია კი არა, შენ მიერ აგებული end-to-end თვისებაა.

5 · Retention და compaction: მეხსიერების ორი ფორმა

დროითი/ზომითი retention ყველაფერს N დღე ინახავს და ძველიდან შლის — log-ი როგორც შემოსაზღვრული ისტორია. Log compaction (შემჭიდროება) სამუდამოდ ინახავს თითო key-ის უახლეს ჩანაწერს და გადაწერილ წინამორბედებს შლის (Kafka docs: compaction). შემჭიდროებული topic-ი changelog-ია: replay და მიმდინარე state ზუსტად აღდგება — ლაბი იდენტურ state-ს 1,766 ჩანაწერიდან აშენებს 30,000-ის ნაცვლად, 17× ნაკლები replay.

ესაა CDC-ის (change data capture) საფუძველი: Debezium database-ის replication log-ს ადევნებს თვალს და ყოველ ცვლილებას topic-ზე აქვეყნებს — DB ნაკადი ხდება, cache-ები და ინდექსები კი replay-თი აღდგენადი consumer-ები. ერთი წინასწარი შენიშვნა: DB-ში ჩაწერა და event-ის გამოქვეყნება ორ ცალკე მოქმედებად შეიძლება ნახევრად ჩავარდეს; transactional outbox ამას იმავე ტრანზაქციაში ჩაწერით და CDC-თი გამოქვეყნებით ასწორებს. სრული განხილვა — შემდეგ გაკვეთილში.

6 · Event-driven არქიტექტურა, event sourcing და CQRS

ორი სტილი (Fowler: Event-Driven): choreography — order-service აქვეყნებს OrderPlaced-ს; inventory, payment, email თითოეული თავად რეაგირებს; კოორდინატორი არ არსებობს — და orchestration — კოორდინატორი ყოველ ნაბიჯს პირდაპირ მართავს. Choreography ავტონომიასა და ჩავარდნების იზოლაციას იძლევა; ფასი ისაა, რომ შეკვეთის მიმდინარე state-ზე ვერავინ მიუთითებს — სიმართლე ხუთი consumer-ის offset-ებზეა გადაფენილი და debugging არქეოლოგია ხდება. ორივე შემთხვევაში event-ები კონტრაქტებია: სქემის registry (schema registry), მხოლოდ backward-compatible ცვლილებები — event-სქემა საჯარო API-ა, რომლის გამომწერებსაც ვერ ჩამოთვლი.

Event sourcing log-ს system-of-record-ად აქცევს: state-ის ნაცვლად („ბალანსი = $90") ინახავ მოვლენათა მიმდევრობას („გახსნა, შეიტანა 100, გაიტანა 10") და state-ს მათზე fold-ით იღებ (Fowler: Event Sourcing). გადახდების ledger-ი კანონიკური მაგალითია — ბუღალტრები append-only ჟურნალებს ხუთი საუკუნეა იყენებენ. CQRS: ჩაწერები event-ებს ამატებენ; კითხვები projection-ებს (მატერიალიზებულ view-ებს) ხვდებიან, რომლებიც ნაკადის fold-ით შენდება და replay-თი თავიდან აშენებადია (Fowler: CQRS). როდისაა ზედმეტი? ჩვეულებრივ CRUD-ს არ სჭირდება: იხდი eventual consistency-ს, უცვლელი ისტორიის სქემა-ვერსიონირებას და replay-ის დროებს. გამოიყენე იქ, სადაც ისტორია თავად ღირებულებაა — ledger-ები, შეკვეთები, მარაგის მოძრაობა.

7 · Lag და failure mode-ები, რომლებიც ინტერვიუებს წყვეტენ

Consumer lag (ჩამორჩენა) — ბოლო დაწერილი offset მინუს ბოლო commit-ებული, per partition — event-მილსადენის ერთადერთი ჯანმრთელობის მეტრიკაა. Latency კარგად გამოიყურება, სანამ lag ჩუმად იზრდება; LinkedIn-მა Burrow სწორედ lag-ის ტრენდის შესაფასებლად ააგო. Alert-ი lag-ის ზრდაზე და დროში გადათარგმნილ lag-ზე („consumer-ი 40 წუთითაა ჩამორჩენილი").

  • Rebalance-ის შტორმები. ნელი consumer-ი heartbeat-ს აცდენს → ჯგუფიდან აგდებენ → rebalance მთელ ჯგუფს აჩერებს → სხვებიც ჩამორჩებიან → კიდევ აგდებენ. შველა: გონივრული max.poll.interval, cooperative rebalancing, static membership.
  • Lag-ის სასიკვდილო სპირალი, ციფრებით. სიმძლავრე 220k event/s, შემოდის 200k/s → მარაგი 20k/s. ერთსაათიანი წყვეტა 720M event-ს აგროვებს; 20k/s მარაგით დაწევას სჭირდება 720M / 20k = 36,000 წმ = 10 საათი. 24-საათიანი retention-ით კიდევ ერთი ინციდენტი — და log-ი წაუკითხავ event-ებს შლის. მარაგი ფუფუნება კი არა, აღდგენის დროის ბიუჯეტია.
  • სქემის მსხვრევა ნაკადის შუაში. Producer-მა ველს სახელი გადაარქვა — ყველა consumer-ი ყველა event-ზე exception-ს ისვრის: ფლოტური poison pill. ამიტომ თავსებადობის შემოწმება CI-შია, registry-ით.
  • დუბლიკატის fan-out. §4-ის redelivery არა-idempotent email-consumer-ს წვდება: მომხმარებელი გადახდის ორ დადასტურებას იღებს; არა-idempotent ledger-consumer-ი გადასახადს ორჯერ წერს. ერთი მიზეზი, სრულიად სხვადასხვა აფეთქების რადიუსი — idempotency-ის ძალისხმევა რადიუსს გაჰყევი.
  • Poison pill. ერთი დაუმუშავებელი event — და partition-ი (და მასში ყველა key) მის უკან სამუდამოდ იჭედება. გამოსავალი: N ცდის შემდეგ DLQ (dead-letter queue) და წინსვლა, იმ key-ის ordering-ის დროებით გატეხვის ფასად (Uber: reliable reprocessing).
  • Compaction-მა ისტორია შეჭამა. ვიღაცამ compaction ჩართო topic-ზე, რომლის სრული ისტორიაც ანალიტიკას მერე დასჭირდა — ყველა შუალედური state უბრუნებლად წაშლილია, დიზაინით. Compaction state-ს ინახავს და არა ისტორიას; topic-ზე გააზრებულად აირჩიე.
LLM შტრიხი Inference-პლატფორმის usage-მილსადენი ეს გაკვეთილია სიტყვასიტყვით: ყოველი მოთხოვნა usage event-ს აგდებს log-ზე (tenant, model, tokens, latency); billing-ი, eval-ი და abuse-detection დამოუკიდებელი consumer group-ებია ერთსა და იმავე topic-ზე; partitioning tenant_id-ით per-tenant ordering-ს იძლევა hot-tenant რისკით, რომელსაც უკვე სახელით იცნობ; და replay მთავარი იარაღია — უკეთესი ქულების მოდელი გამოუშვი, eval-consumer-ი გადაახვიე, გასული თვის ტრაფიკი თავიდან შეაფასე serving-ის ხელშეუხებლად. Billing = payments-ledger შემთხვევა: at-least-once + idempotency event ID-ზე, რადგან tenant-ის ორჯერ დარიცხვა email-twice ჩავარდნაა ინვოისით.

8 · ლაბი — გაუშვი partitioned log შენი ხელით

პრაქტიკა (ქვიზამდე გააკეთე) ყველაფერი ზემოთქმული ახლა დათვლადია: labs/lab-0013-mini-kafka/-ში python3 mini_kafka_lab.py უშვებს სამ ექსპერიმენტს — 50k zipf-key-იანი event 8 partition-ზე (ნული per-key ordering-დარღვევა, ~24k გლობალური ინვერსია, ერთი partition-ი სამართლიანი წილის 1.87×-ზე), consumer-ჯგუფის sweep, სადაც აჩქარება partition-ების რაოდენობაზე მკვდრად ჩერდება და შუა-რბენის crash ~148 დუბლიკატს (commit-after) ან ~258 დანაკარგს (commit-before) იძლევა, და 17×-ად შემჭიდროებული changelog, რომელიც იდენტურ state-ს აშენებს. შემდეგ README-ს E1–E4: იწინასწარმეტყველე, შეცვალე კონფიგი, გაუშვი თავიდან, შეადარე. მცდარი პროგნოზები არის დავალების შედეგი.

9 · პროექტის ნაბიჯი

დააპროექტე usage-metering event-ნაკადი LLM inference-პლატფორმისთვის: დაასახელე topic-ი, partition key და delivery semantics და ორი წინადადებით დაიცავი. დასაცავი მაგალითი: topic usage-events, key tenant_id (per-tenant ordering billing-ის სისწორისთვის, hot-tenant skew მიღებული და მონიტორინგზე აყვანილი), at-least-once მიწოდება billing-consumer-ით, რომელიც idempotent-ია უნიკალურ request_id-ზე — რადგან დაკარგული usage event შემოსავლის გაჟონვაა, დუბლირებული კი გაბრაზებული მომხმარებელი, და idempotency ერთადერთი გამოსავალია, რომელიც ორივეს უძლებს. დაწერე შენი ორი წინადადება — key + semantics + რატომ — და მოიტანე შემდეგ სესიაზე.
