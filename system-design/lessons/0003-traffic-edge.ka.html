ტრაფიკის კიდე: DNS, CDN, Load Balancing

ყოველი მოთხოვნა სამ კარიბჭეს გადის, სანამ შენი კოდი გაეშვება — იცოდე, რას ხედავს თითოეული და რას ვერა.

მისიასთან კავშირი. Anthropic-ის ამოცანა „დააპროექტე token-generation სერვისი, რომელიც უმკლავდება 100,000 მოთხოვნას წამში" იწყება ბევრად ადრე, ვიდრე რომელიმე GPU გამოჩნდება: 100k RPS ბევრ მანქანაზე უნდა დაჯდეს, რაც ნიშნავს, რომ load balancer (დატვირთვის გამანაწილებელი) არის პირველი უჯრა, რომელსაც კლიენტის შემდეგ ხატავ. ინტერვიუერები ზუსტად აქ გსინჯავენ — L4 თუ L7? რომელი ალგორითმი? რა ხდება, როცა backend კვდება? ეს გაკვეთილი დასაცავ პასუხებს გაძლევს. (Exponent-ის 2026 Anthropic გზამკვლევი)

1 · მოგზაურობა: DNS წყვეტს, სად არის „იქ"

სანამ შენი API-ის ერთი ბაიტიც გაეშვება, მოთხოვნა ამ გზას გადის:

client → DNS (name → IP) → CDN edge (სტატიკურია? აქვე პასუხობს, მორჩა)
                          → load balancer (VIP) → reverse proxy → app server

DNS თარგმნის api.example.com-ს IP მისამართად. ჩანაწერის (record) ტიპები, რომლებიც უნდა იცნო (Primer §7):

ჩანაწერიასახავსტიპური გამოყენება
Aსახელი → IPv4 მისამართიმთავარი მუშახელი
CNAMEსახელი → სხვა სახელიალიასი www-სთვის ან CDN hostname-ისთვის
NSზონა → name serverგანსაზღვრავს, ვინ პასუხობს დომენზე
MXდომენი → mail serverელფოსტის მარშრუტიზაცია

მართული DNS (Route 53, Cloudflare) მარშრუტიზაციასაც აკეთებს და არა მხოლოდ ამოხსნას: weighted round-robin (გაუგზავნე 5% canary-ს), latency-based (უპასუხე RTT-ით ყველაზე ახლო რეგიონით), geolocation (EU მომხმარებლები → EU დატაცენტრი, კანონის ან სიახლოვის გამო). ერთი გაფრთხილება ყველას ეხება: DNS პასუხები TTL-ით იქეშება, ამიტომ DNS-დონის failover წუთებით ნელია — არასდროს იყოს შენი დაცვის პირველი ხაზი.

2 · CDN: საერთოდ ნუ ამგზავრებ მოთხოვნას

გაკვეთილი 0001-ის ცხრილი ამბობდა: ოკეანის round trip ~150 ms ღირს. CDN ამ რიცხვს იმით ჯობნის, რომ ოკეანეს საერთოდ არ კვეთს: სტატიკური კონტენტი (JS, სურათები, ვიდეო) მომხმარებელთან ახლო edge სერვერიდან მიეწოდება. კონტენტი edge-ზე ორი გზით ხვდება (Primer §8):

  • Push CDN — ფაილებს რელიზისას შენ ტვირთავ. სრული კონტროლი, პირველი მოთხოვნის აცდენა (miss) არ არსებობს; ერგება მცირე, იშვიათად ცვალებად ნაკრებებს. იხდი ყველგან შენახვაში — იმ ფაილებზეც, რომლებიც იმ რეგიონში არავის სჭირდება.
  • Pull CDN — edge პირველ აცდენაზე origin-იდან იღებს კონტენტს და TTL-ით იქეშებს. ერგება კონტენტის გრძელ კუდებს; ფასი — ნელი პირველი მოთხოვნა და ვადის გასვლამდე მოძველებული (stale) კონტენტი.

რა ეკუთვნის CDN-ს: ყველაფერი სტატიკური და საზიარო. რა არა: პერსონალიზებული ან სწრაფად ცვალებადი პასუხები. და CDN უფასო არ არის — იხდი გატანილ (egress) GB-ებში, ხოლო ქეშის ინვალიდაცია (გასწორებული JS bundle-ის ახლავე მიწოდება) კლასიკური საოპერაციო თავის ტკივილია: ფაილების URL-ებს ვერსია დაურთე, purge-ს ნუ ენდობი.

3 · Load balancer: L4 vs L7 და ალგორითმები

Load balancer შემომავალ მოთხოვნებს იდენტურ app სერვერებზე ანაწილებს. პირველი გადაწყვეტილება — რომელ ფენაზე (layer) მუშაობს (Primer §9):

L4 (transport)L7 (application)
ხედავსმხოლოდ IP-ებს და პორტებსსრულ HTTP-ს: path, headers, cookies
ფასიიაფი, მილიონობით კავშირიTLS-ს ხსნის, HTTP-ს არჩევს — მეტი CPU
გაძლევსსუფთა სიჩქარეს/video და /api სხვადასხვა pool-ში

რომელი backend მიიღებს მოთხოვნას:

  • Round-robin — რიგში შემდეგი სერვერი. სამართლიანი რაოდენობით.
  • Weighted — დიდი მანქანები პროპორციულად მეტს იღებენ.
  • Least-connections — აირჩიე სერვერი ყველაზე ცოტა მიმდინარე (in-flight) მოთხოვნით. სამართლიანი მიმდინარე დატვირთვით.
  • IP-hash / sticky sessions — ერთი კლიენტი ყოველთვის ერთსა და იმავე სერვერს ხვდება, ამიტომ მეხსიერებაში ჩარჩენილი სესია ცოცხლობს.

Stickiness (მიწებება) ხაფანგია: სერვერებს „ფიფქებად" აქცევს — ერთი მოკვდება და მისი მომხმარებლების სესიებიც მასთან ერთად კვდება, ცხელი კლიენტები კი ცხელ სერვერებს ქმნიან. მოწიფული ნაბიჯი არის stateless სერვისი Redis-ში გატანილი სესიებით: ნებისმიერ სერვერს შეუძლია ნებისმიერი მოთხოვნის მომსახურება. ეს statelessness (პლუს shared-nothing — არავითარი ლოკალური მდგომარეობა, რომელიც მეზობელს დასჭირდებოდა) საერთოდაც ჰორიზონტალური მასშტაბირების წინაპირობაა: „კიდევ სერვერების დამატება" მხოლოდ მაშინ მუშაობს, თუ სერვერები ურთიერთჩანაცვლებადია.

4 · ჩავარდნა კიდეზე: health checks და LB წყვილი

Balancer-ი თავის ფასს მაშინ ამართლებს, როცა backend კვდება. ის ყოველ სერვერს სინჯავს (health checks: HTTP GET /health რამდენიმე წამში ერთხელ); N ზედიზედ ჩავარდნის შემდეგ backend როტაციიდან გადის, ერთი გავლილი შემოწმება კი უკან აბრუნებს. აღმოჩენის დრო ≈ interval × N — დაამოკლებ და failover აჩქარდება, მაგრამ ჯანმრთელი-მაგრამ-წამიერად-ნელი სერვერის გაგდებას რისკავ; გაზრდი და მომხმარებლები შეცდომებს უფრო დიდხანს ჭამენ. უფასო პარამეტრი არ არსებობს — მხოლოდ არჩეული გაცვლა.

მაგრამ ახლა თავად load balancer-ია single point of failure (ჩავარდნის ერთი წერტილი). სტანდარტული პასუხი: LB-ები წყვილში — active–passive heartbeat-ით და მოძრავი (floating) IP-ით, რომელსაც გადარჩენილი იჩემებს, ან active–active DNS-ის უკან. ინტერვიუზე წინადადება „და LB წყვილშია, ასე რომ SPOF აღარ არის" ერთი წინადადებაა, რომელიც აჩვენებს, რომ ჩავარდნაზე ავტომატურად ფიქრობ.

5 · რატომ ამტვრევს LLM serving-ი round-robin-ს

აქ არის AI შემობრუნება, რომელზეც ფაზა 4 აშენდება. ვებ-მოთხოვნები დაახლოებით ერთგვაროვანია; LLM მოთხოვნები — არა: ერთი 100 ტოკენს ითხოვს, მეორე 10,000-ს — 100× განსხვავება სამუშაოში, რომელიც იდენტურად გამომზირალ HTTP გამოძახებებს უკან იმალება. Round-robin მოთხოვნების რაოდენობას ათანაბრებს, ამიტომ ერთი უიღბლო GPU გრძელ გენერაციებს აგროვებს, მისი რიგი იზრდება და TTFT p99 ფეთქდება, სანამ მეზობლები უსაქმოდ დგანან. Inference gateway-ები ამიტომ ამარშრუტებენ least outstanding requests-ით (least-connections-ის ბიძაშვილი) ან serving engine-იდან წამოსული უფრო ღრმა queue-depth-aware სიგნალებით — მოგვიანებით კი prefix-cache-aware მარშრუტიზაციითაც. ამ ინტუიციას ქვემოთ, ლაბში, საკუთარი ხელით ააწყობ, ნამდვილს კი ფაზა 4-ში დააპროექტებ.

6 · ლაბი — გაუშვი შენი საკუთარი load balancer

პრაქტიკა (ქვიზამდე გააკეთე) labs/lab-0003-toy-load-balancer/-ში: python3 toy_lb.py --demo უშვებს სამ ნამდვილ HTTP backend-ს პლუს balancer-ს health check-ებით, აგზავნის 30 მოთხოვნას, ბეჭდავს განაწილებას backend-ების მიხედვით — შემდეგ კი შუა პროცესში კლავს ერთ backend-ს, რომ failover შენი თვალით ნახო. შემდეგ README-ს E1–E4 სავარჯიშოები: იწინასწარმეტყველე, შეცვალე კონფიგი, გაუშვი თავიდან, შეადარე. E1/E2 მე-5 სექციის round-robin-vs-least-connections ისტორიას შენივე რიცხვებით იმეორებს.

7 · პროექტის ნაბიჯი

ვუბრუნდებით ხერხემალ-პროექტს — LLM inference gateway-ს. გადაწყვიტე მისი load-balancing სტრატეგია — L4 თუ L7 და რომელი ალგორითმი — და დაწერე ერთი წინადადება, რომელიც არჩევანს მე-5 სექციის შერეული 100-ტოკენიანი/10k-ტოკენიანი დატვირთვის წინააღმდეგ დაიცავს. (დასაცავი მაგალითი: „L7 least-outstanding-requests-ით, რადგან ტოკენების რაოდენობა მოთხოვნების რაოდენობას აზრს უკარგავს და /v1/messages vs /health-სთვის path-ზე დაფუძნებული მარშრუტიზაცია გვჭირდება.") მოიტანე შენი წინადადება შემდეგ სესიაზე; მასწავლებელი მას შეუტევს.
