კონკურენტულობა: რბოლები, ლოკები, event loop-ები და GIL-ი

რატომ არის „ჩემს ლეპტოპზე მუშაობდა" ყველაზე ძვირი წინადადება მრავალნაკადიან კოდში.

მისიასთან კავშირი. Anthropic-ის ყველაზე ხშირად მოხსენიებული საინტერვიუო ამოცანა — inference batching სერვისი, სადაც სინქრონული გამომძახებლები ელოდებიან, სანამ scheduler-ი მათ მოთხოვნებს აჯგუფებს და პასუხებს სწორ მომლოდინეს უბრუნებს — ჯერ კონკურენტულობის (concurrency) ამოცანაა და მხოლოდ შემდეგ ML-ის: საზიარო რიგი, worker pool-ი, ლოკების დისციპლინა და event loop-ი ათასობით ღია კავშირით. ამ გაკვეთილის ყველა მექანიზმი იმ დიზაინში ჩნდება. (Exponent-ის Anthropic გზამკვლევი, Inside vLLM)
ჯერ ლაბი (ADHD წესი) სანამ კითხვას გააგრძელებ, გაუშვი python3 concurrency_lab.py საქაღალდეში labs/lab-p0-03-races-and-gil/. უყურე, როგორ კარგავს მთვლელი განახლებების 70%-ს, როგორ ვარდება deadlock-ი და როგორ ვერ აჩქარებენ thread-ები CPU-ს სამუშაოს. მერე წაიკითხე — ქვემოთ ყველა სექცია ხსნის იმას, რაც ახლახან ნახე.

1 · კონკურენტულობა არ არის პარალელიზმი

კონკურენტულობა (concurrency) სტრუქტურაა: ბევრი ამოცანა ერთდროულად მიმდინარეობს, გადაწნული იმდენ CPU-ზე, რამდენიც გაქვს — შესაძლოა ერთზეც. პარალელიზმი (parallelism) შესრულებაა: ამოცანები სიტყვასიტყვით ერთსა და იმავე მომენტში ეშვება რამდენიმე ბირთვზე. ერთნაკადიანი ვებ-სერვერი 10,000 ღია კავშირით უაღრესად კონკურენტულია და სულაც არა პარალელური. ეს განსხვავება წყვეტს, რომელ ინსტრუმენტს დასწვდები. (OSTEP: Concurrency and Threads)

Thread-ები (გაკვეთილი p0-02) ერთ მისამართთა სივრცეს იზიარებენ — საერთო heap-ი, ცალკე stack-ები. ეს გაზიარება მთელი ღირებულებაცაა და მთელი საფრთხეც: ნებისმიერ thread-ს შეუძლია ჩაწეროს მეხსიერებაში, რომელსაც სხვა კითხულობს.

2 · დაკარგული განახლება: რბოლის ანატომია

counter += 1 ერთი ოპერაცია არ არის. ეს არის წაკითხვა → შეცვლა → ჩაწერა, და scheduler-მა thread-ი შეიძლება ნებისმიერ ორ ნაბიჯს შორის შეაჩეროს:

Thread A: კითხულობს counter-ს (=100)
Thread B: კითხულობს counter-ს (=100)   <- A შეჩერდა; B იმავე მნიშვნელობას კითხულობს
Thread B: წერს 101-ს
Thread A: წერს 101-ს                    <- B-ს განახლება ჩუმად განადგურდა

ორი ინკრემენტი მოხდა. მთვლელი ერთით გაიზარდა. შეცდომა არ დაფიქსირებულა.

ეს არის რბოლის მდგომარეობა (race condition): შედეგი დამოკიდებულია გადაწვნის დროზე, რომელსაც შენ ვერ აკონტროლებ. ყველაზე საშიში თვისება სიჩუმეა — არც exception, არც crash, უბრალოდ მცდარი მონაცემები. ლაბში 4 thread-ი × 50,000 ინკრემენტი განახლებების 50–75%-ს კარგავს, ყოველ გაშვებაზე სხვადასხვა რაოდენობას. (OSTEP: Locks)

ლექსიკონი კოდის უბანი, რომელიც არ უნდა გადაიწნას, არის კრიტიკული სექცია (critical section). ატომურობა (atomicity) ნიშნავს, რომ სექცია სრულდება როგორც ერთი განუყოფელი ერთეული — ან მთლიანად, ან საერთოდ არა, დაკვირვებადი შუალედური მდგომარეობის გარეშე.

3 · ლოკები: სისწორე თავისი ფასით

Mutex-ი (ლოკი) კრიტიკულ სექციას ატომურს ხდის: აიღე შესვლამდე, გაუშვი გამოსვლისას; ერთ დროს მხოლოდ ერთი მფლობელი. ლაბში დალოკილი მთვლელი ზუსტია — და დაახლოებით 4-ჯერ ნელი, რადგან thread-ები ახლა სექციასთან რიგში დგანან. ამ რიგს სახელი უკვე იცი: კონტენცია (contention), და ის პირველად p99-ში ჩნდება.

სენიორული დონის ბერკეტი გრანულარობაა. ერთი მსხვილი ლოკი ყველაფერზე ადვილი გასააზრებელია, მაგრამ მთელ სამყაროს ასერიალებს (ეს სიტყვასიტყვით Python-ის GIL-ია — სექცია 7). წვრილმარცვლოვანი ლოკები უკეთ მასშტაბირდება, მაგრამ deadlock-ის გზებს ამრავლებს. რეალური სისტემები იყენებენ reader–writer ლოკებსაც (ბევრი პარალელური მკითხველი, ერთი მწერალი) და ატომურ ინსტრუქციებს (compare-and-swap) ერთსიტყვიანი განახლებებისთვის.

4 · Deadlock-ი და როგორ არასდროს დაწერო ის

Deadlock-ს ოთხი პირობა სჭირდება ერთდროულად: ურთიერთგამორიცხვა (mutual exclusion), დაჭერა-და-ლოდინი (hold-and-wait) — აიღე ერთი ლოკი და დაელოდე მეორეს, წართმევის აკრძალვა (no preemption) — ლოკს ძალით ვერ წაართმევ, და წრიული ლოდინი (circular wait) — T1-ს უჭირავს A და უნდა B; T2-ს უჭირავს B და უნდა A. ჩაშალე რომელიმე ერთი და deadlock-ი შეუძლებელი გახდება. (OSTEP: Common Concurrency Problems)

ინდუსტრიული გამოსავალი ყველაზე იაფია: ლოკების გლობალური რიგითობა — ყველა კოდის გზა ლოკებს ერთსა და იმავე შეთანხმებულ თანმიმდევრობით იღებს, ამიტომ ციკლი ვერასდროს ჩამოყალიბდება. ლაბი აჩვენებს, როგორ იჭედება ერთი და იგივე ორი ლოკი საპირისპირო რიგით და როგორ სრულდება მყისიერად ფიქსირებული რიგით. სათადარიგო დაცვები: აღება timeout-ით, ან ერთ დროს მაქსიმუმ ერთი ლოკის ჭერა.

5 · Thread pool-ები და ზომის მათემატიკა

თითო მოთხოვნაზე thread-ის გაჩენა პროდაქშენს ვერ უძლებს: stack-ები მეგაბაიტები ღირს, ხოლო შექმნა/მოშლა მაღალ მოთხოვნათა სიხშირეზე დომინანტური ხარჯი ხდება. Thread pool-ი წინასწარ ქმნის N worker-ს, რომლებიც ამოცანებს საზიარო რიგიდან იღებენ (Python: concurrent.futures). ის რიგი თავად ლოკით დაცული სტრუქტურაა — პატერნს producer–consumer ჰქვია.

რამხელა უნდა იყოს N? I/O-შეზღუდული სამუშაოსთვის პასუხი Little-ის კანონია, thread-ებზე გადათარგმნილი:

Gateway-ის სამიზნე: 2,000 req/s; თითო მოთხოვნა 45 ms ელოდება downstream
გამოძახებას და 5 ms CPU სჭირდება.

მიმდინარე მოთხოვნები   L = λ × W = 2,000/s × 0.050 s = 100
Thread-ები (1/მოთხოვნა)  ≈ 100, დაამრგვალე მარაგით → 128
Stack-ის მეხსიერება      100 × 8 MB ნაგულისხმევი stack = 800 MB წმინდა ბიუროკრატია
asyncio ალტერნატივა      100 coroutine × ~რამდენიმე KB  < 1 MB იმავე კონკურენტულობაზე

ეს ბოლო შედარებაა მიზეზი, რის გამოც შემდეგი სექცია არსებობს.

6 · ასინქრონული I/O და event loop-ი

სერვერის „სამუშაოს" უმეტესობა ლოდინია — socket-ებზე, დისკებზე, ბაზებზე, downstream API-ებზე. ლოდინს thread-ი არ სჭირდება. Event loop-ი ერთ thread-ზე მუშაობს, OS-ს ეკითხება „ჩემი 10,000 ფაილური დესკრიპტორიდან რომელია მზად?" (epoll/kqueue) და მხოლოდ იმ ამოცანებს აღვიძებს, რომლებსაც წინსვლა შეუძლიათ. ამოცანები coroutine-ებია, რომლებიც ყოველ ლოდინის წერტილში await-ს აკეთებენ და thread-ს ნებაყოფლობით თმობენ — კოოპერაციული დაგეგმვა, OS-ის პრეემფციულის საპირისპიროდ. (Python: asyncio)

ლაბში 1,000 პარალელური 50 ms-იანი ლოდინი ერთ asyncio thread-ზე ~0.05 წამში მთავრდება; 50-worker-იან thread pool-ს ~1 წამი სჭირდება (1,000/50 × 50 ms). Nginx, Node.js და ყველა სერიოზული API gateway ამ მოდელზეა აგებული. რკინის კანონი: არასდროს დაბლოკო loop-ი — ერთი CPU-მძიმე ან სინქრონულად მბლოკავი გამოძახება ყველა სხვა ამოცანას აშიმშილებს (ლაბის E5 გაიძულებს, ეს თავად გამოიწვიო).

7 · Python-ის GIL-ი — ერთი ლოკი ყველას სამართავად

CPython-ის Global Interpreter Lock-ი ერთ პროცესში ერთდროულად მხოლოდ ერთ thread-ს აძლევს Python-ის ბაიტკოდის შესრულების უფლებას — ერთი მსხვილი ლოკი, რომელიც ინტერპრეტატორის შიდა სტრუქტურებს იცავს. (Python-ის გლოსარიუმი: GIL)

შედეგები, ლაბში გაზომილი:

  • CPU-შეზღუდული thread-ები არ მასშტაბირდება. წმინდა Python-ის მათემატიკის ოთხი thread-ი ≈ თანმიმდევრული დრო (ზოგჯერ უარესიც — GIL-ის გადაცემები ღირს). გამოიყენე პროცესები (თითოს თავისი GIL-ი) ან სამუშაო გადაიტანე C/NumPy/GPU kernel-ებში, რომლებიც GIL-ს ათავისუფლებენ.
  • I/O-შეზღუდული thread-ები კარგად არიან. მბლოკავი syscall-ების დროს GIL-ი თავისუფლდება, ამიტომ socket-ებზე მომლოდინე thread-ები თავისუფლად გადაიწნება.
  • GIL-ი შენს კოდს thread-safe-ს არ ხდის. ლაბის დაკარგული განახლების რბოლა ჩართული GIL-ით ხდება — ის ინტერპრეტატორის შიგნეულს იცავს და არა შენს წაკითხვა-შეცვლა-ჩაწერის მიმდევრობებს.

LLM serving-ისთვის სწორედ ამიტომ უშვებს vLLM-ი და მისი მსგავსები მოდელს GPU kernel-ებში (GIL-ი გაშვებულია), Python-ის ფენას ასინქრონულ კოორდინატორად ტოვებენ, ხოლო CPU-მძიმე ნაბიჯებს, როგორიცაა ტოკენიზაცია, ცალკე worker პროცესებით ამასშტაბირებენ. PEP 703 (free-threaded CPython) GIL-ს აშორებს, მაგრამ არქიტექტურული გაკვეთილი ძალაში რჩება. (Inside vLLM)

8 · მოდელის არჩევა: ერთი დასამახსოვრებელი ცხრილი

მოდელიპარალელური CPU?საუკეთესოაერთეულის ფასიმახვილი კიდე
Thread-ები + ლოკებიკი (CPython-ში — არა)მბლოკავი I/O, საზიარო მდგომარეობა~MB stack, ~10 µs გადართვარბოლები, deadlock-ი, კონტენცია
პროცესებიკიCPU-შეზღუდული სამუშაო, იზოლაცია~10 ms spawn, IPC ასლებისაზიარო მეხსიერების გარეშე, IPC ხარჯი
Async event loop-იარა (ერთი thread-ი)უზარმაზარი რაოდენობის ლოდინები~KB თითო coroutineერთი მბლოკავი გამოძახება ყველას აშიმშილებს

საწარმოო სერვერები სამივეს აერთიანებენ: async loop-ი კიდეზე, thread pool-ი მბლოკავი ბიბლიოთეკებისთვის, პროცესები CPU-სთვის. ეს კომპოზიცია თავად არის სტანდარტული პასუხი კითხვაზე „როგორ უმკლავდება შენი gateway 10k პარალელურ კავშირს?"

9 · მახვილი კიდეები (ჩავარდნის რეჟიმები)

  • შემოწმება-მერე-მოქმედება (TOCTOU). if balance >= amount: withdraw() — ბალანსი შემოწმებასა და მოქმედებას შორის შეიძლება შეიცვალოს. შემოწმება და მოქმედება ერთ კრიტიკულ სექციაში (ან ერთ ატომურ DB ოპერაციაში) უნდა იყოს.
  • რბოლა, რომელიც „არასდროს ხდება". დროზე დამოკიდებული ბაგები თვეობით იმალებიან, მერე კი პროდაქშენის დატვირთვაზე ან სხვა აპარატურაზე ვარდებიან. ლაბის E2 აჩვენებს, როგორ ჩუმდება იგივე რბოლა პრეემფციის ინტერვალის გაზრდისას — ის მაინც იქ არის.
  • Deadlock-ი მხოლოდ დატვირთვისას. ლოკების ციკლს უიღბლო გადაწვნა სჭირდება; დაბალი ტრაფიკი მას იშვიათად პოულობს. Load ტესტები და ლოკების რიგის აუდიტები — კი.
  • მბლოკავი გამოძახება event loop-ზე. ერთი სინქრონული requests.get() ან მძიმე ციკლი async handler-ში ამ loop-ის ყველა კავშირს ყინავს — async პროდაქშენის №1 ბაგი.
  • Pool-ის ამოწურვა. ყველა N worker-ი ნელ downstream-ზეა გაჭედილი → რიგი იზრდება → timeout-ები კასკადურად ვრცელდება. მკურნალობა: timeout-ები, შემოსაზღვრული რიგები და circuit breaker-ები (ფაზა 2, გაკვეთილი 0015).
  • ვარაუდი, რომ GIL-ი გიცავს. ის ბაიტკოდებს ასერიალებს და არა შენს მრავალნაბიჯიან ინვარიანტებს. ნაკადიან Python-ში ლოკები მაინც გჭირდება.

10 · ლაბი — რბოლები, რომლებსაც მოთხოვნით აღადგენ

პრაქტიკა (ქვიზამდე) labs/lab-p0-03-races-and-gil/ → python3 concurrency_lab.py. ოთხი ექსპერიმენტი: დაკარგული განახლების რბოლა (უყურე არადეტერმინიზმს ცდებს შორის), deadlock-ი ლოკების რიგითობის წინააღმდეგ, GIL-ი, რომელიც CPU thread-ებს ახრჩობს, სანამ პროცესები მასშტაბირდება, და 1,000 ყალბი I/O გამოძახება event loop-ზე vs thread pool-ზე. მერე README-ს E1–E5 სავარჯიშოები: იწინასწარმეტყველე, შეცვალე CONFIG-ი, გაუშვი თავიდან, შეადარე. მცდარი პროგნოზები დავალების შედეგია.

11 · პროექტის ნაბიჯი

დახაზე (ქაღალდზე ან ჩანაწერში) ჩვენი LLM inference gateway-ის კონკურენტულობის გეგმა: რომელი ფენაა async event loop-ი (კლიენტების კავშირები, streaming), სად ჩნდება thread pool-ი (მბლოკავი გამოძახებები storage/auth-ზე), სად — პროცესები (tokenizer worker-ები, model worker-ები) და რომელ საზიარო სტრუქტურას სჭირდება ლოკი (batching რიგი gateway-სა და scheduler-ს შორის). მონიშნე ერთი ადგილი, სადაც დაკარგული განახლების რბოლა მდგომარეობას გააფუჭებდა (მინიშნება: მოთხოვნების თვლა rate limit-ებისთვის) და ერთი პოტენციური deadlock-ი (ორი worker-ი, რომლებიც რიგისა და ქეშის ლოკებს ცვლიან). მოიტანე შემდეგ სესიაზე.
