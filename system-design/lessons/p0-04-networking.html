Networking — Where Latency Comes From

TCP, TLS, HTTP/1.1→3, sockets, and connection pooling — a request's latency is mostly round trips you chose to pay.

Mission tie. Every token your inference gateway streams to a user crosses this stack: socket → TCP → TLS → HTTP. When an interviewer asks why TTFT is 800 ms while the model's prefill takes 200 ms, the answer usually lives in this lesson — handshakes, cold connections, and head-of-line blocking. High Performance Browser Networking calls this out bluntly: most web latency is round trips, not bandwidth. (HPBN — latency vs bandwidth primer)

1 · The bill for one HTTPS request

Type a URL, press enter. Before the first response byte arrives, you pay, in sequence: DNS lookup (find the IP), TCP handshake (1 round trip), TLS handshake (1–2 round trips), the request itself (1 round trip), then transfer time for the bytes. The unit of account is the RTT — round-trip time. Same city ≈ 1–10 ms, cross-continent ≈ 50 ms, cross-ocean ≈ 150 ms; physics (speed of light in fiber) sets the floor, and no hardware upgrade lowers it. (HPBN ch. 1)

Bandwidth, by contrast, keeps improving and is rarely the bottleneck for small API payloads. A 10 KB JSON response at 100 Mbps costs ~1 ms of transfer — but 3 RTTs of setup at 50 ms each cost 150 ms. Latency dominates; count round trips first.

2 · TCP — reliability costs a round trip

TCP gives you an ordered, reliable byte stream over an unreliable network. The price of that contract starts with the three-way handshake: SYN → SYN-ACK → ACK. The client can send data after one full round trip — so every new TCP connection costs 1 RTT before byte one. (HPBN — Building Blocks of TCP)

Two throttles then govern how fast bytes flow:

  • Flow control — the receiver advertises a window (rwnd): "I can buffer this much." Protects the receiver.
  • Congestion control — the sender maintains a congestion window (cwnd) that starts small and grows each RTT (slow start: roughly doubling until loss or threshold). Protects the network. A new connection cannot use your full bandwidth for the first several RTTs — one more reason cold connections hurt.
Vocabulary Head-of-line (HOL) blocking: when one lost or slow item stalls everything queued behind it. TCP guarantees ordering, so one lost packet stalls delivery of all later packets on that connection — remember this for HTTP/2 vs HTTP/3.

3 · TLS — identity, keys, and more round trips

TLS adds two things on top of TCP: authentication (the certificate proves you're talking to the real server) and encryption (a symmetric key negotiated per connection). The negotiation costs round trips: TLS 1.2 needs 2 RTTs; TLS 1.3 compressed the handshake into 1 RTT by sending the client's key share in the first flight, and offers 0-RTT resumption for repeat connections (with replay-attack caveats). (HPBN — TLS)

So a cold HTTPS connection costs 1 (TCP) + 1–2 (TLS) = 2–3 RTTs before the HTTP request even leaves. The per-byte encryption cost (AES with hardware support) is nearly free by comparison — the handshake is where the money goes.

4 · Worked example — the time-to-first-byte ladder

One small request, one fresh connection, RTT = 50 ms, server processing ≈ 0. Count round trips to first response byte:

TCP + TLS 1.2 :  1 (TCP) + 2 (TLS) + 1 (request)  =  4 RTT  =  200 ms
TCP + TLS 1.3 :  1 (TCP) + 1 (TLS) + 1 (request)  =  3 RTT  =  150 ms
QUIC (HTTP/3) :  1 (transport+TLS)  + 1 (request)  =  2 RTT  =  100 ms
QUIC 0-RTT    :  0 (resumed)        + 1 (request)  =  1 RTT  =   50 ms
pooled conn   :  0 (already open)   + 1 (request)  =  1 RTT  =   50 ms

Same wire, same server, same bytes: 4x spread — purely protocol choices.

This ladder is experiment 3 of the lab, computed for RTT = 10/50/150 ms. Nothing here is about bandwidth.

5 · HTTP/1.1 → HTTP/2 → HTTP/3

HTTP evolution is one long war on round trips and head-of-line blocking:

VersionTransportConcurrency modelHOL blocking
HTTP/1.1TCP1 request at a time per connection; browsers open ~6 parallel connections; keep-alive reuses themAt HTTP level: one slow response blocks the connection
HTTP/2TCP + TLSMany streams multiplexed on one connection; header compression (HPACK)Fixed at HTTP level, but still present at TCP level: one lost packet stalls all streams
HTTP/3QUIC (over UDP, TLS 1.3 built in)Streams multiplexed with independent delivery; 1-RTT setup, 0-RTT resumption; connections survive IP changesGone at transport level: a lost packet stalls only its own stream

QUIC moved reliability, ordering, and encryption into user space on top of UDP precisely so a lost packet in stream A cannot stall stream B — TCP's in-order contract made that impossible to fix. (RFC 9000 — QUIC, RFC 9114 — HTTP/3, HPBN — HTTP/2)

6 · Sockets and connection pooling

To your code, all of this appears as a socket: a file-descriptor handle on one endpoint of a connection, identified by the 4-tuple (src IP, src port, dst IP, dst port). Servers listen() and accept(); clients connect(). Lesson p0-03's event loops exist largely to juggle thousands of these without a thread each.

Connection pooling is the systems answer to sections 2–4: pay the TCP + TLS setup once, keep the connection open, and reuse it for many requests. Every serious client does this (browsers, requests.Session, gRPC channels, database drivers). Pools bring their own knobs — size, idle timeout, health checks — and their own failure modes (§7). (Primer)

Rule of thumb If a client talks to the same host more than once, an unpooled connection is a bug. The lab's experiment 1 shows even localhost paying ~8x for fresh connections; over a WAN the multiple is worse because each setup burns real RTTs.

7 · Sharp edges — where latency hides in production

  • Nagle's algorithm + delayed ACK — two well-meaning batching features that interact to add up to ~200 ms stalls on small writes. Latency-sensitive services set TCP_NODELAY. (HPBN)
  • Ephemeral port exhaustion — closing thousands of client connections leaves them in TIME_WAIT; open-connect-close loops at high QPS run out of ports. Pooling fixes the cause, not just the symptom.
  • Slow start after idle — an idle pooled connection's congestion window can reset; the first burst after idle crawls. Long-lived does not mean always-warm.
  • Pool exhaustion — all pooled connections busy → requests queue at the client. Looks like "the upstream is slow"; it's your own pool size. Classic misdiagnosis.
  • TCP HOL blocking on HTTP/2 — on lossy networks (mobile), HTTP/2 can be slower than HTTP/1.1 with 6 connections, because one loss stalls every stream. HTTP/3's whole reason to exist.
  • SYN backlog overflow — connection storms (retry storms, thundering herds) fill the server's accept queue; clients see timeouts on connect, not on requests — a distinct signature worth recognizing.

9 · Project step

Apply this to the LLM inference gateway. Decisions to write down (one line each): (1) gateway ↔ model workers use pooled, long-lived connections (gRPC/HTTP2 channels) — never per-request connects; (2) client-facing token streaming runs over SSE or HTTP/2 streams, so one TCP+TLS setup amortizes over hundreds of token frames; (3) your TTFT SLO includes connection setup — a cold TLS 1.2 connection at 150 ms RTT spends 600 ms before the model is even asked (§4), which can exceed prefill itself; (4) retry storms multiply handshakes, not just requests — backoff protects the SYN queue too. The vLLM anatomy post is a good picture of what sits behind the socket you're feeding. (Inside vLLM)

Interview talking points

  • "For small payloads, latency is round trips, not bandwidth — so I count RTTs first: TCP costs 1, TLS 1.2 costs 2 more, TLS 1.3 costs 1, QUIC folds transport and TLS into 1."
  • "A cold HTTPS request at 50 ms RTT pays 150–200 ms before the server even sees it — that's why connection pooling and keep-alive are the first latency optimization, not caching."
  • "HTTP/2 fixed head-of-line blocking at the HTTP layer but not the TCP layer; HTTP/3 runs streams over QUIC so a lost packet only stalls its own stream — that's the whole point of QUIC."
  • "Slow start means a new — or long-idle — connection can't use full bandwidth for several RTTs, so 'we reuse connections' isn't automatically 'we're warm.'"
  • "Pool exhaustion looks exactly like a slow upstream; I check client-side queue time before blaming the service."
  • "In LLM serving, TTFT includes DNS, TCP, and TLS — streaming over one long-lived connection amortizes that setup across every token."
