Distributed Search over 1B Documents

Two systems wearing one trench coat: an offline indexing pipeline and an online query plane. The candidates who fail this rep design only one of them.

Mission tie. This is a reported Anthropic prompt — "distributed search over 1B documents" appears verbatim in the question bank, so treat this rep as live ammunition, and note that the same retrieval infrastructure underpins RAG at production scale (Phase 6, lesson 41): swap "keyword" for "embedding" and the sharding, fanout, and tail-latency math all survive.
REP PROTOCOL Do the 40-minute solo timed rep FIRST — out loud, paper or Excalidraw, no peeking. Prompt: "Design a search system over 1 billion documents." Only then read this lesson, compare, and log every delta (missed component, wrong number, unnamed failure mode) as a learning record.

1 · The prompt, and what it actually tests

The interviewer says one sentence: "Design keyword search over a billion documents." The hidden rubric: do you notice this is two systems in one? An offline indexing pipeline (a batch/streaming data system that turns raw documents into a queryable index) and an online query plane (a low-latency fanout service that answers top-k queries against that index). Candidates who design only the query plane get asked "and where does the index come from?" and stall; candidates who design only the pipeline never touch latency. Say the two-plane split in your first minute — it is the single highest-signal sentence in this rep. Secondary rubric items: index sizing math, the sharding-axis decision (section 6), and fanout tail latency (section 7) — the three places where this prompt quietly measures whether Phase 1 stuck.

2 · Clarify (5 min): freshness is the unlock

RequirementTargetConsequence
Query typeKeyword (AND of terms), rankedInverted index, not table scans
Result shapeTop-k (k = 10) with snippetsRanking + doc store lookup
Latencyp99 ≈ 200 ms end-to-endIndex in RAM/SSD; fanout tail math matters
Corpus1B docs × ~5 KB, growing slowlyMust shard; no single node holds it
QPS~10k peakReplicas per shard, result caching
Index freshnessMinutes to hours OKThe key unlock — see below

Ask the freshness question explicitly and then say out loud that relaxing it is the unlock: if a new document may take minutes to become searchable, indexing can run as an offline batch job producing immutable index files that are shipped to serving nodes on a schedule. The two planes decouple completely — the pipeline can crash, rerun, and rebuild without ever touching query latency. If the interviewer instead demands seconds-fresh results, you've discovered a much harder problem (section 10). Out of scope: crawling politeness (lesson 0024 owns it), auth, ads.

3 · Estimate (5 min): numbers you will spend

Corpus:   1B docs × 5 KB text            ≈ 5 TB raw text
Postings: ~500 distinct terms per doc    → 1B × 500 = 5 × 10¹¹ posting entries
Entry:    ~4 B compressed (delta docID + tf, positions amortized)
Index:    5 × 10¹¹ × 4 B ≈ 2 TB         → index ≈ 40% of corpus (typical: 20–50%)
Shards:   2 TB ÷ ~50 GB index per node  = 40 shards → 50 with headroom
Fanout:   every query hits every shard  → per-shard QPS = full 10k QPS
          10k QPS × 50 shards = 500k shard-queries/s cluster-wide
Replicas: node handles ~2k QPS          → 5 replicas/shard ≈ 250 serving nodes

Now spend each number. 2 TB index → no single node; sharding is forced, not optional. 50 shards → fanout of 50 per query, which triggers the tail-latency math of section 7. Per-shard QPS = full cluster QPS (every query visits every shard — the doc-partitioned tax) → replica count is driven by total QPS, not by a slice of it. 40% index/corpus ratio → compression (section 5) is a capacity decision worth real money, not a nicety. An estimate never referenced again is the classic anti-pattern from lesson 0009 — every one of these four gets reused below.

4 · High-level: draw the two planes separately

INDEXING PLANE (offline, throughput-optimized)
  crawl/ingest → parse + tokenize → build partial indexes (per batch)
      → merge into immutable segments → ship segments to shard replicas

QUERY PLANE (online, latency-optimized)
  client → query service (parse, cache check)
      → fanout to all 50 shards (pick 1 replica each)
      → each shard: top-k′ locally  → gather → merge + rank → top-10 → return

Draw them as two separate diagrams — physically apart on the whiteboard. The indexing plane is a MapReduce-shaped batch system: mappers tokenize document batches and emit partial term→postings maps; reducers merge partials into sorted, compressed segment files; a distribution step ships each segment to the replicas of its shard. The query plane never writes: it loads immutable segments and serves. The only coupling between the planes is the segment handoff — a versioned, atomic "load segment set N+1, drop N" switch. That decoupling is what the relaxed freshness requirement bought you, and saying so ties Step 1 to Step 3 in front of the interviewer.

5 · Deep dive: anatomy of an inverted index

An inverted index maps term → postings list: the sorted list of docIDs containing that term, each entry carrying term frequency (tf) and optionally positions (token offsets — needed for phrase queries and snippet highlighting). Three design facts to narrate:

  • Sorted + delta-encoded + compressed. Postings are sorted by docID, so you store gaps (deltas) instead of raw 8-byte IDs; gaps are small and compress hard with varint/bit-packing. This is where the 40% index ratio of section 3 comes from — uncompressed it would rival the corpus.
  • Skip pointers. An AND query intersects postings lists; skip pointers every √n entries let the intersection leap over runs of the longer list instead of scanning it, turning intersection cost from sum-of-lengths toward length-of-shortest.
  • Immutable segments + background merge. Updates never modify a segment in place: new docs go into new small segments; deletes are tombstone bitmaps; a background process merges small segments into larger ones. Recognize the instinct — this is the LSM tree from lesson 0012 wearing search clothes: sequential writes, lock-free reads, write amplification paid in the background. Lucene, and therefore Elasticsearch, works exactly this way.

6 · Deep dive: document-partitioned vs term-partitioned

The canonical fork of this prompt: shard by document (each shard = full index over a slice of docs) or by term (each shard owns entire postings lists for a slice of the vocabulary)?

DimensionDoc-partitionedTerm-partitioned
Query fanoutEvery query → all 50 shardsOnly shards owning the query's terms (2–3)
Multi-term ANDIntersection is local to each shardPostings shipped cross-shard to intersect — GBs for common terms
Hot termsLoad spreads evenly by designShard owning "the", "http" melts — hot-term skew
Index build / doc addOne shard touched per docEvery doc touches ~500 term shards
Shard failureLose a slice of recall; all queries degrade gracefullySome terms vanish entirely from all results

Term-partitioning looks attractive (tiny fanout!) until the multi-term row: intersecting "distributed AND search" requires shipping one term's postings — potentially millions of entries — to the other term's shard, per query. Hot-term skew finishes it off. This is why virtually everyone ships doc-partitioned — it is the Elasticsearch/Lucene model, and Google's original architecture too. Say the trade-off table out loud, choose doc-partitioned because local intersections and even load beat small fanout, and note the price you accepted: full fanout, which you now must pay for in section 7.

7 · Deep dive: scatter-gather and the tail-latency tax

The coordinator fans a query to all 50 shards; each returns its local top-k′; the coordinator merges the 50·k′ candidates and returns the global top-10. Two subtleties carry senior signal:

Why k′ > k. If every shard scored with identical global statistics, per-shard top-10 would provably contain the global top-10 (each winner lives in exactly one shard). But shards rank with local IDF (their own document frequencies), use early-termination approximations, and carry deletion tombstones — so a globally-strong doc can sit at local rank 12 and never reach the coordinator. Practical fix: request k′ = 2–3×k per shard (a few hundred merge candidates — cheap), or broadcast global term statistics periodically. The lab measures exactly this recall-vs-k′ curve.

Fanout amplifies the tail. A query is as slow as its slowest shard: P(all n shards fast) = pn. With fanout 100 and shards fast 99% of the time, 0.99100 ≈ 0.37 — only 37% of queries avoid every shard's slow 1%, so the per-shard p99 becomes roughly your end-to-end median. To keep end-to-end p99 clean at fanout 100 you need each shard fast at the 99.99th percentile: 0.9999100 ≈ 0.99. Mitigations: hedged requests (if a shard hasn't answered by its p95, fire the same request at another replica and take the first answer — ~5% extra load, dramatic tail cut) and replica-aware load balancing that routes around slow or GC-ing nodes.

8 · Deep dive: ranking — from TF-IDF to two phases

TF-IDF intuition: score a doc by Σ over query terms of tf (how often the term appears here) × idf = log(N/df) (how rare the term is corpus-wide). Rare terms dominate; "the" contributes nothing. BM25 is TF-IDF with two fixes, and you only need the shape: tf saturates (the 50th occurrence adds almost nothing — tf/(tf+k₁) instead of raw tf) and scores are length-normalized (a term hit in a tweet outweighs one in a book — the b·dl/avgdl term). Formula shape, not derivation, is the interview bar.

Two-phase ranking: running an expensive ranker over millions of matching docs blows the latency budget, so phase 1 retrieves the top few hundred candidates with cheap BM25 inside each shard, and phase 2 reranks only those few hundred with the expensive model (dozens of features, or an ML ranker) at the coordinator. Blend in static quality priors — a query-independent score like PageRank or source quality, computed at index time and stored per doc. Finally, cache query results (lesson 0007 callback): query popularity is zipfian, so a modest results cache in front of the coordinator absorbs a large slice of the 10k QPS before any fanout happens.

9 · Failure modes and monitoring

  • One slow shard poisons every query. Full fanout means one bad node (GC pause, failing disk, merge storm) drags the global p99. Mitigate: hedged requests, strict per-shard timeouts, and return partial results with a degraded=true flag rather than failing the query — 49/50 shards is still excellent recall.
  • Segment-merge storms. Background merges are I/O-heavy; an unthrottled merge on a serving node competes with queries and manufactures exactly the slow shard above. Mitigate: throttle merge I/O, or do heavy merges off the serving path and ship the merged segment.
  • Index/serving version skew. Mid-rollout, shards on segment-set N and N+1 return differently-scored results, and merged pages can duplicate or drop docs. Mitigate: version every segment set, have the coordinator pin one version per query, canary new indexes on a shard subset first.
  • Metrics that page a human: per-shard p99 and end-to-end p99 (the gap between them is your fanout tax), hedge-fire rate, indexing lag (freshness SLO), and a recall canary — a fixed set of query→expected-doc pairs replayed continuously; a silent recall drop is otherwise invisible in latency dashboards.

10 · Variants and twists

  • Real-time indexing. "New docs searchable in seconds" → keep a small in-memory delta index on each shard receiving fresh docs, query it alongside the big immutable segments, flush it to a segment periodically. This is precisely Lucene's near-real-time refresh — the LSM memtable instinct again.
  • Phrase queries. "exact phrase" needs the position data from section 5: intersect postings and check adjacent positions; common-bigram indexes ("new york") pre-pay the worst cases.
  • Multilingual. Tokenization is language-specific — CJK languages don't use spaces (segmentation models), German compounds split, stemming/lemmatization per language at both index and query time. Analyzer choice is an index-time commitment: changing it means reindexing.

AI-adjacent note. Embedding-based (vector) search replaces the term-match retrieval phase with approximate nearest-neighbor search over dense vectors, but keeps this lesson's skeleton — sharded ANN indexes, scatter-gather, per-shard top-k′, merge, rerank. Modern RAG stacks run hybrid: BM25 and vector retrieval in parallel, fused at rerank. Full treatment in Phase 6, lesson 41.

11 · Interview talking points

  • Opening: "This is two systems — an offline indexing pipeline and an online query plane — and relaxing freshness to minutes is what lets me keep them decoupled."
  • Spending an estimate: "2 TB of index at 50 GB per node is 40–50 shards — and that fanout number is why I'll spend time on tail latency."
  • The sharding fork: "Doc-partitioned versus term-partitioned: I'll take full fanout in exchange for local intersections and no hot-term shard — the Elasticsearch model."
  • Unprompted tail math: "At fanout 50, p-all-fast is p-per-shard to the 50th power — per-shard p99 becomes my median, so I need hedged requests, not faster averages."
  • Ranking depth on offer: "I can go deep on BM25-plus-rerank or on segment merging — preference?"
  • Closing: "I'd page on end-to-end p99, indexing lag, and a recall canary; with more time: real-time deltas and multilingual analyzers."
