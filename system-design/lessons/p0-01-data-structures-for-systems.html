Data Structures for Systems

Hash tables, B-trees, heaps, tries, Bloom filters, consistent hashing — and why "O(1)" comes with an asterisk your p99 pays for.

Mission tie. Every Anthropic design prompt eventually lands on a data-structure decision: the rate limiter is a hash map with a resize problem, the distributed KV store is a consistent-hash ring, the batching scheduler is a heap, the storage engine is a B-tree or LSM-tree. Interviewers probe one level below the box you drew — this lesson is that level. (System Design Primer, Teach Yourself CS)

1 · Big-O is a starting point, not an answer

Big-O describes how cost scales as n grows; it deliberately ignores constants, caches, and memory layout. Systems care about all three. A linear scan over 100 contiguous integers beats a red-black tree lookup, because sequential memory reads are prefetched while tree traversal is pointer-chasing — a cache miss per level. (TYCS: algorithms + architecture together)

The systems question is never "which is asymptotically fastest" — it is "which access pattern am I paying for, on which storage tier, at which percentile."

2 · Hash tables: O(1)* — the asterisk owns your p99

A hash table hashes the key to a bucket index. Average insert/lookup is O(1). But when the table's load factor crosses a threshold, it must resize: allocate a bigger array and rehash every stored key. That single insert costs O(n) — and some real request pays it. (Primer)

CPython dict: resizes when 2/3 full; capacity roughly doubles.
1,000,000 inserts → resizes near 5k, 10k, 21k, 43k, 87k, 174k, ... entries.
Mean insert  ≈ 50 ns.
The insert that triggers the 699k-key rehash ≈ tens of MILLIONS of ns.
Mean: unchanged. p999: a cliff. This is "amortized O(1)" seen from the tail.
Vocabulary Amortized = averaged over a sequence of operations. Amortized O(1) permits rare O(n) spikes — fine for total runtime, hostile to latency SLOs. Production fixes: pre-size the table, or use incremental rehashing (Redis migrates a few buckets per operation so no single request pays the whole bill).

3 · B-trees: designed for pages, not pointers

Disks and SSDs read fixed-size pages (~4 KB); each read costs ~150 µs. A B-tree makes every node exactly one page holding hundreds of sorted keys, so the tree is extremely wide and therefore extremely shallow. This is why nearly every relational index is a B-tree. (DDIA ch. 3)

Branching factor b ≈ 500 keys per 4 KB page.
height ≈ log_b(N):   N = 10^9 keys → log_500(10^9) ≈ 3.3 → 4 page reads.
Binary tree instead: log2(10^9) ≈ 30 pointer hops = 30 page reads on disk.
At 150 µs per random SSD read:  B-tree ≈ 0.6 ms   binary tree ≈ 4.5 ms.
Same O(log n). 7× fewer I/Os. Bases matter when each level is an I/O.

Because keys sit sorted inside pages, B-trees also answer range queries (WHERE ts BETWEEN …) by walking to the start key and scanning siblings — a hash index simply cannot do this. That is the interview one-liner for "why not hash indexes everywhere."

4 · Heaps: the shape of every scheduler

A binary heap keeps the minimum (or maximum) element at the root: peek O(1), pop/push O(log n), stored flat in an array — no pointers, cache-friendly. Wherever a system repeatedly asks "what is next?", there is a heap: OS run queues, timer wheels' cousins, Dijkstra, top-K rankings, and heapq under Python's asyncio scheduled callbacks.

Systems framing: a heap is how you serve "most urgent first" without re-sorting on every arrival. Re-sorting a list per event is O(n log n) each time; a heap pays O(log n) per event. A request scheduler that picks the earliest deadline among 100k in-flight jobs does it in ~17 comparisons.

5 · Tries: prefixes as first-class keys

A trie stores strings as a tree of characters (or bytes): lookup cost is O(length of key), independent of how many keys exist. Everything sharing a prefix shares a path — so "all keys starting with /api/v2/" is a subtree, not a scan. (Trie)

Where you meet them: router URL matching, autocomplete, IP routing tables (longest-prefix match in radix-trie form), and dictionary-encoded search indexes. The trade: pointer-heavy memory layout; production systems use compressed variants (radix/Patricia tries, adaptive radix trees).

6 · Bloom filters: buying "definitely not" with bits

A Bloom filter is a bit array plus k hash functions. Insert: set k bits. Query: if any of the k bits is 0 → definitely absent; if all are 1 → probably present. False positives happen; false negatives never do. (Bloom filter)

Design rule: m/n bits per element, optimal k ≈ 0.69 × (m/n).
At 10 bits/element → k ≈ 7 → FP ≈ (1 − e^(−kn/m))^k ≈ 0.5^7 ≈ 0.8%.
1 billion URLs as raw 64-byte strings  ≈ 64 GB.
Same set as a Bloom filter @ 10 bits   ≈ 1.25 GB — 50× smaller.
Price: 0.8% of absent URLs answered "maybe" — you verify those in storage.

Classic deployments: LSM storage engines (skip SSTables that can't contain the key — DDIA ch. 3), CDN "have I cached this?" checks, crawlers de-duplicating URLs, databases short-circuiting disk reads for missing rows.

7 · Consistent hashing: adding a server without a stampede

Shard by hash(key) mod N and the arithmetic betrays you: change N and almost every key's answer changes. Consistent hashing places servers and keys on a hash ring; each key belongs to the next server clockwise. Adding a server steals only a slice of the ring. (Consistent hashing, Primer)

SchemeServers 10 → 11Keys that moveCache consequence
hash(key) mod NN changes~10/11 ≈ 91%Mass miss storm; DB gets hammered
Consistent hash ring1 node added~1/11 ≈ 9%One slice warms up; rest unaffected

Real rings use virtual nodes (each server appears ~100–200 times on the ring) to smooth load imbalance. This structure underlies Memcached client sharding, Dynamo/Cassandra partitioning, and every "design a distributed cache" answer you will give.

8 · Sharp edges — where these structures hurt you

  • Rehash pause at peak traffic. The resize triggers exactly when the table is biggest — i.e., at your traffic high-water mark. Pre-size or rehash incrementally.
  • Hash-flooding DoS. Adversarial keys that all collide degrade a hash table to O(n) per op. Languages patched this with randomized/SipHash seeds — know the phrase.
  • Unordered means unordered. Pick a hash index, lose range scans and sorted pagination forever. Schema/index choice is an access-pattern commitment.
  • Bloom filters can't delete. Clearing bits corrupts other keys' membership. Use counting Bloom filters or rebuild periodically.
  • Hot keys break sharding math. Consistent hashing balances key counts, not traffic. One celebrity key still melts one server — see Phase 3's Twitter rep.
  • Heap without decrease-key. Stdlib heaps can't update a buried entry; the standard trick is lazy deletion (push a new entry, ignore stale ones on pop).

Interview talking points

  • "Hash tables are amortized O(1) — the resize rehashes every key, so I pre-size or use incremental rehashing when p99 matters."
  • "Databases index with B-trees because one 4 KB node holds hundreds of keys: a billion rows is ~4 I/Os, and sorted pages give me range scans hash indexes can't."
  • "A Bloom filter gives me 'definitely not present' in ~10 bits per element with ~1% false positives — I put it in front of disk reads and only verify the maybes."
  • "hash mod N remaps ~90% of keys when I add a server; a consistent-hash ring with virtual nodes moves only ~1/N — that's the difference between a cache warm-up and a miss storm."
  • "Consistent hashing balances key counts, not load — hot keys still need replication or request spreading on top."
  • "For small hot sets I'd benchmark a flat array before reaching for trees: big-O ignores the cache hierarchy, and pointer chasing loses to sequential scans at small n."
