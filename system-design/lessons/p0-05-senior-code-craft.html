Senior Code Craft

API contracts, idempotency, testing strategy, profiling — the four habits that separate "it works" from "it survives production."

1 · Senior code is boring on purpose

Junior code optimizes for writing; senior code optimizes for reading, debugging at 3 a.m., and changing safely a year later. The concrete signature: obvious names, small functions, errors handled at every boundary, and — the theme of this lesson — designing for failure before it happens. Networks drop responses, processes die mid-request, callers retry. Senior code assumes all three on every line that touches I/O. The SRE book's framing: hope is not a strategy; reliability is a feature you engineer, not an accident you inherit.

2 · API design: the contract is the product

An API is a promise you cannot easily un-make — clients compile against it. The craft checklist (see Google's API design guide and the Primer's communication section):

  • Nouns for resources, verbs from HTTP: POST /charges, not POST /doCharge. Predictability is a feature.
  • Errors are part of the contract: machine-readable code + human message + what the caller should do (retry? fix input? give up). A bare 500 is an abdication.
  • Never break existing callers: add optional fields freely; never repurpose or remove fields without a versioning story.
  • Pagination and limits from day one — every unbounded list endpoint is a future outage.
Vocabulary An operation is idempotent if performing it N times has the same effect as performing it once. GET and PUT are idempotent by HTTP contract; POST is not — which is exactly why the next three sections exist.

3 · The at-least-once world

Here is the trap every distributed system sets for you: a client sends POST /charges, the server charges the card, and the response is lost (connection reset, timeout, proxy hiccup). The client now cannot distinguish "server never got it" from "server did it and the reply died." Its only sane move is to retry — so real networks deliver requests at least once, not exactly once. Do the math on what that costs:

200 clients, each pays once.  P(success response lost) = 0.30, up to 3 retries.

P(a client retries at least once)      = 0.30            → ~60 clients retry
Each retry after a SUCCESSFUL charge   = another charge!
E[extra charges] ≈ 200 × (0.30 + 0.30² + 0.30³) ≈ 200 × 0.417 ≈ 83 duplicates

Lab measurement: 282 total charges for 200 purchases, 60 customers double-charged.

No bug in the code, no malice — just physics. Retries are mandatory, so safety must be engineered server-side.

4 · Idempotency keys: exactly-once effects

The industry-standard fix, popularized by Stripe's Idempotency-Key header: the client generates a unique key per logical operation (one purchase = one key, however many retries), and the server keeps a small table:

  1. Request arrives with key K. Look up K.
  2. Seen before? Return the stored response. Do not re-execute anything.
  3. New? Execute the operation, store (K → response), return it.

Effect: the operation may be delivered many times but is applied once — at-least-once delivery plus server-side dedup equals exactly-once effect. This one idea shows up in payments, message queues (lesson 8), sagas (lesson 14), and inference-request submission alike. (Stripe: "Designing robust and predictable APIs with idempotency")

5 · Sharp edges of idempotency

Where real implementations go wrong — each of these is a production incident that has actually happened somewhere:

  • Key generated per attempt, not per operation. Client makes a fresh UUID on every retry → dedup never triggers → double charge anyway. The key must survive the retry loop.
  • Concurrent duplicates. Two retries land simultaneously; both see "key unseen" and both execute. Fix: reserve the key atomically (insert-first with a unique constraint), make the second request wait or 409.
  • Storing only the key, not the response. The replay must return the original outcome — including the original failure — or clients see inconsistent worlds.
  • Infinite key retention. The table grows forever; Stripe prunes keys after 24 h. Expiry is a contract decision, not a cleanup detail.
  • "Idempotent" handlers with hidden side effects — sends an email, increments a metric, calls a partner API. Dedup the whole effect or you've deduped nothing.

6 · Retries amplify outages — budget them

Idempotency makes retries safe; it does not make them free. When a server browns out, naive client retries multiply its load precisely when it can least afford it. With failure rate p and up to k attempts, expected sends per logical request form a geometric series:

load multiplier = 1 + p + p² + … + pᵏ⁻¹
p = 0.5, k = 4:  1 + 0.5 + 0.25 + 0.125 = 1.875×   (lab measured: 1.89×)

Nearly double the traffic aimed at a half-dead server — the shove that turns a brownout into a blackout. The SRE prescriptions (Handling Overload, Addressing Cascading Failures): exponential backoff with jitter, per-client retry budgets (e.g., retries ≤ 10% of base traffic), and retrying at most once per layer so a three-layer stack doesn't cube the retries.

7 · Testing strategy: the pyramid buys speed

Senior testing is a portfolio decision, not a virtue contest. The canonical shape (Fowler's practical test pyramid, Google's test sizes):

LayerScopeSpeedCountCatches
Unitone function/class, no I/Omsmanylogic bugs, edge cases
Integrationservice + real DB/queuesecondssomewiring, schemas, SQL
End-to-endwhole system, user flowminutesfewconfig, contracts, deploys

Why this shape: feedback latency compounds. A 10-minute suite gets run before every commit; a 2-hour suite gets run never. Two senior reflexes: every bug fix starts with a failing test that reproduces it (regression insurance), and flaky tests are deleted or fixed, never retried into green — a suite people don't trust is worse than no suite.

8 · Profiling: measure, don't guess

Engineers' intuitions about hotspots are famously wrong — the bottleneck is rarely where the code "looks slow." The rule: no optimization without a profile. Python ships one in the stdlib (cProfile):

python3 -m cProfile -s cumulative my_service.py | head -20

ncalls  tottime  cumtime  filename:lineno(function)
 30000    0.006    0.006  craft_lab.py:49(add_line)      ← hotspot names itself

Read cumtime top-down: the first non-framework line is usually your answer. The lab's example is the classic accidental O(n²) — building a string with += in a loop copies the whole string each pass, so 4× the input costs ~16× the time. One profile view finds in 10 seconds what an afternoon of guessing misses. (For sampling production processes without stopping them, the ecosystem term to know is a sampling profiler — e.g. py-spy — same reading skill, zero pause.)

9 · Reading production Python and Go

Seniors read 10× more code than they write, and they read in a fixed order: entry points → data shapes → error paths. Find main()/the route table; find the core structs/dataclasses (data shapes explain a codebase faster than any comment); then read what happens when things fail — that's where quality lives. Language-specific tells:

  • Python: type hints + dataclasses mark the data spine; context managers (with) mark resource boundaries; a bare except: is a red flag every reviewer should catch.
  • Go (Effective Go): if err != nil after every call makes error paths visible by construction; defer marks cleanup; goroutines + channels mark concurrency boundaries. Go reads flat on purpose — boring is the house style.

Interview talking points

  • "Networks are at-least-once: a lost response forces the client to retry, so any non-idempotent endpoint will eventually execute twice."
  • "My fix is Stripe-style idempotency keys: one key per logical operation, server stores key → response, replays instead of re-executing."
  • "The two classic implementation bugs: regenerating the key on each retry, and not reserving the key atomically against concurrent duplicates."
  • "Retries need budgets and jittered backoff — at 50% failure, four naive attempts nearly double the load on an already-dying server."
  • "I never optimize without a profile; the hotspot is rarely where intuition points, and cProfile finds it in one command."
  • "My test portfolio is pyramid-shaped: many milliseconds-fast unit tests, some integration tests, few end-to-end — because feedback latency compounds."
