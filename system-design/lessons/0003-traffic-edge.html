The Traffic Edge: DNS, CDN, Load Balancing

Every request crosses three gatekeepers before your code runs — know what each one can see, what it costs, and how it fails.

Mission tie. Anthropic's system design rounds, like every serious design interview, open with a scale number — "…handles 100,000 requests per second." That number must land on many machines, which means the first box you draw after the client is the traffic edge: DNS, maybe a CDN, then a load balancer. Interviewers probe exactly here — L4 or L7? which algorithm? what happens when a backend dies? who balances the balancer? This lesson gives you defensible answers to all four. (Exponent's 2026 Anthropic guide)

1 · The journey of a request

Type shop.example.com/checkout and press Enter. Before your application code runs, the request traverses a chain of infrastructure that most candidates hand-wave — and interviewers notice:

client → DNS (name → IP, cached at every step)
       → CDN edge (static asset? answered here, journey over)
       → load balancer (one virtual IP, many real servers)
       → reverse proxy (TLS, compression, rate limits)
       → app server (your code, finally)

Each hop exists to solve one specific problem: DNS solves naming, the CDN solves distance, the load balancer solves one machine isn't enough, the reverse proxy solves don't make app servers do plumbing. Lesson 0001 gave you the physics motivating all of this: an ocean round trip costs ~150 ms, a datacenter round trip ~500 µs. The entire edge is an exercise in either avoiding the slow hop or amortizing it. (System Design Primer, sections 7–10, is the primary source for this whole lesson.)

2 · DNS: the phone book with a cache problem

DNS translates shop.example.com into an IP address via a hierarchy of servers — your OS cache, your ISP's resolver, the root servers, the TLD (.com) servers, and finally the domain's authoritative name server. The record types you must recognize on sight (Primer §7, Cloudflare: what is DNS):

RecordMapsTypical use
Aname → IPv4 addressthe workhorse (AAAA for IPv6)
CNAMEname → another namealias www, or point at a CDN hostname
NSzone → name serverdelegates who answers for a domain
MXdomain → mail serveremail routing, with priorities

The property that dominates DNS design discussions is caching. Every answer carries a TTL (time-to-live), and resolvers all the way down to the user's laptop cache the answer for that long. Consequences worth saying out loud in an interview:

  • DNS lookups are usually free — the vast majority are answered from a nearby cache, which is why the hierarchy scales to the whole internet.
  • DNS changes are slow to propagate. With a 300 s TTL, some users keep hitting the old IP for five minutes after you change the record — so DNS is a terrible primary failover mechanism.
  • DNS is itself a dependency that fails. The 2016 DDoS on the Dyn DNS provider took Twitter, Spotify, and GitHub offline for hours without touching any of their servers — their names simply stopped resolving. Standard mitigation: multiple DNS providers, exactly like any other redundancy story.

3 · DNS as a router, not just a phone book

Managed DNS services (Route 53, Cloudflare, NS1) can return different answers to different queriers, which turns DNS into a coarse global router (Route 53 routing policies):

  • Weighted round-robin — return server A 95% of the time, server B 5%. Classic uses: canary releases (5% of traffic tastes the new version first), gradual migrations between providers, A/B splits at the infrastructure level.
  • Latency-based — answer with the region that has the lowest measured RTT to the querier. A user in Frankfurt gets the EU load balancer's IP; a user in Virginia gets us-east's.
  • Geolocation — route by where the user is, not what's fastest. This is the tool for data-residency law (EU users' data must stay in the EU) and localized content.

Worked micro-example: a payments company runs active datacenters in Dublin and Oregon. Latency-based routing sends each user to the nearer one (~20 ms instead of ~140 ms cross-Atlantic — recall Lesson 0001's numbers). If Dublin fails, Route 53's health-checked failover flips answers to Oregon — but users behind resolvers that cached the Dublin answer stay broken until TTL expiry. Hence the rule of thumb: DNS routing for geography and coarse traffic shaping; load balancers for fast failover. Keep TTLs short (30–60 s) on records you may need to move, and accept the extra lookup traffic as the premium on that insurance.

4 · CDN: don't make the request travel at all

A CDN (content delivery network) beats the 150 ms ocean crossing by not crossing it: static content is copied to hundreds of edge servers (points of presence) and served from the one nearest the user. Two models for how content reaches the edge (Primer §8, Cloudflare: what is a CDN):

Push CDNPull CDN
Content arrivesyou upload it at release timeedge fetches from origin on first miss
First requestfast (already there)slow (miss + origin fetch)
Fitssmall, rarely-changing catalogslong tails; anything with unpredictable popularity
You pay forstorage everywhere, wanted or notorigin egress on misses; some stale-until-TTL content

What belongs on a CDN: images, JS/CSS bundles, fonts, video segments, downloadable files — anything static and shared between users. Netflix pushed this idea to its logical end with Open Connect, its private CDN with cache boxes installed inside ISPs, because video is nearly all of its bytes. What doesn't: personalized pages, cart contents, anything that changes per-request. (Modern CDNs also proxy dynamic traffic for DDoS absorption and TLS termination at the edge, but the caching win is for static content.)

The costs interviewers like to hear acknowledged:

  • Egress fees. CDNs bill per GB delivered. An e-commerce site pushing 2 MB of images per page view at 10 M views/day ships ~20 TB/day; at ~$0.05/GB that's ~$1,000/day. Cheaper than building 300 global POPs — but a real line item that image compression directly reduces.
  • Invalidation is the hard part. You shipped a broken JS bundle and it's cached at 300 edges with a 24 h TTL. Purge APIs are slow and unreliable at the worst moments. The professional pattern is versioned URLs (app.9f3ab2.js): deploy new HTML referencing a new asset name, cache assets effectively forever, and never purge.
  • A cold cache is a thundering herd. After a mass purge (or a new region coming online), thousands of edge misses hit your origin at once. Origins must be sized for miss storms, not the average day.

5 · Load balancer: L4 vs L7

The load balancer presents one address (a VIP — virtual IP) and spreads incoming requests across a pool of identical servers. The first design decision is which network layer it operates at (Primer §9, AWS ELB feature matrix):

L4 (transport layer)L7 (application layer)
Seessource/dest IP and port; forwards packetsthe full HTTP request: path, headers, cookies
Work per requestminimal — NAT-style rewritingterminates TCP and TLS, parses HTTP, opens a second connection
Capacitymillions of connections, near line-rateless per box — you pay CPU for the parsing
Can dospread TCP flowsroute /api vs /static to different pools, cookie stickiness, per-path retries, header rewrites
Real productsAWS NLB, LVS, HAProxy (TCP mode)AWS ALB, Nginx, HAProxy (HTTP mode), Envoy

The trade is performance vs flexibility. Large sites often stack them: an L4 tier at the front (cheap, absorbs floods, spreads across the L7 tier) and an L7 tier behind it (smart routing to services). When asked "L4 or L7?", the winning answer names the trade and picks per workload — payments API with path-based routing and retries: L7; raw TCP throughput for a database proxy or game servers: L4.

6 · Algorithms, stickiness, and why stateless wins

Given a healthy pool, which backend gets the next request?

  • Round-robin — next server in the cycle. Fair by count; assumes requests cost roughly the same.
  • Weighted round-robin — bigger machines get proportionally more. Fair by capacity.
  • Least-connections — pick the server with the fewest in-flight requests. Fair by current load; the self-correcting choice when request costs vary (one slow report query shouldn't starve a server that's stuck with it).
  • IP-hash / sticky sessions — hash the client's IP (or set a cookie at L7) so the same client always lands on the same server, and in-memory session state survives across requests.

Stickiness deserves its own warning label, because it is how small systems grow into unscalable ones. The classic story is an e-commerce shopping cart kept in app-server memory: sticky routing makes it work — until a server is redeployed and every cart on it evaporates mid-purchase, or one corporate NAT (thousands of users, one IP) hashes onto a single unlucky server. Stickiness makes servers non-interchangeable, and interchangeability is the entire premise of horizontal scaling.

The grown-up pattern: keep app servers stateless and externalize sessions to a shared store — Redis holds the cart, keyed by a session ID in the user's cookie. Now any server can serve any request, deploys and autoscaling are non-events, and a server death loses nothing. Generalized, this is the shared-nothing prerequisite for horizontal scaling: no request may depend on local state (memory or disk) of one particular box. Uploaded files go to object storage, sessions to Redis, data to the database — the app tier becomes disposable, which is exactly what you want it to be. (Primer: horizontal scaling)

7 · Health checks, failover, and who balances the balancer

The balancer earns its keep the moment a backend dies. It learns about deaths via health checks — typically an HTTP GET /health every few seconds. After N consecutive failures a backend leaves the rotation; a passing check readmits it. Two tunables, one trade:

detection time ≈ check interval × failure threshold
   interval 5 s × threshold 3  →  up to ~15 s of errors routed to a corpse
   interval 1 s × threshold 2  →  ~2 s detection, but one GC pause can
                                  eject a healthy server (false positive)

There is no free setting — over-eager checks cause flapping (servers bouncing in and out of rotation, each ejection dumping extra load on survivors, which then slow down and get ejected too). Production systems pair fast checks with damping: require several passes before readmission, and never eject below a minimum pool size. Good L7 balancers also do passive detection — a refused connection counts as evidence without waiting for the next probe — and retry the failed request on another backend so the client never sees the death.

Now the uncomfortable question your interviewer is waiting to ask: the load balancer is one box — isn't it a single point of failure? Yes, and the standard answer is an LB pair: two balancers, active–passive, exchanging heartbeats; when the active dies, the passive claims the shared floating IP (via VRRP — keepalived is the classic tool) and traffic continues within seconds. Alternatively active–active pairs behind round-robin DNS. Cloud load balancers (ALB/NLB) are pairs-and-then-some internally — that's what you're paying for. The one-sentence flex in an interview: "the LB is an active–passive pair with a floating IP, so it's not a SPOF."

8 · Reverse proxy: the balancer's quieter sibling

A reverse proxy is a server that fronts your app servers and represents them to the world — a load balancer is just a reverse proxy that happens to balance. Even a single-server deployment wants one (Primer §10, Nginx/F5: reverse proxy), because it centralizes the plumbing your app shouldn't do:

  • TLS termination — certificates live in one place; backends speak plain HTTP on a private network.
  • Compression and static files — Nginx serves /static from disk and gzips responses far more efficiently than your application runtime.
  • Rate limiting and request filtering — drop abusive or malformed traffic before it costs you app-server capacity.
  • A stable public face — backends can be renamed, re-addressed, and redeployed invisibly.

The disadvantage list mirrors the load balancer's: one more hop, one more component to configure, and a SPOF unless paired. In practice the edge layers collapse into few boxes: Cloudflare plays CDN + DNS + L7 proxy at the global edge; an ALB or an Nginx/HAProxy pair plays balancer + reverse proxy at your datacenter's front door.

9 · Worked example: sizing the edge for a flash sale

Tie it together with Lesson 0002-style arithmetic. An e-commerce site expects a flash sale peaking at 50,000 RPS. Page mix: 80% of requests are static (images, JS, CSS), 20% dynamic (search, cart, checkout).

CDN offload:   50,000 × 0.80 = 40,000 RPS die at the edge (pull CDN, long-TTL assets)
Origin sees:   50,000 × 0.20 = 10,000 RPS of dynamic traffic
App servers:   one server ≈ 1,000 RPS  →  10 needed  →  run 13 (~30% headroom)
Balancer:      L7 (route /checkout to its own beefier pool; cookie for cart A/B)
               one modern L7 box ≈ tens of thousands RPS → a pair suffices
Health:        interval 2 s × threshold 2 → ≤4 s detection; passive checks + retry
               hide most failures from users entirely
DNS:           TTL 60 s on shop.example.com, weighted 5% to the canary pool

Note what made this easy: because the app tier is stateless (carts in Redis), "add three more servers mid-sale" is a one-line autoscaling event, not a data-migration project. Every layer of this lesson appears in five lines of arithmetic — this is exactly the shape of a strong interview answer.

10 · One AI note: when request costs vary 100×

Everything above assumed requests cost roughly the same, which is why round-robin is usually fine. LLM inference is the extreme counterexample: one request generates 100 tokens, the next 10,000 — a 100× spread hiding behind identical-looking HTTP calls. Round-robin equalizes counts, so one unlucky GPU server accumulates the long generations and its queue — and tail latency — explodes while neighbors idle. Inference gateways therefore use least-outstanding-requests (least-connections' close cousin) or deeper queue-depth signals from the serving engine. Same principle as the slow-report-query example in section 6, dialed to 100×. We design this properly in Phase 4; the lab below lets you watch it happen today.

11 · Interview talking points

How to deploy this lesson in a live design round:

  • Draw the edge in order, one sentence per box: "DNS resolves and geo-routes; CDN serves static assets from the edge; an L7 balancer pair spreads dynamic traffic over stateless app servers." Thirty seconds, enormous signal.
  • Name the L4/L7 trade before picking. "L4 is faster, L7 can read the request — I want path-based routing and retries here, so L7." Never just name a product.
  • Say "stateless, sessions in Redis" before you scale. Interviewers deduct quietly when candidates add servers to a stateful tier and hope.
  • Volunteer the failure story unprompted: health checks with a detection-time number, LB as a pair, and "DNS failover is slow because of TTL caching — it's my backstop, not my mechanism."
  • Keep three numbers loaded: ~150 ms cross-ocean RTT (why CDNs exist), CDN offload of 70–90% for content-heavy sites, and detection time = interval × threshold.
  • Know the traps by name: sticky sessions (kills interchangeability), cache invalidation (version URLs instead), thundering herd on cold caches, health-check flapping, and the corporate-NAT hot spot under IP-hash.
