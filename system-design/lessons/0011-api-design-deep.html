API Design, Deeply

An API is a promise you keep for years. Semantics, versioning, pagination, and idempotency are where senior candidates separate — because they are exactly the places where a lazy choice becomes a double charge, a broken integration, or a double-shipped order.

Mission tie. Every Anthropic design prompt eventually surfaces an API: "design the public interface for this service" — and the follow-ups are always the same: "what happens when the client times out and retries that POST?" and "how do you ship a breaking change to 10,000 integrations?" Candidates who sketch endpoints keep talking; candidates who name idempotency keys, cursor pagination, and a deprecation policy get the senior signal. The classic systems where these ideas earned their names — Stripe payments, GitHub's API, e-commerce checkout — are this lesson's material. (Exponent's 2026 Anthropic guide)

1 · REST semantics done right

REST's real content is not "pretty URLs" — it is the uniform contract of HTTP methods, defined in RFC 9110. Two properties do all the work: safe (no server state change — free to cache and prefetch) and idempotent (N identical requests ≡ 1 — free to retry). Every retry policy, cache, and proxy on the internet is built on this table; violate it and they all break silently.

MethodSafeIdempotentContract
GETyesyesRead a resource; never mutate on GET (crawlers will "click" it)
PUTnoyesFull replace at a known URI; repeating it lands the same state
DELETEnoyesRemove; a second DELETE may 404, but state is unchanged — still idempotent
PATCHnono*Partial update; idempotent only if the patch itself is (set vs increment)
POSTnonoCreate / process; retrying may create twice — §4 exists because of this row

Status codes are contract too — the coarse-grained ones every client branches on: 200 OK, 201 Created (return a Location), 202 Accepted (async — job started, poll for it), 204 No Content (DELETE's natural reply). Then the pairs interviewers probe: 400 vs 422 (malformed syntax vs well-formed-but-semantically-invalid), 401 vs 403 (unauthenticated — who are you? — vs unauthorized — you may not), 409 Conflict (version/state clash, the optimistic-locking code), 429 Too Many Requests, and 503 + Retry-After (overloaded, and telling clients when to come back instead of letting them decide).

2 · Designing the resource model

Resources are nouns; the method supplies the verb. POST /orders, not POST /createOrder — the moment verbs leak into paths, every operation becomes bespoke and nothing composes. Conventions that survive years (Stripe on payment API design, Microsoft REST API Guidelines):

  • Nest one level, max two. /customers/{id}/orders reads well; /customers/{id}/orders/{id}/items/{id}/discounts welds the URL to today's data model. Deep sub-resources get promoted to top level with a filter: /order-items?order_id=….
  • Filtering and sorting are query params, not new endpoints: GET /orders?status=shipped&sort=-created. One list endpoint with parameters beats five near-duplicate endpoints drifting apart.
  • PATCH vs PUT. PUT replaces the whole representation (client must send every field — old clients silently erase new fields, a classic outage). PATCH sends only changes. Stripe-style APIs use partial update semantics almost everywhere for exactly that forward-compatibility reason.
  • Expandable objects over deep nesting in responses: return customer: "cus_123" by default and inflate it only when the caller asks (?expand=customer) — Stripe's pattern for keeping payloads small without a second endpoint.

3 · Pagination, deeply: offset vs cursor

Worked example — the arithmetic interviewers want to hear. ORDER BY created DESC LIMIT 20 OFFSET 1000000 forces the database to walk and discard 1,000,000 rows before returning 20 — the cost of page N grows linearly with N. At ~1M rows/s of index traversal, page 50,001 costs ~1 second of DB time per request; page 1 costs microseconds. A cursor query — WHERE (created, id) < (:cursor) ORDER BY created DESC, id DESC LIMIT 20 — is a B-tree seek: 3–4 page reads, sub-millisecond, regardless of depth (Use The Index, Luke: No Offset).

Worse than slow: offset is unstable under writes. Offsets are positions, not identities — an insert at the head of a newest-first feed pushes every row down one, so page 2 re-shows page 1's last row (duplicate processing) and deletions make rows skip pages entirely (silent loss). Slack rebuilt its API pagination on cursors for exactly these reasons (Evolving API Pagination at Slack); Stripe's list endpoints are cursor-based (starting_after) throughout (Stripe pagination docs).

Offset (?offset=100&limit=20)Cursor (?starting_after=ord_123)
Cost of deep pagesO(offset) — scans and discardsO(log N) — index seek
Stability under concurrent writesDrifts: duplicates and skipsStable: position pinned to a row
Jump to page 37TrivialNot supported (usually fine)
Cursor contents—Opaque token (encode sort keys; clients must not parse it — you want freedom to change it)
Who uses itAdmin UIs, small static listsStripe, Slack, GitHub (Link headers, docs)

4 · Idempotency keys: the double-charge problem

The scenario that decides payment-API interviews: client calls POST /payments; the server charges the card; the response is lost (timeout, connection reset). The client cannot distinguish "never happened" from "happened, confirmation lost" — so it retries, and the customer is charged twice. Retrying is correct client behavior; POST is simply not idempotent, so the retry is not safe.

Stripe's fix (idempotent requests docs, idempotency blog post): the client generates a unique Idempotency-Key header (a UUID) per logical operation and reuses it on every retry. The server keeps a key store: first time a key arrives, execute and save the response under the key; any later request with the same key replays the stored response without re-executing. Keys expire on a TTL (Stripe prunes after 24 h) — the TTL must exceed the longest plausible retry window, or a late retry finds the key gone and executes again (you will break this yourself in the lab, exercise E2). Result: POST becomes effectively idempotent, and safe retries become a documented client duty.

Worked numbers. 30% response loss, up to 3 retries: expected executions per order = 1 + 0.3 + 0.3² + 0.3³ ≈ 1.42 — a naive endpoint double-charges ~42% of orders under that (admittedly brutal) network. At 100k charges/day that is ~42,000 wrong ledger entries daily. The lab reproduces this: ~90 doubles on 200 orders naive, 0 with keys.

5 · Versioning: breaking changes without breaking clients

Non-breaking changes — adding a field, adding an endpoint, adding an optional param — need no version at all, provided clients follow the must-ignore rule (tolerate unknown fields). Breaking changes — removing/renaming fields, changing types or semantics — need a strategy:

StrategyShapeWinsPays
URL version/v1/orders → /v2/ordersVisible, cacheable, trivially routableCoarse: one flag for the whole API; "v2 forever" migrations stall
Header versionStripe-Version: 2024-06-20Fine-grained; URL stays stableInvisible in logs/browsers; needs tooling
Additive-onlyNever break; only addNo version machinery at allDeprecated fields accumulate forever

Stripe's date-based model is the gold standard (Stripe: APIs as infrastructure): every account is pinned to the API version dated when it first went live; each breaking change ships as a new date with a compatibility layer translating old-pinned requests forward. Integrations written in 2016 still work untouched. Whatever you choose, publish a deprecation policy: announce, emit Deprecation/Sunset headers, give a real window (GitHub gives months and emails affected callers), then remove.

6 · Webhooks: push, verified, and at-least-once

Polling GET /payments/{id} every 2 s to learn a charge settled wastes both sides; a webhook pushes the event to the consumer's URL instead. The contract every serious provider documents (Stripe webhooks, GitHub webhook validation):

  • Signature verification. The provider signs the payload with a shared secret — HMAC-SHA256 in a header (Stripe-Signature, X-Hub-Signature-256). Consumers must verify with a constant-time compare, or anyone who finds the URL can forge "payment succeeded".
  • At-least-once delivery, retried with backoff. Non-2xx or timeout → the provider re-delivers, backing off over hours or days. Therefore the consumer will see duplicates and must dedupe on the event id — consumer-side idempotency, §4's lesson wearing a different hat.
  • Ordering is not guaranteed. invoice.paid can arrive before invoice.created. Consumers treat events as triggers to re-fetch current state, not as a replayable log.
  • Respond fast, process async: ack 2xx immediately, enqueue the real work — slow handlers get timed out and re-delivered, creating their own retry storm.

Interview shorthand: public surface → REST; service-to-service inside the mesh → gRPC; a flexible read layer for many client shapes → GraphQL. They compose — GitHub ships REST and GraphQL publicly while plenty of shops run gRPC behind both.

8 · Rate limits and errors are API surface too

How you fail is part of the promise. Return 429 with Retry-After and quota headers (X-RateLimit-Remaining) so well-behaved clients back off instead of hammering — the limiter algorithms themselves (token bucket et al.) were Lesson 0009's material; here the point is the contract (Stripe: scaling rate limiters). Error bodies deserve a schema: RFC 9457 problem+json standardizes type, title, detail, plus a machine-readable code and a request id for support. Never leak internals — stack traces, SQL fragments, internal hostnames in a 500 are both a debugging crutch and a security gift.

9 · The failure modes that decide interviews

  • The silent breaking change. A field renamed "harmlessly" in a Friday deploy; 10,000 integrations parse null on Monday. No version pin, no deprecation window, no contract tests against real consumer traffic. Fix: consumer-driven contract tests in CI + §5's machinery.
  • Retry storm on a non-idempotent endpoint. Latency spike → clients time out → all retry POST → each retry does the work again and adds load, deepening the spike. Idempotency keys make retries safe; backoff + jitter and a 429/503 with Retry-After make them polite. You need both.
  • Webhook consumer without dedup. At-least-once delivery meets a handler that ships an order per event: redelivery after a slow ack → two shipments of one order. Every provider warns about this in the docs nobody reads (Stripe webhooks: "handle duplicate events").
  • Pagination drift double-processing orders. A nightly batch walks /orders?offset=… while checkout keeps inserting: rows shift, page N re-reads page N−1's tail, and the fulfillment job ships some orders twice while never reaching the oldest ones. The lab makes this happen on your machine — 44 duplicated and 44 missed rows out of 120.

10 · One AI note, then back to fundamentals

An inference API bends REST in three places. Streaming: token-by-token output means one request → many response chunks, so the response becomes SSE (server-sent events) — status code and headers commit before generation finishes, so mid-stream failures arrive as in-band error events, not status codes. Idempotency keys on generation requests: a retried non-idempotent POST here double-bills GPU-minutes instead of double-charging a card — same §4 fix, pricier mistake. Token-metered rate limits: quotas are tokens/min, not requests/min, so 429 handling must read usage headers, not just count calls. Same fundamentals, new units — and that is the whole note.

11 · Interview talking points

How to deploy this lesson live, in order:

  1. Sketch the resource model first — nouns, one nesting level, list endpoints with filter params — and say the conventions out loud (§2). It signals you have shipped a public API before.
  2. Volunteer the retry question before it is asked: "POST /payments isn't idempotent, so timeouts + retries double-charge; I'll require an Idempotency-Key and keep a 24 h key store that replays stored responses" (§4). This is the single highest-value sentence in the lesson.
  3. Pick cursor pagination and give the two reasons: deep-offset cost is O(offset), and offsets drift under concurrent writes (§3). Mention the cursor is opaque.
  4. State a versioning + deprecation policy (§5): "additive changes unversioned; breaking changes behind date-pinned versions, Stripe-style, with Sunset headers and a migration window."
  5. If events leave your system, name the webhook contract (§6): HMAC signatures, at-least-once ⇒ consumer dedup, no ordering guarantee.
  6. Justify the protocol per audience (§7): REST outside, gRPC inside, GraphQL only if many client shapes aggregate many backends.
  7. Close with the error surface (§8): 429 + Retry-After, problem+json bodies, request ids, nothing internal leaked.

13 · Project step

Decision time for the LLM inference gateway: sketch its public API — three endpoints, the idempotency story, and the pagination choice — and defend it in two sentences. A defensible pick: POST /v1/generations (SSE streaming, required Idempotency-Key with a 24 h store so client retries never double-bill GPU time), GET /v1/generations/{id} (poll/refetch after webhook or stream drop), and GET /v1/generations?starting_after=… (opaque cursor, because usage dashboards page deep and generations insert constantly). Date-pinned versioning from day one, token-metered 429s with Retry-After. Write your own sketch — endpoints + idempotency + pagination, two sentences of defense — and bring it to the next session for critique.
