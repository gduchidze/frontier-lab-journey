Design Rep: API Rate Limiter

The classic warm-up prompt, executed as a full 4-step interview rep — five algorithms, one Redis race condition, and the accuracy-vs-latency trade-off that decides senior offers.

Mission tie. "Design a rate limiter" appears in the reported Anthropic question bank as a classic warm-up — often the opener before a harder prompt, which means it is graded on execution speed and crispness, not novelty. A senior candidate finishes the whole loop in 30–35 minutes with algorithm trade-offs quantified in bytes and a named failure policy for the backing store. This lesson is the reference rep for exactly that performance.

1 · The prompt, and what is actually being probed

The prompt is one sentence: "Design a rate limiter for a public API platform." Everything else is yours to structure. Behind the apparent simplicity, interviewers are probing four specific things:

  • Algorithm literacy. Can you name more than one algorithm, and — the real test — quantify their trade-offs in memory, burst behavior, and accuracy instead of reciting names?
  • Distributed-state instincts. The single-box version is trivial. The moment there are N gateway nodes, counting becomes a distributed-systems problem with races, hot keys, and a consistency dial. This is where the rep is won.
  • Failure policy. What happens when the counter store dies? "Fail open vs fail closed" is a one-line question that separates candidates who have operated systems from candidates who have read about them.
  • API manners. Do rejected callers get a response they can act on (429, Retry-After), or do you accidentally design a thundering herd?

Because it is a warm-up, pace matters: budget roughly 5 minutes to clarify, 4 to estimate, 8 for the high-level design, and spend the rest deep in algorithms and distributed counting. Driving fast and out loud is the deliverable.

2 · Step 1 — Clarify (5 min)

Scope it as a concrete business: a public e-commerce API platform (catalog, orders, search endpoints) selling access to third-party integrators. Write the contract down:

TypeRequirement
FunctionalLimit requests per API key (e.g. 600 req/min); different limits per plan tier and per endpoint; reject with 429 + informative headers; limits reconfigurable without redeploy
FunctionalDimensions: per-user (API key) primary; per-IP for unauthenticated traffic; per-endpoint for expensive routes (search vs product-read)
Non-functionalLimiter overhead p99 < 5 ms — it sits on every request; the guard must not become the bottleneck
Non-functionalAvailability ≥ that of the API itself; limiter failure must never take the API down
Non-functionalAccuracy: small transient over-admission tolerable (~1–5%); silent under-admission (throttling paying customers below their plan) is not

Out of scope, said aloud: billing/quotas (monthly caps are a different, DB-backed system), DDoS scrubbing at L3/L4 (edge concern), and authentication itself — we consume an already-verified API key. That last accuracy line is a deliberately senior move: it declares which direction of error is acceptable, and every later trade-off will lean on it.

3 · Step 2 — Estimate (4 min), every number spent

Platform: 10M registered API keys, ~2M active in the peak hour
Traffic:  500M API calls/day ÷ ~100k s/day ≈ 5k QPS avg, ×4 peak ≈ 20k QPS
  → one limiter check per request = 20k checks/s
  → one Redis node ≈ 100k simple ops/s → fits today; 5× growth eats it → plan sharding by key

Latency:  in-DC Redis RTT ≈ 0.5–1 ms; overhead budget p99 < 5 ms
  → at most ONE round trip per check → check+update must be a single atomic call (Lua)

Memory, token bucket: key ~40 B + state 16 B ≈ 60 B with overhead
  → 10M keys × 60 B ≈ 600 MB → the ENTIRE key space fits one Redis node

Memory, sliding window log @ 600 req/min: worst 600 stamps × 8 B ≈ 5 KB/key
  → 2M active keys × 5 KB ≈ 10 GB → ~16× more expensive → log rejected at this scale

Each number is spent immediately: 20k checks/s buys the sharding plan and rules out anything that scans state per check; one-RTT budget buys the Lua/atomic-call decision in section 6; 600 MB vs 10 GB is the arithmetic that will pick the algorithm in section 5 — say that sentence out loud when you get there, and the estimate step retroactively proves it wasn't ritual.

4 · Step 3 — High-level design (8 min)

Request path. Client → load balancer → API gateway → rate-limit middleware → backend services. The limiter consults a counter store (Redis) keyed by {key_id}:{endpoint_class}:{window}; allowed requests pass through, rejected ones return immediately — a rejection must cost microseconds, not a backend call.

The 429 contract. Rejections carry machine-readable pacing information:

HTTP/1.1 429 Too Many Requests
Retry-After: 23
X-RateLimit-Limit: 600
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1721741460

Successful responses carry X-RateLimit-Remaining too, so well-behaved clients throttle themselves before hitting the wall. This one header is quiet thundering-herd prevention (section 9).

Where the limiter sits — three options, signposted:

  • Gateway middleware — one enforcement point, one config surface; the gateway is already on every request path. Default choice.
  • Library in each service — no extra hop, but N languages × M services of duplicated logic and config drift. Rejected here.
  • Sidecar — per-pod local decisions with a shared store behind; buys the local-bucket model of section 7 at the cost of fleet-wide deployment complexity.

Choose gateway middleware because a platform product wants uniform enforcement and a single place to change rules — and note aloud that the sidecar becomes attractive exactly when the accuracy-vs-latency dial of section 7 turns.

5 · Deep dive A — the five algorithms, quantified

Token bucket~16 B (tokens, last-refill)Allows bursts up to bucket size — a configured featureNo — but rolling-window admission can hit capacity + refill ≈ 2× rateExact for its own model

Sliding window counter~24 B (two counters + id)Near-smooth; small approximation errorNo (smoothed away)Approximate — assumes uniform spread in previous window

The boundary spike deserves one concrete sentence in the interview: with a 600/min fixed window, a client can send 600 requests at 11:59:59 and 600 more at 12:00:01 — 1,200 requests in ~2 seconds, double the intended rate, all "legal." The sliding window counter fixes this for 8 more bytes by weighting the previous window: est = cur + prev × (1 − elapsed_fraction). Spend the section-3 arithmetic here: the log's 10 GB vs the counter's tens of MB is why real platforms (Cloudflare among them) ship the approximation and accept ~a-few-percent error.

6 · Deep dive B — distributed counters in Redis

With many gateway nodes, the counter must be shared — and the naive implementation has a race baked in:

count = INCR key          # request 1 crashes here →
IF count == 1: EXPIRE key window   # ...key now lives FOREVER
IF count > limit: reject

Two distinct problems. First, INCR-then-EXPIRE is not atomic: a crash or timeout between the two commands leaves an immortal counter that throttles the key until someone notices. Second, check-then-act across two round trips lets concurrent requests all read count < limit and all pass. The fix for both is making decision + update one atomic unit: a Lua script (Redis executes scripts atomically, nothing interleaves) that refills the bucket, checks, decrements, and sets TTL in one round trip — which also satisfies the one-RTT latency budget from section 3. Redis's own INCR documentation walks this exact rate-limiter pattern and its race.

Hot keys. Sharding by API key spreads load — until one enormous integrator does 5k RPS alone and all of it lands on one shard. Mitigations, in escalating order: dedicated shard for whale keys; split one logical limit into K sub-keys with limit/K each (cheap, slightly lumpy); or per-node local buckets — which is section 7's entire topic.

7 · Deep dive C — the accuracy dial: sync vs eventual

Every distributed limiter sits somewhere on one dial:

  • Synchronous global counting — every check is a Redis round trip. Accuracy: near-exact. Cost: +1 ms on every request, and Redis is on the critical path of the whole platform.
  • Local buckets + async sync — each gateway node admits from a local bucket and reconciles with the shared store every T ms. Checks become in-memory (microseconds). Cost: during the blind window between syncs, N nodes can each admit against a stale global count — transient over-admission bounded roughly by N × arrival-rate × T.

Here is where the Step-1 accuracy requirement gets spent, by name: we declared 1–5% transient over-admission acceptable and under-admission unacceptable — so local buckets with a short sync interval are not a hack, they are the requirement's direct consequence. Choose T from the bound: at 4 nodes and a key bursting at 400 req/s, T = 100 ms risks ~40 extra admits on a 100-limit — ~40% worst case for that key during a burst, a number your lab measures directly. Tighten T for sensitive endpoints, loosen it for cheap ones.

Stripe's published limiter design lands the same way — Redis-backed token buckets, deliberately tolerant of small inaccuracy in exchange for staying off the critical path's worst percentiles.

8 · Deep dive D — rules, config, and dimensions

Limits are policy, and policy changes weekly; the enforcement path cannot require a redeploy. Standard shape: rules live in a small config store (database or config service) — {plan_tier, endpoint_class} → {limit, window, burst} — and gateway nodes cache them in memory with a periodic refresh (tens of seconds is fine; rule changes are not latency-critical). A rules cache miss must default to some rule, never to an error.

Dimensions compose. A single request may be checked against several counters:

  • Per API key — the primary, plan-based limit.
  • Per IP — for unauthenticated endpoints (login, signup) where there is no key yet; this is also the abuse-control dimension.
  • Per endpoint class — search costs 20× a product read, so it gets its own tighter counter under the same key.

Evaluate cheapest-reject-first, and count the cost honestly: three dimensions = three counter updates per request. With Lua, batch them into one script call — one round trip regardless of dimensions — or the 5 ms budget dies by a thousand cuts. Mention the layering aloud: gateway limiter for platform fairness, plus a coarse per-IP layer at the edge in front of it; two cheap layers beat one clever one.

9 · Failure modes and monitoring

FailureWhat happensDesign answer
Redis downEvery check errors — the limiter can take down the API it guardsFail open for a commerce API (dropping all customer traffic to protect a guard rail is backwards), with a local fallback limiter at a conservative rate + loud alert. Fail closed only where the limiter IS the security boundary (auth attempts). State the choice unprompted.
Thundering herd of retriesA window resets, or an outage ends — and every throttled client retries in the same second, re-creating the spike foreverRetry-After with jittered values so retries spread; clients told to use exponential backoff + jitter (your lab-0008 result, now on the server side); token bucket refill smooths re-entry vs fixed window's cliff
Clock skewGateway nodes with drifting clocks disagree about window boundaries → limits enforced unevenlyUse the store's clock (Redis TIME inside the Lua script) as the single time authority; never trust N gateway clocks to agree

Monitoring: the page-a-human metrics are limiter-check p99 latency and store error rate; the dashboards are 429 rate per key/tier (a spike = one customer's broken retry loop) and over-admission drift (sampled comparison of admitted rate vs configured limit). Alert on 429 rate going to zero, too — a silently failed-open limiter looks exactly like a healthy quiet one.

10 · Interviewer twists and variants

  • "Make it a security control." The dial flips: per-IP and per-account limits on login/OTP endpoints exist to slow credential stuffing, so there you fail closed, keep limits low and windows long, and log rejections for the abuse team. Same machinery, opposite failure policy — saying that sentence wins the twist.
  • "Global limiting across datacenters." A synchronous planet-wide counter would put an 80+ ms cross-region RTT on every request — unacceptable. Real answer: per-region quotas (split the global limit by observed traffic share) + async reconciliation shifting the split, accepting transient global over-admission. This is section 7's dial at planetary scale.
  • "How much does accuracy cost?" The recap ladder: fixed window (16 B, 2× spikes) → sliding counter (24 B, ~few % error) → sliding log (5 KB, exact) → synchronous global (exact, +1 ms on everything). Accuracy is bought with memory, then with latency. Name the ladder, place the business on a rung, justify with a requirement.

AI flavor (one paragraph). LLM APIs meter tokens per minute, not just requests — one request may consume 50 or 50,000 tokens. Mechanically it is the same token bucket, refilled in token units, with one wrinkle: output cost is unknown at admission, so providers debit an estimate at request start and reconcile against actual usage at completion — the accuracy dial again, applied to cost.

11 · Interview talking points

  • Opening scope: "Rate limiting protects capacity and fairness in seconds-scale windows; monthly quotas and billing are a different system — scoping to the limiter."
  • Spending the memory number: "A sliding log at this scale is ~10 GB versus ~600 MB for buckets — that arithmetic, not preference, is why I pick token bucket with a sliding-window-counter fallback."
  • The atomicity line: "Naive INCR-then-EXPIRE has two failure modes — an immortal key on crash, and check-then-act races — so the check must be one atomic Lua call. That also keeps us at one round trip."
  • The dial: "Exact global counting and low latency are in direct tension; our requirements tolerate transient over-admission, so I take local buckets synced every ~50–100 ms and I can bound the error: N × rate × interval."
  • Unprompted failure policy: "Before you ask: Redis down means fail open with a conservative local fallback — for a commerce API the limiter must never be the outage. On auth endpoints I'd flip that."
  • Closing: "I'd page on check latency and store errors, dashboard 429s per tier, and alert on 429s flatlining at zero — a dead limiter looks quiet. With more time: multi-region quota splitting and per-key adaptive limits."
