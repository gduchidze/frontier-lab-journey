Lesson 0001 · Phase 1 — Distributed Systems Core

Latency, Throughput, and the Tail

The three words every design answer is graded in — and why the average is a lie.

Mission tie. Anthropic's system design rounds open with prompts like "design a token-generation service that handles 100,000 requests per second." That sentence already contains a throughput target — and your first move as a candidate is to ask for the latency SLO. This lesson gives you those words. (Exponent's 2026 Anthropic guide)

1 · Two axes, not one

Latency is how long one request takes, measured from the caller's point of view (milliseconds). Throughput is how many requests the system completes per second (RPS/QPS — or tokens/second for LLMs). They are independent axes, and systems trade one for the other constantly. (Primer: latency vs throughput)

The canonical trade: batching. Grouping requests and processing them together raises throughput (less per-request overhead, better hardware utilization) but can raise latency (early requests wait for the batch to fill). Hold onto this — the entire discipline of LLM serving is a sophisticated answer to "how do we batch without wrecking latency," and it is the core of the Anthropic prompt. (Inside vLLM)

2 · Percentiles: the average is a lie

Report latency as a distribution, never an average. p50 (median) is the typical experience; p99 is the value that 99% of requests beat — and 1% suffer. The slow end is the tail.

Why obsess over 1%? Two reasons from DDIA ch. 1:

  • The tail hits your best customers. The slowest requests are often the heaviest accounts — the ones with the most data. Amazon famously observed ~1% sales loss per 100 ms of added latency, so they specify SLOs at p999.
  • Fan-out amplifies the tail. If one page load makes N backend calls, the chance of hitting at least one p99-slow call is 1 − 0.99N. At N = 100 that is ≈ 63%. Your "rare" tail becomes the median user experience. This is called tail latency amplification.
Vocabulary An SLO (Service Level Objective) is a target expressed in these terms: "p99 latency < 500 ms at 10k RPS." Designs are judged against SLOs, not vibes. Full definitions live in the Glossary.

3 · Numbers that anchor every estimate

You cannot sanity-check a design without a few physical constants. The full table is in the Estimation Kit — start memorizing these five:

OperationTime
Main memory reference100 ns
Random SSD read (4 KB)150 µs
Round trip inside a datacenter500 µs
HDD seek10 ms
Round trip across an ocean150 ms

Notice the cliff between memory (nanoseconds) and anything involving a network or disk (micro-to-milliseconds). Roughly 2,000 round trips/second fit inside a datacenter; only 6–7 cross an ocean. Most of system design is arranging work so the slow tiers are touched as rarely as possible.

4 · First GPU math (a taste of Phase 4)

The same physics governs LLM serving. Generating one token requires reading essentially all model weights from GPU memory. A 70B-parameter model in bf16 is ~140 GB; an H100 moves ~3.35 TB/s from HBM. So a single request's floor is:

140 GB ÷ 3.35 TB/s ≈ 42 ms per token  →  ~24 tokens/sec, one user,
                                          while the GPU's compute units sit mostly idle.

Decode is memory-bandwidth-bound — which is exactly why serving systems batch many users onto one weight-read. Latency vs throughput, again. We will build this into a full design in lessons 22–26.

5 · Project step

Our spine project is an LLM inference gateway. Its birth certificate is an SLO sheet. In a scratch note (or in your head, then check), write SLOs for it covering: TTFT p50 and p99, TPOT p50, sustained throughput in requests/sec and tokens/sec, and availability. (TTFT/TPOT are defined in the Glossary.) A defensible example: TTFT p99 < 1 s, TPOT p50 < 50 ms, 1,000 RPS sustained, 99.9% availability. There is no single right answer — the skill is stating targets before designing. Bring your sheet to the next session and your teacher will critique it.
