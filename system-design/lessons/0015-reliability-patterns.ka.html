გაკვეთილი 0015 · ფაზა 2 — პროგრამული არქიტექტურა და საიმედოობა · English version

საიმედოობის პატერნები

სისტემები იშვიათად კვდებიან თვითონ ჩავარდნისგან — ისინი საკუთარი დაცვებით კვდებიან. Timeout-ები, retry-ები, breaker-ები, bulkhead-ები, shedding — და როგორ ხდება თითოეული, არასწორად გამოყენებული, თავად ავარია.

მისიასთან კავშირი. Anthropic-ის დიზაინის ამოცანები ყოველთვის ერთსა და იმავე კითხვამდე მიდიან: „და რა ხდება, როცა ის dependency შენელდება?" ვინც ამბობს „დავამატოთ retry", ახლახან retry storm-ი (განმეორებების შტორმი) დააპროექტა; ვინც ასახელებს timeout budget-ს, retry budget-ს, breaker-ის ზღვრებს და shed პოლიტიკას — რიცხვებით — ის აღწერს სისტემას, რომელიც საკუთარ სამშაბათს გადაურჩება. ეს გაკვეთილი არის Release It!-ისა და AWS Builders' Library-ის ლექსიკა. (Exponent-ის 2026 Anthropic გზამკვლევი)

1 · კასკადური ჩავარდნის ანატომია

განაწილებული სისტემის ნაგულისხმევი ქცევა ნაწილობრივი ჩავარდნისას არის ერთი ნელი dependency-ის სრულ ავარიად გადაქცევა. კანონიკური თანმიმდევრობა Release It!-იდან: checkout სერვისი payments პროვაიდერს ეძახის, რომლის p99 ჩუმად 200 ms-დან 10 წამამდე ადის. არაფერი „ჩავარდნილა" — ყველა ზარი ბოლოს ბრუნდება. მაგრამ Little-ის კანონით 100 RPS × 10 წმ = 1,000 პარალელური მოთხოვნა 200-thread-იან pool-ზე. Pool წამებში ივსება; მოთხოვნები, რომლებიც payments-ს საერთოდ არ ეხებიან, რიგში დგებიან მათ უკან — და მთელი საიტი ისეთივე ნელია, როგორც მისი ყველაზე ნელი dependency. Latency გადამდებია. ორი დასკვნა: ნელი უარესია, ვიდრე მკვდარი (მკვდარი სწრაფად ვარდება და thread-ებს ათავისუფლებს; ნელი მძევლად იჭერს), და დაცვები ერთმანეთზე მოქმედებენ — ამიტომ §7 მთლიანად მათს ჩავარდნებზეა. (SRE წიგნი: Addressing cascading failures)

2 · Timeout-ები — ყველა remote ზარს აქვს

ურყევი წესი: ყველა remote ზარს აქვს timeout, ყოველთვის — ზარი timeout-ის გარეშე არის thread, რომელიც სხვის ავარიას აჩუქე (AWS Builders' Library). ორი რიცხვი: connect timeout (მოკლე) და request timeout — ეს უკანასკნელი dependency-ის გაზომილი p99-დან აირჩიე თავისუფალი მარაგით, და არა შეგრძნებით. Timeout-ები ჯაჭვის გასწვრივ უნდა იშლებოდეს (timeout budget — ბიუჯეტი):

HopTimeoutრატომ ეს რიცხვი
კლიენტი → edge2,000 msპროდუქტის დაპირება: checkout 2 წამში პასუხობს ან ბოდიშს იხდის
Edge → checkout სერვისი1,500 msEdge-ს 500 ms რჩება ერთი retry-სთვის ან სუფთა error გვერდისთვის
Checkout → payments პროვაიდერი800 msპროვაიდერის p99 250 ms-ია; 3× მარაგი, ისევ ბიუჯეტში
Checkout → inventory DB300 msp99 40 ms-ია; უფრო ნელი უკვე ანომალიაა

ინვარიანტი ხმამაღლა თქვი: შვილის timeout < მშობლის დარჩენილი ბიუჯეტი, ყველა დონეზე. 5-წამიანი timeout 2-წამიანის ქვეშ დეკორაციაა — მშობელი უკვე გათიშულია.

3 · Retry-ები სწორად

Retry არის ფსონი, რომ ჩავარდნა დროებითი იყო — და ის დამატებითი დატვირთვაა სისტემაზე, რომელმაც ახლახან გითხრა, რომ უჭირს. სამი წესი: (1) მხოლოდ idempotent (გამეორებადი) ოპერაციები და მხოლოდ transient (გარდამავალი) შეცდომები — payment capture-ის გამეორება ორმაგი ჩამოჭრაა; (2) exponential backoff სრული jitter-ით — უბრალო backoff (100, 200, 400 ms) ერთად ჩავარდნილ კლიენტებს სინქრონულ ტალღებად ტოვებს; სრული jitter — ძილი შემთხვევით [0, cap] შუალედში — მათ აშორებს, და AWS-ის სიმულაციამ აჩვენა, რომ ის ჯამურ სამუშაოს დაახლოებით ნახევრამდე ამცირებს; (3) retry budget — retry-ები ≤ 10% საბაზო ტრაფიკისა; ბიუჯეტი რომ ამოიწურება, ვარდები და აღარ იმეორებ.

ნამუშევარი მაგალითი — არითმეტიკა, რომელიც ინტერვიუერებს სმენა სურთ. Retry-ები ფენებს შორის მრავლდებიან: edge 3×, სერვისი 3×, მონაცემთა ფენა 3× — სრული ავარიისას ერთი მოთხოვნა ქვედა dependency-სთან 3 × 3 × 3 = 27 მცდელობა ხდება. 1,000 RPS მოთხოვნაზე მომაკვდავი dependency 27,000 მცდელობა/წმ-ს იღებს — 27-ჯერადი თვითმოწყობილი DDoS, ყველაზე ცუდ მომენტში. ლაბში 2 ფენა × 3 მცდელობაც კი, მხოლოდ 50% ჩავარდნაზე, დატვირთვას 3.3×-მდე წევს — და შტორმი ჩავარდნის ფანჯარას 3 წამით გადააცილებს, რადგან dependency-მ retry-ების შექმნილი ზვავიდან ამოძრომა უნდა მოასწროს. გამოსავალი: retry ერთ ფენაზე, ბიუჯეტით, წინ კი circuit breaker.

4 · Circuit breaker — სწრაფად დავარდი, ფრთხილად გასინჯე

Breaker-ი dependency-ის ზარებს ახვევს და ბოლო შედეგებს ითვლის; როცა ჩავარდნის წილი ზღვარს გადააჭარბებს, ის ირთვება და შემდეგი ზარები მყისიერად ვარდებიან, dependency-ს ხელს არ ახლებენ (Fowler: CircuitBreaker; პატერნი Release It!-მა დააკანონა). State machine, რომლის დახატვაც უნდა შეგეძლოს:

მდგომარეობაქცევა ყოველ ზარზეგადადის, როცა
Closed (ნორმალური)ზარი გადის; შედეგი მოძრავ ფანჯარაში იწერებაჩავარდნის წილი ≥ ზღვარი (მაგ. 50% 20 ზარზე) → Open
Open (ჩართული)მყისიერი უარი — ქსელი არ არის, ვარდნა მიკროწამებშიCooldown გავა (მაგ. 2 წმ) → Half-open
Half-open (სინჯავს)შეზღუდული probe (საცდელი ზარი) გადის; დანარჩენებზე უარიProbe წარმატებულია → Closed; ჩავარდა → Open

რას ყიდულობს fail-fast. Breaker-ის გარეშე ავარიისას ყოველი ზარი სრულ timeout-ს ჭამს: ლაბში 10-წამიანი ავარია 50 ზარი/წმ-ზე 1-წამიანი timeout-ით caller-ს 500 caller-წამი დაბლოკილი thread-დროის უჯდება. Breaker-ით ერთი ფანჯარა იხდის timeout-ს, შემდეგ ყველაფერი მიკროწამებში ვარდება (ლაბში: სულ 14 წამი — 36-ჯერ ნაკლები), dependency კი სიჩუმეს იღებს, რომ გამოჯანმრთელდეს. Half-open probe ელეგანტურობაა: სისტემა გამოჯანმრთელებას ერთი მოთხოვნის ფასად აღმოაჩენს და არა ჯოგით.

5 · Bulkhead-ები და cell-ები

Nygard-მა სახელი გემის ტიხრებიდან აიღო: ერთი დატბორილი განყოფილება გემს არ უნდა ძირავდეს. პროგრამულად: ყოველ dependency-ს (ან tenant-ს) თავისი შემოსაზღვრული pool — thread-ები, connection-ები — რომ §1-ის სცენარი გახდეს „payments-ის 30-thread-იანი pool ამოიწურა; browse და search, საკუთარ pool-ებზე, ვერც ამჩნევენ." მასშტაბზე bulkhead არქიტექტურა ხდება: cell-based არქიტექტურა მთელ სტეკს დამოუკიდებელ cell-ებად ჭრის, ხოლო shuffle sharding ყოველ მომხმარებელს node-ების შემთხვევით მცირე ქვესიმრავლეს ანიჭებს — ხმაურიანი მომხმარებელი მხოლოდ თავის კომბინაციას აფუჭებს, რომელსაც თითქმის არავინ იზიარებს ზუსტად (AWS Builders' Library: Shuffle sharding). Blast radius (დაზიანების რადიუსი) დიზაინის შესავალი მონაცემია და არა პოსტ-ინციდენტური მეტრიკა.

6 · Load shedding და graceful degradation

როცა შემოთავაზებული დატვირთვა სიმძლავრეს აჭარბებს, ვიღაცას მაინც ვერ მოემსახურები — არჩევანი მხოლოდ ვის და როგორ არის. არასწორი პასუხი ნაგულისხმევია: ყველაფერი რიგში ჩააყენე და latency-მ იზარდოს, სანამ ყველა პასუხი თავისი მომლოდინის წასვლის შემდეგ არ მოვა — 100% შრომა, 0% goodput. სწორი პასუხი აშკარაა: უარი admission-ზე (მიღებისას), იაფად და ადრე, 503 + Retry-After-ით (AWS: Using load shedding to avoid overload; SRE: Handling overload). გაიხსენე utilization–latency მრუდი: ~80–90%-ის მერე ლოდინი ვერტიკალურად ადის — admission control 0.9C-ზე ამ კლდეზე აძრომაზე უარია. ლაბში 2× სიმძლავრეზე აწევისას დაუცველი რიგის p99 5.8 წამია და იზრდება; admission control 816 მოთხოვნას აგდებს და გადარჩენილებს ბრტყელ latency-ს უნარჩუნებს.

კლასიმაგალითებიგადატვირთვისას
კრიტიკულიCheckout, payment capture, health check-ებიბოლოს დააგდე — health check-ების დაგდება LB-ს ჯანმრთელი სერვერების მოკვეთას აიძულებს
სტანდარტულიპროდუქტის გვერდები, ძებნაDegradable-ების მერე; სადაც შეიძლება, ძველი cache მიაწოდე
Degradableრეკომენდაციები, analytics, prefetchპირველი დააგდე — ან ფუნქცია მთლიანად გამორთე

Graceful degradation მომხმარებლების ნაცვლად ფუნქციების დაგდებაა: flash sale-ზე რეკომენდაციები მოაშორე, გუშინდელი ბესტსელერების სია cache-იდან მიაწოდე, checkout კი ხელუხლებელი დატოვე. Brownout (შებინდება) > blackout (ჩაქრობა). და იგივე კანონი ყველა შიდა რიგზე: შემოუსაზღვრავი queue latency-ის ბომბია (AWS: Avoiding insurmountable queue backlogs) — ეს გაკვეთილი 0008-ის ლაბში გაზომე.

7 · Failure mode-ები — როცა დაცვები თავს გვესხმიან

  • მოციმციმე (flapping) breaker. ზედმეტად პატარა ფანჯარა (5 ზარი) breaker-ს ხმაურზე ართავს, იღბლიან probe-ზე კეტავს და ისევ ართავს — caller-ები შეცდომებს დამაბნეველ პორციებად იღებენ, dependency კი სუფთა სიჩუმეს ვერასდროს. ლაბში თვითონ გამოიწვევ (E2).
  • გლობალურად გაზიარებული retry budget. ერთი ბიუჯეტი ყველა tenant-ზე ნიშნავს, რომ ერთი ცუდი tenant-ის ჩავარდნები მთელ ბიუჯეტს წვავენ და retry არავის რჩება. ბიუჯეტებსაც bulkhead-ები სჭირდებათ: per-tenant, per-dependency.
  • Fallback, რომელიც სწორედ იმ დღეს ტყდება. Fallback გზა იშვიათად ეშვება, ამიტომ გაუტესტავი და მიტოვებულია — და ზუსტად მაშინ ირთვება, როცა სისტემა ისედაც დეგრადირებულია. ეს არის AWS-ის avoiding fallback არგუმენტის გული: ამჯობინე მუდამ ვარჯიშში მყოფი მექანიზმები (shedding, throttling) მეორე კოდის გზას, რომელმაც ცივად, ხანძრის დროს უნდა იმუშაოს.
  • დეკორატიული timeout. ქვედა timeout, რომელიც caller-ის დარჩენილ ბიუჯეტზე გრძელია, არაფერს იცავს — caller-მა ყურმილი უკვე დაკიდა.
  • იაფი მოთხოვნების shedding. Shedder-ი, რომელიც შემთხვევით აგდებს, $0-იან analytics beacon-სა და $400-იან checkout-ს თანაბარი ენთუზიაზმით აგდებს — ან, უარესი, health check-ებს, და LB ჯანმრთელ სერვერებს კვეთს. პრიორიტეტი shedding-ის მთელი აზრია.
LLM შტრიხი ყველაფერი ზემოთქმული GPU serving-ზე გადადის, ფასები კი იზრდება: GPU worker-ის „thread pool" მისი batch queue-ა, ამიტომ admission control რიგის წინ დგას (უარი, როცა queue depth × service time SLO-ს აჭარბებს); per-tenant bulkhead-ები ერთი მომხმარებლის 100k-token სამუშაოებს ყველას chat ტრაფიკის შიმშილობისგან აჩერებენ; brownout-სვლა კი პატარა მოდელზე დეგრადაციაა — გადატვირთვისას სტანდარტული ტრაფიკი იაფ მოდელზე გადადის, flagship კი premium-ს რჩება. იგივე პატერნები, უფრო ძვირი miss-ები.

8 · ლაბი — თვითონ ჩართე breaker-ი

პრაქტიკა (ქვიზამდე გააკეთე) ყველაფერი ზემოთქმული ახლა შემოწმებადია: labs/lab-0015-circuit-breaker/-ში python3 reliability_lab.py უშვებს სამ ექსპერიმენტს — retry-ის გამრავლება (გულუბრყვილო ფენოვანი retry-ები vs 10%-იანი retry budget 50%-ით ჩავარდნად dependency-ზე, შტორმით, რომელიც თავის ავარიას გადააცილებს), სრული closed/open/half-open breaker-ი 10-წამიან ავარიაზე (უყურე, როგორ აქცევს fail-fast 500 დაკარგულ caller-წამს 14-ად) და load shedding 2× სიმძლავრეზე (დაუცველი რიგი vs admission control, Little-ის კანონი ფასებს ითვლის). შემდეგ README-ს E1–E4: იწინასწარმეტყველე, შეცვალე კონფიგი, გაუშვი თავიდან, შეადარე. მცდარი პროგნოზები არის დავალების შედეგი — მოიტანე ისინი შემდეგ სესიაზე.

9 · პროექტის ნაბიჯი

დაწერე საიმედოობის პოლიტიკა gateway→worker ზარებისთვის LLM inference პლატფორმაში: timeout, retry budget, breaker-ის ზღვრები, shed პოლიტიკა — და ორი წინადადებით დაიცავი. დასაცავი მაგალითი: request timeout = მოთხოვნილი max-tokens-ის p99 decode დროის 1.2×, ერთი retry მხოლოდ idempotent-უსაფრთხო ჩავარდნებზე 10%-იანი per-tenant ბიუჯეტით, breaker 50% ჩავარდნაზე 20 ზარიდან 5-წამიანი cooldown-ით თითო worker-ზე, და admission control batch queue-ს წინ, რომელიც ჯერ free-tier ტრაფიკს აგდებს 503 + Retry-After-ით. დაწერე შენი — ოთხივე რიცხვი, ორი წინადადება დაცვა — და მოიტანე შემდეგ სესიაზე გასარჩევად.
