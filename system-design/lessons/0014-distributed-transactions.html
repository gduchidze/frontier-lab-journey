Distributed Transactions & Coordination

Money moves across services without a global transaction — sagas, outbox, and why 2PC lost; plus consensus, leader election, and the fencing token that saves you.

Mission tie. Anthropic's design prompts love flows where one user action spans several services — "design checkout", "design a booking system" — because the follow-up is a scalpel: "payment succeeded but inventory reservation failed; now what?" Candidates who say "wrap it in a transaction" reveal they've never shipped one; candidates who name the saga, place the outbox, and explain why the lock needs a fencing token keep going. This lesson is the vocabulary of that follow-up, grounded in DDIA chapters 8–9 (Kleppmann, Designing Data-Intensive Applications; Exponent's Anthropic guide).

1 · The problem: one action, N services, no shared ACID

Checkout touches four services: order, payment, inventory, shipping. Each has its own database with lovely local ACID transactions — and no transaction spans them. Same with travel booking: flight + hotel + car are three companies' systems. The business action is atomic in the user's head ("I bought it") but physically it is four commits that can each fail independently.

DDIA ch8 gives the failure taxonomy (Kleppmann, "The Trouble with Distributed Systems"): a remote call can fail before the work happened, after the work happened but before the reply, or the network can just be slow — and from the caller's side these three are indistinguishable. A timeout does not mean "it didn't happen." In distributed systems the default answer to "did my payment go through?" is "maybe" — and every pattern in this lesson is a machine for converting "maybe" into a state you can act on.

Worked example — the arithmetic interviewers want to hear. Four sequential service calls, each 99.9% successful: P(all four succeed) = 0.9994 ≈ 99.6%. So ~0.4% of checkouts stop partway. At 200,000 orders/day that is ~800 orders/day in a half-finished state — charged but unreserved, reserved but unshipped. That is not an edge case; it is a queue you must design a consumer for. The question is never "will partial failure happen" but "who cleans it up, and how fast."

2 · Two-phase commit — and why it lost

The classical answer is 2PC (DDIA ch9, "Consistency and Consensus"). A coordinator runs two rounds: prepare — every participant durably promises "I can commit and will hold my locks until told"; then commit/abort — the coordinator writes its decision to its own log and tells everyone. Atomicity comes from that single decision point.

And that decision point is the flaw. If the coordinator crashes after participants prepared but before they hear the verdict, every participant is in doubt: it cannot commit (maybe the verdict was abort) and cannot abort (maybe it was commit) — so it sits there holding row locks, blocking every other transaction that touches those rows, until the coordinator returns. 2PC is a blocking protocol: it converts one node's failure into everyone's unavailability, which is precisely the coupling microservices exist to avoid. Add the operational reality — heterogeneous services rarely share an XA-capable transaction manager, and the slowest participant sets the pace for all — and you see why cross-service 2PC is effectively dead. It survives inside systems: XA in legacy Java estates, and internally in databases that must commit across their own partitions atomically.

3 · Sagas: a transaction rebuilt from local pieces

The replacement is the saga (Garcia-Molina & Salem, 1987; microservices.io): a sequence of local transactions T1…Tn, where each Ti has a compensating action Ci that semantically undoes it. If T3 fails, run C2, C1 in reverse order. No locks held across services, no coordinator freezing anyone — atomicity is traded for eventual all-or-nothing.

Checkout as a saga: create order → charge payment → reserve inventory → create shipment. If inventory reservation fails after payment charged: refund payment, cancel order. If shipment fails: release inventory, refund, cancel. Note the word semantic: a refund is not a rollback — the charge happened, the customer's statement shows both lines. And some steps have no undo at all: the confirmation email cannot be unsent; you send a correction. Design compensations as first-class business operations, not as database magic.

Two ways to run a saga:

ChoreographyOrchestration
Who decides next stepNo one — each service reacts to the previous service's eventsA central orchestrator sends commands, awaits replies
CouplingLow — services only know event namesOrchestrator knows the whole flow

4 · The outbox pattern: killing the dual write

Sagas (and event-driven anything) hide a trap: a service must commit to its DB and publish an event — two systems, no shared transaction. Commit-then-publish loses events when you crash in between (order exists, nobody ever hears); publish-then-commit invents events for orders that never committed. This is the dual-write problem, and no retry loop fixes a dead process.

The fix is the transactional outbox (microservices.io): write the event into an outbox table in the same local transaction as the order row — one commit, atomically both or neither. A separate relay then delivers: either polling the table or, better, tailing the DB's change stream via CDC — Debezium's outbox event router is the canonical implementation. The relay is at-least-once (it can crash after publishing, before marking the row sent, and republish) — so consumers must be idempotent, deduplicating on the event id. At-least-once delivery + idempotent consumers = effectively-once processing, the honest version of the "exactly-once" everyone asks for.

ApproachLost eventsDuplicatesCost
Dual write (commit, then publish)Yes — crash in the gapPossibleNone — that's the trap
2PC across DB + brokerNoNoBlocking, XA-capable broker required, availability coupling
Outbox + relay/CDCNo — event rides the commitYes — dedup at consumerExtra table, relay to operate, relay lag

Worked numbers. In the lab, a 2% crash rate in the commit-to-publish gap over 10,000 orders silently lost 201 events — 201 paid orders no downstream service ever heard about. The outbox version lost 0 and produced 100 duplicates, which consumer dedup reduced to exactly 10,000 applied. That trade — duplicates you can dedupe instead of losses you can't detect — is the whole pattern in one sentence.

5 · Isolation lost: sagas are not serializable

ACID gave you isolation; sagas give it up. Each Ti commits immediately, so other requests see intermediate states: the order exists but isn't paid; two concurrent sagas read the same stock level and both reserve it (a lost update); a saga later compensated was briefly visible as real (a dirty read that someone acted on). The countermeasures are application-level (microservices.io lists them as saga "countermeasures"):

  • Semantic lock: mark in-flight rows with a state — order: PENDING_PAYMENT, stock: RESERVED not SOLD — and make other transactions treat pending as "hands off." This is a lock encoded in data, visible and queryable.
  • Commutative operations: design steps so order doesn't matter — reserve(3) and reserve(2) as decrements commute; "set stock = 7" does not.
  • Reread / version check: before the pivotal step, re-verify the assumption (price, stock) that was read earlier — optimistic concurrency across the saga.

Interview habit: whenever you propose a saga, name the intermediate state a user might see and the flag that guards it. "The order shows as PENDING_PAYMENT until the saga completes" is the sentence that proves you know isolation was traded away.

6 · Consensus, at working altitude

Some facts must be agreed on by machines that can crash: who is the leader, what is the config, who holds the lock. That is consensus, and the reason quorums are majorities is arithmetic: any two majorities of the same cluster overlap in at least one node, so two conflicting decisions cannot both be ratified, and a decision survives any minority of failures (DDIA ch9).

Raft in five sentences (raft.github.io, Ongaro & Ousterhout): Time is divided into numbered terms, each with at most one leader. A node that stops hearing heartbeats becomes a candidate and wins an election by collecting a majority of votes for the new term. All writes flow through the leader, which appends them to its log and replicates to followers. An entry is committed once a majority holds it — the commit index — and committed entries are guaranteed to survive leader changes. Elections are constrained so only a candidate with an up-to-date log can win, which is how the guarantee holds.

What you actually use it for: the control plane — configuration, service membership, distributed locks, leader election — via etcd (Raft) or ZooKeeper (ZAB). You do not put your order data through consensus: a majority round-trip per write is the wrong price for the data path. Consensus stores hold megabytes that must never be wrong, not terabytes that must be fast.

7 · Distributed locks and the fencing token

"Only one worker may process this order at a time" — so you take a lock in Redis or etcd with a TTL (so a crashed holder can't lock forever). Now the famous failure (Kleppmann, "How to do distributed locking"): worker A acquires the lock, then stalls — GC pause, page fault, network hiccup — past the TTL. The lock expires; worker B acquires it and writes. Then A wakes up, still believing it holds the lock, and writes too. Two writers, corrupted resource, and no amount of TTL tuning fixes it, because a paused process cannot know it is paused.

Kleppmann's fix: the lock service hands out a fencing token — a number that increases monotonically with every grant (A gets 33, B gets 34) — and the resource (storage, DB, file server) records the highest token it has seen and rejects writes with a smaller one. A's stale write arrives carrying 33 < 34 → rejected. The safety check moves from the client (which can be arbitrarily delayed) to the resource (which observes actual write order). In the lab: with a 10 s TTL and pauses averaging 4 s, ~8% of trials outlived the lease — 799 corrupted writes without fencing, 0 with.

Redlock, one paragraph. Redis documents a multi-node locking algorithm, Redlock. Kleppmann argued it depends on timing assumptions (bounded clock drift and pauses) that real systems violate, and issues no fencing tokens — so it is fine for efficiency locks (avoiding duplicate work) but not for correctness locks; antirez disputed the analysis. The interview-safe position: state which of the two you need — if a stale holder can corrupt data, you need fencing at the resource regardless of which lock service you pick.

8 · Leader election: leases, split-brain, and humility

Leader election is the same lock, renewed: a candidate acquires a lease in etcd/ZooKeeper and must keep renewing it; if it dies, the lease lapses and someone else takes over — this is how Kubernetes controllers elect (K8s Leases), following the design of Google's Chubby lock service. The danger is split-brain: during the gap between a leader's last real heartbeat and its lease expiry, the old leader may still be acting while the new one starts — two leaders, briefly, is the normal failure mode of lease-based leadership, not an exotic one.

So the honest posture: leadership is an optimization, not a correctness guarantee. Use the leader to avoid duplicate work (one scheduler assigning jobs, one node compacting), but protect every externally visible effect the same way as §7 — fencing tokens, idempotent writes, or a consensus-backed commit. If your design is only correct when there is exactly one leader at all times, it is incorrect.

9 · Failure modes that decide interviews

  • Compensation that fails. The refund call itself times out — now the saga is stuck half-undone. Compensations must be idempotent and retried until success; whatever still won't succeed lands in a human task queue (in the lab: 10% compensation failure left 159 of 10,000 sagas stuck without retries, 0 with). Design that queue; it is part of the system.
  • Outbox relay lag. Events are never lost but arrive late — downstream sees the order minutes after payment. Monitor relay lag like replication lag; it is your freshness bound.
  • Duplicated compensations. An at-least-once command channel retries a refund whose ack was lost → double refund, unless the refund is keyed by an idempotency key. Same effectively-once rule as §4, applied to the undo path.
  • Two leaders during lease expiry. The paused old leader fires the same batch job as the new one; both write. Fencing tokens make the stale one bounce (§7); without them you get silently doubled side effects.
  • 2PC coordinator crash. If you did use 2PC in the checkout path, a coordinator crash leaves participants in-doubt holding locks — checkout freezes globally. This is the concrete story to tell when asked "why not 2PC."

10 · One AI note, then back to fundamentals

The LLM inference platform needs exactly two of these tools. Its batch-job scheduler runs as multiple replicas with etcd lease-based leader election — one leader assigns jobs to GPU workers, and because leadership is only an optimization (§8), job assignments carry fencing tokens so a paused ex-leader's stale assignment is rejected. And usage billing is a dual-write in disguise: the gateway commits a request log and must emit a billing event — outbox table in the gateway DB, CDC relay, idempotent billing consumer, because "maybe we billed them" is not a state a business survives. Same patterns, GPU-priced stakes. That's the note; §1–§9 are what the interview grades.

11 · Interview talking points

  1. Open with the partial-failure math: "four services at 99.9% each means ~0.4% of checkouts half-finish — hundreds a day at scale — so the design question is who cleans up, not whether."
  2. Dismiss 2PC with the right reason: not "it's slow" but "it blocks — a crashed coordinator leaves participants in-doubt holding locks."
  3. Name the saga and its style: "orchestrated saga: order → payment → inventory → shipment, compensations in reverse; orchestration over choreography because this flow is money-shaped and I want one queryable state machine."
  4. Place the outbox unprompted: "each step's event rides the local commit via an outbox table; the relay is at-least-once, so consumers dedupe on event id — effectively-once."
  5. Admit the isolation loss: "orders sit in PENDING_PAYMENT while the saga runs; nothing downstream treats pending as sold." Unprompted intermediate states are a strong-hire signal.
  6. When locks come up, say fencing before they do: "TTL locks fail under pauses, so the store checks a monotonic fencing token — the resource, not the client, is the last line of defense."
  7. Scope consensus correctly: "etcd for leader election and config — control plane only; consensus never sits on the order data path."

13 · Project step

Design the "provision fine-tune job" flow for the LLM platform as a checkout-shaped saga: list the steps, each step's compensation, and where the outbox sits — then defend it in two sentences. A defensible pick: orchestrated saga — create job record → reserve GPU quota → charge billing hold → enqueue to scheduler, compensating with release-quota / void-hold / mark-job-failed in reverse, every step's event written through an outbox in that service's DB. Orchestration because the flow touches money and quota, and the outbox because a job that exists but was never announced to billing is the exact dual-write hole the lab priced. Write your own two sentences — steps + compensations + outbox placement — and bring them to the next session for critique.
