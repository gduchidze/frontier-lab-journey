სენიორის კოდის ხელობა

API კონტრაქტები, იდემპოტენტურობა, ტესტირების სტრატეგია, პროფაილინგი — ოთხი ჩვევა, რომელიც „მუშაობს"-ს „პროდაქშენში გადარჩება"-სგან განასხვავებს.

მისიასთან კავშირი. Anthropic-ის კოდირების რაუნდები რეალისტური სამუშაო ამოცანებია და არა leetcode: ააწყვე ან გაასწორე ნამდვილი კომპონენტი დროის ლიმიტში — სუფთა კოდი და ტესტირების ინსტინქტები ღიად ფასდება. სისტემური დიზაინის რაუნდში კი „როგორ აიცილებ ორმაგ ჩამოჭრას retry-ზე?" კლასიკური დამატებითი კითხვაა ნებისმიერ გადახდის ან API ამოცანაზე — პასუხი ამ გაკვეთილის idempotency-key რეცეპტია, პირდაპირ Stripe-ის საინჟინრო ბლოგიდან და Google SRE წიგნიდან.
ჯერ ლაბი (ADHD წესი) სანამ კითხვას გააგრძელებ, გაუშვი ლაბი: cd labs/lab-p0-05-profile-and-idempotency && python3 craft_lab.py. ნახავ, როგორ ასახელებს პროფაილერი დამალულ O(n²)-ს, როგორ ეჭრება 60 მომხმარებელს თანხა ორჯერ და როგორ აორმაგებენ retry-ები კვდომადი სერვერის დატვირთვას. მერე წაიკითხე — გაკვეთილი ახსნის, რაც ახლახან ნახე.

1 · სენიორის კოდი განზრახ მოსაწყენია

ჯუნიორის კოდი წერისთვისაა ოპტიმიზებული; სენიორის კოდი — კითხვის, ღამის 3 საათზე გამართვის და ერთი წლის შემდეგ უსაფრთხოდ შეცვლისთვის. კონკრეტული ხელწერა: აშკარა სახელები, პატარა ფუნქციები, შეცდომების დამუშავება ყველა საზღვარზე და — ამ გაკვეთილის თემა — ჩავარდნისთვის დიზაინი მის დადგომამდე. ქსელები კარგავენ პასუხებს, პროცესები კვდებიან მოთხოვნის შუაში, გამომძახებლები იმეორებენ ცდას. სენიორის კოდი სამივეს ვარაუდობს ყველა I/O ხაზზე. SRE წიგნის ფორმულირება: იმედი სტრატეგია არ არის; საიმედოობა საინჟინრო ფუნქციაა და არა შემთხვევით მიღებული მემკვიდრეობა.

2 · API დიზაინი: კონტრაქტი თავად პროდუქტია

API დაპირებაა, რომლის უკან წაღება ძნელია — კლიენტები მასზე შენდებიან. ხელობის ჩეკლისტი (იხ. Google-ის API design guide და Primer-ის კომუნიკაციის სექცია):

  • რესურსები არსებითი სახელებით, ზმნები HTTP-დან: POST /charges და არა POST /doCharge. პროგნოზირებადობა ფუნქციაა.
  • შეცდომებიც კონტრაქტის ნაწილია: მანქანურად წაკითხვადი კოდი + ადამიანური შეტყობინება + რა უნდა ქნას გამომძახებელმა (გაიმეოროს? შეასწოროს? დანებდეს). შიშველი 500 პასუხისმგებლობის მოხსნაა.
  • არასდროს გატეხო არსებული კლიენტები: არასავალდებულო ველები თავისუფლად დაამატე; ველის წაშლა ან გადანიშვნა ვერსიონირების ამბის გარეშე — არასდროს.
  • პაგინაცია და ლიმიტები პირველივე დღიდან — ყოველი შეუზღუდავი სიის endpoint მომავალი ავარიაა.
ლექსიკონი ოპერაცია იდემპოტენტურია (idempotent), თუ მისი N-ჯერ შესრულებას იგივე ეფექტი აქვს, რაც ერთხელ შესრულებას. GET და PUT HTTP კონტრაქტით იდემპოტენტურია; POST — არა, და სწორედ ამიტომ არსებობს შემდეგი სამი სექცია.

3 · at-least-once სამყარო

აი ხაფანგი, რომელსაც ყველა განაწილებული სისტემა გიგებს: კლიენტი აგზავნის POST /charges-ს, სერვერი ბარათს ჭრის და პასუხი იკარგება (connection reset, timeout, proxy-ის ჩავარდნა). კლიენტი ვეღარ არჩევს „სერვერს არ მიუღია"-ს „გააკეთა და პასუხი მოკვდა"-სგან. მისი ერთადერთი გონივრული ნაბიჯი retry-ა — ამიტომ რეალური ქსელები მოთხოვნებს მინიმუმ ერთხელ აწვდიან და არა ზუსტად ერთხელ. დათვალე, რა ღირს ეს:

200 კლიენტი, თითო იხდის ერთხელ.  P(წარმატების პასუხი დაიკარგა) = 0.30, მაქს. 3 retry.

P(კლიენტი ერთხელ მაინც იმეორებს)     = 0.30            → ~60 კლიენტი იმეორებს
ყოველი retry წარმატებული ჩამოჭრის მერე = კიდევ ერთი ჩამოჭრა!
E[ზედმეტი ჩამოჭრები] ≈ 200 × (0.30 + 0.30² + 0.30³) ≈ 200 × 0.417 ≈ 83 დუბლიკატი

ლაბის გაზომვა: 282 ჩამოჭრა 200 შენაძენზე, 60 მომხმარებელი ორჯერ დაიჭრა.

არც ბაგი, არც ბოროტება — უბრალოდ ფიზიკა. Retry-ები სავალდებულოა, ამიტომ უსაფრთხოება სერვერის მხარეს უნდა აშენდეს.

4 · Idempotency key: ზუსტად-ერთხელ ეფექტი

ინდუსტრიის სტანდარტული გამოსავალი, რომელიც Stripe-ის Idempotency-Key ჰედერმა გაავრცელა: კლიენტი აგენერირებს უნიკალურ გასაღებს თითო ლოგიკურ ოპერაციაზე (ერთი შენაძენი = ერთი გასაღები, რამდენი retry-ც უნდა მოჰყვეს), სერვერი კი პატარა ცხრილს ინახავს:

  1. მოდის მოთხოვნა გასაღებით K. მოძებნე K.
  2. ნანახია? დააბრუნე შენახული პასუხი. არაფერი შეასრულო თავიდან.
  3. ახალია? შეასრულე ოპერაცია, შეინახე (K → პასუხი), დააბრუნე.

ეფექტი: ოპერაცია შეიძლება ბევრჯერ მიეწოდოს, მაგრამ ერთხელ გამოიყენება — at-least-once მიწოდება + სერვერული დედუპლიკაცია = exactly-once ეფექტი. ეს ერთი იდეა მეორდება გადახდებში, message queue-ებში (გაკვეთილი 8), saga-ებში (გაკვეთილი 14) და inference მოთხოვნების მიღებაშიც. (Stripe: "Designing robust and predictable APIs with idempotency")

5 · იდემპოტენტურობის ბასრი კიდეები

სად ცდებიან რეალური იმპლემენტაციები — თითოეული ეს პუნქტი სადღაც ნამდვილად მომხდარი პროდაქშენ-ინციდენტია:

  • გასაღები გენერირდება თითო ცდაზე და არა თითო ოპერაციაზე. კლიენტი ყოველ retry-ზე ახალ UUID-ს აკეთებს → დედუპლიკაცია ვერასდროს ირთვება → მაინც ორმაგი ჩამოჭრა. გასაღებმა retry ციკლს უნდა გაუძლოს.
  • ერთდროული დუბლიკატები. ორი retry ერთდროულად ჩამოდის; ორივე ხედავს „გასაღები უცნობია"-ს და ორივე სრულდება. გამოსავალი: გასაღების ატომური დაჯავშნა (insert-first უნიკალური constraint-ით), მეორემ დაიცადოს ან მიიღოს 409.
  • ინახება მხოლოდ გასაღები და არა პასუხი. Replay-მ თავდაპირველი შედეგი უნდა დააბრუნოს — თავდაპირველი ჩავარდნის ჩათვლით — თორემ კლიენტები შეუთავსებელ სამყაროებს ხედავენ.
  • გასაღებების უსასრულო შენახვა. ცხრილი სამუდამოდ იზრდება; Stripe გასაღებებს 24 საათში შლის. ვადა კონტრაქტის გადაწყვეტილებაა და არა დალაგების დეტალი.
  • „იდემპოტენტური" ჰენდლერი დამალული გვერდითი ეფექტებით — აგზავნის იმეილს, ზრდის მეტრიკას, ეძახის პარტნიორის API-ს. დადედუპლიკატება მთელ ეფექტს სჭირდება, თორემ არაფერი დაგიდედუპლიკატებია.

6 · Retry-ები ავარიას აძლიერებენ — დაუწესე ბიუჯეტი

იდემპოტენტურობა retry-ებს უსაფრთხოს ხდის, უფასოს — ვერა. როცა სერვერი იჩოქებს, კლიენტების გულუბრყვილო retry-ები მის დატვირთვას ზუსტად მაშინ ამრავლებენ, როცა ეს ყველაზე ნაკლებად შეუძლია. ჩავარდნის ალბათობით p და მაქსიმუმ k ცდით მოსალოდნელი გაგზავნები გეომეტრიული მწკრივია:

დატვირთვის მამრავლი = 1 + p + p² + … + pᵏ⁻¹
p = 0.5, k = 4:  1 + 0.5 + 0.25 + 0.125 = 1.875×   (ლაბმა გაზომა: 1.89×)

თითქმის გაორმაგებული ტრაფიკი ნახევრად მკვდარ სერვერზე — ბიძგი, რომელიც brownout-ს blackout-ად აქცევს. SRE-ს რეცეპტები (Handling Overload, Addressing Cascading Failures): ექსპონენციური backoff jitter-ით, კლიენტზე retry ბიუჯეტი (მაგ. retry ≤ საბაზო ტრაფიკის 10%) და მაქსიმუმ ერთი retry თითო ფენაზე, რომ სამფენიანმა სტეკმა retry-ები კუბში არ აიყვანოს.

7 · ტესტირების სტრატეგია: პირამიდა სისწრაფეს ყიდულობს

სენიორული ტესტირება პორტფელის გადაწყვეტილებაა და არა სათნოების შეჯიბრი. კანონიკური ფორმა (Fowler-ის practical test pyramid, Google-ის test sizes):

ფენაარეალისიჩქარერაოდენობაიჭერს
Unitერთი ფუნქცია/კლასი, I/O-ს გარეშეmsბევრილოგიკის ბაგები, კიდის შემთხვევები
Integrationსერვისი + ნამდვილი DB/queueწამებიზომიერადშეერთებები, სქემები, SQL
End-to-endმთელი სისტემა, მომხმარებლის ნაკადიწუთებიცოტაკონფიგი, კონტრაქტები, დეპლოი

რატომ ეს ფორმა: უკუკავშირის დაყოვნება გროვდება. 10-წუთიანი suite ყოველ commit-ამდე ეშვება; 2-საათიანი — არასდროს. ორი სენიორული რეფლექსი: ყოველი ბაგ-ფიქსი იწყება ჩავარდნილი ტესტით, რომელიც მას იმეორებს (რეგრესიის დაზღვევა), და flaky ტესტი ან სწორდება, ან იშლება — არასდროს „გამწვანდება" გამეორებით: suite, რომელსაც გუნდი არ ენდობა, არარსებულზე უარესია.

8 · პროფაილინგი: გაზომე, ნუ გამოიცნობ

ინჟინრების ინტუიცია hotspot-ებზე სახელგანთქმულად ცდება — ბოთლის ყელი იშვიათადაა იქ, სადაც კოდი „ნელა გამოიყურება". წესი: ოპტიმიზაცია პროფაილის გარეშე — არასდროს. Python-ს stdlib-შივე მოჰყვება (cProfile):

9 · პროდაქშენ Python-ისა და Go-ს კითხვა

სენიორები 10-ჯერ მეტ კოდს კითხულობენ, ვიდრე წერენ, და კითხულობენ ფიქსირებული თანმიმდევრობით: შესვლის წერტილები → მონაცემთა ფორმები → შეცდომების გზები. იპოვე main()/route ცხრილი; იპოვე მთავარი struct-ები/dataclass-ები (მონაცემთა ფორმები კოდბეისს ნებისმიერ კომენტარზე სწრაფად ხსნიან); მერე წაიკითხე, რა ხდება ჩავარდნისას — ხარისხი იქ ცხოვრობს. ენობრივი ნიშნები:

  • Python: type hint-ები + dataclass-ები მონაცემთა ხერხემალს ნიშნავენ; context manager-ები (with) — რესურსების საზღვრებს; შიშველი except: წითელი დროშაა, რომელიც ყველა რევიუერმა უნდა დაიჭიროს.
  • Go (Effective Go): if err != nil ყოველი გამოძახების მერე შეცდომების გზებს კონსტრუქციითვე ხილულს ხდის; defer — დასუფთავებას; goroutine-ები + channel-ები — კონკურენტულობის საზღვრებს. Go განზრახ ბრტყლად იკითხება — მოსაწყენი იქ სახლის სტილია.

10 · ლაბი — retry შტორმები და hotspot-ები

პრაქტიკა (ქვიზამდე გააკეთე) labs/lab-p0-05-profile-and-idempotency/ → python3 craft_lab.py. სამი ექსპერიმენტი: O(n²) კონკატენაცია გაზომილი და cProfile-ით დაპროფაილებული; 200-კლიენტიანი retry შტორმი idempotency key-ებით და მათ გარეშე (დათვალე ორმაგი ჩამოჭრები); retry გაძლიერება retry ბიუჯეტის წინააღმდეგ. მერე README-ს E1–E4: იწინასწარმეტყველე, შეცვალე CONFIG, გაუშვი თავიდან, შეადარე. მცდარი პროგნოზები დავალების შედეგია.

11 · პროექტის ნაბიჯი

ჩვენი ხერხემალი-პროექტი — LLM inference gateway — იღებს გენერაციის მოთხოვნებს, რომლებიც ნამდვილ ფულად GPU-წამებში ჯდება. დაწერე (ჩანაწერიც კმარა) gateway-ის იდემპოტენტურობის კონტრაქტი: რა არის გასაღები (კლიენტის ჰედერი? მოთხოვნის ჰეში?), ზუსტად რა ინახება replay-სთვის ნაკადური (streaming) გენერაციის პასუხად, რამდენ ხანს ცოცხლობენ გასაღებები და რას იღებს ერთდროული დუბლიკატი გენერაციის შუაში. Streaming „შენახული პასუხის replay"-ს მართლაც საინტერესოს ხდის — მოიტანე პასუხი შემდეგ სესიაზე და დაიცავი.

საინტერვიუო თეზისები

  • „ქსელები at-least-once-ია: დაკარგული პასუხი კლიენტს retry-ს აიძულებს, ამიტომ ნებისმიერი არაიდემპოტენტური endpoint ოდესმე ორჯერ შესრულდება."
  • „ჩემი გამოსავალი Stripe-ის სტილის idempotency key-ებია: ერთი გასაღები თითო ლოგიკურ ოპერაციაზე, სერვერი ინახავს key → პასუხს და replay-ს აკეთებს ხელახლა შესრულების ნაცვლად."
  • „ორი კლასიკური იმპლემენტაციის ბაგი: გასაღების ხელახლა გენერაცია ყოველ retry-ზე და გასაღების არაატომური დაჯავშნა ერთდროული დუბლიკატების წინააღმდეგ."
  • „Retry-ებს ბიუჯეტი და jitter-იანი backoff სჭირდება — 50% ჩავარდნაზე ოთხი გულუბრყვილო ცდა უკვე კვდომადი სერვერის დატვირთვას თითქმის აორმაგებს."
  • „პროფაილის გარეშე არასდროს ვაოპტიმიზებ; hotspot-ი იშვიათადაა იქ, სადაც ინტუიცია მიუთითებს, და cProfile მას ერთი ბრძანებით პოულობს."
  • „ჩემი ტესტების პორტფელი პირამიდისებურია: ბევრი მილიწამიანი unit ტესტი, ზომიერად integration, ცოტა end-to-end — რადგან უკუკავშირის დაყოვნება გროვდება."
