Payment System — Exactly-Once Money Movement

Every distributed-systems failure you have studied — lost messages, duplicate deliveries, partial writes, ambiguous timeouts — reappears here, except now each one costs literal money. This is the senior-bar correctness rep.

Mission tie. Billing and usage metering for any API platform — including LLM APIs, where every token generated must be counted once, priced once, and charged exactly once — is this exact design wearing a different label.
REP PROTOCOL Do a 40-minute solo timed rep FIRST — out loud, whiteboard or paper, no peeking. Prompt: "Design a payment system for an e-commerce platform." Then read this reference, compare, and log your deltas as learning records. The deltas are the lesson.

1 · The prompt, and what is actually tested

"Design a payment system for an e-commerce platform." The hidden rubric is not architecture breadth — it is correctness under retries. At some point the interviewer will inject the killer scenario: "the network dies right after the charge succeeded — your service never saw the response. Now what?" If your design blindly retries, you double-charge; if it gives up, you lose track of real money that really moved. Everything in this lesson exists to make that question boring. Secondary signals: knowing you build on top of a payment service provider (PSP), an append-only ledger instead of mutable balances, and reconciliation as the final safety net. Latency talk is a junior tell here — this rep is graded on the money never lying.

2 · Clarify (5–8 min): buy the right problem

RequirementDecision
Pay-in flowCustomer pays via a PSP (Stripe/Adyen). We do not touch card networks ourselves: storing or transmitting raw card data pulls us into full PCI DSS compliance scope — audits, certified infrastructure, liability. The PSP tokenizes the card; we only ever see a token.
Pay-out flowMove sellers' earnings to their bank accounts (batched, e.g. daily).
RefundsFull and partial, as new money movements — never edits to old ones.
ReconciliationNightly diff of PSP settlement files against our internal ledger.

Non-functional requirements, stated as absolutes: no double charge, ever; no lost money, ever; auditability > latency. And say this explicitly: for the money path we rank consistency over availability — if we cannot be certain a charge is recorded, we would rather reject the payment and show an error than guess. A payment page that is briefly down costs a sale; a wrong ledger costs trust, audits, and lawsuits.

3 · Estimate (~5 min): small numbers, high stakes

1M payments/day ÷ ~100k s/day ≈ 10 TPS average, ×5 peak ≈ 50 TPS
Ledger: ~5 entries/payment × ~200 B ≈ 1 KB/payment
  → ~1 GB/day ≈ 365 GB/year;  retention 7+ years → ~2.5 TB total

Say this out loud: payments are LOW-QPS, high-stakes — the exact opposite trade-off of the feed rep. The feed tolerated staleness and lost updates to win throughput; here 50 peak TPS fits comfortably on one well-run PostgreSQL primary, so throughput buys you nothing and correctness is the entire game. Now spend the numbers: 50 TPS means single-primary ACID transactions are affordable — no exotic sharding needed, so don't propose it. 2.5 TB over a 7-year legal retention window means the ledger is partitioned by time and archived to cold object storage, but never deleted or rewritten. And 1M payments/day means even a 0.01% bug rate is 100 wrong charges daily — which is why section 8's reconciliation exists no matter how good the code is.

4 · High-level design (~10 min): boxes with failure semantics

client → payment service → payment executor → PSP adapter → PSP (Stripe)
              │                    │
              ▼                    ▼
         wallet service        ledger service
  • Payment service — accepts payment requests, owns the payment state machine, enforces idempotency. Arrow from client: at-least-once (clients retry) → must dedupe.
  • Payment executor — drives one payment order through the PSP. Arrow to PSP adapter: ambiguous on timeout (charge may have happened) → never blind-retry without a key.
  • PSP adapter — one thin client per provider; translates our model to their API. Inbound webhook arrows: at-least-once, unordered → verify, dedupe, reorder.
  • Wallet service — merchant balances, derived from the ledger. Ledger service — the append-only source of truth (section 6).

The payment state machine, drawn explicitly:

INITIATED → PENDING → SUCCEEDED   (terminal, immutable)
                    ↘ FAILED      (terminal, immutable)

Terminal states are immutable: a SUCCEEDED payment is never edited — a refund is a new movement referencing it. Every transition is a database write, so a crashed executor can resume any payment from its persisted state.

5 · Deep dive: idempotency end-to-end

The retry problem, solved layer by layer. The client generates an idempotency key — a UUID, created when the user lands on checkout, sent as a header (Idempotency-Key: 8e03…) on every attempt of that one logical payment. The payment service inserts it into a table with a unique constraint. First request: insert succeeds, payment proceeds, and the final response is stored against the key. Retry: insert violates the constraint → look up and return the stored response — same status code, same body, no re-execution. If the first attempt is still in flight, the retry gets 409/wait, not a second execution.

Then the same trick again one layer down: the executor sends its own idempotency key with the PSP call — this is exactly Stripe's mechanism: pass Idempotency-Key, and a retried charge returns the recorded result of the first attempt instead of charging again. Now the ambiguous-timeout answer writes itself: the charge may have happened, so retry with the SAME key — safe either way.

Be precise about the theory: exactly-once delivery is impossible over an unreliable network. The honest formulation is at-least-once delivery + idempotent processing = effectively exactly-once effect. Say it in those words; interviewers listen for whether you know the difference.

6 · Deep dive: the double-entry ledger

Rule, unchanged since Venetian bookkeeping: every movement of money is a balanced set of debit and credit entries — money is never created or destroyed, only moved between accounts. Entries are append-only and immutable: no UPDATE, no DELETE, ever. The system invariant, checkable at any moment: sum(debits) = sum(credits), always. One $10 payment with a 59¢ processing fee:

AccountDebitCredit
psp_clearing (money captured by PSP)1000¢—
merchant_payable (owed to the seller)—941¢
fee_revenue (our cut)—59¢
Totals1000¢1000¢

Balances are derived by summing an account's entries — never stored as the primary truth. And corrections: posted 941¢ to the wrong merchant? You do not edit the row. You append a reversing entry (credit back 941¢) plus a correct new entry. The mistake stays visible forever — that is the audit trail, and it is why auditors and regulators require this structure.

7 · Deep dive: staying consistent with the PSP

Your database and Stripe's database are two systems with no shared transaction — the dual-write problem from lesson 0014, now with money. Three tools close the gap:

The outbox pattern. Never write "payment SUCCEEDED" to your DB and then separately publish an event — a crash between the two loses the event. Instead, one local transaction writes both the state change and an event row into an outbox table; a relay process reads the outbox and publishes (at-least-once — consumers dedupe by event id). Atomic, crash-proof.

Webhook handling. PSPs push results (payment_succeeded, payment_failed) as webhooks. Treat them as hostile input: verify the signature (anyone can POST to your endpoint), dedupe by event id (PSPs deliver at-least-once), and tolerate out-of-order arrival — a refund_succeeded can land before the payment_succeeded it depends on. The state machine rejects illegal transitions; hold early events and re-apply.

The pending-timeout sweeper. Webhooks get lost too. A background job finds payments stuck in PENDING beyond a threshold and queries the PSP directly for the true status — reconciliation at the single-payment scale, closing the last gap.

8 · Deep dive: reconciliation — the ultimate safety net

Every night the PSP produces a settlement file: every charge, refund, and fee they processed for you, with amounts. A nightly job diffs it against the internal ledger, transaction by transaction. Three mismatch classes:

MismatchTypical causeAction
In theirs, missing in oursCrash after PSP success, before our ledger write; lost webhookBackfill ledger entry; investigate the crash path
In ours, missing in theirsWe recorded optimistically; PSP call actually failedReverse our entry; alert — customer thinks they paid
Amount driftFee/rounding computed differently; currency conversion mismatchAdjusting entry; fix the fee formula

Why bother if sections 5–7 are implemented perfectly? Because "perfectly" is not a property real systems have: deploys have bugs, PSPs have incidents, operators run manual fixes. Reconciliation is the mechanism that bounds how long any error can survive — with a nightly run, no discrepancy lives past 24 hours. Interviewers score it as the mark of someone who has operated a money system, not just drawn one.

9 · Failure modes & monitoring

  • PSP timeout ambiguity. The charge MAY have happened. Never blind-retry without an idempotency key; never mark FAILED without querying the PSP. This is the interviewer's injected scenario — answer it before they ask.
  • Partial refund races. Two concurrent $6 refunds on a $10 payment must not both pass the "≤ remaining" check. Serialize per payment: row lock or version check (optimistic concurrency).
  • Currency and rounding. Store minor units as integers — 1000 cents, NEVER floats: binary floats cannot represent 0.1, and tiny errors compound over millions of entries until sum(debits) ≠ sum(credits). The lab makes you watch this happen.

Metrics that page a human: payment success rate (drop = PSP incident or our bug — this pages before Twitter knows), reconciliation break count (nonzero = money is lying somewhere), stuck-PENDING age (growing = webhooks or sweeper failing). CPU and latency dashboards are secondary here.

10 · Variants & twists

  • Payouts / marketplace splits. One pay-in fans out to seller + platform fee + tax — more legs in the same balanced ledger entry; payouts batched daily through the PSP's transfer API, same idempotency discipline.
  • Multi-currency. One account per (owner, currency); never mix currencies in one account. Record the FX rate used in the entry itself, so history is reproducible.
  • Chargebacks. The card network forcibly reverses a charge — arrives weeks later as a webhook, modeled as a new money movement plus a dispute-evidence workflow, never as an edit.

AI angle — usage metering for API billing. Billing an LLM API is this design with tokens as the currency: meter usage events (at-least-once delivery → dedupe by event id), aggregate into an append-only usage ledger, price idempotently per invoice line, and reconcile metered totals against invoiced totals nightly. Same skeleton, same failure modes.

11 · Interview talking points

  • Scoping: "We integrate a PSP rather than touching card networks — raw card data would put us in PCI DSS scope, which is a compliance project, not a systems one."
  • Ranking: "For the money path I'm choosing consistency over availability — I'd rather fail a checkout than record it wrong."
  • The killer question, pre-empted: "If the PSP times out, the charge may have happened — so every call carries an idempotency key, and retrying with the same key is safe either way."
  • Honesty about exactly-once: "Exactly-once delivery is impossible; at-least-once plus idempotent processing gives exactly-once effect — that's the real guarantee."
  • Ledger: "Append-only double-entry with sum(debits) = sum(credits) as a checkable invariant; corrections are reversing entries, never UPDATEs."
  • Closing: "Even with all that, I'd run nightly reconciliation against PSP settlement files — it bounds any error's lifetime to 24 hours."
