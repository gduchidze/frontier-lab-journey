გაკვეთილი 0014 · ფაზა 2 — პროგრამული არქიტექტურა და საიმედოობა · English version

განაწილებული ტრანზაქციები და კოორდინაცია

ფული სერვისებს შორის გლობალური ტრანზაქციის გარეშე მოძრაობს — saga-ები, outbox და რატომ წააგო 2PC-მ; პლუს consensus, leader election და fencing token, რომელიც გიხსნის.

მისიასთან კავშირი. Anthropic-ის დიზაინის ამოცანებს უყვართ ნაკადები, სადაც ერთი მოქმედება რამდენიმე სერვისს ეხება — „დააპროექტე checkout" — რადგან დამატებითი კითხვა სკალპელია: „გადახდა შესრულდა, მაგრამ inventory-ის დაჯავშნა ჩავარდა; ახლა რა?" ვინც ამბობს „ტრანზაქციაში გავახვიოთ", ჩერდება; ვინც ასახელებს saga-ს, სვამს outbox-ს და ხსნის, რატომ სჭირდება lock-ს fencing token, აგრძელებს. საფუძველი: DDIA, თავები 8–9 (Kleppmann, Designing Data-Intensive Applications; Exponent-ის Anthropic გზამკვლევი).

1 · პრობლემა: ერთი მოქმედება, N სერვისი, საერთო ACID არ არსებობს

Checkout ოთხ სერვისს ეხება: order, payment, inventory, shipping. თითოეულს თავისი database აქვს ლოკალური ACID ტრანზაქციებით — და არცერთი ტრანზაქცია მათ არ აერთიანებს. ბიზნეს-მოქმედება მომხმარებლის თავში ატომურია („ვიყიდე"), ფიზიკურად კი ოთხი დამოუკიდებელი commit-ია, რომელთაგან თითოეული ცალკე შეიძლება ჩავარდეს.

DDIA-ს მე-8 თავი partial failure-ის (ნაწილობრივი ჩავარდნის) ტაქსონომიას იძლევა: დაშორებული გამოძახება შეიძლება ჩავარდეს სამუშაოს შესრულებამდე, შესრულების შემდეგ პასუხამდე, ან ქსელი უბრალოდ ნელია — და გამომძახებლისთვის ეს სამი განურჩეველია. Timeout არ ნიშნავს „არ მოხდა". განაწილებულ სისტემებში კითხვაზე „გავიდა ჩემი გადახდა?" ნაგულისხმევი პასუხია „იქნებ" — ამ გაკვეთილის ყველა pattern მანქანაა, რომელიც „იქნებ"-ს ქმედით მდგომარეობად აქცევს.

ნამუშევარი მაგალითი — არითმეტიკა, რომელიც ინტერვიუერს სურს. ოთხი თანმიმდევრული გამოძახება, თითო 99.9% წარმატებით: 0.9994 ≈ 99.6%. ანუ checkout-ების ~0.4% შუა გზაზე ჩერდება. დღეში 200,000 შეკვეთაზე ეს ~800 ნახევრად დასრულებული შეკვეთაა დღეში — გადახდილი-მაგრამ-დაუჯავშნავი. კითხვა არასდროსაა „მოხდება თუ არა" — კითხვაა „ვინ ალაგებს და რამდენად სწრაფად".

2 · Two-phase commit — და რატომ წააგო

კლასიკური პასუხია 2PC (DDIA, თავი 9). Coordinator ორ რაუნდს ატარებს: prepare — ყველა მონაწილე მყარად ჰპირდება „შემიძლია commit და lock-ებს გადაწყვეტილებამდე ვიჭერ"; შემდეგ commit/abort — coordinator-ი გადაწყვეტილებას საკუთარ ლოგში წერს და ყველას ატყობინებს.

და სწორედ ეს წერტილია ნაკლი: თუ coordinator-ი prepare-ის შემდეგ და ვერდიქტამდე მოკვდა, ყველა მონაწილე in doubt (გაურკვევლობაში) რჩება — commit-ს ვერ აკეთებს და abort-საც ვერ, ამიტომ ზის და row lock-ებს იჭერს, ბლოკავს ყველას, სანამ coordinator-ი დაბრუნდება. 2PC მბლოკავი პროტოკოლია: ერთი კვანძის ჩავარდნას ყველას მიუწვდომლობად აქცევს — ზუსტად ის coupling-ი (შეჭიდულობა), რომლის თავიდან ასაცილებლადაც microservice-ები არსებობს. ის ცოცხლობს სისტემების შიგნით: XA ძველ Java სამყაროში და databases-ის შიდა cross-partition commit-ებში.

3 · Saga: ტრანზაქცია, აწყობილი ლოკალური ნაჭრებისგან

ჩამნაცვლებელია saga (Garcia-Molina & Salem, 1987; microservices.io): ლოკალური ტრანზაქციების მიმდევრობა T1…Tn, სადაც თითოეულ Ti-ს აქვს compensating action (მაკომპენსირებელი მოქმედება) Ci, რომელიც მას სემანტიკურად აუქმებს. თუ T3 ჩავარდა — გაუშვი C2, C1 უკუმიმდევრობით.

Checkout როგორც saga: შეკვეთის შექმნა → გადახდა → inventory-ის დაჯავშნა → shipment. თუ დაჯავშნა გადახდის შემდეგ ჩავარდა: refund (თანხის დაბრუნება), შეკვეთის გაუქმება. ყურადღება სიტყვაზე სემანტიკური: refund rollback-ი არაა — გადახდა მოხდა, ამონაწერზე ორივე ხაზი ჩანს. ზოგ ნაბიჯს undo საერთოდ არ აქვს: დადასტურების email-ს ვერ გააუგზავნინებ უკან — შესწორებას აგზავნი. Compensation-ები სრულფასოვანი ბიზნეს-ოპერაციებია და არა database-ის მაგია.

4 · Outbox pattern: dual write-ის მოკვლა

Saga-ები მალავენ ხაფანგს: სერვისმა უნდა ჩაწეროს DB-ში და გამოაქვეყნოს event-ი — ორი სისტემა, საერთო ტრანზაქცია არ არსებობს. Commit-მერე-publish კარგავს event-ებს, როცა შუაში კვდები (შეკვეთა არსებობს, არავის სმენია); publish-მერე-commit იგონებს event-ებს შეკვეთებზე, რომლებიც არასდროს ჩაწერილა. ეს dual-write პრობლემაა და მკვდარ პროცესს ვერანაირი retry ვერ უშველის.

გამოსავალია transactional outbox (microservices.io): ჩაწერე event-ი outbox ცხრილში იმავე ლოკალურ ტრანზაქციაში, რომელშიც შეკვეთის row — ერთი commit, ატომურად ორივე ან არცერთი. ცალკე relay აწვდის: ან ცხრილს ამოწმებს, ან, უკეთესია, DB-ის ცვლილებების ნაკადს CDC-ით უსმენს — კანონიკური იმპლემენტაციაა Debezium-ის outbox router. Relay at-least-once-ია (შეიძლება publish-ის შემდეგ, მონიშვნამდე მოკვდეს და თავიდან გამოაქვეყნოს) — ამიტომ consumer-ები idempotent-ები უნდა იყვნენ, event id-ზე dedup-ით. At-least-once + idempotent consumer = effectively-once — „exactly-once"-ის გულახდილი ვერსია.

მიდგომადაკარგული event-ებიდუბლიკატებიფასი
Dual write (commit, მერე publish)კი — crash შუალედშიშესაძლებელიაარაფერი — ეგაა ხაფანგი
2PC DB-სა და broker-ს შორისარაარამბლოკავი, XA broker-ია საჭირო, availability coupling
Outbox + relay/CDCარა — event-ი commit-ს მიჰყვებაკი — dedup consumer-თანზედმეტი ცხრილი, relay-ის ოპერირება, relay lag

ციფრები ლაბიდან. 2% crash-ის ალბათობამ commit-სა და publish-ს შორის 10,000 შეკვეთაზე ჩუმად დაკარგა 201 event-ი — 201 გადახდილი შეკვეთა, რომელზეც downstream-ს არასდროს სმენია. Outbox-ის ვერსიამ დაკარგა 0 და აწარმოა 100 დუბლიკატი, რომლებიც consumer-ის dedup-მა ზუსტად 10,000 გამოყენებულ event-ად აქცია. ეს გაცვლა — დუბლიკატები, რომელთა dedup შეიძლება, დანაკარგების ნაცვლად, რომელთა შემჩნევაც კი არ შეიძლება — მთელი pattern-ია ერთ წინადადებაში.

5 · დაკარგული isolation: saga-ები serializable არ არიან

ACID-მა isolation მოგცა; saga-ები მას თმობენ. თითოეული Ti მაშინვე commit-დება, ამიტომ სხვა მოთხოვნები შუალედურ მდგომარეობებს ხედავენ: შეკვეთა არსებობს, მაგრამ გადახდილი არაა; ორი პარალელური saga ერთსა და იმავე მარაგს კითხულობს და ორივე ჯავშნის (lost update); მოგვიანებით კომპენსირებული saga ცოტა ხნით ნამდვილად ჩანდა (dirty read). კონტრზომები აპლიკაციის დონეზეა (microservices.io-ს countermeasures):

  • Semantic lock: მიმდინარე row-ებს მდგომარეობა მიაწერე — order: PENDING_PAYMENT, მარაგი: RESERVED და არა SOLD — და სხვებმა pending-ს „ხელი არ ახლონ".
  • კომუტაციური ოპერაციები: ნაბიჯები ისე დააპროექტე, რომ რიგს მნიშვნელობა არ ჰქონდეს — reserve(3) და reserve(2) დეკრემენტებად კომუტირებენ; „set stock = 7" — არა.
  • ხელახალი წაკითხვა / ვერსიის შემოწმება: გადამწყვეტ ნაბიჯამდე ხელახლა შეამოწმე ადრე წაკითხული დაშვება (ფასი, მარაგი).

6 · Consensus, სამუშაო სიმაღლიდან

ზოგ ფაქტზე მანქანები უნდა შეთანხმდნენ, რომლებიც შეიძლება მოკვდნენ: ვინაა ლიდერი, რა არის კონფიგი, ვინ იჭერს lock-ს. ეს consensus-ია, და quorum-ები უმრავლესობებია არითმეტიკის გამო: ერთი კლასტერის ნებისმიერი ორი უმრავლესობა ერთმანეთს კვეთს მინიმუმ ერთ კვანძში, ამიტომ ორი ურთიერთსაწინააღმდეგო გადაწყვეტილება ერთად ვერ დამტკიცდება (DDIA, თავი 9).

Raft ხუთ წინადადებაში (raft.github.io, Ongaro & Ousterhout): დრო დანომრილ term-ებად იყოფა, თითოეულში მაქსიმუმ ერთი ლიდერია. კვანძი, რომელიც heartbeat-ებს ვეღარ ისმენს, კანდიდატი ხდება და არჩევნებს ხმების უმრავლესობით იგებს. ყველა ჩაწერა ლიდერზე გადის, რომელიც ლოგში ამატებს და follower-ებზე ახდენს რეპლიკაციას. ჩანაწერი committed-ია, როცა უმრავლესობას აქვს — ეს commit index-ია — და committed ჩანაწერები ლიდერის ცვლილებას გადაურჩებიან. არჩევნები ისეა შეზღუდული, რომ მხოლოდ განახლებული ლოგის მქონე კანდიდატი იგებს.

რაში იყენებ რეალურად: control plane — კონფიგურაცია, membership, lock-ები, leader election — etcd-თი (Raft) ან ZooKeeper-ით (ZAB). შეკვეთების მონაცემებს consensus-ში არ ატარებ: უმრავლესობის round-trip ყოველ ჩაწერაზე data path-ისთვის არასწორი ფასია.

7 · განაწილებული lock-ები და fencing token

„ამ შეკვეთას ერთდროულად მხოლოდ ერთი worker ამუშავებს" — ამიტომ Redis-ში ან etcd-ში lock-ს TTL-ით იღებ. ახლა ცნობილი ჩავარდნა (Kleppmann, „How to do distributed locking"): worker A lock-ს იღებს, მერე ჭედავს — GC pause, page fault — TTL-ზე მეტ ხანს. Lock ვადადება; worker B იღებს და წერს. მერე A იღვიძებს, ჯერ კიდევ სჯერა, რომ lock უჭირავს, და ისიც წერს. ორი მწერალი, გაფუჭებული რესურსი — და TTL-ის ვერანაირი აწყობა ამას ვერ უშველის, რადგან გაჩერებულმა პროცესმა ვერ იცის, რომ გაჩერებულია.

Kleppmann-ის გამოსავალი: lock-სერვისი გასცემს fencing token-ს — რიცხვს, რომელიც ყოველ გაცემაზე მონოტონურად იზრდება (A-მ მიიღო 33, B-მ 34) — და რესურსი იმახსოვრებს უმაღლეს ნანახ token-ს და უარყოფს ჩაწერებს უფრო მცირეთი. A-ს დაგვიანებული ჩაწერა 33 < 34 → უარყოფილია. უსაფრთხოების შემოწმება კლიენტიდან (რომელიც ნებისმიერად შეიძლება დაყოვნდეს) რესურსზე გადადის. ლაბში: 10 წმ TTL და საშუალოდ 4 წმ pause-ებით ცდების ~8%-მა lease-ს გადააჭარბა — 799 გაფუჭებული ჩაწერა fencing-ის გარეშე, 0 მასთან ერთად.

Redlock, ერთი აბზაცი. Redis-ი მრავალკვანძიან ალგორითმს Redlock-ს აღწერს. Kleppmann-ი ამტკიცებდა, რომ ის დროის დაშვებებზეა დამოკიდებული, რომლებსაც რეალური სისტემები არღვევენ, და fencing token-ებს არ გასცემს — ანუ ვარგისია efficiency lock-ებისთვის (ორმაგი სამუშაოს თავიდან აცილება), მაგრამ არა correctness lock-ებისთვის; antirez-ი არ დაეთანხმა. ინტერვიუზე უსაფრთხო პოზიცია: თქვი, რომელი გჭირდება — თუ მოძველებულ მფლობელს მონაცემების გაფუჭება შეუძლია, fencing რესურსთან გჭირდება, lock-სერვისის მიუხედავად.

8 · Leader election: lease-ები, split-brain და თავმდაბლობა

Leader election იგივე lock-ია, განახლებადი: კანდიდატი etcd/ZooKeeper-ში lease-ს (იჯარას) იღებს და მუდმივად უნდა აახლოს; თუ მოკვდა, lease ქრება და სხვა იკავებს — ასე ირჩევენ Kubernetes-ის controller-ები (K8s Leases), Google-ის Chubby-ის დიზაინით. საფრთხე split-brain-ია: ძველი ლიდერის ბოლო heartbeat-სა და lease-ის ვადის გასვლას შორის ძველი ჯერ კიდევ მოქმედებს, ახალი კი უკვე იწყებს — ორი ლიდერი, ცოტა ხნით, lease-ზე დაფუძნებული ლიდერობის ნორმალური ჩავარდნაა.

ამიტომ გულახდილი პოზიცია: ლიდერობა ოპტიმიზაციაა და არა კორექტულობის გარანტია. ლიდერი გამოიყენე ორმაგი სამუშაოს თავიდან ასაცილებლად, მაგრამ ყოველი გარეგანი ეფექტი §7-ის წესით დაიცავი — fencing token-ები, idempotent ჩაწერები. თუ შენი დიზაინი მხოლოდ მაშინაა სწორი, როცა ზუსტად ერთი ლიდერია ყოველთვის — შენი დიზაინი არასწორია.

9 · Failure mode-ები, რომლებიც ინტერვიუებს წყვეტენ

  • Compensation, რომელიც თავად ვარდება. Refund-ის გამოძახებაც timeout-დება — saga გაჭედილია ნახევრად გაუქმებული. Compensation-ები idempotent-ები უნდა იყვნენ და წარმატებამდე მეორდებოდნენ; რაც მაინც არ გამოდის, ადამიანების რიგში ხვდება (ლაბში: 10%-იანმა compensation-ის ჩავარდნამ retry-ის გარეშე 10,000-დან 159 saga გაჭედა, retry-ით — 0). ეს რიგი სისტემის ნაწილია — დააპროექტე.
  • Outbox relay-ის lag. Event-ები არ იკარგება, მაგრამ გვიან მოდის. Relay lag replication lag-ივით დააკვირდი — ეს შენი სიახლის ზღვარია.
  • გაორმაგებული compensation-ები. At-least-once არხი იმეორებს refund-ს, რომლის ack დაიკარგა → ორმაგი refund, თუ idempotency key არ აქვს.
  • ორი ლიდერი lease-ის ვადის გასვლისას. გაჩერებული ძველი ლიდერი იმავე batch job-ს უშვებს, რასაც ახალი; ორივე წერს. Fencing token მოძველებულს აბრუნებს (§7); მის გარეშე — ჩუმად გაორმაგებული ეფექტები.
  • 2PC coordinator-ის crash. Checkout-ში 2PC რომ გქონოდა, coordinator-ის crash მონაწილეებს in-doubt ტოვებს lock-ებით ხელში — checkout გლობალურად იყინება. ეს არის კონკრეტული ამბავი კითხვაზე „რატომ არა 2PC".

10 · ერთი AI შტრიხი, მერე ისევ საფუძვლები

LLM inference პლატფორმას ზუსტად ორი ეს ხელსაწყო სჭირდება. Batch-job scheduler-ი რამდენიმე რეპლიკად eskarbis etcd-ის lease-ით ირჩევს ლიდერს — და რადგან ლიდერობა მხოლოდ ოპტიმიზაციაა (§8), job-ის მინიჭებებს fencing token-ები აქვთ, რომ გაჩერებული ექს-ლიდერის მოძველებული მინიჭება უარიყოს. ხოლო usage billing შენიღბული dual-write-ია: gateway წერს request-ის ლოგს და billing event-ი უნდა გაუშვას — outbox ცხრილი gateway-ის DB-ში, CDC relay, idempotent billing consumer, რადგან „იქნებ დავარიცხეთ" ბიზნესისთვის მდგომარეობა არაა. იგივე pattern-ები, GPU-ს ფასის ფსონები. სულ ესაა შტრიხი; ინტერვიუ §1–§9-ს აფასებს.

11 · ინტერვიუს სალაპარაკო პუნქტები

  1. დაიწყე partial-failure-ის მათემატიკით: „ოთხი სერვისი თითო 99.9%-ით ნიშნავს, რომ checkout-ების ~0.4% ნახევრდება — ასობით დღეში — ამიტომ საკითხავია ვინ ალაგებს და არა მოხდება თუ არა."
  2. 2PC სწორი მიზეზით უარყავი: არა „ნელია", არამედ „ბლოკავს — ჩავარდნილი coordinator-ი მონაწილეებს in-doubt ტოვებს lock-ებით ხელში."
  3. დაასახელე saga და მისი სტილი: „ორკესტრირებული saga, compensation-ები უკუმიმდევრობით; orchestration, რადგან ნაკადი ფულს ეხება და ერთი queryable state machine მინდა."
  4. Outbox შეუკითხავად დასვი: „ყოველი ნაბიჯის event ლოკალურ commit-ს outbox-ით მიჰყვება; relay at-least-once-ია, consumer-ები event id-ზე dedup-ენ — effectively-once."
  5. Isolation-ის დათმობა აღიარე: „შეკვეთები PENDING_PAYMENT-ში სხედან, სანამ saga მუშაობს; downstream pending-ს გაყიდულად არ თვლის."
  6. Lock-ებზე fencing-ი მათზე ადრე თქვი: „TTL lock-ები pause-ებზე ვარდებიან, ამიტომ საცავი მონოტონურ fencing token-ს ამოწმებს — რესურსი და არა კლიენტია დაცვის ბოლო ხაზი."
  7. Consensus სწორად შემოფარგლე: „etcd leader election-ისა და კონფიგისთვის — მხოლოდ control plane; consensus შეკვეთების data path-ზე არასდროს ზის."

12 · ლაბი — დაკარგე ფული, მერე შეწყვიტე კარგვა

პრაქტიკა (ქვიზამდე გააკეთე) ყველაფერი ზემოთქმული ახლა შემოწმებადია: labs/lab-0014-sagas-and-fencing/-ში python3 saga_lab.py უშვებს სამ ექსპერიმენტს — გულუბრყვილო dual write კარგავს 10,000-დან 201 გადახდილ შეკვეთას, outbox კი ნულს (dedup დუბლიკატებს effectively-once-ად აქცევს); checkout saga crash-ის ინექციით ყოველ ნაბიჯზე (უყურე, როგორ ჩნდება გაჭედილი saga-ების რიგი, როცა compensation-ებიც ვარდებიან, და როგორ ქრება retry-ით); და TTL lock, რომელსაც GC pause-ები აფუჭებენ, სანამ fencing token არ გააჩერებს. შემდეგ README-ს E1–E4: იწინასწარმეტყველე, შეცვალე კონფიგი, გაუშვი თავიდან, შეადარე. მცდარი პროგნოზები დავალების შედეგია — მოიტანე შემდეგ სესიაზე.

13 · პროექტის ნაბიჯი

დააპროექტე „provision fine-tune job" ნაკადი LLM პლატფორმისთვის, როგორც checkout-ის ფორმის saga: ჩამოწერე ნაბიჯები, თითოეულის compensation და სად ზის outbox — და ორი წინადადებით დაიცავი. დასაცავი მაგალითი: ორკესტრირებული saga — job-ის ჩანაწერის შექმნა → GPU კვოტის დაჯავშნა → billing hold-ის დარიცხვა → scheduler-ის რიგში ჩაყენება, უკუმიმდევრობით release-quota / void-hold / mark-job-failed compensation-ებით, და ყოველი ნაბიჯის event ამ სერვისის DB-ის outbox-ში იწერება. Orchestration, რადგან ნაკადი ფულსა და კვოტას ეხება, ხოლო outbox — რადგან job, რომელიც არსებობს, მაგრამ billing-მა არ იცის, ზუსტად ის dual-write ხვრელია, რომელიც ლაბმა დააფასა. დაწერე შენი ორი წინადადება და მოიტანე შემდეგ სესიაზე გასარჩევად.
