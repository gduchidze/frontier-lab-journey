Distributed Key-Value Store — a mini-Dynamo

Get and put over 100 TB, on hardware that fails daily. This rep assembles consistent hashing, quorums, vector clocks, and anti-entropy into one machine — the design behind Dynamo, Cassandra, and Riak.

Mission tie. "Design a distributed KV store" is a classic Anthropic warm-up prompt — and the KV-cache manager in your spine-project inference platform is literally a distributed KV problem: route a prefix hash to the worker holding its cache blocks, replicate hot prefixes, evict cold ones.
REP PROTOCOL Do the 40-minute solo timed rep FIRST — out loud, whiteboard or paper, no peeking. Then read this reference, compare, and log every delta (missed component, wrong number, unnamed failure mode) as a learning record. The deltas are the lesson; the text below is just the answer key.

1 · The prompt, and what it tests

"Design a distributed key-value store: get(key) and put(key, value) over a dataset far larger than one machine, highly available, with tunable consistency." One sentence, and it is the rep where CAP (lesson 0004) stops being theory: every component you draw is a concrete answer to "what happens during a partition?" The interviewer is scoring four things: how you partition data across nodes, how you replicate for durability and availability, how you make the consistency trade-off explicit and tunable, and how the system heals itself when nodes die — because at this scale they always are dying. The API is trivially small; the whole interview lives underneath it. Budget the standard rep shape: clarify 5, estimate 5, high-level 10, deep dives 20.

That is a choice: if this were a ledger, I'd flip to a CP design and pay availability during partitions." Naming the fork before designing is the senior move; the interviewer now knows every later decision is deliberate.

3 · Estimate (5 min): numbers you will spend

Dataset:      100 TB ÷ 10 KB/value  ≈ 10^10 = 10 billion keys
Replicated:   100 TB × 3 (N=3)      = 300 TB raw
Nodes:        300 TB ÷ 2 TB usable  = 150 nodes
Traffic:      1M ops/s peak ÷ 150   ≈ 7k ops/s per node (×3 for write fan-out)

Now spend each number. 150 nodes → with commodity hardware, a node failure is a daily event, not an incident: failure handling (gossip, hinted handoff) must be automatic, and rebalancing must not stampede. 10 billion keys → no node can hold an index of everything; placement must be computable from the key itself — that is why consistent hashing, not a lookup table. 7k ops/s per node → CPU is comfortable; the binding constraints are storage and recovery bandwidth, so the deep dives go to replication and healing, not to compute.

4 · High-level: the ring

There is no master. A client sends get/put to any node (or a thin coordinator tier); that node becomes the coordinator for the request. Placement is consistent hashing — lesson 0006's tool, verified in lab-0006: hash node IDs and keys onto one circular space; a key belongs to the first node clockwise. Adding or removing a node moves only ~1/N of the keys, not everything — the property that makes 150-node membership churn survivable. Each physical node projects ~100–200 virtual nodes onto the ring, so when a node dies its load scatters in small slices across many survivors instead of dumping wholesale onto one clockwise neighbor — vnodes are what make rebalancing smooth rather than lumpy, and they let a beefy node take proportionally more vnodes.

Replication: the key's home node plus the next N−1 distinct physical successors on the ring form its preference list — with N=3, every key lives on 3 nodes. The coordinator forwards writes to all N and answers according to the quorum rules below. Everything else in this design — quorums, hinted handoff, Merkle repair — is machinery for keeping those 3 copies honest.

5 · Deep dive: quorums — N, R, W

Three knobs: N replicas per key, a write is acknowledged after W replicas confirm, a read consults R replicas and returns the freshest version seen. The load-bearing theorem: if R + W > N, every read set intersects every write set. The argument, drawn out: the write landed on some W-subset of the N replicas; the read consults some R-subset; two subsets of an N-element set with |R| + |W| > N cannot be disjoint — pigeonhole — so at least one node in the read set holds the acknowledged write, and taking the freshest version surfaces it. Set R + W ≤ N and the sets can miss each other entirely: the read may consult only replicas the write never reached. Your lab measures exactly this gap.

Config (N=3)OptimizesCost
R=1, W=3Fast reads (one replica answers)Slow, fragile writes — one dead replica blocks puts
R=3, W=1Fast writesSlow reads; a lone-replica write risks loss if it dies
R=2, W=2Balanced — the common defaultBoth paths tolerate exactly one dead replica

Sloppy quorum: a strict quorum refuses writes when fewer than W preference-list replicas are alive. Dynamo instead writes to the next healthy nodes past the preference list; the stand-in stores the value with a hint ("this belongs to node D") and hands it back when D recovers — hinted handoff. Availability preserved; the price is that quorum overlap is no longer guaranteed during failures.

6 · Deep dive: conflict resolution — LWW vs vector clocks

An AP system accepts writes on both sides of a partition, so conflicting versions will exist. Last-write-wins (LWW) resolves by timestamp: highest wall-clock wins, the other write silently vanishes. Cheap, and Cassandra defaults to it — but clocks skew, and "concurrent" has no true order, so LWW is a policy for losing data quietly. Acceptable for a cache; unacceptable for a shopping cart.

Vector clocks keep causality instead. Worked example, Dynamo's own: a cart is written via node A → version [A:1]. The client updates it via A again → [A:2]. Now a partition: one client adds a book via A → [A:3]; concurrently another adds a lamp via node B, descending from [A:2] → [A:2, B:1]. Compare: [A:3] and [A:2, B:1] — neither dominates the other componentwise, so the system knows these are concurrent, not ordered. It keeps both as siblings and returns both on the next read; the client merges (union the carts — book and lamp survive) and writes back [A:4, B:1], collapsing the siblings. The trade: LWW loses updates invisibly; vector clocks push merge complexity to the reader. Say which your use case tolerates.

7 · Deep dive: anti-entropy — repair without a master

Replicas drift — dropped messages, dead nodes, sloppy quorums. Three healing mechanisms, cheapest first:

  • Read repair. The coordinator reads R replicas, notices one returned a stale version, and pushes the freshest version to it in the background. Free healing — but only for keys that get read.
  • Merkle trees. For cold keys, replicas periodically compare a hash tree of their key range: leaves hash buckets of keys, parents hash their children. If two roots match, the ranges are identical — one comparison. If not, recurse only into the differing child: divergent keys are found in O(log n) hash exchanges instead of shipping 10 billion entries. This is how Dynamo and Cassandra sync replicas without drowning the network.
  • Gossip. Membership and failure detection with no master: every second each node exchanges its view (who's alive, who owns what) with a random peer; information spreads epidemically in O(log n) rounds. A node missing heartbeats gets marked suspect → dead, triggering hinted handoff for its writes.

8 · Deep dive: the storage engine on each node

Each node persists its key range with an LSM-tree — three-sentence recap of lesson 0012: writes land in an in-memory memtable, which flushes to immutable sorted SSTable files when full; reads check memtable then SSTables newest-first; background compaction merges SSTables and drops overwritten versions. LSM buys the sequential-write throughput a write-heavy KV node needs (full treatment: lesson 0012). Two companions: a WAL — every write appends to a write-ahead log before the memtable ack, so a crash replays the log and loses nothing; and a bloom filter per SSTable — a probabilistic "definitely not here / maybe here" bitmap that lets a get skip most SSTables without touching disk, which at 10 billion keys is the difference between one disk seek and ten.

10 · Variants and twists

The cache variant — the Primer's "query cache for a search engine" question — is this same design with durability deleted: same ring, same vnodes, same replication if you want hit-rate resilience, but no WAL, no SSTables, values carry TTLs, and each node evicts with LRU — exactly the policy you built in lab-0007. Memcached is this design with N=1; Redis Cluster is this design with async replication. If the interviewer says "cache", drop section 8 and spend the minutes on eviction and TTL-vs-invalidation instead.

The CP twist: "what if we need strong consistency?" Swap quorum replication for consensus — each shard runs Raft (lesson 0014), writes commit through a leader, partitioned minorities refuse writes. That is etcd or Spanner territory: linearizable, and unavailable on the minority side — say the price out loud.

AI note

An inference platform's prefix-cache router is this lesson wearing GPU clothes: consistent-hash prompt prefixes across workers so repeated prefixes hit the worker already holding their KV blocks, evict cold prefixes LRU-style, and accept staleness — it's the cache variant, not the durable one.

11 · Interview talking points

  • Framing: "Requirement says never refuse writes → I'm building AP, Dynamo-style. If this were payments I'd build CP — want me to design that instead?"
  • Spending a number: "150 nodes means failure is a Tuesday, so recovery must be automatic — that's why gossip and hinted handoff, not an ops runbook."
  • The theorem: "R=2, W=2, N=3: read and write sets of 2 can't be disjoint in a set of 3, so reads see acknowledged writes — unless a sloppy quorum stretched the ring, which is the availability price."
  • Conflicts: "LWW is fine for the cache variant; for carts I keep siblings with vector clocks and let the client merge — Dynamo's actual choice, losing no adds."
  • Unprompted failure mode: "Before you ask: a partition creates split-brain writes; after healing, my sibling-count metric spikes and read repair converges the replicas."
  • Closing: "I'd page on hint-queue depth and divergence rate; with more time — cross-region replication and compaction scheduling."
