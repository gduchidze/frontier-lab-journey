Security & Multi-Tenancy

Who are you, what may you do, where do the secrets live, and how do you keep tenant A out of tenant B's data and tenant B's latency — the defensive vocabulary that turns "add auth" into a design.

Mission tie. Anthropic's design prompts — "design the API for a multi-tenant SaaS platform" — always grow a security limb: "how does one customer's traffic not affect another's? how do you know this request is allowed to read this row?" Candidates who wave at "we'll use JWTs" stall; candidates who name authN vs authZ, the token's audience check, the tenant-isolation model, and the rate-limit dimension keep going. This lesson gives you those names with the classic systems — a B2B SaaS app, a banking API, a Slack-like workspace product — where each one is load-bearing. (Exponent's 2026 Anthropic guide)

1 · The perimeter is dead — zero trust

The old model was a castle: a hard firewall around a soft interior, and once you were "inside the network" you were trusted. That model died the day a single phished laptop or one compromised service could roam the whole internal network freely. The replacement is zero trust: there is no inside. Every hop authenticates, every hop is encrypted in transit, and every identity gets the least privilege that lets it do its job (Google, BeyondCorp: A New Approach to Enterprise Security).

Three principles carry the whole lesson:

  • Authenticate every hop. The browser proves identity to the gateway; the gateway proves identity to the order service; the order service proves identity to the database. "It came from inside" is not a credential.
  • Encrypt in transit, always. TLS on the edge and between internal services — the network is assumed hostile, because sometimes it is.
  • Least privilege. The reporting service that only reads gets a read-only role. A leaked credential should unlock a room, not the building.

Wrapping these is defense in depth: layered controls so no single failure is catastrophic. Auth at the edge, authz at the service, row filters at the database, secrets in a vault, rate limits in front — if one layer is bypassed, the next still holds. The canonical checklist of what those layers must resist is the OWASP Top 10; the verification-requirements companion is the OWASP ASVS. Everything below is one of these layers.

2 · AuthN vs AuthZ — two words people fuse into one

The single most common vocabulary slip in an interview is treating authentication and authorization as one thing. They are separate systems that fail separately.

Authentication (authN)Authorization (authZ)
QuestionWho are you?What may you do?
ProducesA verified identity (a user, a service)An allow / deny decision on an action
RunsOnce at the edge, per session/requestAt every resource, per action, per tenant
Classic failureStolen password, forged tokenValid user reads another tenant's row
ModelsPasswords, OAuth, mTLS, passkeysRBAC, ABAC, row-level security, scopes

The trap the interviewer is hunting for: a request can be perfectly authenticated and still unauthorized. A logged-in user of tenant A presenting a flawless token still must be denied tenant B's data — and that denial is authZ's job, checked at the resource, not authN's job at the door. Say this split out loud early; it structures the rest of your answer.

Where identity lives between requests. Two shapes. Session cookies: the server stores session state, the cookie holds only an opaque ID; revocation is a row delete, and the cookie is scoped to one origin with HttpOnly; Secure; SameSite. Tokens (bearer): the client holds a signed credential and sends it on every request, often to many services; stateless to verify, but harder to revoke. Cookies suit a single web app; tokens suit APIs and service meshes (OWASP Session Management Cheat Sheet).

3 · OAuth2 & OIDC — delegated access, then identity

OAuth2 answers one precise question: how does an app act on a user's behalf without holding the user's password? It is a delegated authorization framework, not a login protocol. Four roles (RFC 6749, and the readable index at oauth.net):

RoleWho it isExample (Slack-style app)
Resource ownerThe user who owns the dataThe employee whose messages they are
ClientThe app wanting accessA calendar bot requesting message scope
Authorization serverIssues tokens after consentThe workspace's identity provider
Resource serverHolds the data, checks the tokenThe messages API

Authorization-code flow with PKCE, in words (the flow to know cold): the client redirects the user to the authorization server with a code_challenge (the hash of a random secret it just generated). The user logs in and consents. The auth server redirects back with a short-lived authorization code. The client then calls the token endpoint, presenting the code plus the original code_verifier; the server hashes the verifier, checks it equals the challenge it stored, and only then returns tokens. PKCE (RFC 7636) closes the attack where a malicious app intercepts the redirected code: a stolen code is useless without the verifier that never left the real client. PKCE is now recommended for all clients, not just mobile.

Client-credentials flow is the other one to name: no user at all — a service authenticates as itself to get a token for machine-to-machine calls (a nightly batch job calling the billing API). If there is a human and a consent screen, it's authorization-code; if it's a backend acting on its own behalf, it's client-credentials.

OIDC is the identity layer OAuth2 lacks. OAuth2 grants access; it never actually tells the client who logged in. OpenID Connect adds a signed ID token (a JWT with the user's identity claims) issued alongside the access token, so "Sign in with Google" can both authenticate the user and be handed an access token (OIDC Core spec). One-line memory hook: access token = what you may do; ID token = who you are; OIDC = OAuth2 + identity.

4 · JWT, honestly

A JSON Web Token is three base64url parts joined by dots — header.payload.signature (RFC 7519). The header names the algorithm; the payload holds claims (sub, iss, aud, exp, plus your own); the signature is an HMAC or RSA/ECDSA signature over the first two parts. The win is stateless verification: any service holding the signing key (or public key) can verify the token locally — no round trip to an auth database on every request. That is why JWTs dominate service meshes and horizontally-scaled APIs.

But stateless verification is exactly the catch: you cannot un-issue a JWT. Once signed and handed out, it is valid until it expires, whatever happens to the user meanwhile. The honest mitigations:

  • Short TTL + refresh tokens. Access tokens live minutes; a long-lived refresh token (revocable, stored server-side) mints new ones. A fired employee loses access within the access-token TTL, not instantly — state the window out loud.
  • Denylist for the "revoke now" case: a small store of blacklisted token IDs (jti) checked on each request. This re-introduces state, which is the whole point — you trade a lookup for real-time revocation on the tokens that matter.

The pitfalls interviewers probe:

  • alg=none. Early libraries honored a header claiming "no algorithm" and skipped signature verification — so an attacker strips the signature, sets alg to none, and forges any claims they like. A strict verifier must pin the expected algorithm and reject anything else (Auth0: critical JWT vulnerabilities). You will exploit this exact bug in the lab.
  • Key confusion (RS256 → HS256). If the verifier lets the token pick the family, an attacker signs with HMAC using the public RSA key as the secret; a naive library "verifies" it. Pin the algorithm.
  • Audience unchecked. A token minted for service X, if X never verifies aud, is happily accepted by service Y. Verifying the signature is not enough — verify iss, aud, and exp too.
  • Secrets in claims. The payload is signed, not encrypted — it is base64, readable by anyone holding the token. Never put passwords, PII, or keys in claims.

The alternative: opaque tokens + introspection. Instead of a self-describing JWT, hand out a random string that means nothing on its own; the resource server calls the auth server's introspection endpoint (RFC 7662) to ask "is this live, and what does it grant?" Revocation is instant (delete the row), but you pay a network hop per verification.

JWT (self-contained)Opaque token + introspection
VerifyLocal, offline, signature onlyNetwork call to auth server
RevocationHard — TTL or denylistInstant — delete the record
Scales to N servicesCheaply (no central bottleneck)Auth server becomes hot path
Leaks on decodeAll claims are readableNothing — it's just an ID
Best forHigh-fanout internal APIsSensitive, revocation-critical access

Interview move: default to short-TTL JWTs for scale, add a denylist for the revoke-now cases, and reach for opaque tokens when the data is sensitive enough that instant revocation beats offline verification (Okta: validating access tokens).

5 · Service-to-service — proving identity between machines

Inside the mesh, "the call came from the order service" needs proof, not a hostname. Three mechanisms, weakest to strongest:

MechanismHow it proves identityWeakness
Shared API keyCaller sends a secret stringOne side only; leaks are silent, rotation is manual
Signed token (JWT)Caller presents a signed identity claimBearer — a stolen token is fully usable
mTLSBoth sides present certificates; each verifies the otherCert lifecycle is real work (that's what the mesh is for)

Mutual TLS is the zero-trust default for internal traffic: ordinary TLS proves the server's identity to the client; mTLS additionally makes the client present a certificate, so both ends are cryptographically identified and the channel is encrypted. The operational cost — issuing, distributing, and rotating thousands of short-lived certs — is exactly why service meshes exist. SPIFFE/SPIRE gives every workload a verifiable identity document (a SVID) and rotates it automatically (SPIFFE overview); Istio can turn on mTLS mesh-wide and handle rotation without app changes (Istio security concepts). Say "mesh-managed mTLS with short-lived certs" and you have named the whole pattern. API keys still have a place — third-party developer access at the public edge — but inside the mesh they are the weak option because they authenticate only one direction and leak without a trace.

6 · Secrets management

The first failure is upstream of any algorithm: secrets in the code, or in an .env file committed to git. Once a credential is in git history it is compromised forever — rotate it, don't just delete the file. The correct model is a central secrets manager (HashiCorp Vault, AWS Secrets Manager, cloud KMS-backed stores) with four properties:

  • Central store, not scattered files. One audited system holds secrets; apps fetch at boot or on demand with their own identity.
  • Short-lived dynamic credentials. The best database password is one Vault generates on request and revokes minutes later — a leaked credential has already expired. Static long-lived secrets are the ones that hurt.
  • Audit log. Every read is recorded: who fetched which secret, when. After an incident you can answer "what was exposed?"
  • Rotation as routine. Secrets rotate on a schedule and on demand, without a deploy.

Data at rest: envelope encryption. You don't encrypt a terabyte with the master key. Instead a KMS holds a root key-encryption key (KEK) that never leaves the hardware; each dataset gets its own data-encryption key (DEK) which encrypts the data, and the DEK is stored next to the data encrypted by the KEK. To read, you ask KMS to unwrap the DEK; to rotate the master, you re-wrap DEKs, not re-encrypt petabytes (AWS KMS: envelope encryption). This is how per-tenant encryption keys become practical — a DEK per tenant, all wrapped by one KEK.

7 · Rate limiting as a security control

Rate limiting is usually filed under "capacity," but half its value is security. It is the cheapest defense against a family of abuse:

  • Brute force — guessing one account's password. A limiter of 5 attempts / minute / account turns a million-guess attack into a two-century one.
  • Credential stuffing — replaying breached username/password pairs across your login (OWASP: credential stuffing). This one is spread thin across many accounts, so a per-account limit misses it — you also need per-IP and global-anomaly limits.
  • Scraping — draining your catalog or user directory through the public API.
  • Abuse economics. The frame that lands in interviews: your job is to make the attack cost more than it yields. Every 429 you return raises the attacker's cost per attempt; you don't need to block them, only to make them unprofitable.

The three dimensions you key the limiter on, and why you usually need more than one:

DimensionCatchesBlind spot
Per-API-keyOne customer's runaway integrationAttacker rotating through many keys
Per-IPA single abusive hostNAT (a whole office = one IP); botnets (many IPs)
Per-tenantOne tenant starving the platformCross-tenant distributed abuse

Two design notes that separate a real answer from a textbook one. First, return 429 Too Many Requests with a Retry-After header — a well-behaved client backs off, a bad one identifies itself. Second, protect the limiter itself: do the cheap rejection before the expensive work. Checking a token bucket must not require a database read or a full auth handshake, or the check becomes the DoS. Push the cheapest possible reject (IP/key bucket in memory or Redis) as early in the request path as you can, ahead of auth where the endpoint allows it. Lesson 18 takes this further into full overload and load-shedding behavior (Google SRE: Handling Overload) — here the point is only that the limiter is a security layer, and it must survive the attack it's defending against.

8 · Multi-tenancy isolation models

One platform, many customers (tenants) whose data and performance must not bleed into each other. The core decision — memorize this table, it is the spine of any SaaS design answer (AWS SaaS Lens: tenant isolation):

ModelLayoutIsolationCost / densityOps burden
SiloDB (or stack) per tenantStrongest — physical separationWorst — pay per tenant, idle or notMigrations & backups ×N tenants
BridgeShared DB, schema per tenantMedium — logical separationBetter — one cluster, many schemasMigrations ×N schemas, one fleet
PoolShared tables, tenant_id columnWeakest — a filter is the only wallBest — full density, one everythingLowest ops, highest blast radius

The arithmetic interviewers want to hear. Say 500 tenants. Silo: a small managed Postgres runs ~$50/mo, so 500 idle-or-not instances ≈ 500 × $50 = $25,000/mo in database floor alone, plus 500 backup jobs and 500 migration runs per release. Pool: one 3-node cluster at ~$2,000/mo ≈ $6,000/mo shared across all 500, one migration per release. That's roughly a 4× infrastructure gap and an N× vs constant ops gap — and it flips the moment one tenant is a bank that will pay for a silo and walk over a shared table. There is no best model; there is a best model per tenant tier, and mature SaaS runs a mix (pool for the long tail, silo for the whales).

Hardening the pool with row-level security. The pool model's whole risk is that isolation rides on every query remembering WHERE tenant_id = ?. Postgres RLS moves that wall into the database: attach a policy like USING (tenant_id = current_setting('app.tenant_id')) and the engine appends the filter to every query automatically — a helper that forgets the filter still cannot see another tenant's rows (PostgreSQL: Row Security Policies). It turns "hope every developer remembers" into "the database enforces it."

Noisy neighbor = performance isolation. Data isolation is only half. In the pool model tenants share workers, connections, and queues, so one tenant's batch job can starve everyone else's latency — the noisy neighbor. The fixes are the performance twin of RLS: per-tenant quotas (token buckets per tenant, §7), fair queuing (round-robin or weighted dequeue so no tenant monopolizes the worker pool), and bulkheads (isolate resource pools so one tenant's saturation can't sink the ship — from Release It!). Isolation is two invariants, not one: tenant A can't read tenant B's data, and tenant A can't consume tenant B's capacity. You will make both fail and then fix them in the lab.

9 · The failure modes that decide interviews

Security bugs are rarely exotic; they are one missing check. The classics interviewers reach for:

  • Missing tenant_id on one query. The signature SaaS bug: four of five query helpers filter by tenant, the fifth forgets, and tenant A sees tenant B's orders. It passes every test that uses one tenant. Defense: RLS or a default-scoped data layer so the filter is structural, never per-query discipline.
  • JWT verified but audience unchecked. The signature is valid, exp is fine, so the service accepts a token that was minted for a different service. Always check aud and iss, not just the signature.
  • Long-lived leaked API key, no rotation. A key committed to a public repo two years ago still works, because nothing expires and nothing rotates. Short-lived credentials would have closed the window automatically.
  • Rate limiter keyed on IP behind NAT. A 300-person office shares one egress IP; your per-IP login limit locks out the entire company on a few fat-fingered passwords. Key sensitive limits on the account, and treat IP as one signal among several.
  • One tenant's batch job starves the pool. No fair queuing, so a single tenant's bulk export fills the shared worker queue and every other tenant's p99 explodes — a data-isolated system with zero performance isolation. Per-tenant quotas plus fair dequeue restore it.

10 · One AI note, then back to fundamentals

Everything above transfers unchanged to a multi-tenant inference platform. Per-tenant token quotas + fair scheduling on the GPU queue is exactly §8's noisy-neighbor problem — one tenant's flood of long-context requests must not blow up another tenant's TTFT, so the batching scheduler dequeues fairly (weighted by tier) rather than FIFO. Prompt and output logs are sensitive data: treat them like PII — encrypt at rest with per-tenant DEKs (§6), set a retention policy, and never let them into JWT claims or shared logs. And model access is an authZ scope: "may tenant A call the 70B model?" is a scope check on the access token (§4), identical to any RBAC decision. Same patterns, GPU-priced blast radius. That's the note; §1–§9 are what the interview is grading.

11 · Interview talking points

How to deploy this lesson live, in order:

  1. Split authN from authZ in the first sentence. "The gateway authenticates the caller; each service authorizes the action against the tenant." This alone signals you won't conflate them (§2).
  2. Name the token shape and its revocation story. "Short-TTL JWTs for internal fan-out, a denylist for revoke-now, opaque tokens on the sensitive path" — pattern plus its weakness (§4).
  3. Say mTLS for east-west traffic and let the mesh own cert rotation (§5): "SPIFFE identities, mesh-managed mTLS, no app changes."
  4. Put secrets in a vault out loud (§6): "dynamic short-lived DB creds from Vault, envelope encryption with per-tenant DEKs — nothing sensitive in code or env."
  5. Frame rate limiting as security (§7): "per-account limit for brute force, per-IP and global for credential stuffing, cheap reject before auth so the limiter survives the attack."
  6. Pick a tenancy model and defend the mix (§8): "pool with RLS for the long tail, silo for regulated whales — here's the 4× cost math." Then name performance isolation, not just data.
  7. Raise the failure modes before the interviewer does (§9): "the risk is a query that forgets tenant_id — that's why isolation lives in RLS, not in code review."

Hands-on (do this before the quiz) Everything above is now verifiable in labs/lab-0017-tenant-isolation/: python3 tenant_lab.py runs three experiments — a JWT forgery drill (valid passes; tampered and expired fail; an alg=none token that a naive verifier accepts and a strict verifier rejects, with the forged-admit count going to zero), a noisy-neighbor run (tenant B floods 10×; a FIFO shared queue explodes tenant A's p99, then per-tenant token buckets + round-robin fair dequeue restore it and throttle B with counted 429s), and a pool-model leak drill (five query helpers, one forgets its tenant_id filter; an invariant checker scans results and names the guilty helper). 13 · Project step

Decision time for the LLM inference platform: choose its tenancy model and its authentication scheme, and defend both in two sentences. A defensible pick: pool model with Postgres RLS and per-tenant token-bucket quotas plus fair GPU-queue scheduling (full density for the long tail, hard data and performance walls), fronted by short-TTL OIDC access tokens with audience checks and mesh-managed mTLS between the gateway, router, and workers.
