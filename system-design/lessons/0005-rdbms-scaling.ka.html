RDBMS-ის მასშტაბირება: Replication, Sharding, Indexes

ერთი ყუთი Postgres საოცრად შორს მიგიყვანს — აი ზუსტი თანმიმდევრობა, რომლითაც უნდა დახარჯო მისი სიცოცხლეები, და რატომ არის sharding კიბის ბოლო საფეხური და არა პირველი.

მისიასთან კავშირი. Anthropic-ის დიზაინის ამოცანები ხშირად შენიღბულ გიგანტურ append-heavy ცხრილს შეიცავს: „დააპროექტე საუბრების საცავი Claude-ის მასშტაბის ჩატ-პროდუქტისთვის." ინტერვიუერი ორ კონკრეტულ წინადადებას ელოდება — ერთს, რომელიც ასახელებს ინდექსს (index) ყოველი დაპირებული query-ის უკან, და მეორეს, რომელიც იცავს შენს shard key-ს celebrity პრობლემისგან. ორივე წინადადება ამ გაკვეთილის შედეგია.

1 · ACID — და როდის გჭირდება ის სინამდვილეში

ACID არის ტრანზაქციის (transaction) კონტრაქტი: Atomicity (მრავალნაბიჯიანი ცვლილება ან მთლიანად ჯდება, ან საერთოდ არა), Consistency (შეზღუდვები და ინვარიანტები ძალაშია ცვლილებამდეც და მის შემდეგაც), Isolation (პარალელური ტრანზაქციები ვერ ხედავენ ერთმანეთის ნახევრად შესრულებულ სამუშაოს), Durability (რაც დაკომიტდა, კრახი (crash) ვეღარ წაშლის). ტრანზაქციები გჭირდება, როცა რამდენიმე რიგი ერთად უნდა შეიცვალოს და ნაწილობრივი მდგომარეობა დაუშვებელია — მომხმარებლის token კვოტის შემცირება და მოხმარების ჩანაწერის დამატება, ფულის გადარიცხვა, უნიკალური username-ის დაკავება. ისინი არ გჭირდება ერთრიგიანი append-ისთვის, როგორიცაა ერთი ჩატ-შეტყობინების ჩასმა; serializable isolation-ში იქ ფულის გადახდა ისეთი ინვარიანტის დაზღვევაა, რომელიც არ გაქვს. (DDIA, თავი 7)

2 · ინდექსები: B-tree, რომლის დახატვაც უნდა შეგეძლოს

ინდექსი (index) არის B-tree: დახარისხებული გასაღებები განიერ, თხელ ხეში — რამდენიმე ასეული შვილი თითო კვანძზე, ასე რომ 300 მილიონი რიგი ~4 სიღრმის ხეა. წერტილოვანი წაკითხვა (point read) ფესვიდან ფოთლამდე O(log n) გვერდის შეხებით ჩადის ცხრილის სკანირების ნაცვლად; დიაპაზონური query დიაპაზონის დასაწყისამდე ჩადის და შემდეგ ფოთლებს თანმიმდევრულად კითხულობს. ფასი არის write amplification (ჩაწერის გაძლიერება): ყოველი ინდექსი კიდევ ერთი B-tree-ა, რომელიც ყოველ insert-სა და update-ზე უნდა განახლდეს — ხუთინდექსიანი ცხრილი თითო ჩაწერილ რიგზე ექვს სტრუქტურას წერს.

  • კომპოზიტური ინდექსის რიგითობის წესი. ინდექსი (user_id, created_at)-ზე ტელეფონის წიგნია, დახარისხებული ჯერ გვარით, მერე სახელით. ის იდეალურად ემსახურება query-ს „equality user_id-ზე, შემდეგ range/sort created_at-ზე" — და თითქმის უსარგებლოა „range მხოლოდ created_at-ზე"-სთვის, ზუსტად ისე, როგორც ტელეფონის წიგნი უსარგებლოა ყველა ანას საპოვნელად. Equality სვეტები ჯერ, range/sort სვეტი ბოლოს.
  • Covering index (მფარავი ინდექსი). თუ ინდექსი შეიცავს ყველა სვეტს, რომელსაც query ეხება, ძრავა ცხრილს საერთოდ არ ეწვევა — წაკითხვა მთლიანად ინდექსიდან სრულდება.
დიზაინის დისციპლინა დიზაინში ყოველმა query-მ თავისი ინდექსი უნდა დაასახელოს. „მომხმარებლის ბოლო 50 შეტყობინებას ვიღებთ — (user_id, created_at) ინდექსით" — ეს პასუხია. „შეტყობინებებს ვიღებთ" — ეს full table scan-ია, რომელიც ინტერვიუს დამაზუსტებელ კითხვაზე გამოაშკარავდება.

3 · მასშტაბირების კიბე — თანმიმდევრობით, გამოტოვების გარეშე

როცა ერთი primary გაჯერდება, ადი ამ თანმიმდევრობით; ყოველი საფეხური უფრო იაფი და უფრო შექცევადია, ვიდრე მომდევნო. (Primer §12: RDBMS)

საფეხურირას ყიდულობსრა ღირს
1 · Read replica-ებიწაკითხვის გამტარუნარიანობა, იაფადReplication lag (გაკვეთილი 0004); ჩაწერები ისევ ერთ ყუთზეა
2 · ქეშირება (caching)შთანთქავს ცხელ წასაკითხ ნაკრებსInvalidation-ის ბაგები, მოძველებული წაკითხვები
3 · Federation (ფედერაცია)დაყოფა ფუნქციით: users DB, messages DB, billing DBბაზათაშორისი join-ები აღარ არსებობს; თითო სერვისის ჭერი რჩება
4 · Sharding (დანაწილება)ჩაწერის მასშტაბირება, ბოლოს და ბოლოსყველაფერი მე-4 სექციიდან

დენორმალიზაცია (denormalization) კიბის გვერდით მიდის: მონაცემს აკოპირებ, რომ წაკითხვა, რომელიც 3-მაგისტრალიანი join იყო, ერთ lookup-ად იქცეს — ყოველ ჩაწერას ორჯერ (ან სამჯერ) იხდი, რომ ყოველი წაკითხვა გააიაფო. Materialized view ეს გარიგებაა ბუღალტერიით ერთად: წინასწარ გამოთვლილი query-ის შედეგი, რომელსაც ბაზა შენს მაგივრად აახლებს. წაკითხვაზე მძიმე სისტემები (ჩატის ისტორია ~100:1 წაკითხვა/ჩაწერაა) ამ გარიგებას მუდმივად ყიდულობენ.

4 · Sharding: უკანასკნელი საშუალება, პუნქტობრივად

Sharding ერთი ცხრილის რიგებს ანაწილებს მანქანებზე shard key-ით. Hash sharding დატვირთვას თანაბრად შლის, მაგრამ ანადგურებს დიაპაზონურ სკანირებას (მეზობელი გასაღებები სხვადასხვა მანქანაზე ხვდებიან); range sharding დიაპაზონებს ლოკალურად ინახავს, მაგრამ ცხელ დიაპაზონებს იწვევს — დაანაწილე created_at-ით და დღევანდელი shard ჩაწერების 100%-ს იღებს. Sharding-ის დღესვე ოთხი ხარჯი მოდის:

  • ცხელი გასაღებები (celebrity პრობლემა). ჰეშირება გასაღებებს აბალანსებს და არა დატვირთვას. ერთი გასაღები 100× ტრაფიკით ერთ shard-ს ავარვარებს, სანამ სამი უქმად დგას — ლაბი ამას Zipf დატვირთვით თვალსაჩინოს ხდის.
  • Resharding-ის ტკივილი. გულუბრყვილო hash(key) % N-ით N-ის შეცვლა თითქმის ყველა რიგს გადაანაწილებს. მიგრაციის გეგმა პირველივე დღეს დაწერე (consistent hashing მოგვიანებით გაკვეთილში დაბრუნდება).
  • Shard-თაშორისი query-ები scatter-gather ხდება. ნებისმიერი კითხვა, რომელიც shard key-ზე არ არის მიბმული, ყველა shard-ზე უნდა გაიშალოს და შედეგები გაერთიანდეს — და გაკვეთილმა 0001 გითხრა, რას უშვება fan-out კუდის დაყოვნებას.
  • Auto-increment ID-ები ტყდება. ყოველი shard სიამოვნებით აგენერირებს საკუთარ 1, 2, 3…-ს — კონფლიქტური მიმდევრობები. გჭირდება ცენტრალური ან სტრუქტურირებული ID გენერაცია (UUID-ები, Snowflake-ის სტილის ID-ები).

AI არომატი — სად დანაწილდეს ჩატ-პროდუქტის messages ცხრილი? დაანაწილე conversation_id-ით: ერთი ჩატის დახატვა ერთ shard-ს ხვდება, მაგრამ „ჩამომითვალე ჩემი ყველა საუბარი" ყოველ გვერდით პანელზე შესვლისას ყველა shard-ზე scatter-gather-ობს. დაანაწილე user_id-ით: მთელი ისტორიის გვერდი ერთი shard-ია, მაგრამ უზომოდ აქტიური ანგარიში — ორგანიზაციის workspace, API-ს მბომბავი ბოტი — ცხელი shard ხდება, და მომავალი გაზიარებული-საუბრის ფუნქცია კოლოკაციას კარგავს. Claude-ის მსგავსი პროდუქტისთვის, სადაც საუბრები ერთ მომხმარებელს ეკუთვნის, user_id ჩვეულებრივ იმარჯვებს; შემდეგ celebrity შემთხვევას ხმამაღლა იცავ (ვეშაპების sub-partition ან იზოლაცია).

5 · ლაბი — დაანაწილე და დააინდექსე შენი ხელით

პრაქტიკა (ქვიზამდე გააკეთე) labs/lab-0005-shard-and-index/-ში: python3 shard_index_lab.py აშენებს 300,000-რიგიან messages ცხრილს და უშვებს სამ ექსპერიმენტს — წერტილოვანი/დიაპაზონური query-ები ინდექსით და მის გარეშე (EXPLAIN QUERY PLAN-ის ქვითრებით), კომპოზიტური რიგითობის წესი, როგორც ხელშესახები დროის სხვაობა, და 4-shard-იანი დაყოფა, სადაც Zipf დატვირთვა ცხელ shard-ს აჩენს და top-10 query იძულებით scatter-gather-ს ითხოვს. შემდეგ README-ს E1–E4 სავარჯიშოები: იწინასწარმეტყველე, შეცვალე კონფიგი, გაუშვი თავიდან, შეადარე.

6 · პროექტის ნაბიჯი

ჩვენი ხერხემალი-პროექტი, LLM inference gateway, ინახავს request_log ცხრილს: (api_key_id, model, created_at, prompt_tokens, completion_tokens, latency_ms). აირჩიე მისი shard key და დაასაბუთე ზუსტად ორი წინადადებით — ერთი ასახელებს დომინანტურ query ნიმუშს და მის ინდექსს, მეორე იცავს ცხელი-გასაღების შემთხვევისგან (ერთი enterprise API key შეიძლება მთელი ტრაფიკის 30% იყოს). ორივე წინადადება შემდეგ სესიაზე მოიტანე; მასწავლებელი მეორეს შეუტევს.
