TinyURL & Pastebin: IDs, Redirects, Read-Heavy Caching

The classic warm-up rep. Small surface, deep interior: unique ID generation at scale and a brutally read-heavy serving path — patterns you will reuse in every design that follows.

Mission tie. This is the canonical warm-up prompt, and its two hard problems never leave you: distributed unique ID generation (post IDs in the news feed, message IDs in chat, event IDs in payment ledgers) and read-heavy caching (every feed, catalog, and profile page you will ever design). Nail the reasoning here and lessons 0020–0028 inherit it for free.
REP PROTOCOL — do this first Before reading a single section below: 40-minute solo timed design, out loud, using the 4-step framework from lesson 0009 (clarify → estimate → high-level → deep dive). Prompt: "Design a URL shortener like TinyURL; extend it to a Pastebin." Only then read this reference solution and log your deltas as a learning record.

1 · The prompt, and what is really being tested

"Design TinyURL" sounds like a toy: two endpoints, one table. That is the trap. The interviewer is testing whether you can find the hard problems inside a simple surface:

  • Uniqueness at scale. Billions of short codes, no duplicates, no coordination bottleneck — the distributed ID generation problem in its purest form.
  • An extreme read/write ratio. A link is written once and read thousands of times. Your whole architecture should visibly bend around that asymmetry.
  • One consequential HTTP detail. 301 vs 302 — a single status code that trades server load against analytics.

Pastebin is the same skeleton plus one twist: the payload is no longer a 100-byte URL but a multi-kilobyte (sometimes multi-megabyte) blob — which forces the metadata-vs-blob storage split. A candidate who says "same design, bigger values" misses the point; a candidate who says "same control plane, different data plane" gets it.

3 · Estimate (~5 min): numbers you will spend

Writes:  100M links/month ÷ 2.6M s/month ≈ 40 links/s avg  → ×3 peak ≈ 120/s
Reads:   100:1 ratio → 4,000 redirects/s avg → ×3 peak ≈ 12k QPS
Storage: 100M/mo × 60 months = 6B links × 500 B/row ≈ 3 TB metadata
Codes:   need 6B unique → base62, 62^7 ≈ 3.5 × 10^12 → 7 chars, ~600× headroom
Cache:   reads/day ≈ 4k × 86,400 ≈ 350M; 80/20 rule → cache ~20% of
         daily objects: 0.2 × 350M × 500 B ≈ 35 GB → one Redis box, 3 for HA
Pastebin blobs: 10M/mo × 10 KB ≈ 100 GB/month ≈ 6 TB over 5 years

Now spend every number:

  • 120 writes/s → any single Postgres survives this. Say so: "the write path is trivial — this interview is about reads and ID uniqueness."
  • 12k redirect QPS → too hot for comfort on one DB doing random-row lookups at tight p99 → cache + read replicas.
  • 3 TB / 6B rows → fits one machine's disk, but index pressure and availability argue for sharding by code — and the code itself is a perfect shard key (uniform, immutable).
  • 35 GB cache → the entire hot set fits in memory. This is why the read path can be almost DB-free.
  • 6 TB of blobs → object storage, not the relational DB. The split is now justified by a number, not by fashion.

4 · High-level design: API, the 301/302 decision, data model

POST /urls          {long_url, custom_alias?, ttl?}   → {code}      201
GET  /{code}                                          → 301/302 Location: long_url
POST /pastes        {content, syntax?, ttl?}          → {paste_id}  201
GET  /pastes/{id}                                     → metadata + content

The canonical talking point — deliver it unprompted:

StatusBehaviorWinsLoses
301 Moved PermanentlyBrowsers and CDNs cache the mapping; repeat clicks never reach youMassive load reduction; lower latency for the userYou go blind: no per-click analytics; hard to change/expire the target
302 FoundEvery click comes back to your serversFull analytics; expiry and target changes take effect immediatelyYou pay for every click at 12k QPS

Pick based on the section-2 question: analytics required → 302 (and section 10 shows how to count cheaply); pure redirect utility → 301 and let the internet cache for you.

Data model. urls(code PK, long_url, user_id, created_at, expires_at) — one row, ~500 B, code is both primary key and shard key. Pastebin: pastes(paste_id PK, blob_url, syntax, size, created_at, expires_at) — metadata in the DB, content at blob_url in object storage (section 8).

5 · Deep dive: encoding — base62 and where codes come from

Base62 (0–9a–zA–Z) is the alphabet because it is URL-safe without percent-encoding (base64's + and / are not).

62^6 ≈ 5.7 × 10^10   (57B — covers 6B links with only ~10× headroom)
62^7 ≈ 3.5 × 10^12   (3.5T — ~600× headroom → pick 7)

Two families of code generation, with opposite failure modes:

  • Hash-then-truncate. MD5/SHA the long URL, keep the first 7 base62 chars. No coordination, but truncation invites birthday collisions: expect the first one around √(62^7) ≈ 1.9M keys — far, far before the 3.5T space "fills up." So every insert needs a uniqueness check and a retry-with-salt path. (The lab measures this curve empirically.)
  • Counter-based. A global counter, base62-encoded: unique by construction, no collision checks. But sequential codes are enumerable — a competitor can walk your keyspace, scrape every "private" link, and read your growth rate off the counter. Mitigate by encrypting/permuting the counter (a bijection keeps uniqueness) or by handing counter ranges to app servers.

Say the security sentence out loud: "sequential codes leak business metrics and make private links guessable — that alone can disqualify the naive counter."

6 · Deep dive: distributed ID generation

The counter is now your single point of coordination. Three industrial escapes:

Snowflake (Twitter). A 64-bit ID assembled locally on each node:

1 bit  sign (always 0)
41 bits timestamp, ms since custom epoch  → ~69 years
10 bits machine ID                        → 1,024 workers
12 bits per-ms sequence                   → 4,096 IDs/ms/worker

No coordination on the hot path, and IDs are time-sortable — which is why feeds and chats love them. The Achilles heel is the clock: if NTP steps a node's clock backwards, the same (timestamp, sequence) pair can be minted twice. Correct behavior is to refuse to issue IDs until the clock catches up — availability traded for uniqueness. (The lab simulates exactly this skew.)

Key Generation Service (KGS). Pre-generate random 7-char keys offline into a pool; app servers lease batches. Uniqueness is guaranteed by construction, codes stay short and unordered. Cost: the pool is a service you must replicate (section 9).

Instagram's variant: shard-local Postgres sequences packed into the ID with the timestamp — Snowflake's layout without extra infrastructure (see primary sources).

Senior observation: Snowflake at 11 base62 chars is too long for a short link — that is why TinyURL favors KGS or a permuted counter, while Snowflake wins wherever length doesn't matter but sort order does.

7 · Deep dive: the read path at 12k QPS

Redirects are the product. Three layers, cheapest first:

  • Cache-aside on hot links. GET /{code} → Redis lookup → on miss, read DB, populate with TTL, LRU eviction. Section 3 said the 35 GB hot set fits in memory: expect 80–95% hit rate, so the DB sees ~1–2k QPS instead of 12k.
  • Negative caching. Random-code scanners and dead links generate misses that all fall through to the DB — a cache is useless against keys that don't exist. Cache the 404 itself with a short TTL (say 60 s); a Bloom filter over live codes is the heavier-duty version. Without this, an attacker's miss traffic becomes your DB's load test.
  • CDN-edge redirects. If you chose 301, the redirect response is immutable and tiny — let the CDN serve it from the edge, keyed by code. Reads then scale like static content. Note the coupling: this is only available because you gave up per-click analytics; the 301/302 choice propagates all the way to the edge.

Consistency footnote worth saying: a newly shortened link that takes one second to appear on every replica is fine — read-your-writes for the creator (route their first GET to the primary or seed the cache on write) is the only consistency anyone actually notices.

8 · Deep dive: expiry, cleanup, and the Pastebin storage split

Expiry has two halves. Lazy deletion: on read, if expires_at < now, return 404/410 and delete — correctness is enforced at the only moment it matters. Background sweep: a low-priority job scans for expired rows in batches (and deletes orphaned blobs) — reclaims space for links nobody reads again. You need both: lazy alone leaks storage forever; sweep alone serves dead links between passes.

Pastebin's storage split. Metadata (id, syntax, size, expiry) is small, uniform, and queried on every view → relational DB. Content is a variable-size immutable blob written once → object storage (the S3 model from lesson 0012). Why not one store?

  • 10 MB rows wreck DB page cache locality and bloat replication streams for data that never changes.
  • Object storage gives eleven-nines-style durability and per-GB pricing that a relational store can't match at 6 TB and growing.
  • The view path becomes: DB for metadata (cacheable) → fetch blob by URL, possibly straight from CDN. Large pastes stop competing with small ones — a non-functional requirement from section 2, satisfied by an architecture choice.

Syntax highlighting is a render concern: highlight client-side, or cache the rendered HTML as just another derived blob. Never store only the rendered form — raw text is the source of truth.

9 · Failure modes and monitoring

FailureBlast radiusMitigation
Hash collision on insertTwo URLs, one code — wrong redirectsUnique constraint on code; on violation, re-hash with a salt and retry; alert if retry rate climbs
Cache stampede on a viral linkOne hot key expires → thousands of concurrent misses hammer one DB rowRequest coalescing (one flight per key), stale-while-revalidate, jittered TTLs — lesson 0007's toolkit
Counter / KGS node diesAll new shortens fail; reads unaffectedLease counter ranges (or key batches) to app servers so they survive minutes of outage; replicate the pool with failover
Clock skew on a Snowflake nodeDuplicate IDs — silent data corruptionRefuse to mint until clock catches up; page if refusal window exceeds seconds

Dashboards that page a human: redirect p99 (the SLO), cache hit rate (a dip predicts DB overload before latency shows it), 404 rate (a spike = scanner or a bad partner integration), and keyspace/pool consumption rate (running out of pre-generated keys is a silent, fully predictable outage — the most embarrassing kind).

10 · Variants and twists interviewers reach for

"Add click counts — now it's write-heavy." The wrong answer is UPDATE clicks SET n = n + 1 per redirect: 12k row-lock contentions/s on one viral row. The right shape: emit a click event to a log/queue (Kafka, lesson 0013), aggregate in batches every few seconds, and serve counts from the aggregate. If approximate is acceptable, say so — sampled counting or sketches (e.g., count-min) cut the write volume by orders of magnitude. This flips the whole system from read-heavy to write-heavy, which is exactly why interviewers ask it.

"Stop abuse of the shortener." Free URL shorteners are spam infrastructure by default. Rate limit POST /urls per user/IP (lesson 0018 drops in wholesale), cap redirects/s per code if a link is used for attack amplification, and keep a denylist of destination domains checked at shorten time — checking at redirect time too costs one cache lookup and catches later-poisoned targets.

AI flavor (one paragraph, as promised). A modern twist: classify destination URLs for phishing/malware with an ML model. The design lesson is placement — inference is 10–100 ms, so it sits async after shorten (accept, then quarantine on a bad verdict), never inline in the redirect path. Same async-vs-inline reasoning as the payment-fraud check you will meet in lesson 0027.

11 · Interview talking points

  • Framing: "The write path is trivial at 120/s — the design lives in ID uniqueness and the read path. Let me spend the time there."
  • Spending a number: "100:1 reads at 12k peak QPS is why a 35 GB cache, not a bigger database, is the first thing I add."
  • The 301/302 line: "301 lets browsers and CDNs absorb repeat clicks; 302 keeps every click observable. This is an analytics-requirements question, not a technical one — which do we need?"
  • Encoding: "62^7 is 3.5 trillion codes, but with hash-truncate the birthday bound means collisions around two million keys — so the space is huge and collisions are still a day-one problem."
  • ID generation: "Snowflake gives coordination-free, time-sorted IDs but trusts the clock and yields 11 chars; KGS gives short random codes but is a service to operate. For a shortener I'd take KGS; for feed IDs, Snowflake."
  • Unprompted failure mode: "The scary one is cache stampede on a viral link — here's request coalescing before you ask."
  • Closing: "I'd page on redirect p99, cache hit rate, and key-pool consumption. With more time: analytics pipeline and multi-region."
