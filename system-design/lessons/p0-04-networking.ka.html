ქსელები — საიდან მოდის დაყოვნება

TCP, TLS, HTTP/1.1→3, სოკეტები და კავშირების პულინგი — მოთხოვნის დაყოვნება ძირითადად ის round trip-ებია, რომლებიც შენ თვითონ აირჩიე გადაიხადო.

მისიასთან კავშირი. ყოველი ტოკენი, რომელსაც შენი inference gateway მომხმარებელს უგზავნის, ამ სტეკს კვეთს: სოკეტი → TCP → TLS → HTTP. როცა ინტერვიუერი გეკითხება, რატომ არის TTFT 800 ms, როცა მოდელის prefill მხოლოდ 200 ms-ს იღებს, პასუხი ჩვეულებრივ ამ გაკვეთილში ცხოვრობს — ჰენდშეიქები (handshakes), ცივი კავშირები და head-of-line ბლოკირება. High Performance Browser Networking ამას პირდაპირ ამბობს: ვების დაყოვნების უმეტესობა round trip-ებია და არა გამტარუნარიანობა. (HPBN — latency vs bandwidth)
ჯერ ლაბი (ADHD-მეგობრული რიგი) გახსენი labs/lab-p0-04-latency-anatomy/ და გაუშვი python3 latency_anatomy_lab.py კითხვის დაწყებამდე. ის რეალურ სოკეტებზე ზომავს კავშირის ხელახალ გამოყენებას და ითვლის ამ გაკვეთილის ყველა ცხრილს. გაკვეთილი წაიკითხე გამონატანთან ერთად.

1 · ერთი HTTPS მოთხოვნის ანგარიში

აკრიფე URL, დააჭირე enter-ს. სანამ პასუხის პირველი ბაიტი მოვა, თანმიმდევრობით იხდი: DNS ძიება (IP-ის პოვნა), TCP ჰენდშეიქი (1 round trip), TLS ჰენდშეიქი (1–2 round trip), თავად მოთხოვნა (1 round trip), და ბოლოს ბაიტების გადაცემის დრო. ანგარიშსწორების ერთეული არის RTT — round-trip time. ერთი ქალაქი ≈ 1–10 ms, კონტინენტის გადაკვეთა ≈ 50 ms, ოკეანის გადაღმა ≈ 150 ms; იატაკს ფიზიკა (სინათლის სიჩქარე ოპტიკურ ბოჭკოში) აწესებს და მას ვერანაირი აპარატურა ვერ დასწევს. (HPBN, თავი 1)

გამტარუნარიანობა კი მუდმივად უმჯობესდება და პატარა API პასუხებისთვის იშვიათად არის ვიწრო ადგილი. 10 KB JSON 100 Mbps-ზე ~1 ms გადაცემა ჯდება — მაგრამ 3 setup RTT, თითო 50 ms, 150 ms ღირს. დაყოვნება დომინირებს; ჯერ round trip-ები დათვალე.

2 · TCP — საიმედოობა round trip-ად ღირს

TCP გაძლევს მოწესრიგებულ, საიმედო ბაიტების ნაკადს არასაიმედო ქსელზე. ამ კონტრაქტის ფასი სამსაფეხურიანი ჰენდშეიქით (three-way handshake) იწყება: SYN → SYN-ACK → ACK. კლიენტს მონაცემების გაგზავნა ერთი სრული round trip-ის შემდეგ შეუძლია — ანუ ყოველი ახალი TCP კავშირი პირველ ბაიტამდე 1 RTT ღირს. (HPBN — Building Blocks of TCP)

შემდეგ ბაიტების სიჩქარეს ორი მუხრუჭი მართავს:

  • ნაკადის კონტროლი (flow control) — მიმღები აცხადებს ფანჯარას (rwnd): „ამდენის დაბუფერება შემიძლია." იცავს მიმღებს.
  • გადატვირთვის კონტროლი (congestion control) — გამგზავნი ინახავს congestion window-ს (cwnd), რომელიც პატარად იწყება და ყოველ RTT-ზე იზრდება (slow start: დაახლოებით ორმაგდება დანაკარგამდე ან ზღვრამდე). იცავს ქსელს. ახალი კავშირი პირველი რამდენიმე RTT-ის განმავლობაში შენს სრულ გამტარუნარიანობას ვერ იყენებს — კიდევ ერთი მიზეზი, რატომ გვტკივა ცივი კავშირები.
ლექსიკონი Head-of-line (HOL) ბლოკირება: როცა ერთი დაკარგული ან ნელი ელემენტი აჩერებს ყველაფერს, რაც მის უკან დგას რიგში. TCP თანმიმდევრობას იძლევა გარანტიით, ამიტომ ერთი დაკარგული პაკეტი ამ კავშირზე ყველა მომდევნო პაკეტის მიწოდებას აჩერებს — დაიმახსოვრე ეს HTTP/2 vs HTTP/3-სთვის.

3 · TLS — ვინაობა, გასაღებები და კიდევ round trip-ები

TLS TCP-ს ორ რამეს ამატებს: ავთენტიფიკაციას (სერტიფიკატი ამტკიცებს, რომ ნამდვილ სერვერს ელაპარაკები) და დაშიფვრას (თითო კავშირზე შეთანხმებული სიმეტრიული გასაღები). შეთანხმება round trip-ებად ღირს: TLS 1.2-ს 2 RTT სჭირდება; TLS 1.3-მა ჰენდშეიქი 1 RTT-მდე შეკუმშა — კლიენტის key share პირველივე გზავნილში მიდის — და განმეორებით კავშირებზე 0-RTT აღდგენას (resumption) გვთავაზობს (replay-შეტევის დათქმებით). (HPBN — TLS)

ასე რომ, ცივი HTTPS კავშირი ჯდება 1 (TCP) + 1–2 (TLS) = 2–3 RTT მანამ, სანამ HTTP მოთხოვნა საერთოდ გავა. ბაიტზე დაშიფვრის ფასი (AES აპარატურული მხარდაჭერით) ამასთან შედარებით თითქმის უფასოა — ფული ჰენდშეიქში მიდის.

4 · გათვლილი მაგალითი — time-to-first-byte კიბე

ერთი პატარა მოთხოვნა, ერთი ახალი კავშირი, RTT = 50 ms, სერვერის დამუშავება ≈ 0. დათვალე round trip-ები პასუხის პირველ ბაიტამდე:

TCP + TLS 1.2 :  1 (TCP) + 2 (TLS) + 1 (მოთხოვნა)  =  4 RTT  =  200 ms
TCP + TLS 1.3 :  1 (TCP) + 1 (TLS) + 1 (მოთხოვნა)  =  3 RTT  =  150 ms
QUIC (HTTP/3) :  1 (ტრანსპორტი+TLS) + 1 (მოთხოვნა)  =  2 RTT  =  100 ms
QUIC 0-RTT    :  0 (აღდგენილი)      + 1 (მოთხოვნა)  =  1 RTT  =   50 ms
pooled კავშირი:  0 (უკვე ღიაა)      + 1 (მოთხოვნა)  =  1 RTT  =   50 ms

იგივე სადენი, იგივე სერვერი, იგივე ბაიტები: 4x სხვაობა — მხოლოდ პროტოკოლის არჩევანი.

ეს კიბე ლაბის მე-3 ექსპერიმენტია, გამოთვლილი RTT = 10/50/150 ms-ზე. აქ არაფერია გამტარუნარიანობაზე.

5 · HTTP/1.1 → HTTP/2 → HTTP/3

HTTP-ის ევოლუცია ერთი გრძელი ომია round trip-ებისა და head-of-line ბლოკირების წინააღმდეგ:

ვერსიატრანსპორტიპარალელურობის მოდელიHOL ბლოკირება
HTTP/1.1TCP1 მოთხოვნა ერთდროულად თითო კავშირზე; ბრაუზერები ~6 პარალელურ კავშირს ხსნიან; keep-alive მათ ხელახლა იყენებსHTTP დონეზე: ერთი ნელი პასუხი კავშირს ბლოკავს
HTTP/2TCP + TLSბევრი ნაკადი (stream) მულტიპლექსირებულია ერთ კავშირზე; ჰედერების შეკუმშვა (HPACK)HTTP დონეზე მოგვარებულია, მაგრამ TCP დონეზე რჩება: ერთი დაკარგული პაკეტი ყველა ნაკადს აჩერებს
HTTP/3QUIC (UDP-ზე, ჩაშენებული TLS 1.3-ით)ნაკადები მულტიპლექსირებულია დამოუკიდებელი მიწოდებით; 1-RTT setup, 0-RTT აღდგენა; კავშირები IP-ის ცვლილებას უძლებენტრანსპორტის დონეზე გაქრა: დაკარგული პაკეტი მხოლოდ თავის ნაკადს აჩერებს

QUIC-მა საიმედოობა, თანმიმდევრობა და დაშიფვრა UDP-ის თავზე, user space-ში გადაიტანა სწორედ იმისთვის, რომ A ნაკადში დაკარგულმა პაკეტმა B ნაკადი ვერ გააჩეროს — TCP-ის „მკაცრი რიგის" კონტრაქტი ამის გამოსწორებას შეუძლებელს ხდიდა. (RFC 9000 — QUIC, RFC 9114 — HTTP/3, HPBN — HTTP/2)

6 · სოკეტები და კავშირების პულინგი

შენი კოდისთვის ეს ყველაფერი სოკეტად (socket) ჩანს: კავშირის ერთი ბოლოს file-descriptor სახელური, იდენტიფიცირებული 4-ეულით (src IP, src port, dst IP, dst port). სერვერები აკეთებენ listen()-ს და accept()-ს; კლიენტები — connect()-ს. p0-03 გაკვეთილის event loop-ები დიდწილად იმისთვის არსებობს, რომ ათასობით ასეთი სახელური თითო ნაკადის გარეშე აჟონგლიორონ.

კავშირების პულინგი (connection pooling) სისტემური პასუხია §2–4-ზე: გადაიხადე TCP + TLS setup ერთხელ, შეინახე კავშირი ღია და გამოიყენე ბევრ მოთხოვნაზე. ამას ყველა სერიოზული კლიენტი აკეთებს (ბრაუზერები, requests.Session, gRPC არხები, ბაზის დრაივერები). პულებს საკუთარი სახელურები აქვთ — ზომა, idle timeout, health check — და საკუთარი ჩავარდნის რეჟიმებიც (§7). (Primer)

ცერის წესი თუ კლიენტი ერთსა და იმავე ჰოსტს ერთზე მეტჯერ ელაპარაკება, პულის გარეშე კავშირი ბაგია. ლაბის 1-ლი ექსპერიმენტი აჩვენებს, რომ ლოკალჰოსტიც კი ~8x-ს იხდის ახალ კავშირებზე; WAN-ზე მამრავლი უარესია, რადგან ყოველი setup რეალურ RTT-ებს წვავს.

7 · ბასრი კიდეები — სად იმალება დაყოვნება პროდაქშენში

  • Nagle-ის ალგორითმი + delayed ACK — ორი კეთილგანწყობილი „ბეჩინგის" ფუნქცია, რომლებიც ერთმანეთს ხვდებიან და პატარა ჩანაწერებზე ~200 ms-მდე შეფერხებას ამატებენ. დაყოვნებაზე მგრძნობიარე სერვისები აყენებენ TCP_NODELAY-ს. (HPBN)
  • ეფემერული პორტების ამოწურვა — ათასობით კლიენტური კავშირის დახურვა მათ TIME_WAIT-ში ტოვებს; „გახსენი-დაუკავშირდი-დახურე" ციკლები მაღალ QPS-ზე პორტებს ამოწურავს. პულინგი მიზეზს კურნავს და არა მხოლოდ სიმპტომს.
  • Slow start უსაქმობის შემდეგ — უსაქმო pooled კავშირის congestion window შეიძლება განულდეს; პირველი აფეთქება უსაქმობის შემდეგ დაცოცავს. გრძელვადიანი არ ნიშნავს ყოველთვის თბილს.
  • პულის ამოწურვა — ყველა pooled კავშირი დაკავებულია → მოთხოვნები კლიენტთან რიგში დგებიან. ჰგავს „upstream-ი ნელია"-ს; სინამდვილეში შენი პულის ზომაა. კლასიკური მცდარი დიაგნოზი.
  • TCP HOL ბლოკირება HTTP/2-ზე — ხარვეზიან ქსელებში (მობილური) HTTP/2 შეიძლება HTTP/1.1-ზე 6 კავშირით ნელიც იყოს, რადგან ერთი დანაკარგი ყველა ნაკადს აჩერებს. HTTP/3-ის არსებობის მთელი მიზეზი.
  • SYN backlog-ის გადავსება — კავშირების შტორმები (retry storm, thundering herd) სერვერის accept რიგს ავსებენ; კლიენტები timeout-ს connect-ზე ხედავენ და არა მოთხოვნებზე — გამორჩეული ხელწერა, რომლის ცნობაც ღირს.

E2 და E3 ზუსტი არითმეტიკაა — შენი ხელით ნათვლელი პროგრამას მილიწამამდე უნდა დაემთხვეს.

9 · პროექტის ნაბიჯი

გამოიყენე ეს LLM inference gateway-ზე. ჩასაწერი გადაწყვეტილებები (თითო ერთი ხაზი): (1) gateway ↔ model worker-ები pooled, გრძელვადიან კავშირებს იყენებენ (gRPC/HTTP2 არხები) — არასდროს თითო-მოთხოვნაზე connect; (2) მომხმარებლისკენ ტოკენების სტრიმინგი SSE-ზე ან HTTP/2 ნაკადებზე მიდის, ასე ერთი TCP+TLS setup ასობით ტოკენ-ფრეიმზე ნაწილდება; (3) შენი TTFT SLO კავშირის setup-საც შეიცავს — ცივი TLS 1.2 კავშირი 150 ms RTT-ზე 600 ms-ს ხარჯავს მანამ, სანამ მოდელს საერთოდ ჰკითხავენ (§4), რაც prefill-საც კი შეიძლება აჭარბებდეს; (4) retry შტორმები ჰენდშეიქებს ამრავლებენ და არა მხოლოდ მოთხოვნებს — backoff SYN რიგსაც იცავს. vLLM-ის ანატომიის პოსტი კარგი სურათია იმისა, თუ რა დგას სოკეტის უკან, რომელსაც კვებავ. (Inside vLLM)

საინტერვიუო თეზისები

  • „პატარა payload-ებზე დაყოვნება round trip-ებია და არა გამტარუნარიანობა — ამიტომ ჯერ RTT-ებს ვითვლი: TCP ღირს 1, TLS 1.2 კიდევ 2, TLS 1.3 — 1, QUIC კი ტრანსპორტსა და TLS-ს 1-ში კეცავს."
  • „ცივი HTTPS მოთხოვნა 50 ms RTT-ზე 150–200 ms-ს იხდის მანამ, სანამ სერვერი მას საერთოდ დაინახავს — ამიტომ connection pooling და keep-alive პირველი დაყოვნების ოპტიმიზაციაა და არა ქეშირება."
  • „HTTP/2-მა head-of-line ბლოკირება HTTP ფენაზე გაასწორა, TCP ფენაზე — ვერა; HTTP/3 ნაკადებს QUIC-ზე უშვებს და დაკარგული პაკეტი მხოლოდ საკუთარ ნაკადს აჩერებს — ეს არის QUIC-ის მთელი აზრი."
  • „Slow start ნიშნავს, რომ ახალი — ან დიდხანს უსაქმო — კავშირი რამდენიმე RTT სრულ გამტარუნარიანობას ვერ იყენებს, ასე რომ ‚კავშირებს ვიყენებთ ხელახლა' ავტომატურად არ ნიშნავს ‚თბილები ვართ'-ს."
  • „პულის ამოწურვა ზუსტად ნელ upstream-ს ჰგავს; სერვისის დადანაშაულებამდე კლიენტის მხარეს რიგში დგომის დროს ვამოწმებ."
  • „LLM serving-ში TTFT შეიცავს DNS-ს, TCP-სა და TLS-ს — ერთ გრძელვადიან კავშირზე სტრიმინგი ამ setup-ს ყოველ ტოკენზე ანაწილებს."
