Advanced Scheduling and the GPU Fleet

The advanced track begins. In production, WHERE a pod lands is a cost and reliability decision — especially when nodes carry $30k GPUs. Today: how the scheduler really decides, and every knob that overrides it. You'll simulate a GPU fleet on your Mac.

Advanced track ground rules Prereq: lessons 1–10 done. GPU-only mechanics are taught honestly: theory + simulation + capacity math here; the concepts transfer 1:1 the day you touch real A100s. Lab 11 rebuilds your cluster with an advanced config (3 nodes, Cilium CNI, ingress ports) — rebuilding from your own chart IS the first exercise.

Knowledge

1. The scheduler is a two-phase pipeline

For each pending pod: Filter (drop nodes that can't host it: insufficient resources, taints, affinity misses, volume zone) → Score (rank survivors: least/most-allocated, spread, image locality) → bind. Everything below is you injecting constraints into those phases. (docs)

2. Taints repel, tolerations permit, affinity attracts

MechanismDirectionGPU-fleet use
Taint + tolerationNode says "keep off unless invited"nvidia.com/gpu:NoSchedule on GPU nodes so web pods never occupy them
Node affinityPod says "I need nodes like this"Pin to gpu-type=h100 pools; prefer cheaper A10 pools when possible
Pod anti-affinityPod says "not next to those pods"Spread replicas across nodes/zones for failure isolation
Topology spread"Even distribution across a topology key"Cheaper, more flexible successor to anti-affinity for spreading
PriorityClass + preemption"Evict lesser pods if I don't fit"Prod inference preempts batch/experiment jobs on the GPU pool

3. GPUs are extended resources served by device plugins

nvidia.com/gpu: 1 is not built into K8s — the NVIDIA device plugin (DaemonSet) discovers GPUs and advertises them on the node's status; the scheduler then treats them like any countable resource: integer-only, no overcommit, request = limit. Fractional sharing exists one level down:

Sharing techWhat it isTrade
MIGHardware partition of A100/H100 into isolated slices (e.g. nvidia.com/mig-3g.20gb)True isolation; fixed slice sizes
Time-slicingDevice plugin advertises N virtual GPUs per physicalNo isolation — noisy neighbors; fine for dev
MPSCUDA-level concurrent process sharingBetter utilization, shared fault domain

You saw requestGPUType: "nvidia.com/mig-4g.71gb" in production-stack's values.yaml — now you know what it means. (GPU Operator docs)

4. Bin-pack or spread? The $ question

Web workloads spread (resilience). GPU inference usually bin-packs: fill node 1 completely before waking node 2, so the cluster autoscaler / Karpenter can scale empty $30/hr nodes to zero. Wrong default = five nodes each 20% busy = 5× the bill. Senior answer: bin-pack fleets per pool, anti-affinity only across failure domains you actually pay to protect.

Fake a GPU pool

Schedule against it

Deploy "gpu" pods requesting example.com/gpu: 1 with toleration + affinity → both land on worker2 → third stays Pending: Insufficient example.com/gpu. The exact failure you'll debug in every GPU cluster.

Preempt

PriorityClasses prod-inference > batch-experiment; fill the "GPU" node with batch pods; deploy a prod pod → watch eviction in Events.

Spread the engines

Add topologySpreadConstraints (zone key = hostname here) to your vLLM chart template; 2 replicas land on different workers; verify with -o wide.

Check yourself

GPU pods request nvidia.com/gpu but sit Pending on a node WITH free GPUs. Classic cause?

GPU pools are almost always tainted; a pod without the matching toleration is filtered out before scoring. Extended resources aren't fragmentable — they're integers.

Why bin-pack GPU inference instead of spreading?

Packed = whole idle nodes = autoscaler reclaims them = money. Pods never share weights across containers, and TP happens inside one pod's GPUs.

MIG vs time-slicing for two paying tenants on one A100?

Tenants need isolation — MIG partitions memory/SMs in hardware. Time-slicing shares everything: one tenant's OOM or burst hits the other.
