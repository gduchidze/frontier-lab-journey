SRE for Inference: SLOs, Alerts, Chaos

Building the stack was engineering; running it at 3 AM is a different discipline. Today you define what "working" means numerically, alert on it without paging yourself to death, write the runbook future-you will thank you for, and then break everything on purpose.

Knowledge

1. SLI → SLO → error budget

SLI: a measured ratio of good events ("chat requests with TTFT < 4s"). SLO: the target over a window ("99% over 30 days"). Error budget: 100% − SLO — the amount of badness you're allowed, which is what makes deploys, chaos tests, and risk-taking rational instead of forbidden. For this stack:

SLISLO (local-realistic)Source
Availability (non-5xx via router)99.5% / 30drouter metrics
TTFT < 10s (CPU-realistic)95% of requestsvllm:time_to_first_token_seconds_bucket
Queue sanitywaiting < 10 for 99% of minutesvllm:num_requests_waiting

The senior move: SLOs are user-experience statements, so TTFT/availability — never "CPU below 80%" (that's a cause, not an experience).

2. Alert on symptoms, page on burn rate

Threshold alerts ("TTFT > 4s for 5m") either page constantly or too late. Multi-window burn-rate alerts fire when the error budget is being consumed abnormally fast:

# page: burning 30d budget 14.4× too fast, confirmed on two windows
(error_rate_1h  > 14.4 × (1 − SLO))
and
(error_rate_5m  > 14.4 × (1 − SLO))

Fast-burn (1h/5m, 14.4×) pages a human; slow-burn (6h/30m, 6×) files a ticket. Two windows kill flapping. This is the Google SRE Workbook recipe, and it's asked in interviews by name.

Alert taxonomy for THIS stack — cause-alerts kept as diagnostics, not pages: engines down (up == 0), queue climbing, KV usage pinned > 0.95, HPA at maxReplicas for > 15m.

3. Runbooks: written before the incident, linked from the alert

An alert without a runbook is homework assigned at 3 AM. Skeleton that works:

ALERT: HighTTFTBurnRate
1. CONFIRM   dashboards X, Y — real or metric artifact?
2. SCOPE     one engine or all? one tenant? started when? deploy nearby? (Argo history!)
3. MITIGATE  scale out | shed load (rate-limit at edge) | rollback last change
4. VERIFY    burn rate falling on the 5m window
5. FOLLOW-UP file postmortem; link evidence

Mitigate-then-diagnose. Restoring service outranks understanding it — understanding happens in the postmortem, which is blameless: systems and processes fail, not people.

4. Chaos engineering: hypothesis, not vandalism

Format: "Given [steady load], when [engine pod dies], we expect [router drops it < 5s, zero 5xx, TTFT bump < 30s]." Run, observe, compare, fix the gap. You've done baby chaos (lesson 8); today it's systematic, under load, with the dashboard recording and expectations written BEFORE.

Check yourself

Why burn-rate alerts instead of "P90 TTFT > 4s for 5m"?

Burn rate ties paging to how fast user-promised reliability is being spent — brief blips don't page, slow bleeds eventually do. Threshold alerts can't do either. (They require MORE upfront SLO work, not less.)

Mid-incident, cause unknown, users hurting. Correct next move?

Scale out / shed / roll back — restore the experience, then investigate. Metrics and logs preserve evidence for you; users don't wait for root cause.

A paused (hung) node is nastier than a killed one because…

Dead nodes fail fast and clean. Hung nodes ride on stale heartbeats (~40s+ before NotReady, minutes before eviction) while requests routed there time out. Gray failures are the expensive ones.
