The Production Edge: Ingress, Gateway, TLS

NodePort got you to the demo. Production traffic arrives at one hardened entry point with hostnames, TLS, timeouts, and auth — and LLM streaming breaks the default settings of every proxy ever shipped. Today you build a real edge.

Knowledge

1. The edge stack, bottom to top

Service (L4) → Ingress (L7 rules: host/path → Service) interpreted by an ingress controller (nginx, Traefik, Envoy) → fronted in cloud by a LoadBalancer. One controller, one IP, N services behind hostnames: api.company.com → router, chat.company.com → UI. (docs)

2. Gateway API is the successor — learn both

Gateway API splits the Ingress monolith into role-oriented resources: GatewayClass (infra provider) → Gateway (ops: listeners, TLS) → HTTPRoute (app teams: routing, traffic weights). Weighted backendRefs give you canary at the edge — remember that for lesson 14. New platforms are Gateway-API-first; existing fleets still run Ingress. Interview-ready = both.

3. TLS: cert-manager turns certificates into a controller problem

cert-manager watches Certificate/Issuer CRDs and keeps Secrets filled with valid certs (Let's Encrypt via ACME in prod, self-signed CA locally). The Ingress references the Secret; renewal is nobody's pager. Same reconciliation pattern as everything else in K8s — that's the deep lesson.

5. The edge is where auth lives

Engines stay dumb; the edge (or an AI gateway like LiteLLM in front of the router) enforces API keys, per-tenant rate limits, and usage metering. vLLM's --api-key is the last line, not the first — lesson 13 wires it.

Lab (labs/lab-12): hostnames and TLS on your stack

Install ingress-nginx (kind flavor)

The advanced cluster maps 80/443. Apply the kind-specific manifest, wait for the controller.

Route by hostname

Ingress with LLM-tuned annotations: chat.llm.local → open-webui, api.llm.local → vllm-router. Add both hosts to /etc/hosts → 127.0.0.1. Chat in the browser at http://chat.llm.local — no port numbers, like a real product.

cert-manager + self-signed CA

Install cert-manager, create a CA ClusterIssuer, annotate the Ingress — TLS Secrets appear. curl -k https://api.llm.local/v1/models, then inspect the cert chain with openssl s_client.

Prove the streaming fix

Stream a long generation through the edge with buffering ON vs OFF (flip the annotation) — watch tokens lump vs flow. This 2-minute demo is the whole "LLM edge" war story, self-inflicted safely.

Check yourself

Users say "the answer appears all at once at the end". Edge suspect #1?

Buffering proxies hold the response until complete — streaming works end-to-end but LOOKS broken. Engine delay would push TTFT, not lump the whole answer.

Gateway API over Ingress — the structural win?

Gateway/HTTPRoute split ops from app teams and make weights/matches first-class API — no annotation soup. Data plane is the same proxies; certs still need an issuer.

Where should per-tenant rate limiting live?

Reject cheap at the edge before a request consumes expensive prefill compute. Engines can't see tenants; clients can't be trusted.
