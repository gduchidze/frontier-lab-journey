Security and Multi-Tenancy

An inference platform holds two crown jewels: expensive compute anyone would love to borrow, and prompts full of user data. Today: defense in depth for both — network, identity, resources, and the engine's own front door.

Knowledge

1. The 4C mental model

Cloud → Cluster → Container → Code: each layer only as strong as the outer ones. Today works Cluster (RBAC, NetworkPolicy, quotas) and Container/Code (image hygiene, engine auth). (docs)

2. NetworkPolicy: default-deny, then allow the graph you drew

Flat pod network = any compromised pod can curl your engines. NetworkPolicy is a pod-level firewall — but only if the CNI enforces it (kind's default kindnet does NOT; that's why lab 11 installed Cilium). The production pattern:

1. default-deny ingress in the namespace
2. allow: ui → router (port 8000)
3. allow: router → engines (port 8000)
4. allow: monitoring → engines (/metrics)
5. everything else stays dead — including engine → engine

Your architecture diagram from lesson 8 IS the allowlist. If you can draw it, you can enforce it.

3. RBAC as an auditable surface

Least privilege isn't a vibe, it's a query:

kubectl auth can-i --list --as=system:serviceaccount:llm:vllm-router -n llm
kubectl auth can-i delete pods --as=system:serviceaccount:ui:default -n llm

Senior habit: audit ServiceAccounts quarterly; anything with cluster-admin or * verbs justifies itself or dies. Your router's 3-verb Role from lesson 6 is the model citizen.

4. Tenancy = namespace + quota + limits + netpol, as a unit

ObjectControlsLLM-platform example
ResourceQuotaNamespace totalsTeam A gets 8 GPUs, 200Gi RAM — hard cap
LimitRangePer-pod defaults/boundsNo limitless pods sneak onto GPU nodes
NetworkPolicyWho talks to whomTenant A cannot reach tenant B's engines
PriorityClassWho wins contentionProd > staging > experiments (lesson 11)

Namespaces are soft tenancy (one API server, one kernel). Hostile tenants ⇒ separate clusters or hard sandboxes (Kata/gVisor) — know the line.

5. Secrets and images, the honest version

K8s Secrets are base64 in etcd — protection comes from RBAC + etcd encryption-at-rest + short-lived tokens. Production pulls from a manager (External Secrets Operator + Vault/cloud SM); git holds references, never values (or sealed-secrets if it must hold ciphertext). Images: pin digests, scan (trivy), minimal bases, no :latest in prod — your vllm-cpu:local would be registry/vllm-cpu@sha256:….

6. The engine's own door

vllm serve --api-key (from a Secret, injected as env) means even a pod that sneaks past netpol still can't prompt the model. Defense in depth: edge auth (lesson 12) + network (this) + engine key.

Lab (labs/lab-13): lock the stack

Default-deny, watch it break

Apply deny-all ingress to llm. UI dies, scrapes die, router→engine dies. Feel the blast radius — 30 seconds of controlled outage teaches more than an hour of reading.

Allow the drawn graph

Add the four allow policies (ui→router, router→engines, monitoring→engines, edge→ui/router). Everything returns. Then prove isolation: kubectl run attacker in default ns → curl engine → timeout. Screenshot that timeout; it's a portfolio artifact.

Quota the tenant

ResourceQuota (cpu 12, mem 20Gi, pods 10) + LimitRange defaults on llm. Try to scale engines to 5 — watch quota refuse politely in Events.

Engine API key

Secret → env → --api-key arg in your chart (values-driven). Curl without key → 401; UI updated with the key → works. Audit RBAC with auth can-i --list for every ServiceAccount in llm/ui.

Check yourself

NetworkPolicies applied on stock kind (kindnet) do…

Policies are just objects; enforcement is the CNI's job. kindnet doesn't implement them — the #1 false sense of security in local clusters. Cilium/Calico enforce.

Why is base64 in a Secret not encryption?

base64 is decode-able by anyone holding the bytes. Real protection: RBAC on the object, etcd encryption at rest, external secret managers.

Two paying customers demand hard isolation. Namespaces enough?

Namespace tenancy is soft — one API server, shared kernels, escape routes via misconfig. Hard requirements ⇒ cluster-per-tenant or hardware/VM sandboxes (and dedicated nodes still share the control plane).
