Autoscaling and the Capstone

The final loop: the cluster reacts to load by itself. Then you package ten lessons of work into a portfolio project that speaks for you in interviews.

Knowledge: closing the control loop

1. HPA: desired replicas as a function of metrics

The HorizontalPodAutoscaler adjusts a Deployment's replicas to hold a metric near target: desired = ceil(current × metricValue / target). It needs a metrics source — the metrics-server for CPU/memory (not installed by default on kind).

2. CPU% is the wrong metric for LLMs — know why anyway

An inference engine can be "full" (queue growing) while CPU sits at 60%, or pegged at 100% while healthy. The right signals are engine-native: vllm:num_requests_waiting, KV cache usage — exactly what production-stack's roadmap means by "autoscaling based on vLLM-specific metrics". Custom-metric scaling needs KEDA or prometheus-adapter bridging Prometheus → HPA. Today: CPU-based HPA (the mechanism), KEDA as documented stretch goal (the production answer). Knowing why CPU is wrong is the interview answer.

3. Scaling limits nobody escapes

Scale-out adds capacity but each new replica must load the model (minutes on CPU — this is why real platforms obsess over model-load time, use PV-cached weights, and pre-warm). Scale down too eagerly and you kill engines mid-generation. HPA's behavior stanza (stabilization windows) exists for exactly this.

Lab part A: reactive cluster

Install metrics-server (kind needs a flag)

helm repo add metrics-server https://kubernetes-sigs.github.io/metrics-server/
helm install metrics-server metrics-server/metrics-server \
  -n kube-system --set args={--kubelet-insecure-tls}
sleep 30 && kubectl top nodes && kubectl top pods -n llm

expect: CPU/memory numbers per node and pod (insecure-tls flag is kind-only: node certs lack SANs)

Create the HPA

Save as k8s/vllm-hpa.yaml:

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: vllm
  namespace: llm
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: vllm
  minReplicas: 1
  maxReplicas: 2
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
# let the HPA own replica count: set replicaCount back to 1 first
helm upgrade slm charts/slm-stack --set vllm.replicaCount=1
kubectl apply -f k8s/vllm-hpa.yaml
kubectl get hpa -n llm -w

expect: TARGETS shows live cpu% vs 70%; REPLICAS 1

Trigger it

Terminal 1: kubectl get hpa,pods -n llm -w. Terminal 2: run the lesson-9 k6 test (or the curl loop) for 3–4 minutes.

expect: cpu% crosses 70 → REPLICAS 1→2 → second pod loads model → joins router automatically (service discovery!) → load spreads. After traffic stops: ~5 min stabilization, then back to 1

The win Load rises → HPA scales → router discovers → Grafana shows both engines → drain → scale-in. Every layer of the course in one autonomous feedback loop, with no human in it. That's a production stack.

Stretch (optional, separate session): scale on queue depth with KEDA

helm repo add kedacore https://kedacore.github.io/charts
helm install keda kedacore/keda -n keda --create-namespace

Then a ScaledObject with a prometheus trigger on sum(vllm:num_requests_waiting), threshold ~5. See KEDA prometheus scaler docs. This is the real-world pattern — document it in the README even if you only prototype it.

Check yourself

Why is CPU utilization a poor autoscaling signal for LLM engines?

Engine fullness = admission/queue state (waiting requests, KV cache), which decouples from CPU%. Scale on num_requests_waiting via KEDA/prometheus-adapter.

Why set a scale-down stabilization window on the HPA?

Traffic dips briefly → instant scale-in would kill an engine mid-generation and force a minutes-long model reload when traffic returns. The window makes scale-in deliberate.

New replica scales up but serves nothing for minutes. Why?

The pod isn't Ready until the engine loads the model and /health passes — so no traffic reaches it. This model-load latency is THE scaling bottleneck of LLM serving (and why weight-caching PVs matter).
