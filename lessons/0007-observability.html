SLM Production Stack · Lesson 7 · Phase 7 · ~2 h

Observability: Prometheus + Grafana

Every serious inference platform answers three questions continuously: how fast (TTFT, latency), how loaded (running/pending requests, KV cache usage), how healthy (up, error rate). Today you build the system that answers them.

Knowledge: the scrape model

1. Pull, not push

Prometheus scrapes: every N seconds it GETs /metrics from each target and stores the time series. vLLM already exposes this endpoint (you curled it in lesson 3) — zero instrumentation work. The router exposes its own per-engine stats too.

2. Metric types you'll actually read

TypeShapevLLM example
GaugeValue right now, up and downvllm:num_requests_running, vllm:kv_cache_usage_perc
CounterOnly ever increases → read as rate()vllm:prompt_tokens_total, vllm:request_success_total
HistogramBuckets → read as histogram_quantile()vllm:time_to_first_token_seconds, vllm:e2e_request_latency_seconds

Full metric list: vLLM production metrics docs · condensed table in the observability reference.

3. PromQL, the 3 patterns that cover 90%

# current load per engine (gauge, as-is)
vllm:num_requests_running

# throughput: tokens/sec over 1m window (counter → rate)
rate(vllm:generation_tokens_total[1m])

# P90 TTFT over 5m (histogram → quantile)
histogram_quantile(0.9, rate(vllm:time_to_first_token_seconds_bucket[5m]))

4. ServiceMonitor: telling Prometheus what to scrape, the K8s way

The kube-prometheus-stack Helm chart installs Prometheus + Grafana + the Prometheus Operator. The Operator watches for ServiceMonitor objects — a manifest saying "scrape Services matching these labels on this port". Declarative scrape config, same philosophy as everything else you've built.

Lab: lights on

Install kube-prometheus-stack

helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update
helm install kps prometheus-community/kube-prometheus-stack \
  --namespace monitoring --create-namespace \
  --set grafana.adminPassword=admin \
  --set prometheus.prometheusSpec.serviceMonitorSelectorNilUsesHelmValues=false
kubectl get pods -n monitoring -w     # wait until all Running (2-3 min)

expect: prometheus, grafana, operator, kube-state-metrics, node-exporter pods Running

That last --set flag By default the chart only scrapes ServiceMonitors carrying its own release label. Setting the selector to nil-uses-false = "scrape ANY ServiceMonitor". Saves you the classic "my target never appears" hour.

Point Prometheus at vLLM

Save as k8s/vllm-servicemonitor.yaml:

apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: vllm
  namespace: llm
  labels: { app: vllm }
spec:
  selector:
    matchLabels: { app: vllm }
  endpoints:
  - port: "8000"
    path: /metrics
    interval: 15s
Port must be NAMED ServiceMonitor's port: refers to a Service port name, not number. Add name: http to the port in your vllm Service template, use port: http here. This mismatch is the #1 ServiceMonitor gotcha.
kubectl apply -f k8s/vllm-servicemonitor.yaml

Verify the target, run first queries

kubectl port-forward -n monitoring svc/kps-kube-prometheus-stack-prometheus 9090:9090

Open localhost:9090/targets → your vllm endpoints should show UP (give it ~30s). Then in the Graph tab, run each of the 3 PromQL patterns from the knowledge section. Generate some traffic through the router first so the graphs move:

for i in $(seq 1 10); do curl -s localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"Tell me a 50 word story."}],"max_tokens":80}' > /dev/null & done

Grafana: build the inference dashboard

kubectl port-forward -n monitoring svc/kps-grafana 3000:80

Login at localhost:3000 (admin / admin). Two paths — do both:

a) Import the reference dashboard. production-stack ships one: download observability/vllm-dashboard.json from the repo → Grafana → Dashboards → Import → upload. Instant panels: TTFT distribution, running/pending requests, healthy instances.

b) Build one panel yourself (this is the skill): New dashboard → Add visualization → datasource Prometheus → query:

histogram_quantile(0.9, sum(rate(vllm:time_to_first_token_seconds_bucket[5m])) by (le))

Title it "P90 TTFT", unit seconds, save dashboard as SLM Stack.

expect: live TTFT curve that spikes when you fire the traffic loop

Load-and-watch drill (10 min)

Split screen: Grafana left, terminal right. Fire 20 concurrent requests. Narrate what you see: pending queue rises → running plateaus at engine capacity → TTFT P90 climbs → drains. This narration is exactly what an SRE interview asks you to do with a dashboard.

Check yourself

Why must counters be wrapped in rate() to be useful?

A counter is a lifetime total — its slope (rate) is the meaningful signal. rate() also handles restarts (counter resets) gracefully.

ServiceMonitor applied, target never appears in Prometheus. First check?

Named-port mismatch (or label-selector mismatch) causes silent non-discovery. Grafana is downstream — irrelevant to target discovery.

TTFT P90 rising while num_requests_waiting grows means…

Waiting queue growth = admission backlog; time in queue lands in TTFT. Network latency is milliseconds, and PagedAttention already prevents fragmentation.
