Load Testing: Get the Numbers

"It works" is a demo. "It sustains 0.9 req/s at P90 TTFT under 4s before saturating at 6 concurrent requests" is engineering. Today you measure your stack with the industry's own tools and read the results properly.

Knowledge: the four numbers of LLM serving

MetricDefinitionWho cares
TTFTTime To First Token — request sent → first token backUsers (feels like "responsiveness")
TPOT / ITLTime Per Output Token (inter-token latency) after the firstUsers (feels like "typing speed")
ThroughputTokens/sec (or requests/sec) across the whole systemWhoever pays for hardware
GoodputThroughput that meets latency SLOsEveryone, eventually

The fundamental trade: batching harder raises throughput but worsens per-request latency. A load test's job is to find your knee of that curve — the saturation point.

Always report percentiles, not means: P50 tells the typical story, P99 tells the on-call story. One 60-second outlier hides completely in a mean.

Lab: two tools, one saturation curve

Tool 1 — vllm bench serve (the official benchmark)

It ships inside your image, so run it as a container on the same network path. Terminal 1: kubectl port-forward -n llm svc/vllm-router 8080:80. Terminal 2:

docker run --rm --network host --entrypoint vllm vllm-cpu:local \
  bench serve \
  --backend openai-chat \
  --base-url http://host.docker.internal:8080 \
  --endpoint /v1/chat/completions \
  --model Qwen/Qwen3.5-0.8B \
  --dataset-name random \
  --random-input-len 128 --random-output-len 64 \
  --num-prompts 30 \
  --request-rate 0.5 \
  --save-result --result-dir /tmp

expect (after a few min): a summary table — Request throughput, Output token throughput, Mean/Median/P99 TTFT, TPOT, ITL

Reading it --request-rate 0.5 = Poisson arrivals at 0.5 req/s (open-loop, like real traffic). inf would fire everything at once (closed-loop stress). The vllm-bench-serve skill in this workspace documents every flag.

Find the saturation point

Repeat the benchmark at --request-rate 0.25 → 0.5 → 1 → 2. Record into a table (put it in NOTES.md or the capstone README):

rate   req/s achieved   P50 TTFT   P90 TTFT   output tok/s
0.25   ____             ____       ____       ____
0.5    ____             ____       ____       ____
1      ____             ____       ____       ____
2      ____             ____       ____       ____

expect: achieved req/s stops following offered rate at some point, and P90 TTFT hockey-sticks — that knee IS your capacity

Keep Grafana open: num_requests_waiting going nonzero-and-climbing is saturation seen from the inside — the same knee, two views.

Tool 2 — k6 (the SRE's load tool)

What the doronmak repo uses (load_tests/load_test.js). Different lens: user-scenario scripting, SLO thresholds, pass/fail gates for CI.

brew install k6

Save as load_tests/chat.js:

import http from 'k6/http';
import { check } from 'k6';

export const options = {
  stages: [
    { duration: '1m', target: 4 },   // ramp to 4 virtual users
    { duration: '2m', target: 4 },   // hold
    { duration: '30s', target: 0 },  // drain
  ],
  thresholds: {
    http_req_failed: ['rate<0.01'],        // <1% errors
    http_req_duration: ['p(90)<30000'],    // P90 end-to-end < 30s (CPU-realistic)
  },
};

export default function () {
  const res = http.post(
    'http://localhost:8080/v1/chat/completions',
    JSON.stringify({
      model: 'Qwen/Qwen3.5-0.8B',
      messages: [{ role: 'user', content: 'Reply with one short sentence.' }],
      max_tokens: 32,
    }),
    { headers: { 'Content-Type': 'application/json' }, timeout: '120s' },
  );
  check(res, { 'status 200': (r) => r.status === 200 });
}
k6 run load_tests/chat.js

expect: live VU/RPS readout; final report shows checks passed, thresholds ✓ or ✗ — a ✗ exits nonzero, which is how CI gates on performance

Watch the whole system under load

Run the k6 test again with Grafana beside it. Confirm the story end to end: k6 VUs up → router QPS up → engine running requests plateau → waiting queue grows → TTFT P90 climbs → k6 P90 threshold nears its limit. One load, five instruments, one coherent narrative.

The win You have real numbers: capacity (req/s at the knee), latency (P50/P90 TTFT and TPOT), and an automated SLO gate. Put the table and a Grafana-under-load screenshot in the capstone README — measurable results are what make a portfolio project credible.

Check yourself

Why report P90 TTFT instead of mean TTFT?

Users experience the tail; a handful of slow requests vanish into a mean. SLOs are defined on percentiles for exactly this reason.

Offered rate 2/s, achieved 0.9/s, waiting queue growing. System is…

Achieved < offered with a growing queue is the definition of saturation. Warm-up ends; this doesn't. Nothing here rejects requests — they just wait.

Raising max batch size usually trades…

Bigger batches keep hardware busier (throughput ↑) but each request shares compute with more neighbors (TPOT ↑). Accuracy is untouched.
