Helm: Package the Stack

Your vLLM YAML works, but it's copy-paste infrastructure. Helm turns manifests into a parameterized, versioned, installable package — the way both reference repos ship, and the way you'll install Prometheus and Grafana next lesson.

Knowledge: Helm in four ideas

1. Chart = templates + values

A chart is a directory: Chart.yaml (name, version), values.yaml (the knobs), templates/ (manifests with {{ .Values.* }} holes). helm install renders templates with values and applies the result. (docs)

2. Values are the public API of your infrastructure

Look at how the pros expose knobs — doronmak's values.yaml:

vllm:
  replicaCount: 1
  image: { repository: doronmak/vllm-cpu-image, tag: latest }
  model: { name: facebook/opt-125m, dtype: auto }
  resources: { limits: { memory: 10Gi, cpu: "4" } }
  env:
    VLLM_TARGET_DEVICE: "cpu"
    VLLM_CPU_KVCACHE_SPACE: "5"

And production-stack's modelSpec list — one entry per model, each with replicaCount, requestCPU/GPU, pvcStorage. Same idea, industrial scale. A good values.yaml is what separates a demo chart from a production one.

3. Releases are stateful installs

A release = chart + values installed under a name. helm upgrade diffs and applies; helm rollback reverts; helm history shows every revision. Versioned infrastructure changes — interview gold.

4. Render-before-apply is the debugging superpower

helm template . renders locally without touching the cluster. Broken chart? Look at what it actually generates. You'll use this constantly.

Define the values

Replace charts/slm-stack/values.yaml:

namespace: llm

vllm:
  replicaCount: 1
  image:
    repository: vllm-cpu
    tag: local
    pullPolicy: IfNotPresent
  model:
    name: Qwen/Qwen3.5-0.8B
    maxModelLen: 4096
    dtype: bfloat16
  kvCacheSpaceGiB: 4
  resources:
    requests: { cpu: "4", memory: 6Gi }
    limits:   { cpu: "6", memory: 8Gi }
  pvc:
    size: 10Gi
  service:
    port: 8000

Template the Deployment

Create charts/slm-stack/templates/vllm-deployment.yaml by converting your k8s/vllm.yaml. The pattern — every hardcoded value becomes a reference:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm
  namespace: {{ .Values.namespace }}
  labels: { app: vllm }
spec:
  replicas: {{ .Values.vllm.replicaCount }}
  selector:
    matchLabels: { app: vllm }
  template:
    metadata:
      labels: { app: vllm }
    spec:
      containers:
      - name: vllm
        image: "{{ .Values.vllm.image.repository }}:{{ .Values.vllm.image.tag }}"
        imagePullPolicy: {{ .Values.vllm.image.pullPolicy }}
        args:
          - "--model"
          - {{ .Values.vllm.model.name | quote }}
          - "--max-model-len"
          - {{ .Values.vllm.model.maxModelLen | quote }}
          - "--dtype"
          - {{ .Values.vllm.model.dtype | quote }}
        env:
        - name: VLLM_CPU_KVCACHE_SPACE
          value: {{ .Values.vllm.kvCacheSpaceGiB | quote }}
        ports:
        - containerPort: 8000
        resources:
          {{- toYaml .Values.vllm.resources | nindent 10 }}
        startupProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 10
          failureThreshold: 60
        readinessProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 10
        livenessProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 20
          failureThreshold: 3
        volumeMounts:
        - { name: hf-cache, mountPath: /root/.cache/huggingface }
        - { name: dshm, mountPath: /dev/shm }
      volumes:
      - name: hf-cache
        persistentVolumeClaim: { claimName: hf-cache }
      - name: dshm
        emptyDir: { medium: Memory, sizeLimit: 2Gi }

Then create templates/vllm-service.yaml and templates/vllm-pvc.yaml the same way (port and size from values). Peek at doronmak's templates when unsure — it's the same conversion.

Render locally, then install

helm template charts/slm-stack | less     # inspect generated YAML — find your values in it

# replace the raw-YAML deploy with the chart-managed release:
kubectl delete -f k8s/vllm.yaml
helm install slm charts/slm-stack
helm list
kubectl get pods -n llm -w

expect: release slm deployed; vllm pod READY 1/1 fast (PVC kept the weights? No — new PVC name means re-download unless names match. Did you name it hf-cache? Then yes: fast.)

Feel the power: change a value, upgrade, roll back

helm upgrade slm charts/slm-stack --set vllm.model.maxModelLen=2048
kubectl get pods -n llm -w              # rolling replacement
helm history slm
helm rollback slm 1
helm history slm

expect: history shows 3 revisions; rollback restored maxModelLen=4096

The win Your stack is now an installable, versioned package: helm install slm charts/slm-stack recreates everything from zero. That command in a README is portfolio language.

Check yourself

helm upgrade broke the release. Fastest safe recovery?

Rollback re-applies the stored previous revision — atomic, audited. Uninstall/reinstall loses release history and may drop PVCs; manual edits drift from the chart.

Chart renders invalid YAML somewhere. Best first diagnostic?

Render locally, see exactly what the template produced (usually an indentation bug — nindent). Template files aren't valid YAML on their own, so applying them directly is meaningless.
