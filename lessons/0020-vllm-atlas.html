Spotlight 1 — Structured outputs (use this everywhere)

curl -s localhost:8080/v1/chat/completions -H "Content-Type: application/json" -d '{
  "model": "Qwen/Qwen3.5-0.8B",
  "messages": [{"role":"user","content":"Extract: The Mac M4 has 10 cores and 16GB RAM."}],
  "response_format": {
    "type": "json_schema",
    "json_schema": { "name": "hw", "schema": {
      "type": "object",
      "properties": { "chip": {"type":"string"}, "cores": {"type":"integer"}, "ram_gb": {"type":"integer"} },
      "required": ["chip","cores","ram_gb"] } } }
}'

The grammar masks invalid tokens at sampling time — output cannot violate the schema. Near-zero overhead after grammar compile. Retry-on-parse-error dies today. (docs)

Spotlight 2 — Multi-LoRA: fine-tunes without fleets

vllm serve BASE --enable-lora \
  --lora-modules customer-a=/adapters/a customer-b=/adapters/b
# request routes by model name:
{"model": "customer-a", ...}   # base weights + A's adapter, batched WITH B's traffic

Adapters are ~10–200 MB vs GB-scale bases; one engine batches requests across adapters simultaneously; dynamic load API adds tenants at runtime. The economics: 50 fine-tuned customers ≈ one base fleet + 50 small files, not 50 fleets. This single feature justifies platforms. (docs)

Spotlight 3 — Offline batch: the other half of the engine

from vllm import LLM, SamplingParams
llm = LLM(model="Qwen/Qwen3.5-0.8B", max_model_len=4096)
out = llm.generate(prompts, SamplingParams(temperature=0, max_tokens=64, seed=7))

No server, no HTTP, scheduler free to maximize pure throughput — evals and data generation run 2–5× cheaper this way than hammering your own API. Wrap it in lesson 19's Job and you have the canonical ML batch pattern.

Spotlight 4 — Model load time is your autoscaling constant

Replica usefulness = download + load + warmup. Attack all three: weights on RWX/PVC or init-container pre-pull (lesson 19), streaming loaders (tensorizer / Run:ai streamer: stream-to-GPU, skip CPU staging), and --load-format choices. Platforms measure "cold-replica-to-first-token" as a first-class SLO — put it in your design doc.

Check yourself

How do structured outputs guarantee valid JSON?

Constrained decoding: the grammar zeroes out logits of tokens that would violate the schema — invalid output is unrepresentable, not retried away.

40 per-customer fine-tunes of one 8B base. Cheapest serving shape?

Multi-LoRA batches all customers on shared base weights; adapters are megabytes. Forty fleets = forty× hardware; merging destroys per-customer behavior.

Nightly eval of 100k prompts: server API or offline LLM class?

Offline mode drops HTTP/streaming/fairness constraints and lets the scheduler pack purely for throughput — meaningfully cheaper at this scale. Prod-parity smoke tests can stay on the API.
