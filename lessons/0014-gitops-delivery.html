GitOps and Progressive Model Delivery

You've been the deployment tool: helm upgrade by hand. Production removes the human: git is the desired state, a controller reconciles the cluster to it, and shipping a new model becomes a reviewable pull request with a rollback button.

Knowledge

1. GitOps = the reconciliation loop, one level up

A Deployment reconciles pods to spec. Argo CD reconciles the whole cluster to a git repo: watch repo → diff live state (drift detection) → sync. Consequences: every change is a commit (audit log for free), rollback = git revert, cluster credentials leave laptops, and clusters are rebuildable from repo + secrets alone. Your capstone README quickstart was proto-GitOps; this automates it.

2. The Application CRD and app-of-apps

spec:
  source:  { repoURL: …, path: charts/slm-stack, targetRevision: main }
  destination: { server: https://kubernetes.default.svc, namespace: llm }
  syncPolicy:
    automated: { prune: true, selfHeal: true }   # drift gets reverted

selfHeal means your manual kubectl edit gets stomped within minutes — correct behavior, disorienting the first time. Platforms use app-of-apps: one root Application spawning children (engines, monitoring, edge) — the whole platform as one tree.

3. Models are versioned artifacts — treat rollouts like code deploys

A model change (new weights, new revision, new quantization) is a riskier deploy than code: quality regressions don't throw exceptions. production-stack's values expose modelURL + modelRevision — pin the revision (HF commit hash), never float on a repo's main. Rollout strategies, in order of blast radius:

StrategyMechanicsCatch for LLMs
RollingDeployment default, pod by podMixed model versions serve simultaneously mid-roll
Blue/greenFull v2 stack, flip the Service/route2× GPU cost during overlap — real money
CanaryWeighted routes (Gateway API / Argo Rollouts): 5% → 50% → 100%Needs quality metrics per arm, not just error rate
Shadowv2 gets mirrored traffic, responses discardedFull duplicate compute; gold for eval, silent to users

The senior insight: canary analysis for models must include quality signals (eval-set scores, thumbs-down rate, refusal rate) alongside TTFT/error rate. Latency-healthy garbage passes a naive canary.

4. CI/CD around the loop

CI builds/scans the image, runs offline evals on the candidate model, bumps the chart values in a PR. CD is just Argo noticing the merge. The pipeline never holds cluster credentials — the security win that sells GitOps to auditors.

Check yourself

Argo keeps reverting your HPA's replica changes. Root cause?

Two controllers own one field. Fix: remove replicas from the chart when HPA rules, or ignoreDifferences on that path. Classic production collision.

Why must model canaries track more than latency and errors?

A quality regression returns 200s at great TTFT. Canary analysis needs quality signals — eval scores, feedback rate — or it promotes garbage.

Pin modelRevision to a commit hash because…

Floating revisions mean the same manifest deploys different weights next week — unreproducible, unauditable. Same reason images pin digests.
