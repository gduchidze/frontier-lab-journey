Knowledge: Kubernetes in four ideas

1. A cluster is machines pretending to be one computer

A cluster = a control plane (the brain: API server, scheduler) + nodes (the muscle: machines that run your containers). You tell the API server what you want; the scheduler decides which node runs it. At a company, nodes are GPU servers. On your Mac, nodes are Docker containers created by kind — "Kubernetes in Docker". Same API, same skills.

2. The Pod is the atom

A Pod is the smallest deployable unit: one or more containers sharing a network identity. You almost never create Pods directly — they're mortal. When a Pod dies, it's gone. (docs)

3. Deployments keep the desired state true

A Deployment says "I want 3 replicas of this Pod, always." Kubernetes runs a reconciliation loop: kill a Pod, and a new one appears. This is declarative infrastructure — you declare the goal, K8s does the work. This one idea is why K8s won. (docs)

4. Services give mortal Pods an immortal address

Pods get random IPs and die. A Service is a stable name + IP that load-balances across whatever Pods currently match its label selector. When your router sends traffic to "the vLLM engines" in phase 6, it will talk to a Service, never to Pods. (docs)

Career note Interviews for ML infra roles routinely ask: "What happens when you run kubectl apply?" and "Deployment vs Pod vs Service?" After today's lab you'll have touched all three.

Lab: cluster up, app served

Goal: a two-node cluster on your Mac, an nginx web server running as a Deployment, reachable through a Service. ~45–60 min including installs.

Start Docker, install tools

Open Docker Desktop (kind lives inside Docker), then:

brew install kind helm
kind version && helm version

expect: kind v0.2x.x, helm v3.x

Create the cluster

Save as k8s/kind-config.yaml in this workspace (same pattern as the doronmak repo — a control-plane node plus a worker, with port 30080 mapped to your Mac for later):

kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
name: llm-stack
nodes:
  - role: control-plane
    extraPortMappings:
      - containerPort: 30080
        hostPort: 30080
  - role: worker
kind create cluster --config k8s/kind-config.yaml
kubectl get nodes

expect: 2 nodes, STATUS Ready (control-plane may take ~30s)

Look at what you built

docker ps                          # your "machines" are containers
kubectl get pods -A                # K8s runs itself as pods too
kubectl cluster-info

Notice kube-apiserver, etcd, kube-scheduler in namespace kube-system — the control plane from idea #1, running as Pods.

Run a Deployment

kubectl create deployment hello --image=nginx:alpine --replicas=2
kubectl get pods -o wide

expect: 2 pods hello-xxxx Running, spread across nodes

Now prove the reconciliation loop (idea #3). Delete one Pod and watch K8s resurrect it:

kubectl delete $(kubectl get pods -l app=hello -o name | head -1)
kubectl get pods -w        # Ctrl+C to stop watching

Note: -o name already prints pod/NAME — the resource type is included, so delete gets no extra pod argument.

expect: a NEW pod (different suffix) appears within seconds

Expose it with a Service

kubectl expose deployment hello --port=80
kubectl get svc hello
kubectl port-forward svc/hello 8080:80

In a second terminal:

curl -s localhost:8080 | head -4

expect: <!DOCTYPE html> ... Welcome to nginx!

The win You just did, in miniature, exactly what phase 4 does with vLLM: Deployment runs the workload, Service gives it a stable address, port-forward lets you talk to it. Swap nginx for vllm and curl / for curl /v1/chat/completions — same shape.

Clean up (keep the cluster!)

kubectl delete deployment hello
kubectl delete svc hello

The cluster stays — lesson 2 builds on it. (If you ever need to nuke it: kind delete cluster --name llm-stack.)

Check yourself

Answer from memory — retrieval is the point. Click an option.

A Pod in your Deployment crashes at 3 AM. What happens?

The Deployment's reconciliation loop notices actual state (1 replica) ≠ desired state (2 replicas) and creates a new Pod. Services never manage Pods — they only route to them.

Why does the router talk to a Service instead of Pod IPs?

Pods are mortal and get new IPs each time. The Service is the stable name that load-balances across whichever Pods currently match its selector.

In kind, what is a "node" physically?

kind = Kubernetes IN Docker. Each node is a container. That's why docker ps showed your "machines". Real clusters use VMs or bare metal — same API either way.
