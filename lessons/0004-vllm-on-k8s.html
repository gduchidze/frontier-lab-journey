SLM Production Stack · Lesson 4 · Phase 4 · ~90 min

vLLM Meets Kubernetes

The convergence lesson. Your engine image from lesson 3, deployed with the production patterns from lesson 2: namespace, PVC for model weights, probes tuned for slow model loads, and a Service the whole stack will talk to.

Prerequisites Cluster running · image vllm-cpu:local built (lesson 3) · namespace llm exists (lesson 2).

Knowledge: three new pieces

1. Getting a local image into kind

Your cluster nodes are containers — they can't see your Mac's Docker images. kind load docker-image copies an image into every node. In real clusters this step is "push to a registry" (ECR, GCR, Artifact Registry); kind lets you skip the registry while learning:

kind load docker-image vllm-cpu:local --name llm-stack

Then the manifest must say imagePullPolicy: IfNotPresent — otherwise K8s tries to pull vllm-cpu:local from Docker Hub and fails with ImagePullBackOff.

2. PersistentVolumeClaims: weights survive Pods

Without persistence, every Pod restart re-downloads ~2 GB of weights. A PVC requests storage that outlives Pods (docs). Mount it at /root/.cache/huggingface — the HF cache path — and restarts become fast. Both reference repos do exactly this (doronmak: vllm-models PVC; production-stack: pvcStorage per model). kind ships a default standard StorageClass, so a bare PVC "just works".

3. Probe timing for LLM engines

vLLM exposes /health (200 = engine up). The trap: model load on CPU can take minutes. So: startup probe with a huge window (failureThreshold: 60 × periodSeconds: 10 = 10 min grace), then tight readiness/liveness. Compare with the vllm-deploy-k8s skill templates and doronmak's chart — all three use this shape.

Lab: deploy the engine

Load the image into the cluster

kind load docker-image vllm-cpu:local --name llm-stack
# verify it landed on a node:
docker exec llm-stack-worker crictl images | grep vllm-cpu

expect: vllm-cpu local listed

Write the manifests

Save as k8s/vllm.yaml:

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: hf-cache
  namespace: llm
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 10Gi
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm
  namespace: llm
  labels: { app: vllm }
spec:
  replicas: 1
  selector:
    matchLabels: { app: vllm }
  template:
    metadata:
      labels: { app: vllm }
    spec:
      containers:
      - name: vllm
        image: vllm-cpu:local
        imagePullPolicy: IfNotPresent
        args:
          - "--model"
          - "Qwen/Qwen3.5-0.8B"
          - "--max-model-len"
          - "4096"
          - "--dtype"
          - "bfloat16"
        env:
        - name: VLLM_CPU_KVCACHE_SPACE
          value: "4"
        ports:
        - containerPort: 8000
          name: http
        resources:
          requests: { cpu: "4", memory: 6Gi }
          limits:   { cpu: "6", memory: 8Gi }
        startupProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 10
          failureThreshold: 60
        readinessProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 10
        livenessProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 20
          failureThreshold: 3
        volumeMounts:
        - name: hf-cache
          mountPath: /root/.cache/huggingface
        - name: dshm
          mountPath: /dev/shm
      volumes:
      - name: hf-cache
        persistentVolumeClaim: { claimName: hf-cache }
      - name: dshm
        emptyDir:
          medium: Memory
          sizeLimit: 2Gi
---
apiVersion: v1
kind: Service
metadata:
  name: vllm
  namespace: llm
spec:
  selector: { app: vllm }
  ports:
  - port: 8000
    targetPort: 8000
Why /dev/shm? PyTorch uses shared memory for inter-process tensors; the default container /dev/shm is 64 MB. Both reference repos mount a Memory-backed emptyDir. Undersized shm = cryptic crashes.

Deploy and watch the startup probe earn its keep

kubectl apply -f k8s/vllm.yaml
kubectl get pods -n llm -w

The Pod will sit READY 0/1 for several minutes (downloading + loading). That's the startup probe holding liveness back. Watch the engine work:

kubectl logs -n llm -l app=vllm -f

expect: eventually "Application startup complete" → pod flips to READY 1/1

Chat through the cluster

kubectl port-forward -n llm svc/vllm 8000:8000

Second terminal:

curl -s localhost:8000/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"You are running inside Kubernetes. How does that feel?"}],"max_tokens":80}' \
  | python3 -m json.tool

expect: Qwen answers — served by a Pod, through a Service

Prove the PVC works

kubectl delete pod -n llm -l app=vllm
kubectl get pods -n llm -w

expect: new pod becomes READY far faster — weights read from PVC, no re-download

The win A production-shaped inference service: declarative, self-healing, persistent, probed. This YAML file is the heart of your portfolio repo — everything else in the course wraps around it.

Check yourself

Pod shows ImagePullBackOff for vllm-cpu:local. Most likely cause?

Local-only images must be kind load-ed. K8s falls back to pulling from a registry, where the image doesn't exist. PVC issues surface differently (Pending / CreateContainerError).

Why mount the PVC at /root/.cache/huggingface specifically?

It's the Hugging Face cache convention. Persist it and any restart or new replica reuses downloaded weights.
