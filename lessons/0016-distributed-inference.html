Distributed and Disaggregated Inference

When the model outgrows one GPU — or the workload outgrows one architecture — serving becomes a distributed-systems problem. This is the lesson interviewers reach for when the req says "senior". Mostly theory + capacity math; your deliverable is a design doc.

Knowledge

1. The memory math that forces all of this

weights ≈ params × bytes/param        70B @ bf16 ≈ 140 GB   → no single GPU
KV/token ≈ 2 × layers × kv_heads × head_dim × bytes
   Llama-70B bf16 ≈ ~320 KB/token → 8k-ctx request ≈ ~2.6 GB of KV — EACH

Weights don't fit → split the model. KV dominates at scale → cache becomes a first-class distributed resource. Every architecture below is one of those two forces.

2. The parallelism zoo

Composition rule of thumb: TP up to 8 (one node), then PP across nodes, then DP replicas for throughput. Multi-node coordination runs on Ray. PP's tax: pipeline bubbles — stages idling while waiting; microbatching shrinks but never removes them. (docs)

3. Disaggregated prefill: split the two workloads onto different metal

Lesson 15's tension (prefill interferes with decode) scales into an architecture: prefill instances (compute-optimized) process prompts, then ship KV to decode instances (bandwidth-optimized) via a KV-transfer connector (NIXL, Mooncake, LMCache). Wins: independent scaling of the two phases, tight TPOT SLOs without prefill jitter, right hardware per phase. Cost: KV shipping latency + a scheduler that must place both halves. This is production-stack's "disaggregated prefill" roadmap line and the architecture behind DeepSeek/Mooncake-style serving. (docs)

4. The KV hierarchy: cache as infrastructure

LMCache (production-stack's sibling) tiers KV: GPU HBM → CPU RAM → local disk → remote store. Long system prompts, RAG contexts, and multi-turn histories get computed once and re-attached anywhere. Combine with the router's session/prefix awareness (lesson 6) and the cluster starts behaving like a CDN for computed attention. Tutorial 05 in production-stack wires exactly this.

5. Failure domains grow with the topology

DP replica dies → router drops it, capacity −1/N (you demoed this in lesson 8). TP shard dies → that whole replica dies (all-reduce needs every rank). PP stage dies → pipeline dead. Disagg prefill dies → TTFT degrades but decode drains. Senior design = stating blast radius per failure BEFORE it happens.

Check yourself

Why does TP stay inside one node while PP crosses nodes?

TP synchronizes at every layer — interconnect bandwidth is the ceiling, so NVLink-class links or bust. PP only passes activations at stage boundaries; Ethernet survives that.

Disaggregated prefill's core economic win?

Prefill demand (prompt tokens/s) and decode demand (concurrent streams) move independently; separate pools let you buy the right hardware for each and stop cross-interference.

One GPU dies in a TP=8 replica. Immediate effect?

All-reduce needs all ranks — TP is a single failure domain. In-flight generations on it are lost (the router retries new ones elsewhere). Design capacity with N-1 replicas in mind.
