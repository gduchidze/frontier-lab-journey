SLM Production Stack · გაკვეთილი 4 · ფაზა 4 · ~90 წთ

vLLM ხვდება Kubernetes-ს

შერწყმის გაკვეთილი. შენი ძრავის image გაკვეთილ 3-დან, დაპლოებული გაკვეთილ 2-ის production პატერნებით: namespace, PVC (PersistentVolumeClaim — მუდმივი საცავის მოთხოვნა) მოდელის წონებისთვის, ნელი ჩატვირთვისთვის მორგებული probe-ები და Service, რომელსაც მთელი სტეკი დაელაპარაკება.

წინაპირობები Cluster გაშვებულია · image vllm-cpu:local აშენებულია (გაკვეთილი 3) · namespace llm არსებობს (გაკვეთილი 2).

ცოდნა: სამი ახალი ნაწილი

1. ლოკალური image-ის შეტანა kind-ში

შენი cluster-ის node-ები კონტეინერებია — ისინი ვერ ხედავენ შენი Mac-ის Docker image-ებს. kind load docker-image image-ს ყველა node-ში აკოპირებს. რეალურ კლასტერებში ეს ნაბიჯი არის "push to a registry" (ECR, GCR, Artifact Registry); kind სწავლისას registry-ის გამოტოვების საშუალებას გაძლევს:

kind load docker-image vllm-cpu:local --name llm-stack

ამის შემდეგ მანიფესტში უნდა ეწეროს imagePullPolicy: IfNotPresent — წინააღმდეგ შემთხვევაში K8s შეეცდება vllm-cpu:local-ის Docker Hub-იდან ჩამოტვირთვას და ჩავარდება ImagePullBackOff-ით.

2. PersistentVolumeClaim-ები: წონები Pod-ებზე დიდხანს ცოცხლობენ

Persistence-ის გარეშე ყოველი Pod-ის გადატვირთვა ~2 GB წონებს ხელახლა ტვირთავს. PVC ითხოვს საცავს, რომელიც Pod-ებზე მეტხანს ცოცხლობს (დოკუმენტაცია). დაამაუნთე ის /root/.cache/huggingface-ზე — HF cache-ის path — და გადატვირთვები სწრაფი გახდება. ორივე reference რეპო ზუსტად ამას აკეთებს (doronmak: vllm-models PVC; production-stack: pvcStorage თითო მოდელზე). kind-ს მოჰყვება default standard StorageClass, ამიტომ უბრალო PVC "უბრალოდ მუშაობს".

3. Probe-ების timing LLM ძრავებისთვის

vLLM გამოაქვეყნებს /health-ს (200 = engine მუშაობს). ხაფანგი: მოდელის ჩატვირთვას CPU-ზე შესაძლოა წუთები დასჭირდეს. ამიტომ: startup probe უზარმაზარი ფანჯრით (failureThreshold: 60 × periodSeconds: 10 = 10 წთ ვადა), შემდეგ მჭიდრო readiness/liveness. შეადარე vllm-deploy-k8s skill-ის შაბლონებსა და doronmak-ის chart-ს — სამივე ამ ფორმას იყენებს.

ლაბორატორია: ძრავის deploy

ჩატვირთე image კლასტერში

kind load docker-image vllm-cpu:local --name llm-stack
# verify it landed on a node:
docker exec llm-stack-worker crictl images | grep vllm-cpu

მოსალოდნელია: vllm-cpu local სიაშია

დაწერე მანიფესტები

შეინახე როგორც k8s/vllm.yaml:

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: hf-cache
  namespace: llm
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 10Gi
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm
  namespace: llm
  labels: { app: vllm }
spec:
  replicas: 1
  selector:
    matchLabels: { app: vllm }
  template:
    metadata:
      labels: { app: vllm }
    spec:
      containers:
      - name: vllm
        image: vllm-cpu:local
        imagePullPolicy: IfNotPresent
        args:
          - "--model"
          - "Qwen/Qwen3.5-0.8B"
          - "--max-model-len"
          - "4096"
          - "--dtype"
          - "bfloat16"
        env:
        - name: VLLM_CPU_KVCACHE_SPACE
          value: "4"
        ports:
        - containerPort: 8000
          name: http
        resources:
          requests: { cpu: "4", memory: 6Gi }
          limits:   { cpu: "6", memory: 8Gi }
        startupProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 10
          failureThreshold: 60
        readinessProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 10
        livenessProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 20
          failureThreshold: 3
        volumeMounts:
        - name: hf-cache
          mountPath: /root/.cache/huggingface
        - name: dshm
          mountPath: /dev/shm
      volumes:
      - name: hf-cache
        persistentVolumeClaim: { claimName: hf-cache }
      - name: dshm
        emptyDir:
          medium: Memory
          sizeLimit: 2Gi
---
apiVersion: v1
kind: Service
metadata:
  name: vllm
  namespace: llm
spec:
  selector: { app: vllm }
  ports:
  - port: 8000
    targetPort: 8000
რატომ /dev/shm? PyTorch shared memory-ს იყენებს პროცესთაშორისი ტენზორებისთვის; კონტეინერის default /dev/shm 64 MB-ია. ორივე reference რეპო Memory-backed emptyDir-ს ამაუნთებს. ზედმეტად პატარა shm = გაუგებარი კრაშები.

Deploy და უყურე, როგორ ამართლებს startup probe თავის არსებობას

kubectl apply -f k8s/vllm.yaml
kubectl get pods -n llm -w

Pod რამდენიმე წუთი READY 0/1-ზე იჯდება (ჩამოტვირთვა + ჩატვირთვა). ეს startup probe-ია, რომელიც liveness-ს აკავებს. უყურე ძრავის მუშაობას:

kubectl logs -n llm -l app=vllm -f

მოსალოდნელია: საბოლოოდ "Application startup complete" → pod გადადის READY 1/1-ზე

ჩატი კლასტერის გავლით

kubectl port-forward -n llm svc/vllm 8000:8000

მეორე ტერმინალი:

curl -s localhost:8000/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"You are running inside Kubernetes. How does that feel?"}],"max_tokens":80}' \
  | python3 -m json.tool

მოსალოდნელია: Qwen პასუხობს — Pod-ის მიერ, Service-ის გავლით

დაამტკიცე, რომ PVC მუშაობს

kubectl delete pod -n llm -l app=vllm
kubectl get pods -n llm -w

მოსალოდნელია: ახალი pod ბევრად სწრაფად ხდება READY — წონები PVC-დან იკითხება, ხელახლა ჩამოტვირთვა აღარ არის

მოგება Production-ის ფორმის inference სერვისი: დეკლარაციული, თვითაღდგენადი, persistent, probe-ებით დაცული. ეს YAML ფაილი შენი პორტფოლიო-რეპოს გულია — კურსში ყველაფერი დანარჩენი მის გარშემო ეწყობა.

შეამოწმე საკუთარი თავი

Pod აჩვენებს ImagePullBackOff-ს vllm-cpu:local-ისთვის. ყველაზე სავარაუდო მიზეზი?

მხოლოდ ლოკალური image-ები kind load-ით უნდა ჩაიტვირთოს. K8s გადადის registry-დან ჩამოტვირთვაზე, სადაც image არ არსებობს. PVC-ის პრობლემები სხვაგვარად ვლინდება (Pending / CreateContainerError).

რატომ ვამაუნთებთ PVC-ს ზუსტად /root/.cache/huggingface-ზე?

ეს Hugging Face-ის cache-ის კონვენციაა. შეინახე ის მუდმივად და ნებისმიერი გადატვირთვა ან ახალი რეპლიკა ჩამოტვირთულ წონებს ხელახლა გამოიყენებს.
