SLM Production Stack · მოწინავე · გაკვეთილი 18 · ~3 სთ + წერილობითი ანგარიში

ღირებულება, სიმძლავრე და სენიორის Capstone

ამ კურსში ყველა არქიტექტურული გადაწყვეტილება ფარულად ფინანსური გადაწყვეტილება იყო. ბოლო გაკვეთილი ფულს ცხადად აჩვენებს — $/მილიონ ტოკენზე, build vs buy (აშენება თუ ყიდვა), capacity planning (სიმძლავრის დაგეგმვა) — და მთავრდება არტეფაქტით, რომელიც წრეს კრავს: პლატფორმის დიზაინის დოკუმენტით, რომელსაც იმიტირებულ ინტერვიუზე დაიცავ.

ცოდნა

1. ტოკენის ერთეულოვანი ეკონომიკა

$/1M output tokens ≈ (GPU $/hr × num_GPUs) / (throughput tok/s × 3600) × 1e6

H100 @ ~$2.5/hr, 70B fp8 TP=2 doing ~2500 tok/s:
  (2.5 × 2) / (2500 × 3600) × 1e6  ≈  $0.56 / 1M output tokens — at 100% utilization

გადამწყვეტი ცვლადი არის utilization (რესურსების ათვისება). საშუალო დატვირთვის 20%-ზე შენი რეალური ღირებულება 5-ჯერ აღემატება „ხელზე ნაანგარიშებ" რიცხვს — და 20–40% ტიპურია დღიური ციკლის მქონე ჩატის ტრაფიკისთვის. ყველაფერი, რაც ააგე, სწორედ ამ რიცხვის ასაწევად არსებობს: continuous batching (გაკვეთილი 3), routing (6), autoscaling (10), bin-packing (11), disaggregation (16).

2. ბერკეტები, დალაგებული იმის მიხედვით, თუ რამდენად ხშირად იმარჯვებენ

ბერკეტიტიპური მოგებასად ისწავლე
მოდელის სწორი ზომა (0.8B იქ, სადაც 70B რეფლექსი იყო; distill/fine-tune)10–100×მთელი კურსის ამოსავალი წერტილი
Utilization: autoscale + bin-pack + scale-to-zero არაპიკურ საათებში2–5×10, 11
Quantization (fp8 წონები + KV)~2× სიმძლავრე/GPU15
Prefix/KV caching საზიარო-კონტექსტიან სამუშაოებზე1.5–10× prefill-ზე6, 15, 16
Spot/preemptible batch-ისთვის + stateless რეპლიკებისთვის~3× ამ პულებზე11 (პრიორიტეტები მას უსაფრთხოს ხდის)
Batch tier: ღამის რიგი არაინტერაქტიული სამუშაოსთვისავსებს utilization-ის ჩავარდნებს17 (SLO tiers)

3. Build vs buy — რელიგიის გარეშე

API cost/mo  = tokens/mo × $/token           (scales with usage, zero floor)
Self-host/mo = GPUs × $/hr × 730 + eng_time  (scales with PEAK capacity, high floor)

Breakeven (გატოლების წერტილი) ჩვეულებრივ თვეში სტაბილურ ასობით მილიონ ტოკენზე მოდის — მაგრამ გადაწყვეტილება იშვიათად არის წმინდა მათემატიკა: მონაცემთა მართვის წესები (prompt-ები შენს VPC-ს ტოვებს?), latency-ის კონტროლი, მოდელის კასტომიზაცია (LoRA ფლოტები) და rate-limit-ებზე სუვერენიტეტი — ყველაფერი ეს build-ისკენ იხრება; მკვეთრად ცვალებადი/დაბალი მოცულობა და ინფრასტრუქტურის გუნდის არარსებობა — buy-სკენ. სენიორის პასუხი ცხრილია და არა ლოზუნგი — და „დაიწყე API-ებზე, გაზომე გამოყენება, breakeven-ის მრუდთან გადახედე" ჩვეულებრივ ყველაზე პატიოსანი რეკომენდაციაა.

4. Capacity planning როგორც განმეორებადი პროცედურა

1. Forecast: peak RPS, tokens in/out distribution, growth, tenant mix
2. Measure:  bench ONE replica on target hardware (lesson 9 — you know how)
3. Divide:   replicas = peak_demand / per-replica goodput (at SLO, not at max!)
4. Headroom: ×1.3–1.5 for N-1 failures, deploys, cache warmup (lesson 10's lesson)
5. Re-run:   every model change, quantization change, workload drift

ერთი რეპლიკის goodput SLO-ზე (სასარგებლო გამტარუნარიანობა) — მე-9 გაკვეთილის მრუდის „მუხლი" და არა პიკი — არის ის რიცხვი, რომელიც ცხრილში ჯდება. შენ ის უკვე გაზომე; ინჟინრების უმეტესობას ეს არასდროს გაუკეთებია.

5. დიზაინის დოკუმენტი — სენიორის მთავარი ნაშრომი

სამუშაო სტრუქტურა: კონტექსტი და მოთხოვნები → SLO-ები → არქიტექტურა (+ ორი უარყოფილი დიზაინი და მიზეზები) → სიმძლავრისა და ღირებულების მოდელი → Failure modes და blast radius-ები → უსაფრთხოება/tenancy → Rollout და ოპერაციები → ღია კითხვები. თუ 11–17 გაკვეთილებმა თითო არტეფაქტი დაგიტოვა, ეს დოკუმენტი ძირითადად მათ აწყობს — გეგმა თავიდანვე ეს იყო.

შეამოწმე შენი თავი

„ხელზე ნაანგარიშები" $0.56/1M ტოკენს ამბობს; ანგარიშ-ფაქტურა $2.80-ს გულისხმობს. ყველაზე სავარაუდო მიზეზი?

5× სხვაობა ≈ 20% utilization — კლასიკური რიცხვი დღიური ციკლის ტრაფიკისთვის. გამოთვლა უშვებდა, რომ GPU-ები არასდროს უქმდებიან; სინამდვილეში ძირითადად უქმად დგანან. ტექსტზე egress კაპიკებია.

Capacity plan მოთხოვნას რომელ ერთი-რეპლიკის რიცხვზე ყოფს?

„მუხლს" მიღმა ისეთ ტოკენებს ემსახურები, რომლებიც SLO-ს არღვევენ — სიმძლავრე, რომელიც „არსებობს", მაგრამ ვერ გაიყიდება. გაჯერების throughput-ზე დაგეგმვა პიკზე SLO-ს დარღვევებს თავიდანვე დებს გეგმაში.

ყველაზე ძლიერი ერთი არგუმენტი self-hosting-ის სასარგებლოდ, API-ს მოხერხებულობის მიუხედავად?

მონაცემთა მართვის მოთხოვნა ბინარულია და ზოგი მყიდველისთვის — არადათმობადი; ღირებულება მოცულობასთან ერთად იცვლება, ხოლო API-ვენდორები fine-tune-ებსაც ჰოსტავენ. როცა მოთხოვნა data-residency-ია, მათემატიკა მეორეხარისხოვანი ხდება.
