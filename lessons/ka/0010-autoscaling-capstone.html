SLM Production Stack · გაკვეთილი 10 · ფაზა 10 · ~2 სთ

Autoscaling და Capstone

ბოლო ციკლი: კლასტერი დატვირთვაზე თავად რეაგირებს. შემდეგ ათი გაკვეთილის ნამუშევარს კრავ პორტფოლიო-პროექტად, რომელიც ინტერვიუებზე შენ მაგივრად ილაპარაკებს.

ცოდნა: მართვის ციკლის ჩაკეტვა

1. HPA: სასურველი replica-ების რაოდენობა როგორც მეტრიკების ფუნქცია

HorizontalPodAutoscaler (ჰორიზონტალური ავტომასშტაბირების მექანიზმი) არეგულირებს Deployment-ის replicas-ს ისე, რომ მეტრიკა სამიზნესთან ახლოს დარჩეს: desired = ceil(current × metricValue / target). მას სჭირდება მეტრიკების წყარო — metrics-server CPU/მეხსიერებისთვის (kind-ზე ნაგულისხმევად დაყენებული არ არის).

2. CPU% არასწორი მეტრიკაა LLM-ებისთვის — მაინც იცოდე რატომ

ინფერენსის engine შეიძლება „სავსე" იყოს (რიგი იზრდება), როცა CPU 60%-ზეა, ან 100%-ზე იყოს მიჭერილი და მაინც ჯანმრთელი. სწორი სიგნალები engine-ის საკუთარი მეტრიკებია: vllm:num_requests_waiting, KV cache-ის გამოყენება — ზუსტად ამას გულისხმობს production-stack-ის roadmap ფრაზით „autoscaling based on vLLM-specific metrics". Custom-მეტრიკებზე მასშტაბირებას სჭირდება KEDA ან prometheus-adapter, რომელიც Prometheus → HPA ხიდს დებს. დღეს: CPU-ზე დაფუძნებული HPA (მექანიზმი), KEDA — დოკუმენტირებული stretch-მიზანი (production-პასუხი). ცოდნა იმისა, თუ რატომ არის CPU არასწორი, სწორედ ინტერვიუს პასუხია.

3. მასშტაბირების ლიმიტები, რომლებსაც ვერავინ გაექცევა

Scale-out ამატებს სიმძლავრეს, მაგრამ ყოველმა ახალმა replica-მ მოდელი უნდა ჩატვირთოს (CPU-ზე — წუთები; ამიტომაც არიან რეალური პლატფორმები მოდელის ჩატვირთვის დროზე შეპყრობილნი, იყენებენ PV-ში დაქეშილ წონებს და წინასწარ თბობას). ზედმეტად სწრაფად რომ დაპატარავდე (scale down), engine-ებს გენერაციის შუაში კლავ. HPA-ს behavior სექცია (სტაბილიზაციის ფანჯრები) ზუსტად ამისთვის არსებობს.

ლაბორატორია, ნაწილი A: რეაქტიული კლასტერი

დააყენე metrics-server (kind-ს flag სჭირდება)

helm repo add metrics-server https://kubernetes-sigs.github.io/metrics-server/
helm install metrics-server metrics-server/metrics-server \
  -n kube-system --set args={--kubelet-insecure-tls}
sleep 30 && kubectl top nodes && kubectl top pods -n llm

მოსალოდნელია: CPU/მეხსიერების რიცხვები თითო node-სა და pod-ზე (insecure-tls flag მხოლოდ kind-ისთვისაა: node-ების სერტიფიკატებს SAN-ები აკლიათ)

შექმენი HPA

შეინახე როგორც k8s/vllm-hpa.yaml:

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: vllm
  namespace: llm
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: vllm
  minReplicas: 1
  maxReplicas: 2
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
# let the HPA own replica count: set replicaCount back to 1 first
helm upgrade slm charts/slm-stack --set vllm.replicaCount=1
kubectl apply -f k8s/vllm-hpa.yaml
kubectl get hpa -n llm -w

მოსალოდნელია: TARGETS აჩვენებს ცოცხალ cpu%-ს 70%-თან შედარებით; REPLICAS 1

აამოქმედე

ტერმინალი 1: kubectl get hpa,pods -n llm -w. ტერმინალი 2: გაუშვი მე-9 გაკვეთილის k6 ტესტი (ან curl-ის ციკლი) 3–4 წუთის განმავლობაში.

მოსალოდნელია: cpu% სცდება 70-ს → REPLICAS 1→2 → მეორე pod ტვირთავს მოდელს → router-ს ავტომატურად უერთდება (service discovery!) → დატვირთვა ნაწილდება. ტრაფიკის შეჩერების შემდეგ: ~5 წთ სტაბილიზაცია, შემდეგ ისევ 1

მოგება დატვირთვა იზრდება → HPA ამასშტაბებს → router აღმოაჩენს → Grafana ორივე engine-ს აჩვენებს → დაცლა → scale-in. კურსის ყველა ფენა ერთ ავტონომიურ უკუკავშირის ციკლში, ადამიანის გარეშე. სწორედ ეს არის production-სტეკი.

Stretch (არასავალდებულო, ცალკე სესია): მასშტაბირება რიგის სიღრმეზე KEDA-თი

helm repo add kedacore https://kedacore.github.io/charts
helm install keda kedacore/keda -n keda --create-namespace

შემდეგ — ScaledObject prometheus-ტრიგერით sum(vllm:num_requests_waiting)-ზე, ზღვარი ~5. იხილე KEDA-ს prometheus scaler-ის დოკუმენტაცია. ეს რეალური სამყაროს პატერნია — README-ში მაინც აღწერე, თუნდაც მხოლოდ პროტოტიპი გააკეთო.

უთხარი შენს მასწავლებელს

როცა ჩანაწერი მზადაა, სთხოვე აგენტს: იმიტირებული ინტერვიუს „დაკითხვა" სტეკზე („მოთხოვნის გზა ამიწერე", „engine-ი ღამის 2 საათზე OOM-ავს — რას აკეთებ?", „მეორე მოდელს როგორ დაამატებდი?") და სასწავლო ჩანაწერის ბოლო ჩანაწერი. შემდეგ იფიქრე სიბრძნის ნაბიჯზე: გამოაქვეყნე რეპოზიტორია r/LocalLLaMA-ზე ან vLLM-ის Slack-ში რეალური უკუკავშირისთვის.

შეამოწმე შენი თავი

რატომ არის CPU-ს გამოყენება ცუდი autoscaling-სიგნალი LLM engine-ებისთვის?

Engine-ის სისავსე = მიღების/რიგის მდგომარეობა (მომლოდინე მოთხოვნები, KV cache), რომელიც CPU%-ისგან დამოუკიდებელია. იმასშტაბე num_requests_waiting-ზე KEDA/prometheus-adapter-ის მეშვეობით.

რატომ ვაყენებთ HPA-ზე scale-down-ის სტაბილიზაციის ფანჯარას?

ტრაფიკი მოკლედ ეცემა → მყისიერი scale-in engine-ს გენერაციის შუაში მოკლავდა და ტრაფიკის დაბრუნებისას მოდელის რამდენიმეწუთიან ხელახალ ჩატვირთვას გამოიწვევდა. ფანჯარა scale-in-ს გააზრებულს ხდის.

ახალი replica მასშტაბირებით ემატება, მაგრამ წუთების განმავლობაში ვერაფერს ემსახურება. რატომ?

Pod არ არის Ready, სანამ engine მოდელს არ ჩატვირთავს და /health არ გაივლის — ამიტომ ტრაფიკი მას ვერ აღწევს. მოდელის ჩატვირთვის ეს დაყოვნება არის LLM serving-ის მასშტაბირების მთავარი ვიწრო ადგილი (და ამიტომაც არის მნიშვნელოვანი წონების დამქეშავი PV-ები).

HPA-ს დოკუმენტაცია + KEDA-ს prometheus scaler. საით მიდის დარგი: production-stack-ის 2026 roadmap-ის დისკუსია.
