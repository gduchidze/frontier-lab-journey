SLM Production Stack · გაკვეთილი 6 · ფაზა 6 · ~2 სთ

მასშტაბირება: რეპლიკები და მოთხოვნების Router

ერთ engine-ს (ინფერენსის ძრავას) მომსახურების ზღვარი აქვს. დღეს ორს გაუშვებ და გაიგებ, რატომ სჭირდება LLM serving-ს უბრალო Service-ზე ჭკვიანი load balancer-ი — მოთხოვნების router-ი (მარშრუტიზატორი), vLLM production-stack-ის გული.

ჯერ მეხსიერება შეამოწმე ორი vLLM რეპლიკა ≈ 12–16 Gi მოთხოვნილი 16 Gi Mac-ზე. ამ ლაბორატორიამდე: Docker Desktop → Settings → Resources → მიეცი Docker-ს ≥ 12 GB. ასევე შეამცირე რესურსები თითო რეპლიკაზე (ნაბიჯი 1). თუ შენი მანქანა ძლიერ swap-ავს, გაიარე ეს გაკვეთილი მოდელად facebook/opt-125m-ით — მარშრუტიზაციის გაკვეთილები იდენტურია.

ცოდნა: რატომ არ კმარა უბრალო Service

1. LLM მოთხოვნები ვებ-მოთხოვნები არ არის

ვებ-მოთხოვნა მილიწამები ღირს; LLM მოთხოვნა — წამებიდან წუთებამდე მდგომარეობის მქონე გამოთვლა, მზარდი KV cache-ით. შემთხვევითი round-robin (რასაც Service აკეთებს) უგულებელყოფს: engine-ის დატვირთვას (რიგის სიღრმეს), KV cache-ის ხელახლა გამოყენებას (ერთი და იმავე მომხმარებლის საუბრის პრეფიქსი უკვე დაქეშილია engine A-ზე) და მოდელის განთავსებას (engine B სხვა მოდელს ემსახურება). შედეგი: თავიდან აცილებადი დაყოვნება (latency) და cache-იდან გამოდევნები.

2. Router აპლიკაციის დონის load balancer-ია

production-stack-ის vllm-router engine-ების წინ დგას და ამატებს იმას, რაც Service-ს არ შეუძლია:

შესაძლებლობარატომ არის მნიშვნელოვანი
K8s service discoveryAPI-ს უყურებს engine Pod-ების label-ის მიხედვით — რეპლიკები ავტომატურად ერთვებიან/გამოდიან
მარშრუტიზაციის ალგორითმებიround-robin, session-ID affinity, prefix-aware (ერთი პრეფიქსი → ერთი engine → KV cache-ის მოხვედრა)
მეტრიკები თითო engine-ზეQPS, TTFT, pending requests თითოეულ engine-ზე — კვებავს მე-7 ფაზის dashboard-ებს
მოდელის მარშრუტიზაციაერთი endpoint-ი, ბევრი მოდელი: მოთხოვნის model ველი ირჩევს backend-ს

3. Session affinity = KV cache-ის ეკონომიკა

ჩატის საუბარი ყოველ სვლაზე მთელ ისტორიას ხელახლა აგზავნის. თუ N-ური სვლა იმ engine-ზე მოხვდება, რომელმაც N−1-ური მოემსახურა, პრეფიქსი უკვე KV cache-შია (prefix caching-ის ჩართვისას) → TTFT მკვეთრად ეცემა. მიბმული (sticky) მარშრუტიზაცია სასიამოვნო წვრილმანი კი არა, გამოთვლითი ხარჯების ბერკეტია. სწორედ ეს არის "KV cache reuse" ფრაზა production-stack-ის README-ში — და ML infra ვაკანსიების აღწერებში.

ლაბორატორია: ორი engine, ერთი router

შეამცირე რესურსები რეპლიკაზე, გაზარდე 2-მდე

დაარედაქტირე charts/slm-stack/values.yaml: replicaCount: 2, resources → requests: { cpu: "2", memory: 5Gi }, limits: { cpu: "4", memory: 6Gi }, kvCacheSpaceGiB: 2. კიდევ ერთი ცვლილება — ორი Pod-ი ვერ გაიზიარებს ReadWriteOnce PVC-ს სხვადასხვა node-ზე. უმარტივესი ლოკალური გამოსავალი: deployment template-ში PVC volume ჩაანაცვლე per-pod emptyDir-ით (წონები თითო Pod-ზე ერთხელ ხელახლა ჩამოიტვირთება; ლოკალურად შეიგუე) — ან ორივე რეპლიკა ერთ node-ზე მიამაგრე nodeSelector-ით. აირჩიე, შემდეგ:

helm upgrade slm charts/slm-stack
kubectl get pods -n llm -o wide -w    # wait for both READY 1/1 (patience: CPU load × 2)

მოსალოდნელია: vllm-xxxx და vllm-yyyy — ორივე Running READY 1/1

დააკვირდი უბრალო Service-ის round-robin-ს

kubectl port-forward -n llm svc/vllm 8000:8000 &
for i in 1 2 3 4 5 6; do
  curl -s localhost:8000/v1/chat/completions -H "Content-Type: application/json" \
    -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"hi"}],"max_tokens":5}' > /dev/null
done
kubectl logs -n llm -l app=vllm --tail=5 --prefix | grep "POST /v1"

მოსალოდნელია: მოთხოვნები ჩანს ორივე Pod-ის ლოგებში (შენიშვნა: port-forward ერთ Pod-ს ემაგრება — კლასტერშიდა ტრაფიკი ნაწილდება; შესადარებლად ორივე ლოგი შეამოწმე)

გაუშვი router-ი

შეინახე როგორც k8s/router.yaml:

apiVersion: v1
kind: ServiceAccount
metadata:
  name: vllm-router
  namespace: llm
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: llm
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: vllm-router-pod-reader
  namespace: llm
subjects:
- kind: ServiceAccount
  name: vllm-router
  namespace: llm
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm-router
  namespace: llm
  labels: { app: vllm-router }
spec:
  replicas: 1
  selector:
    matchLabels: { app: vllm-router }
  template:
    metadata:
      labels: { app: vllm-router }
    spec:
      serviceAccountName: vllm-router
      containers:
      - name: router
        image: lmcache/lmstack-router:latest
        args:
          - "--host"
          - "0.0.0.0"
          - "--port"
          - "8000"
          - "--service-discovery"
          - "k8s"
          - "--k8s-namespace"
          - "llm"
          - "--k8s-label-selector"
          - "app=vllm"
          - "--routing-logic"
          - "roundrobin"
          - "--engine-scrape-interval"
          - "15"
          - "--request-stats-window"
          - "60"
        ports:
        - containerPort: 8000
        resources:
          requests: { cpu: 250m, memory: 256Mi }
          limits:   { cpu: "1",  memory: 512Mi }
---
apiVersion: v1
kind: Service
metadata:
  name: vllm-router
  namespace: llm
spec:
  selector: { app: vllm-router }
  ports:
  - port: 80
    targetPort: 8000
ორ რამეს მიაქციე ყურადღება (1) RBAC ბლოკი: router-ი Pod-ებს K8s API-ით უყურებს, ამიტომ მას ცხადი ნებართვა სჭირდება — შენი პირველი შეხება კლასტერის უსაფრთხოებასთან. (2) Image tag: apply-მდე გადაამოწმე მიმდინარე image/args router-ის README-სთან — flag-ები იცვლება. თუ დაეჭვდები, სთხოვე მასწავლებელს გადამოწმება.
kubectl apply -f k8s/router.yaml
kubectl logs -n llm -l app=vllm-router -f    # expect: discovers 2 engine pods

გაატარე ტრაფიკი მასში

kill %1 2>/dev/null   # stop old port-forward
kubectl port-forward -n llm svc/vllm-router 8080:80

მეორე ტერმინალი — იგივე API, ახლა router-ის უკან:

curl -s localhost:8080/v1/models | python3 -m json.tool
for i in 1 2 3 4; do
  curl -s localhost:8080/v1/chat/completions -H "Content-Type: application/json" \
    -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"one word: ping"}],"max_tokens":5}' | python3 -c "import json,sys; print(json.load(sys.stdin)['choices'][0]['message']['content'])"
done

მოსალოდნელია: 4 პასუხი; router-ის ლოგებში ჩანს, რომ მოთხოვნები engine-ებს შორის მონაცვლეობს

გადადი სესიაზე მიბმულ მარშრუტიზაციაზე

k8s/router.yaml-ში შეცვალე --routing-logic მნიშვნელობით session და დაამატე --session-key / x-user-id არგუმენტები, შემდეგ:

kubectl apply -f k8s/router.yaml
kubectl rollout status deployment/vllm-router -n llm
# same header → same engine every time:
for i in 1 2 3 4; do
  curl -s localhost:8080/v1/chat/completions -H "Content-Type: application/json" \
    -H "x-user-id: giorgi" \
    -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"ping"}],"max_tokens":5}' > /dev/null
done
kubectl logs -n llm -l app=vllm --tail=10 --prefix | grep "POST /v1"

მოსალოდნელია: ოთხივე მოთხოვნა ერთი Pod-ის ლოგში — affinity მუშაობს

მოგება ახლა production-stack-ის საბაზისო არქიტექტურას უშვებ: N engine + აღმომჩენი, affinity-ს მცოდნე router-ი. აქედან მოყოლებული ყველაფერი (UI, მონიტორინგი, დატვირთვის ტესტები) vllm-router.llm.svc.cluster.local-ს ელაპარაკება.

შეამოწმე საკუთარი თავი

რატომ ამცირებს session affinity time-to-first-token-ს ჩატში?

ჩატი ისტორიას ყოველ სვლაზე ხელახლა აგზავნის; იმავე engine-ზე მოხვედრა ნიშნავს, რომ საზიარო პრეფიქსის KV cache ხელახლა გამოიყენება ხელახალი გამოთვლის ნაცვლად — და სწორედ ეს ხელახალი გამოთვლაა TTFT-ის უმეტესი ნაწილი.

რატომ დასჭირდა router-ის Deployment-ს ServiceAccount + Role?

service discovery = K8s API-სთან შეკითხვა label-ს შესაბამისი Pod-ების შესახებ. ნაგულისხმევ service account-ებს ეს არ შეუძლიათ. router-ი engine-ებს არასოდეს გადატვირთავს — ეს Deployment controller-ის საქმეა.

პირველწყარო
