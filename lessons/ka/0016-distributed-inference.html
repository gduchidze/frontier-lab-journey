განაწილებული და დისაგრეგირებული Inference

როცა მოდელი ერთ GPU-ს გადააჭარბებს — ან დატვირთვა ერთ არქიტექტურას — serving განაწილებული სისტემების პრობლემად იქცევა. ეს ის გაკვეთილია, რომელსაც ინტერვიუერები მაშინ იშველიებენ, როცა ვაკანსიაში „senior" წერია. ძირითადად თეორია + ტევადობის მათემატიკა; შენი შედეგი დიზაინის დოკუმენტია.

ცოდნა

1. მეხსიერების მათემატიკა, რომელიც ამ ყველაფერს გვაიძულებს

weights ≈ params × bytes/param        70B @ bf16 ≈ 140 GB   → no single GPU
KV/token ≈ 2 × layers × kv_heads × head_dim × bytes
   Llama-70B bf16 ≈ ~320 KB/token → 8k-ctx request ≈ ~2.6 GB of KV — EACH

წონები არ ეტევა → გაყავი მოდელი. KV cache მასშტაბზე დომინირებს → cache პირველი რანგის განაწილებულ რესურსად იქცევა. ქვემოთ მოცემული ყველა არქიტექტურა ამ ორი ძალიდან ერთ-ერთია.

კომბინირების ემპირიული წესი: TP 8-მდე (ერთი კვანძი), შემდეგ PP კვანძებს შორის, შემდეგ DP რეპლიკები throughput-ისთვის. მრავალკვანძიან კოორდინაციას Ray უზრუნველყოფს. PP-ის გადასახადი: pipeline-ის ბუშტები (bubbles) — საფეხურები უქმად დგანან ლოდინისას; microbatching ამცირებს მათ, მაგრამ ვერასდროს აქრობს. (docs)

3. Disaggregated prefill: გაყავი ორი დატვირთვა სხვადასხვა რკინაზე

მე-15 გაკვეთილის დაძაბულობა (prefill ხელს უშლის decode-ს) არქიტექტურად იზრდება: prefill ინსტანსები (გამოთვლებზე ოპტიმიზებული) ამუშავებენ პრომპტებს, შემდეგ KV-ს გზავნიან decode ინსტანსებზე (გამტარუნარიანობაზე ოპტიმიზებული) KV-transfer კონექტორით (NIXL, Mooncake, LMCache). მოგება: ორი ფაზის დამოუკიდებელი მასშტაბირება, მკაცრი TPOT SLO-ები prefill-ის რყევის გარეშე, სწორი აპარატურა თითო ფაზაზე. ფასი: KV-ის გადაზიდვის latency + scheduler-ი, რომელმაც ორივე ნახევარი უნდა განათავსოს. ეს არის production-stack-ის „disaggregated prefill" roadmap-ის ხაზი და DeepSeek/Mooncake-ის სტილის serving-ის უკან მდგარი არქიტექტურა. (docs)

4. KV იერარქია: cache როგორც ინფრასტრუქტურა

LMCache (production-stack-ის და-პროექტი) KV-ს იარუსებად ალაგებს: GPU HBM → CPU RAM → ლოკალური დისკი → დაშორებული საცავი. გრძელი system prompt-ები, RAG კონტექსტები და მრავალსვლიანი ისტორიები ერთხელ ითვლება და ნებისმიერ ადგილას ხელახლა ერთვება. დაუმატე router-ის session/prefix ცნობიერება (გაკვეთილი 6) და კლასტერი გამოთვლილი attention-ის CDN-ივით იწყებს ქცევას. production-stack-ის Tutorial 05 ზუსტად ამას აწყობს.

5. მტყუნების დომენები ტოპოლოგიასთან ერთად იზრდება

DP რეპლიკა კვდება → router-ი აგდებს მას, ტევადობა −1/N (ეს მე-8 გაკვეთილში აჩვენე). TP shard-ი კვდება → მთელი ის რეპლიკა კვდება (all-reduce-ს ყველა rank-ი სჭირდება). PP საფეხური კვდება → pipeline მკვდარია. Disagg prefill კვდება → TTFT უარესდება, მაგრამ decode ბოლომდე მიდის. სენიორული დიზაინი = აფეთქების რადიუსის (blast radius) ჩამოყალიბება თითო მტყუნებაზე მანამ, სანამ ის მოხდება.

ლაბორატორია (labs/lab-16): ტევადობის მათემატიკა + დიზაინის დოკუმენტი

ქაღალდის ლაბორატორია — შედეგები არის ის, რაც „წაკითხული მაქვს"-ს „დაპროექტება შემიძლია"-სგან განასხვავებს.

სამუშაო ფურცელი: სამი ზომის ამოცანა

docs/distributed-design.md-ში აჩვენე შენი არითმეტიკა: (a) Llama-3.1-70B bf16 H100-80GB-ზე — მინიმალური TP წონებისთვის + KV-ის თავისუფალი ადგილი 8k ctx × 32 კონკურენტულზე; (b) იგივე fp8 წონებით + fp8 KV-ით — რა მცირდება, რა არის ახლა შემზღუდავი ფაქტორი; (c) 405B კვანძებზე — TP×PP განლაგება და სად გვტკივა ბუშტები.

წაიკითხე ნამდვილი

production-stack-ის tutorial 05 (KV offloading) + router-ის KV-aware routing-ის ჩანაწერები. შენს დოკუმენტში თითოეული config ბლოკი ამ გაკვეთილის ცნებას შეუსაბამე.

დიზაინი: ინტერვიუს კითხვა

იმავე დოკუმენტში, ბოლო სექცია: „ჩატის პროდუქტი, 30k კონკურენტული მომხმარებელი, 8k საშუალო კონტექსტი, მკაცრი TPOT SLO, H100-ის ბიუჯეტი არსებობს. მონოლითური DP ფლოტი vs დისაგრეგირებული P/D — აირჩიე, დაასაბუთე, ჩამოაყალიბე blast radius-ები და ის მეტრიკები, რომლებიც გეტყოდა, რომ არასწორად აირჩიე." ერთი გვერდი, მკაფიო პოზიციით.

დაიცავი

სთხოვე მასწავლებელს, შეუტიოს დოკუმენტს. ერთხელ გადაამუშავე. (არასავალდებულო cloud სესია: იქირავე 2×A10 ერთი საათით, გაუშვი ნამდვილი TP=2 — დაგეგმვა მასწავლებელს სთხოვე.)

შეამოწმე თავი

რატომ რჩება TP ერთი კვანძის შიგნით, სანამ PP კვანძებს კვეთს?

TP ყოველ ფენაზე სინქრონდება — ჭერი interconnect-ის გამტარუნარიანობაა, ამიტომ NVLink-ის კლასის ბმულები ან არაფერი. PP აქტივაციებს მხოლოდ საფეხურების საზღვრებზე გადასცემს; ამას Ethernet-იც უძლებს.

Disaggregated prefill-ის მთავარი ეკონომიკური მოგება?

Prefill-ის მოთხოვნა (prompt tokens/s) და decode-ის მოთხოვნა (კონკურენტული ნაკადები) დამოუკიდებლად მოძრაობს; ცალკეული პულები საშუალებას გაძლევს, თითოეულისთვის სწორი აპარატურა იყიდო და ურთიერთხელშეშლა შეწყვიტო.

TP=8 რეპლიკაში ერთი GPU კვდება. მყისიერი ეფექტი?

All-reduce-ს ყველა rank-ი სჭირდება — TP ერთი მტყუნების დომენია. მასზე მიმდინარე გენერაციები იკარგება (router-ი ახლებს სხვაგან ცდის). ტევადობა N-1 რეპლიკის გათვალისწინებით დააპროექტე.
