UI: Open WebUI სტეკზე

აქამდე ყველაფერი curl-ის ენაზე ლაპარაკობდა. დღეს სტეკი სახეს იძენს: Open WebUI — სრულფასოვანი ჩატის frontend-ი — კლასტერში გაშლილი, router-ზე მიერთებული და შენს ბრაუზერამდე მიწვდენილი იმ NodePort-ით, რომელიც ჯერ კიდევ 1-ელ გაკვეთილში დააკონფიგურირე.

ცოდნა: სამი იდეა

1. OpenAI API-ის დივიდენდი ანაზღაურდება

Open WebUI OpenAI API-ზე ლაპარაკობს. მიმართე OPENAI_API_BASE_URL ნებისმიერ თავსებად სერვერზე — vLLM, შენი router — და ის ჩამოთვლის მოდელებს, ასტრიმებს ჩატს, მართავს საუბრებს. ნულოვანი მორგებული კოდი. სწორედ ამიტომ უწოდა მე-3 გაკვეთილმა ამ API-ს "დივიდენდი": ეკოსისტემის ყველა ინსტრუმენტი უფასოდ ერთვება. (ამ წყვილს vLLM-ის საკუთარი დოკუმენტაციაც აჩვენებს: Open WebUI deployment guide.)

2. კლასტერშიდა DNS აკეთებს გადაბმას

UI router-ს Service-ის DNS-ით სწვდება: http://vllm-router.llm.svc.cluster.local/v1. არც IP-ები, არც კონფიგურაციის ძებნა — მე-2 გაკვეთილის პატერნი, რომელიც ახლა namespace-ებს შორისაც ამართლებს.

3. NodePort: მიწვდომადობის მესამე გზა

port-forward (dev გვირაბი) და ClusterIP (მხოლოდ კლასტერშიდა) უკვე იცი. NodePort (კვანძის პორტი) ხსნის პორტს (30000–32767) ყველა node-ზე. შენი 1-ელი გაკვეთილის kind კონფიგმა node-ის პორტი 30080 → Mac-ის პორტ 30080-ზე გადაანაწილა — ზუსტად დღევანდელი დღისთვის ჩათესილი. production-ში Ingress/LoadBalancer იქნებოდა; NodePort ამის პატიოსანი ლოკალური ეკვივალენტია. (დოკუმენტაცია)

ლაბორატორია: მიაწოდე სახე

გაუშვი Open WebUI

შეინახე როგორც k8s/webui.yaml:

apiVersion: v1
kind: Namespace
metadata:
  name: ui
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: webui-data
  namespace: ui
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 2Gi
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: open-webui
  namespace: ui
  labels: { app: open-webui }
spec:
  replicas: 1
  selector:
    matchLabels: { app: open-webui }
  template:
    metadata:
      labels: { app: open-webui }
    spec:
      containers:
      - name: open-webui
        image: ghcr.io/open-webui/open-webui:main
        env:
        - name: OPENAI_API_BASE_URL
          value: "http://vllm-router.llm.svc.cluster.local/v1"
        - name: OPENAI_API_KEY
          value: "not-needed-locally"
        - name: WEBUI_AUTH
          value: "False"
        - name: ENABLE_OLLAMA_API
          value: "False"
        ports:
        - containerPort: 8080
        resources:
          requests: { cpu: 250m, memory: 512Mi }
          limits:   { cpu: "1",  memory: 1Gi }
        readinessProbe:
          httpGet: { path: /health, port: 8080 }
          periodSeconds: 10
        volumeMounts:
        - name: data
          mountPath: /app/backend/data
      volumes:
      - name: data
        persistentVolumeClaim: { claimName: webui-data }
---
apiVersion: v1
kind: Service
metadata:
  name: open-webui
  namespace: ui
spec:
  type: NodePort
  selector: { app: open-webui }
  ports:
  - port: 8080
    targetPort: 8080
    nodePort: 30080
WEBUI_AUTH=False მხოლოდ ლოკალურადაა დასაშვები შენს ლეპტოპზე ნორმალურია, მაგრამ დისკვალიფიკაციაა ყველგან, სადაც სხვები წვდებიან. თუ ამას ოდესმე საკუთარი მანქანის გარეთ აჩვენებ, ჩართე ავთენტიფიკაცია და TLS-ის უკან დააყენე. აღნიშნე ეს პორტფოლიოს README-ში — შემფასებლები უსაფრთხოების გააზრებას ამჩნევენ.
kubectl apply -f k8s/webui.yaml
kubectl get pods -n ui -w

მოსალოდნელია: open-webui Pod-ი READY 1/1 ~1 წუთში (პირველი image pull ~1 GB-ია, მოთმინება)

გახსენი — port-forward აღარ გჭირდება

ბრაუზერი → http://localhost:30080. მიჰყევი, რატომ მუშაობს ეს, ნაბიჯ-ნაბიჯ: ბრაუზერი → Mac:30080 → kind extraPortMapping → control-plane node:30080 → NodePort Service → UI Pod. ხმამაღლა თქვი; K8s ქსელში მოთხოვნის გზის კვალდაკვალ გავლა გასაუბრებების კლასიკაა.

მოსალოდნელია: Open WebUI იტვირთება; მოდელის ამრჩევში ჩანს Qwen/Qwen3.5-0.8B (წამოღებული router-ის /v1/models-იდან)

ისაუბრე ჩატში, მერე ფარდის უკან ჩაიხედე

UI-ში მოკლე საუბარი გამართე. შემდეგ, გაყოფილი ეკრანით Grafana-სთან ერთად (მე-7 გაკვეთილის dashboard-ი):

kubectl logs -n llm -l app=vllm-router -f --tail=5

მოსალოდნელია: UI-ს ყოველი შეტყობინება = POST /v1/chat/completions router-ის ლოგებში; Grafana-ში TTFT პანელი ინძრევა

მოგება მომხმარებლისკენ მიმართული პროდუქტი შენივე ინფერენსის ინფრასტრუქტურაზე: ბრაუზერი → NodePort → UI → router → engine-ები, წონები PVC-ებზე და მეტრიკები Grafana-სკენ მიედინება. ეს წინადადება — რომლის დაცვაც ახლა სტრიქონ-სტრიქონ შეგიძლია — შენი CV-ის პროექტის რეზიუმეა.

მდგრადობის დემო (5 წთ, გააკეთე — სახალისოა)

საუბრის შუაში მოკალი ერთი engine:

kubectl delete pod -n llm $(kubectl get pods -n llm -l app=vllm -o name | head -1 | cut -d/ -f2)

განაგრძე ჩატი. router-ი აღმოაჩენს მკვდარ engine-ს და გადარჩენილზე ამისამართებს; Deployment-ი მსხვერპლს აღადგენს. მომხმარებელი ვერაფერს ამჩნევს. ყველა ფენა, რომელიც ააგე, ახლა თვალსაჩინოდ აკეთებს თავის საქმეს.

შეამოწმე საკუთარი თავი

როგორ პოულობს Open WebUI მოდელებს თავის ამრჩევში საჩვენებლად?

წმინდა OpenAI-API კლიენტის ქცევაა — ის endpoint-ს ეკითხება, რა მოდელები არსებობს. K8s API-ს მხოლოდ router-ი ეხება.

რატომ აღწევს localhost:30080 UI-მდე port-forward-ის გარეშე?

ორი მექანიზმის ჯაჭვია: NodePort ხსნის 30080-ს node-ზე; 1-ელი გაკვეთილის extraPortMappings Mac:30080 → node:30080-ზე ამისამართებს. kind-ის ამ გადანაწილების გარეშე NodePort მხოლოდ node-ის კონტეინერებიდან იქნებოდა მიწვდომადი.

პირველწყარო

vLLM-ის დოკუმენტაცია — Open WebUI და Open WebUI-ის დოკუმენტაცია (env ცვლადების ცნობარი).
