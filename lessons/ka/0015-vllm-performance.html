vLLM-ის წარმადობის ინჟინერია

საბაზისო კურსი ძრავას კარგ „შავ ყუთად" განიხილავდა. სენიორებს იმაში უხდიან, რომ ის გახსნან: quantization (კვანტიზაცია), batching-ის სახელურები, chunked prefill, speculative decoding — და, უპირველეს ყოვლისა, დისციპლინირებული მეთოდი ტიუნინგისთვის, ფლაგების ქაოსური ცვლის ნაცვლად.

ცოდნა

1. ძირეული პრინციპები: prefill გამოთვლებით შეზღუდულია, decode — გამტარუნარიანობით

Prefill (პრომპტის წინასწარი დამუშავება) მთელ პრომპტს პარალელურად ამუშავებს — FLOPs-ით შეზღუდული. Decode (ტოკენების გენერაცია) ყოველ ნაბიჯზე თითო ტოკენს აწარმოებს თითო მიმდევრობაზე და ყოველ ნაბიჯზე ხელახლა კითხულობს წონებს + KV cache-ს — მეხსიერების გამტარუნარიანობით შეზღუდული. ქვემოთ მოცემული ყველა ოპტიმიზაცია ან ერთ მხარეს ესხმის თავს, ან მათ ურთიერთხელშეშლას. თქვი რომელს — და სენიორად ჟღერ; გამოტოვე — და სახელურები ასტროლოგიაა.

2. Quantization-ის ტაქსონომია

ოჯახირა მცირდებაეფექტიშენიშვნები
FP8 (W8A8)წონები + აქტივაციები~2× მეხსიერება, prefill-ის აჩქარება H100+-ზეთითქმის უდანაკარგო; production-ის ნაგულისხმევი Hopper-ზე
AWQ / GPTQ (4-bit)მხოლოდ წონები~4× წონების მეხსიერება, decode-ის აჩქარება (ნაკლებია წასაკითხი)კალიბრაციაზე დაფუძნებული; მცირე ხარისხობრივი დანახარჯი; შესანიშნავია დიდი მოდელების ჩასატევად
KV cache FP8 (--kv-cache-dtype fp8)KV cache~2× cache → ~2× batch/კონტექსტის ტევადობატევადობის ბოთლყელს პირდაპირ ესხმის თავს
GGUFწონები (llama.cpp-ის სამყარო)CPU-სთვის მოსახერხებელი ფორმატებიvLLM-ის მხარდაჭერა არსებობს; ყურადღება მიაქციე წყაროს ხარისხს

გადაწყვეტილების ფორმა: მეხსიერება გაკლია → წონების quantization; batch/კონტექსტი გაკლია → KV quantization; Hopper GPU-ები → FP8 ნაგულისხმევად; და ყოველთვის ხელახლა გაუშვი შენივე ევალები — quantization-ით ხარისხის დანაკარგი დატვირთვის ტიპზეა დამოკიდებული. (docs)

3. Batching-ის სახელურები

სახელურირას მართავსროდის გაზარდო
--max-num-seqsმაქს. მიმდევრობა თითო ნაბიჯზერიგი იზრდება, სანამ KV-ის გამოყენება დაბალია
--max-num-batched-tokensტოკენების ბიუჯეტი თითო ნაბიჯზეthroughput-ზე ორიენტირებული დატვირთვები; შეამცირე TPOT-ის დასაცავად
--gpu-memory-utilizationწინასწარ გამოყოფილი წილი (წონები+KV)ნაგულისხმევი 0.9; დატოვე თავისუფალი ადგილი პიკებისთვის/სხვა pod-ებისთვის
--max-model-lenუარესი შემთხვევის KV თითო მიმდევრობაზეშეამცირე, რომ მეტი პარალელური მიმდევრობა დაუშვა

4. Chunked prefill: აღარ მისცე გრძელ პრომპტებს decode-ის დაშიმშილების საშუალება

გულუბრყვილო დაგეგმვა უზარმაზარ prefill-ს ერთ ნაბიჯად უშვებს — ყველა decode-ის ფაზაში მყოფი მომხმარებელი ჩერდება (TPOT-ის ნახტომი). Chunked prefill (ნაწილებად დაყოფილი prefill) პრომპტებს ბიუჯეტის ზომის ნაჭრებად ჭრის და ყოველ ნაბიჯზე decode-ის ტოკენებთან ერთად ურევს: ოდნავ ნელი prefill-ის დასრულება, სამაგიეროდ მკვეთრად უფრო გლუვი ITL. თანამედროვე vLLM-ში (V1) ნაგულისხმევად ჩართულია; სახელური ტოკენების ბიუჯეტია. ეს არის კანონიკური „throughput vs tail latency" ინტერვიუს კითხვა ერთ ფუნქციაში. (optimization docs)

5. Speculative decoding: დახარჯე FLOPs, რომ იყიდო latency

Decode პარალელურ აპარატურას ერთ ტოკენზე ფლანგავს ერთ ჯერზე. გამოსავალი: იაფი drafter (პატარა მოდელი, n-gram lookup, ან გაწვრთნილი head-ები: EAGLE/MTP) k ტოკენს სთავაზობს; დიდი მოდელი k-ვეს ერთ პარალელურ გავლაში ამოწმებს; მიღებული პრეფიქსი რჩება, არასწორი პროგნოზები იჭრება — greedy რეჟიმში გამოსავალი დამტკიცებადად იდენტურია. იგებს, როცა მიღების მაჩვენებელი მაღალია (კოდი, შაბლონური ტექსტი) და batch პატარაა (latency-კრიტიკული); აგებს გაჯერებისას (ვერიფიკაციის FLOPs batch-ის throughput-ს ეჯიბრება). (docs)

6. მეთოდი (ეს არის ნამდვილი გაკვეთილი)

1. Fix a workload (input/output lens, concurrency) — bench is meaningless otherwise
2. Baseline: vllm bench serve → save JSON
3. ONE knob per run. Re-bench. Log: knob, TTFT P50/P90, TPOT, tok/s, KV usage
4. Accept/revert against YOUR SLO (not "bigger number")
5. Write the tuning report — the artifact that outlives the cluster

ლაბორატორია (labs/lab-15): ნამდვილი ტიუნინგის კვლევა CPU-ზე

CPU-ის ციფრები GPU-ისგან განსხვავდება; მეთოდი კი იდენტურია — სწორედ მას ვარჯიშობ.

Baseline

ერთი ძრავა, ფიქსირებული დატვირთვა (128 in / 64 out, rate 0.5): გაუშვი bench, შეინახე JSON, ჩაწერე ცხრილის სტრიქონი.

სახელური 1: max-num-seqs-ის გადარბენა

4 → 8 → 16. დააკვირდი, როგორ უარესდება TPOT batch-ის ზრდასთან ერთად, სანამ tok/s იზრდება — ფუნდამენტური კომპრომისი, საკუთარი ხელით გაზომილი.

სახელური 2: prefix caching

--enable-prefix-caching + საზიარო პრეფიქსის დატვირთვა (ერთი და იგივე 400-ტოკენიანი system prompt, სხვადასხვა კითხვები). შეადარე TTFT-ის განაწილებები ჩართულით/გამორთულით. harness-ისთვის გამოიყენე vllm-prefix-cache-bench skill-ი.

სახელური 3: max-model-len როგორც ტევადობის ბერკეტი

4096 → 2048 მაღალ კონკურენტულობასთან ერთად; დააკვირდი, როგორ უმჯობესდება დაშვება (მომლოდინე რიგი უფრო სწრაფად იცლება) კონტექსტის ჭერის ხარჯზე.

ანგარიში

docs/tuning-report.md: დატვირთვის სპეციფიკაცია, გაშვებების ცხრილი, ერთი დიაგრამა (თუნდაც ASCII), 3 დასკვნა და სექცია „რას ვცდიდი H100-ებზე" (FP8, kv fp8, spec decode — ნავარაუდევი ეფექტებით და დასაბუთებით). GPU-ის სექცია ამტკიცებს, რომ თეორია გადმოიტანე.

შეამოწმე თავი

TPOT ნახტომებს აკეთებს ყოველთვის, როცა გრძელპრომპტიანი მოთხოვნა შემოდის. ყველაზე შესაფერისი ფუნქცია?

ნახტომი prefill-ის მიერ ნაბიჯების მონოპოლიზაციაა. Chunking ზღუდავს prefill-ს თითო ნაბიჯზე, ასე რომ decode-ის ტოკენები დინებას აგრძელებენ. Spec decode და KV quantization prefill-ის ხელშეშლას არ ეხება.

70B მოდელი ეტევა, მაგრამ batch პაწაწინა რჩება; რიგი იზრდება. პირველი ბერკეტი?

წონები ეტევა — შეზღუდვა KV მეხსიერებაა თითო მიმდევრობაზე. fp8 KV ≈ აორმაგებს პარალელურ ტევადობას. max-num-seqs-ის აწევა მეხსიერების გარეშე არაფერს ცვლის (დაშვება KV-ით არის შეზღუდული); 2-bit წონები ხარისხს ანადგურებს.

Speculative decoding ყველაზე ნაკლებად ეხმარება, როცა…

გაჯერებისას ის „უქმად მდგარი პარალელური FLOPs", რომელსაც spec decode იყენებს, აღარ არსებობს — ვერიფიკაცია batch-ის throughput-ს ეჯიბრება. შაბლონური გამოსავალი (მაღალი მიღება) და მეჩხერი ტრაფიკი მისი საუკეთესო შემთხვევებია.
