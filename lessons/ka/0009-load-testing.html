დატვირთვის ტესტირება: მიიღე რიცხვები

„მუშაობს" — ეს დემოა. „უძლებს 0.9 req/s-ს P90 TTFT-ით 4 წამზე ნაკლები და სატურაციას აღწევს 6 პარალელურ მოთხოვნაზე" — ეს ინჟინერიაა. დღეს შენს სტეკს ინდუსტრიის საკუთარი ხელსაწყოებით გაზომავ და შედეგებს სწორად წაიკითხავ.

ცოდნა: LLM serving-ის ოთხი რიცხვი

მეტრიკაგანმარტებავისთვის არის მნიშვნელოვანი
TTFTTime To First Token (დრო პირველ ტოკენამდე) — მოთხოვნა გაიგზავნა → პირველი ტოკენი დაბრუნდამომხმარებლებისთვის (აღიქმება როგორც „რეაგირების სისწრაფე")
TPOT / ITLTime Per Output Token (ტოკენთაშორისი დაყოვნება) პირველი ტოკენის შემდეგმომხმარებლებისთვის (აღიქმება როგორც „ბეჭდვის სიჩქარე")
Throughputტოკენი/წმ (ან მოთხოვნა/წმ) მთელი სისტემის მასშტაბით (გამტარუნარიანობა)მისთვის, ვინც აპარატურაში იხდის
GoodputThroughput, რომელიც აკმაყოფილებს latency SLO-ებსსაბოლოოდ — ყველასთვის

ფუნდამენტური კომპრომისი: უფრო აგრესიული batching ზრდის throughput-ს, მაგრამ აუარესებს თითოეული მოთხოვნის latency-ს (დაყოვნებას). დატვირთვის ტესტის ამოცანაა იპოვოს ამ მრუდის შენი გადატეხის წერტილი — სატურაციის წერტილი.

ყოველთვის მოახსენე პროცენტილები და არა საშუალო: P50 ტიპურ სურათს გვიჩვენებს, P99 — on-call ინჟინრის რეალობას. ერთი 60-წამიანი გამონაკლისი საშუალო მნიშვნელობაში სრულიად იკარგება.

ლაბორატორია: ორი ხელსაწყო, ერთი სატურაციის მრუდი

ხელსაწყო 1 — vllm bench serve (ოფიციალური ბენჩმარკი)

ის შენს image-შივე მოდის, ამიტომ გაუშვი კონტეინერად იმავე ქსელურ გზაზე. ტერმინალი 1: kubectl port-forward -n llm svc/vllm-router 8080:80. ტერმინალი 2:

docker run --rm --network host --entrypoint vllm vllm-cpu:local \
  bench serve \
  --backend openai-chat \
  --base-url http://host.docker.internal:8080 \
  --endpoint /v1/chat/completions \
  --model Qwen/Qwen3.5-0.8B \
  --dataset-name random \
  --random-input-len 128 --random-output-len 64 \
  --num-prompts 30 \
  --request-rate 0.5 \
  --save-result --result-dir /tmp

მოსალოდნელია (რამდენიმე წუთის შემდეგ): შემაჯამებელი ცხრილი — Request throughput, Output token throughput, Mean/Median/P99 TTFT, TPOT, ITL

როგორ წავიკითხოთ --request-rate 0.5 = Poisson-ის განაწილებით შემომავალი მოთხოვნები 0.5 req/s სიხშირით (open-loop, რეალური ტრაფიკის მსგავსად). inf ყველაფერს ერთდროულად გაისროდა (closed-loop სტრეს-ტესტი). ამ workspace-ის vllm-bench-serve skill-ში ყველა flag-ია დოკუმენტირებული.

იპოვე სატურაციის წერტილი

გაიმეორე ბენჩმარკი --request-rate მნიშვნელობებით 0.25 → 0.5 → 1 → 2. შედეგები ჩაწერე ცხრილში (მოათავსე NOTES.md-ში ან capstone-ის README-ში):

rate   req/s achieved   P50 TTFT   P90 TTFT   output tok/s
0.25   ____             ____       ____       ____
0.5    ____             ____       ____       ____

მოსალოდნელია: მიღწეული req/s გარკვეულ წერტილში აღარ მიჰყვება შეთავაზებულ სიხშირეს, ხოლო P90 TTFT მკვეთრად ავარდება — ეს გადატეხის წერტილი შენი რეალური სიმძლავრეა (capacity)

Grafana ღია დატოვე: num_requests_waiting-ის ნულიდან აწევა და შემდგომი ზრდა სატურაციაა შიგნიდან დანახული — იგივე გადატეხის წერტილი, ორი ხედი.

ხელსაწყო 2 — k6 (SRE-ს დატვირთვის ხელსაწყო)

ამას იყენებს doronmak-ის რეპოზიტორია (load_tests/load_test.js). სხვა კუთხე: მომხმარებლის სცენარების სკრიპტინგი, SLO ზღვრები (thresholds), pass/fail კარიბჭეები CI-სთვის.

brew install k6

შეინახე როგორც load_tests/chat.js:

import http from 'k6/http';
import { check } from 'k6';

export const options = {
  stages: [
    { duration: '1m', target: 4 },   // ramp to 4 virtual users
    { duration: '2m', target: 4 },   // hold
    { duration: '30s', target: 0 },  // drain
  ],
  thresholds: {
    http_req_failed: ['rate<0.01'],        // <1% errors
    http_req_duration: ['p(90)<30000'],    // P90 end-to-end < 30s (CPU-realistic)
  },
};

export default function () {
  const res = http.post(
    'http://localhost:8080/v1/chat/completions',
    JSON.stringify({
      model: 'Qwen/Qwen3.5-0.8B',
      messages: [{ role: 'user', content: 'Reply with one short sentence.' }],
      max_tokens: 32,
    }),
    { headers: { 'Content-Type': 'application/json' }, timeout: '120s' },
  );
  check(res, { 'status 200': (r) => r.status === 200 });
}
k6 run load_tests/chat.js

მოსალოდნელია: VU/RPS-ის ცოცხალი ჩვენება; საბოლოო ანგარიშში ჩანს გავლილი check-ები, ზღვრები ✓ ან ✗ — ✗ პროცესს არანულოვანი კოდით ასრულებს, სწორედ ასე აკეთებს CI performance-ზე კარიბჭეს

დააკვირდი მთელ სისტემას დატვირთვის ქვეშ

გაუშვი k6 ტესტი კიდევ ერთხელ, გვერდით გახსნილი Grafana-თი. დაადასტურე ამბავი თავიდან ბოლომდე: k6-ის VU-ები იზრდება → router-ის QPS იზრდება → engine-ის აქტიური მოთხოვნები პლატოზე გადის → მომლოდინეთა რიგი იზრდება → TTFT P90 ადის → k6-ის P90 ზღვარი თავის ლიმიტს უახლოვდება. ერთი დატვირთვა, ხუთი ინსტრუმენტი, ერთი თანმიმდევრული ნარატივი.

მოგება შენ გაქვს რეალური რიცხვები: სიმძლავრე (req/s გადატეხის წერტილში), latency (P50/P90 TTFT და TPOT) და ავტომატიზებული SLO კარიბჭე. ცხრილი და დატვირთვის ქვეშ გადაღებული Grafana-ს სქრინშოტი მოათავსე capstone-ის README-ში — გაზომვადი შედეგები არის ის, რაც პორტფოლიო-პროექტს დამაჯერებელს ხდის.

შეამოწმე შენი თავი

რატომ უნდა მოვახსენოთ P90 TTFT და არა საშუალო TTFT?

მომხმარებლები სწორედ „კუდს" გრძნობენ; რამდენიმე ნელი მოთხოვნა საშუალოში ქრება. SLO-ები ზუსტად ამ მიზეზით პროცენტილებზეა განსაზღვრული.

შეთავაზებული სიხშირე 2/s, მიღწეული 0.9/s, მომლოდინეთა რიგი იზრდება. სისტემა…

მიღწეული < შეთავაზებულზე მზარდი რიგით — ეს სატურაციის განმარტებაა. გათბობა მთავრდება; ეს — არა. აქ არაფერი უარყოფს მოთხოვნებს — ისინი უბრალოდ ელოდებიან.

მაქსიმალური batch-ის ზომის გაზრდა, როგორც წესი, ცვლის…

უფრო დიდი batch-ები აპარატურას მეტად დაკავებულს ინახავს (throughput ↑), მაგრამ თითოეული მოთხოვნა გამოთვლით რესურსს მეტ მეზობელს უზიარებს (TPOT ↑). სიზუსტეს არაფერი ეხება.

პირველწყარო

vLLM-ის ბენჩმარკინგის დოკუმენტაცია (bench serve-ის flag-ები, მეტრიკების განმარტებები) და k6-ის დოკუმენტაცია (thresholds, stages). შენი k6 სკრიპტი შეადარე doronmak-ის load_test.js-ს.
