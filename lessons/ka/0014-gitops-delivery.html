GitOps და მოდელების პროგრესული მიწოდება

აქამდე deploy-ინსტრუმენტი შენ იყავი: helm upgrade ხელით. პროდაქშენი ადამიანს პროცესიდან იღებს: git არის სასურველი მდგომარეობა, controller კლასტერს მას უჯერებს (reconciliation), ახალი მოდელის გამოშვება კი ხდება განხილვადი pull request rollback-ღილაკით.

ცოდნა

1. GitOps = reconciliation-ციკლი, ერთი დონით მაღლა

Deployment pod-ებს spec-ს უჯერებს. Argo CD კი მთელ კლასტერს უჯერებს git-რეპოზიტორიას: უყურებს repo-ს → ადარებს ცოცხალ მდგომარეობას (drift detection — გადახრის აღმოჩენა) → sync. შედეგები: ყველა ცვლილება commit-ია (აუდიტის ლოგი უფასოდ), rollback = git revert, კლასტერის credentials ლეპტოპებს ტოვებს, კლასტერები კი მხოლოდ repo + secrets-ისგან აღდგენადია. შენი capstone README-ის quickstart პროტო-GitOps იყო; ეს გაკვეთილი მას აავტომატებს.

2. Application CRD და app-of-apps

spec:
  source:  { repoURL: …, path: charts/slm-stack, targetRevision: main }
  destination: { server: https://kubernetes.default.svc, namespace: llm }
  syncPolicy:
    automated: { prune: true, selfHeal: true }   # drift gets reverted

selfHeal ნიშნავს, რომ შენს ხელით გაკეთებულ kubectl edit-ს წუთებში გადააწერენ — სწორი ქცევაა, თუმცა პირველად თავბრუდამხვევი. პლატფორმები იყენებენ app-of-apps-ს: ერთი root Application შვილებს აჩენს (engines, monitoring, edge) — მთელი პლატფორმა ერთ ხედ.

3. მოდელები ვერსიონირებული არტეფაქტებია — rollout-ებს კოდის deploy-სავით მოექეცი

მოდელის ცვლილება (ახალი წონები, ახალი revision, ახალი quantization) კოდზე უფრო სარისკო deploy-ია: ხარისხის რეგრესიები exception-ებს არ ისვრიან. production-stack-ის values ააშკარავებს modelURL + modelRevision-ს — დააფიქსირე revision (HF commit hash), არასდროს "იტივტივო" repo-ს main-ზე. Rollout-სტრატეგიები, აფეთქების რადიუსის მიხედვით:

სტრატეგიამექანიკახრიკი LLM-ებისთვის
RollingDeployment-ის ნაგულისხმევი, pod-ის მიყოლებითRoll-ის შუაში მოდელის შერეული ვერსიები ერთდროულად ემსახურება მომხმარებლებს
Blue/greenსრული v2 სტეკი, გადართე Service/route2× GPU-ხარჯი გადაფარვისას — ნამდვილი ფული
Canaryშეწონილი მარშრუტები (Gateway API / Argo Rollouts): 5% → 50% → 100%სჭირდება ხარისხის მეტრიკები თითო arm-ზე და არა მხოლოდ error rate
Shadowv2 იღებს დაზერკალებულ ტრაფიკს, პასუხები იყრებასრული დუბლირებული compute; ოქროა eval-ისთვის, უხილავია მომხმარებლებისთვის

სენიორული ინსაითი: მოდელების canary-ანალიზი უნდა მოიცავდეს ხარისხის სიგნალებს (eval-set ქულები, thumbs-down-ის მაჩვენებელი, უარის თქმის მაჩვენებელი) TTFT/error rate-ის გვერდით. ლატენტურობით ჯანმრთელი ნაგავი გულუბრყვილო canary-ს იოლად გაივლის.

4. CI/CD ციკლის გარშემო

CI აწყობს/ასკანერებს image-ს, უშვებს offline eval-ებს კანდიდატ მოდელზე და chart-ის values-ს PR-ში ცვლის. CD უბრალოდ ის არის, რომ Argo merge-ს ამჩნევს. Pipeline-ს კლასტერის credentials არასდროს უჭირავს — უსაფრთხოების ის მოგება, რომელიც GitOps-ს აუდიტორებთან ყიდის.

ლაბორატორია (labs/lab-14): კლასტერი git-ს მიჰყვება

Push და ინსტალაცია

ატვირთე ეს repo GitHub-ზე (ის უკვე git-რეპოზიტორიაა). დააინსტალირე Argo CD, გახსენი მისი UI (port-forward, საწყისი admin secret).

Application → შენი chart

Application, რომელიც charts/slm-stack-ზე მიუთითებს, automated sync + selfHeal. უყურე, როგორ იშვილებს არსებულ release-ს; tree view = მთელი შენი სტეკი გრაფად.

Deploy commit-ით

შეცვალე maxModelLen values.yaml-ში → commit → push → უყურე, როგორ ამჩნევს Argo ცვლილებას, ადარებს, sync-ავს და Deployment-ს გადაატრიალებს. kubectl საერთოდ არ მონაწილეობს. შემდეგ git revert → push → rollback თავისით deploy-დება.

იგრძენი selfHeal

kubectl scale deployment vllm -n llm --replicas=3 ხელით → უყურე, როგორ აბრუნებს Argo შენს drift-ს უკან. განიხილე მასწავლებელთან: როდის არის selfHeal არასწორი? (HPA მას ებრძვის — ამიტომაც replicas არ უნდა იყოს tracked, როცა მას HPA ფლობს; Argo-ს ზუსტად ამისთვის აქვს ignoreDifferences.)

დააპროექტე მოდელის canary (ქაღალდზე)

დაწერე docs/model-rollout.md: როგორ გააკეთებდი canary-ს Qwen3.5-0.8B → ჰიპოთეტური 1.5B ამ სტეკზე — მარშრუტების წონები, მეტრიკები თითო arm-ზე, დაწინაურების კრიტერიუმები, შეწყვეტის კრიტერიუმები. 1 გვერდი. ეს დოკუმენტი ინტერვიუს პასუხია, რომელსაც სიტყვასიტყვით გამოიყენებ.

შეამოწმე შენი თავი

Argo მუდმივად აბრუნებს უკან შენი HPA-ის replica-ცვლილებებს. ძირეული მიზეზი?

ორი controller ერთ ველს ფლობს. გამოსავალი: ამოიღე replicas chart-იდან, როცა HPA განაგებს, ან ignoreDifferences ამ path-ზე. კლასიკური საპროდაქშენო შეჯახება.

რატომ უნდა აკონტროლებდეს მოდელის canary მეტს, ვიდრე ლატენტურობა და შეცდომებია?

ხარისხის რეგრესია 200-ებს აბრუნებს შესანიშნავი TTFT-ით. Canary-ანალიზს ხარისხის სიგნალები სჭირდება — eval-ქულები, უკუკავშირის მაჩვენებელი — თორემ ნაგავს დააწინაურებს.

დააფიქსირე modelRevision commit hash-ზე, რადგან…

მოძრავი revision-ები ნიშნავს, რომ იგივე მანიფესტი მომავალ კვირას სხვა წონებს deploy-ავს — არარეპროდუცირებადი, არააუდიტირებადი. იმავე მიზეზით, რითაც image-ები digest-ებს აფიქსირებენ.
