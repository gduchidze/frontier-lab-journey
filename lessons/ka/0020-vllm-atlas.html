განხილვა 1 — Structured outputs (გამოიყენე ყველგან)

curl -s localhost:8080/v1/chat/completions -H "Content-Type: application/json" -d '{
  "model": "Qwen/Qwen3.5-0.8B",
  "messages": [{"role":"user","content":"Extract: The Mac M4 has 10 cores and 16GB RAM."}],
  "response_format": {
    "type": "json_schema",
    "json_schema": { "name": "hw", "schema": {
      "type": "object",
      "properties": { "chip": {"type":"string"}, "cores": {"type":"integer"}, "ram_gb": {"type":"integer"} },
      "required": ["chip","cores","ram_gb"] } } }
}'

გრამატიკა არავალიდურ ტოკენებს sampling-ის დროსვე ნიღბავს — გამონატანი სქემას ვერ დაარღვევს. თითქმის ნულოვანი overhead გრამატიკის კომპილაციის შემდეგ. Retry-on-parse-error დღეს კვდება. (docs)

განხილვა 2 — Multi-LoRA: fine-tune-ები ფლოტების გარეშე

vllm serve BASE --enable-lora \
  --lora-modules customer-a=/adapters/a customer-b=/adapters/b
# request routes by model name:
{"model": "customer-a", ...}   # base weights + A's adapter, batched WITH B's traffic

ადაპტერები ~10–200 MB-ია GB-მასშტაბის ბაზისების ფონზე; ერთი engine სხვადასხვა ადაპტერის მოთხოვნებს ერთდროულად ჰკრავს batch-ებში; დინამიური load API ტენანტებს runtime-ზე ამატებს. ეკონომიკა: 50 fine-tuned კლიენტი ≈ ერთი ბაზისური ფლოტი + 50 პატარა ფაილი, და არა 50 ფლოტი. ეს ერთი ფუნქცია მთელ პლატფორმებს ამართლებს. განხილვა 3 — Offline batch: engine-ის მეორე ნახევარი

from vllm import LLM, SamplingParams
llm = LLM(model="Qwen/Qwen3.5-0.8B", max_model_len=4096)
out = llm.generate(prompts, SamplingParams(temperature=0, max_tokens=64, seed=7))

არც სერვერი, არც HTTP — scheduler თავისუფალია, წმინდა throughput მაქსიმუმამდე აწიოს: eval-ები და მონაცემთა გენერაცია ასე 2–5-ჯერ იაფი ჯდება, ვიდრე საკუთარი API-ის „ბომბარდირება". შეახვიე მე-19 გაკვეთილის Job-ში და მიიღებ ML-batch-ის კანონიკურ პატერნს.

განხილვა 4 — მოდელის ჩატვირთვის დრო შენი autoscaling-ის კონსტანტაა

რეპლიკის სარგებლიანობა = ჩამოტვირთვა + ჩატვირთვა + warmup. სამივეს შეუტიე: წონები RWX/PVC-ზე ან init-container-ის წინასწარი pre-pull (გაკვეთილი 19), streaming loader-ები (tensorizer / Run:ai streamer: სტრიმინგი პირდაპირ GPU-ში, CPU-ეტაპის გამოტოვება), და --load-format-ის არჩევანი. პლატფორმები „cold-replica-to-first-token"-ს პირველი რიგის SLO-დ ზომავენ — ჩასვი შენს დიზაინის დოკუმენტში.

მულტიმოდალურობა, ბოლოს და ბოლოს

Qwen3.5-0.8B არის image-text-to-text და შენ მას 19 გაკვეთილის მანძილზე მხოლოდ ტექსტით ემსახურებოდი. გაუგზავნე სურათი (base64 data URL image_url content-ნაწილში): „რა არის ამ სურათზე?" თუ შენი build ამ არქიტექტურის მულტიმოდალურ გზას წინ უსწრებს, დააფიქსირე შეცდომა + საჭირო flag/ვერსია — ესეც ნამდვილი პლატფორმის სამუშაოა.

Sampling-საფარი

იგივე prompt: temp 0 + seed (ორჯერ — იდენტურია?), temp 1.8 (ქაოსი), logprobs: 5 (თავდაჯერებულობის ინსპექცია), repetition penalty ციკლისკენ მიდრეკილ prompt-ზე. ერთი ცხრილი ტურის დოკუმენტში.

Offline batch Job-ში

ჩამაუნთე 20-prompt-იანი JSONL ConfigMap-ით; Job უშვებს vllm-cpu:local-ს განხილვა 3-ის python-ით (entrypoint override), რომელიც completion-ებს pod-ის ლოგში წერს. Offline engine + K8s Job = გაკვეთილი 19 × გაკვეთილი 20.

Alias + ხარვეზების რუკა

დაამატე --served-model-name qwen-prod chart-ის values-ით; UI-ის ამომრჩევი განახლდება — მოდელის იდენტობა HF-ის path-ს მოწყდა (rollout-ის წინაპირობა მე-14 გაკვეთილიდან). შემდეგ docs/vllm-gap-map.md, იგივე სამმდგომარეობიანი გულახდილობა, რაც lab 19-ში.

Constrained decoding: გრამატიკა ანულებს იმ ტოკენების logits-ს, რომლებიც სქემას დაარღვევდნენ — არავალიდური გამონატანი წარმოუდგენელია და არა retry-თ გამოსწორებული.

40 კლიენტზე მორგებული fine-tune ერთი 8B ბაზისისა. ყველაზე იაფი serving-ფორმა?

Multi-LoRA ყველა კლიენტს საზიარო ბაზისურ წონებზე ჰკრავს batch-ებში; ადაპტერები მეგაბაიტებია. ორმოცი ფლოტი = ორმოცჯერ მეტი აპარატურა; შერწყმა კლიენტების ინდივიდუალურ ქცევას ანადგურებს.

ღამის eval 100k prompt-ზე: სერვერის API თუ offline LLM კლასი?

Offline-რეჟიმი აგდებს HTTP-ის/streaming-ის/სამართლიანობის შეზღუდვებს და scheduler-ს წმინდა throughput-ისთვის დაწყობის საშუალებას აძლევს — ამ მასშტაბზე შესამჩნევად იაფია. Prod-პარიტეტის smoke-ტესტები API-ზე დარჩეს.
