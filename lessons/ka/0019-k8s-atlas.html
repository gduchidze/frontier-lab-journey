განხილვა 1 — Graceful shutdown LLM-engine-ებისთვის

შეწყვეტის თანმიმდევრობა: pod მოინიშნება Terminating-ად → ამოღება Service-ის endpoint-ებიდან → ეშვება preStop hook → SIGTERM → grace period → SIGKILL. ნაგულისხმევი grace: 30 წმ. 70B engine-ს გრძელი გამონატანის შუაში წუთები სჭირდება. Production-პატერნი:

terminationGracePeriodSeconds: 300
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh","-c","sleep 15"]  # let endpoint removal propagate
                                            # before the engine even sees SIGTERM

პლუს PDB, რომ drain-ებმა ფლოტი N−1-ზე დაბლა არასდროს ჩამოიყვანონ. ამის გარეშე ყოველი node-ის განახლება = მომხმარებლისთვის ხილული ჩავარდნები. ეს 10-ხაზიანი diff-ია, რომელიც შენს დიზაინის დოკუმენტში ცალკე აბზაცად ღირს.

განხილვა 2 — Job-ები AI-გამოთვლების ნახევარია

Eval-ები, batch scoring, embedding-ების backfill-ები, distillation-მონაცემების გენერაცია — არცერთ მათგანს Deployment არ უნდა. Job-ები გაძლევენ დასრულების თვალყურის დევნებას, retry-ს backoffLimit-ით, parallelism-ის fan-out-ს და (Indexed completion mode-ით) ინდექსს თითო pod-ზე — ღარიბის rank-ს. დაამატე Kueue, როცა რამდენიმე გუნდის Job-ები ერთსა და იმავე GPU-ებზე კონკურირებენ. შენი ლაბორატორია სტეკზე ნამდვილ batch-inference Job-ს უშვებს.

განხილვა 3 — Operator-ები: პატერნი ყველაფრის უკან, რაც დააყენე

CRD = ახალი არსებითი სახელი API-ში (ServiceMonitor, Certificate, Application). Operator = controller, რომელიც ამ არსებით სახელს ჭეშმარიტად აქცევს — ციკლში, სამუდამოდ. მთელი ხრიკი ეს არის — იგივე reconciliation (შეთანხმების ციკლი), რომელსაც პირველ გაკვეთილში აკვირდებოდი, მიმართული დომენის ლოგიკაზე. production-stack-ის operator/ დირექტორია engine-ფლოტებს სწორედ ასე მართავს. ლაბორატორიაში სათამაშო CRD-ს დაწერ, რომ მისტიკა გაქარწყლდეს; kubebuilder კი ნამდვილი ინსტრუმენტია, როცა გუნდს ის დასჭირდება.

განხილვა 4 — Init containers წონების წინასწარი ჩატვირთვისთვის

Scaling-ის დაყოვნება (მე-10 გაკვეთილის ტკივილი) ძირითადად წონების ჩატვირთვაა. Init-container-ის პატერნი: downloader-image PVC/hostPath cache-ს ავსებს, ᲡᲐᲜᲐᲛ engine-კონტეინერი დაიწყება; RWX storage-თან ან node-ლოკალურ cache-ებთან კომბინაციაში ახალი რეპლიკები ჩამოტვირთვას საერთოდ გამოტოვებენ. Streaming loader-ები (Run:ai model streamer, tensorizer — გაკვეთილი 20) იმავე წამებს ესხმიან თავს.

განხილვა 5 — Helm vs Kustomize (ინტერვიუს მარადიული კითხვა)

Helm: packaging + templating + lifecycle (დაინსტალირებადი პროდუქტი, values API — ის, რაც შენ ააგე). Kustomize: patch-ებზე დაფუძნებული env-overlay-ები ნულოვანი შაბლონებით (ფლოტის კონფიგები, შესანარჩუნებელი chart-ის გარეშე). რეალური პლატფორმები: Helm მესამე მხარის software-ისთვის, Kustomize-ის overlay-ები (ზოგჯერ დარენდერებული Helm-ის ᲖᲔᲛᲝᲓᲐᲜ) env-სპეციფიკური ცვლილებებისთვის. ზუსტად ამ წინადადების თქმა უმეტეს ოთახში დებატს ხურავს.

Job (parallelism 2, completions 6), რომლის pod-ები router-ს ConfigMap-იდან აღებული prompt-ებით curl-ავენ და გამონატანს საზიარო emptyDir→log-ში წერენ. უყურე, როგორ ითვლის kubectl get jobs -w completion-ებს. ამოხსნა: solutions/k8s-advanced/batch-job.yaml.

Graceful drain-ის წვრთნა

დაამატე preStop + 300 წმ grace + PDB (minAvailable: 1) engine-ის chart-ს. შემდეგ kubectl drain llm-stack-worker --ignore-daemonsets --delete-emptydir-data დატვირთვის შუაგულში: PDB eviction-ის ტემპს არეგულირებს, engine ზრდილობიანად იცლება. ბოლოს kubectl uncordon.

kubectl debug

kubectl debug -it <engine-pod> -n llm --image=busybox --target=vllm — ჩაიხედე /proc-ში, ნახე engine-ის სამყარო sidecar-იდან, რომელიც წამის წინ არ არსებობდა.

სათამაშო CRD

Apply-ით დააყენე ModelEndpoint CRD + ერთი CR (ამოხსნა მოცემულია). kubectl get modelendpoints — შენი საკუთარი არსებითი სახელი API-ში. Controller ჯერ არ არის; მასწავლებელთან განიხილე, რას გააკეთებდა reconcile-ციკლი (ეს საუბარი სწორედ operator-ის ინტერვიუა).

ხარვეზების რუკა

docs/k8s-gap-map.md: ატლასის ყოველ სტრიქონზე — ვიცი / შემიძლია გამოვიყენო / სასწავლი მაქვს + ერთი წინადადება იმაზე, როდის დასჭირდებოდა ეს შენს სტეკს. ეს გულახდილი დოკუმენტი შენს გამეორების გეგმას მართავს.

შეამოწმე შენი თავი

Node-ის განახლებამ გენერაციები შუა ტოკენზე მოკლა, PDB-ის მიუხედავად. რა აკლია?

PDB არეგულირებს, ᲠᲐᲛᲓᲔᲜᲘ pod გამოირიცხება — და არა რამდენად ნაზად კვდება თითოეული. მიმდინარე სამუშაოს terminationGracePeriodSeconds + preStop-drain სჭირდება.

ღამის eval 10k prompt-ზე 4 worker-ით. სწორი პრიმიტივი?

დასრულებამდე გაშვება + parallelism + pod-ის ინდექსი sharding-ისთვის = Job; CronJob მას განრიგში სვამს. Deployment-ები არასდროს მთავრდებიან; StatefulSet-ები იდენტობას ხსნიან და არა დასრულებას.

რა აქცევს CRD-ს operator-ად?

CRD ცალკე ინერტული საცავია (შენი ლაბორატორია ამას ამტკიცებს). Operator ᲡᲬᲝᲠᲔᲓ reconcile-ციკლია, რომელიც ამ ობიექტებს უყურებს და რეალობას მათ არგებს.
