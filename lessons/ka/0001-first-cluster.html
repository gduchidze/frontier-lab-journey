SLM Production Stack · გაკვეთილი 1 · ფაზა 1 · ~90 წთ

რუკა და შენი პირველი კლასტერი

სანამ ენობრივ მოდელს მოემსახურები, გჭირდება ადგილი, სადაც მას გაუშვებ. დღეს გაიგებ, რა არის სინამდვილეში Kubernetes, დაინახავ მთლიან სტეკს, რომელსაც ააწყობ, და გაუშვებ შენს პირველ workload-ს ნამდვილ cluster-ში (კლასტერში) საკუთარ მანქანაზე.

საით მიდიხარ

ეს არის სისტემა, რომელიც კურსის ბოლოსთვის აწყობილი გექნება — იგივე არქიტექტურა, რაც vLLM production-stack-ს აქვს, ოფიციალური რეფერენსი, რომელსაც რეალური inference პლატფორმები იყენებენ:

ყველაფერი, რაც ამ ყუთებშია, Kubernetes-ის შიგნით მუშაობს. სწორედ ამიტომ იწყება კურსი K8s-ით: ის მთელი სტეკის ოპერაციული სისტემაა.

ცოდნა: Kubernetes ოთხ იდეაში

1. კლასტერი არის მანქანები, რომლებიც ერთ კომპიუტერად ასაღებენ თავს

cluster (კლასტერი) = control plane (ტვინი: API server, scheduler) + node-ები (კუნთები: მანქანები, რომლებზეც შენი კონტეინერები მუშაობს). API server-ს ეუბნები რა გინდა; scheduler წყვეტს, რომელი node გაუშვებს. კომპანიაში node-ები GPU სერვერებია. შენს Mac-ზე node-ები Docker კონტეინერებია, რომლებსაც kind ქმნის — "Kubernetes in Docker". იგივე API, იგივე უნარები.

2. Pod არის ატომი

Pod (პოდი — უმცირესი გასაშვები ერთეული) არის ერთი ან რამდენიმე კონტეინერი, რომლებიც საერთო ქსელურ იდენტობას იზიარებენ. Pod-ებს პირდაპირ თითქმის არასდროს ქმნი — ისინი მოკვდავები არიან. როცა Pod კვდება, ის სამუდამოდ ქრება. (დოკუმენტაცია)

3. Deployment-ები სასურველ მდგომარეობას ინარჩუნებენ

Deployment (დეპლოიმენტი) ამბობს: "მინდა ამ Pod-ის 3 რეპლიკა, ყოველთვის." Kubernetes უშვებს reconciliation loop-ს (შეჯერების ციკლს): მოკალი Pod, და ახალი გაჩნდება. ეს არის დეკლარაციული ინფრასტრუქტურა — შენ აცხადებ მიზანს, K8s ასრულებს სამუშაოს. სწორედ ამ ერთმა იდეამ მოუტანა K8s-ს გამარჯვება. (დოკუმენტაცია)

4. Service-ები მოკვდავ Pod-ებს უკვდავ მისამართს აძლევენ

Pod-ები შემთხვევით IP-ებს იღებენ და კვდებიან. Service (სერვისი) არის სტაბილური სახელი + IP, რომელიც დატვირთვას ანაწილებს ყველა იმ Pod-ზე, რომელიც ამჟამად მის label selector-ს ემთხვევა. როცა ფაზა 6-ში შენი router ტრაფიკს "vLLM engine-ებთან" გაგზავნის, ის Service-ს დაელაპარაკება და არასდროს — Pod-ებს. (დოკუმენტაცია)

კარიერული შენიშვნა ML infra პოზიციების ინტერვიუებზე რეგულარულად კითხულობენ: "რა ხდება, როცა kubectl apply-ს უშვებ?" და "Deployment vs Pod vs Service?" დღევანდელი ლაბორატორიის შემდეგ სამივეს შეხებული იქნები.

ლაბორატორია: კლასტერი აწეულია, აპლიკაცია — გაშვებული

მიზანი: ორ-node-იანი cluster შენს Mac-ზე, nginx ვებ-სერვერი, რომელიც Deployment-ად მუშაობს და Service-ის გავლით არის მისაწვდომი. ~45–60 წთ, ინსტალაციების ჩათვლით.

გაუშვი Docker, დააინსტალირე ხელსაწყოები

გახსენი Docker Desktop (kind Docker-ის შიგნით ცხოვრობს), შემდეგ:

brew install kind helm
kind version && helm version

მოსალოდნელია: kind v0.2x.x, helm v3.x

შექმენი კლასტერი

შეინახე როგორც k8s/kind-config.yaml ამ workspace-ში (იგივე პატერნი, რაც doronmak-ის რეპოშია — control-plane node პლუს worker, პორტი 30080 შენს Mac-ზეა გადამისამართებული მოგვიანებით გამოსაყენებლად):

kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
name: llm-stack
nodes:
  - role: control-plane
    extraPortMappings:
      - containerPort: 30080
        hostPort: 30080
  - role: worker
kind create cluster --config k8s/kind-config.yaml
kubectl get nodes

მოსალოდნელია: 2 node, STATUS Ready (control-plane-ს შესაძლოა ~30წმ დასჭირდეს)

დააკვირდი, რა ააგე

docker ps                          # your "machines" are containers
kubectl get pods -A                # K8s runs itself as pods too
kubectl cluster-info

შენიშნე kube-apiserver, etcd, kube-scheduler namespace-ში kube-system — ეს არის control plane #1 იდეიდან, თავადაც Pod-ებად გაშვებული.

გაუშვი Deployment

kubectl create deployment hello --image=nginx:alpine --replicas=2
kubectl get pods -o wide

მოსალოდნელია: 2 pod hello-xxxx Running, node-ებზე გადანაწილებული

ახლა დაამტკიცე reconciliation loop (#3 იდეა). წაშალე ერთი Pod და უყურე, როგორ აღადგენს მას K8s:

kubectl delete pod $(kubectl get pods -l app=hello -o name | head -1)
kubectl get pods -w        # Ctrl+C to stop watching

მოსალოდნელია: ᲐᲮᲐᲚᲘ pod (განსხვავებული სუფიქსით) წამებში გამოჩნდება

გახსენი ის Service-ით

kubectl expose deployment hello --port=80
kubectl get svc hello
kubectl port-forward svc/hello 8080:80

მეორე ტერმინალში:

curl -s localhost:8080 | head -4

მოსალოდნელია: <!DOCTYPE html> ... Welcome to nginx!

მოგება შენ ახლა მინიატურაში გააკეთე ზუსტად ის, რასაც ფაზა 4 აკეთებს vLLM-ით: Deployment უშვებს workload-ს, Service აძლევს მას სტაბილურ მისამართს, port-forward კი საშუალებას გაძლევს დაელაპარაკო. ჩაანაცვლე nginx → vllm და curl / → curl /v1/chat/completions — იგივე ფორმაა.

მოწესრიგება (კლასტერი დაიტოვე!)

kubectl delete deployment hello
kubectl delete svc hello

კლასტერი რჩება — გაკვეთილი 2 მასზე აშენდება. (თუ ოდესმე მთლიანად წაშლა დაგჭირდა: kind delete cluster --name llm-stack.)

შეამოწმე საკუთარი თავი

უპასუხე მეხსიერებიდან — გახსენება არის მთავარი. დააკლიკე ვარიანტს.

შენს Deployment-ში Pod ღამის 3 საათზე კრაშდება. რა ხდება?

Deployment-ის reconciliation loop ამჩნევს, რომ ფაქტობრივი მდგომარეობა (1 რეპლიკა) ≠ სასურველი მდგომარეობა (2 რეპლიკა) და ქმნის ახალ Pod-ს. Service-ები Pod-ებს არასდროს მართავენ — ისინი მხოლოდ ტრაფიკს მიმართავენ მათკენ.

რატომ ელაპარაკება router Service-ს და არა Pod-ების IP-ებს?

Pod-ები მოკვდავები არიან და ყოველ ჯერზე ახალ IP-ებს იღებენ. Service არის სტაბილური სახელი, რომელიც დატვირთვას ანაწილებს ყველა იმ Pod-ზე, რომელიც ამჟამად მის selector-ს ემთხვევა.

kind-ში რა არის "node" ფიზიკურად?

kind = Kubernetes IN Docker. თითოეული node კონტეინერია. სწორედ ამიტომ აჩვენა docker ps-მა შენი "მანქანები". რეალური კლასტერები VM-ებს ან bare metal-ს იყენებენ — API ორივე შემთხვევაში იგივეა.

პირველადი წყარო

წაიკითხე ეს დღეს საღამოს (15 წთ): Kubernetes docs — Overview & Concepts. ეს კანონიკური წყაროა და უჩვეულოდ კარგად არის დაწერილი. გადაათვალიერე kind quickstart იმისთვის, რაც ახლახან გააკეთე.
