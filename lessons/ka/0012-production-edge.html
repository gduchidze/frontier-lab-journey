საპროდაქშენო Edge: Ingress, Gateway, TLS

NodePort-მა დემომდე მიგიყვანა. საპროდაქშენო ტრაფიკი ერთ გამაგრებულ შესასვლელ წერტილში შემოდის — hostname-ებით, TLS-ით, timeout-ებითა და ავთენტიფიკაციით — LLM streaming (ნაკადური გადაცემა) კი ამსხვრევს ყველა ოდესმე გამოშვებული proxy-ის ნაგულისხმევ პარამეტრებს. დღეს ნამდვილ edge-ს (ქსელის კიდეს) ააგებ.

ცოდნა

1. Edge-სტეკი ქვემოდან ზემოთ

Service (L4) → Ingress (L7 წესები: host/path → Service), რომელსაც ინტერპრეტირებს ingress controller (nginx, Traefik, Envoy) → ღრუბელში წინ დგას LoadBalancer. ერთი controller, ერთი IP, N სერვისი hostname-ების უკან: api.company.com → router, chat.company.com → UI. (დოკუმენტაცია)

2. Gateway API მემკვიდრეა — ისწავლე ორივე

Gateway API Ingress-ის მონოლითს ჰყოფს როლებზე ორიენტირებულ რესურსებად: GatewayClass (ინფრასტრუქტურის პროვაიდერი) → Gateway (ოპერაციების გუნდი: listener-ები, TLS) → HTTPRoute (აპლიკაციის გუნდები: მარშრუტიზაცია, ტრაფიკის წონები). შეწონილი backendRefs გაძლევს canary-ს (ეტაპობრივი გამოშვება) პირდაპირ edge-ზე — დაიმახსოვრე ეს მე-14 გაკვეთილისთვის. ახალი პლატფორმები Gateway-API-first არიან; არსებული ინფრასტრუქტურა ჯერ კიდევ Ingress-ზე მუშაობს. ინტერვიუსთვის მზადყოფნა = ორივეს ცოდნა.

3. TLS: cert-manager სერტიფიკატებს controller-ის ამოცანად აქცევს

cert-manager აკვირდება Certificate/Issuer CRD-ებს და Secret-ებს ვალიდური სერტიფიკატებით ავსებულს ინახავს (პროდაქშენში Let's Encrypt ACME-ს გავლით, ლოკალურად — self-signed CA). Ingress მიუთითებს Secret-ზე; განახლება არავის pager-ს აღარ არეკინებს. იგივე reconciliation (შეჯერების) პატერნია, რაც K8s-ში ყველაფერ დანარჩენს აქვს — ეს არის ღრმა გაკვეთილი.

4. LLM-ტრაფიკი ამსხვრევს proxy-ის ნაგულისხმევ პარამეტრებს

ეს ცხრილი ამ გაკვეთილის ღირებულების ნახევარია — ეს არის ის, რასაც "AI infra გამოცდილება" ნიშნავს edge-ზე:

ნაგულისხმევირას ამტვრევსგამოსწორება (nginx-ingress annotations)
60 წმ read timeoutგრძელი გენერაციები ნაკადის შუაში კვდებაproxy-read-timeout: "600"
Response buffering ჩართულიSSE ტოკენები ბოლოში ერთ გროვად ჩამოდის — streaming ვიზუალურად მკვდარიაproxy-buffering: "off"
1 MB body limitგრძელკონტექსტიანი / მულტიმოდალური payload-ები 413-ით უარიყოფაproxy-body-size: "10m"
Round-robin LBეწინააღმდეგება შენს session-affinity router-სტრაფიკი router-ის Service-ისკენ წარმართე და engine-ების დაბალანსება მას მიანდე
Retries timeout-ზეძვირადღირებული გენერაციების დუბლირებაIdempotency keys / retry-ის გამორთვა POST-ისთვის

5. Edge არის იქ, სადაც ავთენტიფიკაცია ცხოვრობს

Engine-ები "ბრიყვები" რჩებიან; edge (ან AI gateway, მაგალითად LiteLLM router-ის წინ) ახორციელებს API keys-ს, per-tenant rate limit-ებსა და მოხმარების აღრიცხვას. vLLM-ის --api-key ბოლო ხაზია და არა პირველი — მე-13 გაკვეთილი მას შეაერთებს.

ლაბორატორია (labs/lab-12): hostname-ები და TLS შენს სტეკზე

დააინსტალირე ingress-nginx (kind ვარიანტი)

გაძლიერებული კლასტერი 80/443 პორტებს ამისამართებს. გამოიყენე kind-სპეციფიკური მანიფესტი, დაელოდე controller-ს.

მარშრუტიზაცია hostname-ის მიხედვით

Ingress LLM-ზე მორგებული annotation-ებით: chat.llm.local → open-webui, api.llm.local → vllm-router. ორივე host დაამატე /etc/hosts-ში → 127.0.0.1. ისაუბრე ბრაუზერში http://chat.llm.local მისამართზე — პორტის ნომრების გარეშე, როგორც ნამდვილ პროდუქტში.

cert-manager + self-signed CA

დააინსტალირე cert-manager, შექმენი CA ClusterIssuer, დაამატე annotation Ingress-ს — TLS Secret-ები გაჩნდება. curl -k https://api.llm.local/v1/models, შემდეგ სერტიფიკატების ჯაჭვი შეამოწმე openssl s_client-ით.

დაამტკიცე streaming-ის გამოსწორება

გაატარე გრძელი გენერაცია edge-ის გავლით buffering ჩართული vs გამორთული (გადართე annotation) — უყურე, როგორ გროვდება ტოკენები ერთად ან მიედინება ნაკადად. ეს 2-წუთიანი დემო არის მთელი "LLM edge" საბრძოლო ისტორია, უსაფრთხოდ საკუთარ თავზე მოწყობილი.

შეამოწმე შენი თავი

მომხმარებლები ამბობენ: "პასუხი ბოლოში ერთბაშად ჩნდება". Edge-ის ეჭვმიტანილი #1?

Buffering-ის მქონე proxy-ები პასუხს დასრულებამდე იკავებენ — streaming ბოლომდე მუშაობს, მაგრამ გატეხილს ჰგავს. Engine-ის დაყოვნება TTFT-ს გაზრდიდა და არა მთელ პასუხს ერთ გროვად აქცევდა.

Gateway API Ingress-თან შედარებით — სტრუქტურული უპირატესობა?

Gateway/HTTPRoute ოპერაციების გუნდს აპლიკაციის გუნდებისგან მიჯნავს და წონებსა და match-ებს API-ის სრულფასოვან ნაწილად აქცევს — annotation-ების "წვნიანის" გარეშე. Data plane იგივე proxy-ებია; სერტიფიკატებს კვლავ issuer სჭირდება.

სად უნდა ცხოვრობდეს per-tenant rate limiting?

იაფად უარყავი edge-ზე, სანამ მოთხოვნა ძვირადღირებულ prefill გამოთვლას მოიხმარს. Engine-ები tenant-ებს ვერ ხედავენ; კლიენტებს ვერ ენდობი.
