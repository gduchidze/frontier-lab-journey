SLM Production Stack · გაკვეთილი 2 · ფაზა 2 · ~90 წთ

Production-ის ფორმის მანიფესტები

გაკვეთილი 1-ში გენერატორ ბრძანებებს იყენებდი. რეალური გუნდები YAML-ს git-ში ინახავენ. დღეს მანიფესტებს (manifest — სასურველი მდგომარეობის დეკლარაცია) ხელით დაწერ — namespace-ებით, resource limit-ებით და health probe-ებით — ზუსტად ის სამი ბერკეტი, რომელიც vLLM deployment-ს ან ააწყობს, ან ჩაშლის.

წინაპირობა გაკვეთილი 1-ის cluster-ი llm-stack გაშვებულია (kubectl get nodes → 2 Ready). თუ არა: kind create cluster --config k8s/kind-config.yaml.

ცოდნა: ხუთი იდეა

1. ყველაფერი მანიფესტია

მანიფესტი არის YAML, რომელიც სასურველ მდგომარეობას აცხადებს: apiVersion + kind + metadata + spec. kubectl apply -f მას API server-ს უგზავნის; controller-ები კი მას რეალობად აქცევენ. git-ში დაკომიტებული მანიფესტები = შენს ინფრასტრუქტურას აქვს code review, ისტორია, rollback. სწორედ ამიტომ არის "GitOps" სამსახურის მოთხოვნა. სრული ანოტირებული ჩონჩხები: manifest anatomy reference.

2. Namespace-ები კლასტერს ანაწილებენ

Namespace (სახელთა სივრცე) არის კლასტერის დასახელებული ნაჭერი. ჩვენი სტეკი ცხოვრობს llm-ში; მონიტორინგი იცხოვრებს monitoring-ში. DNS კლასტერის შიგნით: <service>.<namespace>.svc.cluster.local — დაიმახსოვრე ეს, router-ისა და UI-ის გაკვეთილები მასზეა დამოკიდებული.

3. Request-ები და limit-ები

Request = ის, რასაც scheduler შენთვის იჯავშნის ("დასათავსებლად 4 CPU მჭირდება"). Limit = ჭერი, რომლის გადაკვეთისთვისაც მოგკლავენ (მეხსიერება → OOMKilled). (დოკუმენტაცია) 16 GB Mac-ზე ეს აკადემიური საკითხი არ არის: vLLM CPU-ს ~6 Gi მოუნდება. request-ები ზედმეტად მაღლა დააყენე → Pod სამუდამოდ Pending-ში გაიჭედება. ეს #1 GPU-cluster უნარიცაა — GPU-ები იმავე გზით მოითხოვება (nvidia.com/gpu: 1).

4. სამი probe

ProbeკითხვაჩავარდნისასvLLM-ის კონტექსტში
startupჩატვირთვა დაასრულე?ლოდინი გრძელდება (ზღვრამდე)მოდელი ჯერ კიდევ იტვირთება/ჩამოიტვირთება — შეიძლება წუთები დასჭირდეს
readinessშეგიძლია ახლა ტრაფიკის მიღება?ამოღება Service-ის endpoint-ებიდანEngine დაკავებულია/გადატვირთულია — აქ routing შეწყდეს
livenessსაერთოდ ცოცხალი ხარ?კონტეინერი გადაიტვირთებაEngine გაიჭედა — მოკვლა და ხელახლა ჩატვირთვა

startup probe-ის გარეშე ნელა ჩამტვირთავ მოდელს liveness ციკლურად კლავს: კლასიკური CrashLoopBackOff deploy-ისას. (დოკუმენტაცია)

5. ConfigMap-ები და Secret-ები

კონფიგურაცია image-ის გარეთ ცხოვრობს: ConfigMap ჩვეულებრივი პარამეტრებისთვის, Secret ავტორიზაციის მონაცემებისთვის (შენი Hugging Face token ფაზა 4-ში). ინექცია ხდება env ცვლადებად ან ფაილებად. token-ები არასდროს ჩააშენო image-ებში ან git-ში დაკომიტებულ YAML-ში.

ლაბორატორია: hello-ს ხელახლა აწყობა, production-ის ფორმით

იგივე nginx, რაც გაკვეთილ 1-ში — მაგრამ ახლა namespace-ში, რესურსებით შემოსაზღვრული, probe-ებით და ConfigMap-ით კონფიგურირებული. მერე კი განზრახ გავაფუჭებთ.

შექმენი აპლიკაციის მანიფესტები

შეინახე როგორც k8s/lab2.yaml (რამდენიმე დოკუმენტი, ----ით გამოყოფილი):

apiVersion: v1
kind: Namespace
metadata:
  name: llm
---
apiVersion: v1
kind: ConfigMap
metadata:
  name: hello-config
  namespace: llm
data:
  index.html: |
    <h1>Served by Kubernetes, configured by ConfigMap</h1>
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello
  namespace: llm
spec:
  replicas: 2
  selector:
    matchLabels: { app: hello }
  template:
    metadata:
      labels: { app: hello }
    spec:
      containers:
      - name: web
        image: nginx:alpine
        ports:
        - containerPort: 80
        resources:
          requests: { cpu: 100m, memory: 64Mi }
          limits:   { cpu: 500m, memory: 128Mi }
        startupProbe:
          httpGet: { path: /, port: 80 }
          periodSeconds: 2
          failureThreshold: 15
        readinessProbe:
          httpGet: { path: /, port: 80 }
          periodSeconds: 5
        livenessProbe:
          httpGet: { path: /, port: 80 }
          periodSeconds: 10
        volumeMounts:
        - name: content
          mountPath: /usr/share/nginx/html
      volumes:
      - name: content
        configMap: { name: hello-config }
---
apiVersion: v1
kind: Service
metadata:
  name: hello
  namespace: llm
spec:
  selector: { app: hello }
  ports:
  - port: 80
    targetPort: 80

Apply და გადამოწმება

kubectl apply -f k8s/lab2.yaml
kubectl get pods -n llm -w      # wait for 2/2 READY 1/1, Ctrl+C
kubectl port-forward -n llm svc/hello 8080:80

მეორე ტერმინალი:

curl -s localhost:8080

მოსალოდნელია: ConfigMap-ის HTML და არა nginx-ის სტანდარტული გვერდი

გააფუჭე readiness probe — უყურე, როგორ იწრიტება ტრაფიკი

kubectl exec -n llm deploy/hello -- rm /usr/share/nginx/html/index.html 2>/dev/null; \
kubectl get pods -n llm -w

მოიცადე — ამან ᲔᲠᲗᲘ pod-ის დამაუნთებული volume-იდან წაშალა? არა: ConfigMap-ის mount-ები read-only-ა. სცადე — rm ჩავარდება. კარგია. გააფუჭე სხვაგვარად — მიმართე probe იმ path-ზე, რომელიც არ არსებობს:

kubectl patch deployment hello -n llm --type=json -p='[
 {"op":"replace","path":"/spec/template/spec/containers/0/readinessProbe/httpGet/path","value":"/nope"}]'
kubectl get pods -n llm -w

მოსალოდნელია: ახალი pod-ები ჩნდება, მაგრამ READY 0/1-ზე რჩება — rollout ჩერდება, ᲫᲕᲔᲚᲘ pod-ები კი აგრძელებენ მომსახურებას

ეს არის დამცავი ბადე მოქმედებაში: Kubernetes უარს ამბობს მომუშავე Pod-ების გაფუჭებულებით ჩანაცვლებაზე. შეამოწმე, რატომ არ არის ready:

kubectl describe pod -n llm -l app=hello | grep -A3 "Readiness"
kubectl rollout undo deployment/hello -n llm
kubectl get pods -n llm

მოსალოდნელია: undo-ს შემდეგ ისევ 2 pod READY 1/1

გააფუჭე მეხსიერება — გაიცანი OOMKilled

გაუშვი Pod, რომელიც თავის limit-ზე მეტს გამოყოფს:

kubectl apply -f - <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: oom-demo
  namespace: llm
spec:
  restartPolicy: Never
  containers:
  - name: stress
    image: polinux/stress
    command: ["stress", "--vm", "1", "--vm-bytes", "128M", "--vm-hang", "0"]
    resources:
      limits:
        memory: 64Mi
EOF

sleep 10 && kubectl get pod oom-demo -n llm
kubectl describe pod oom-demo -n llm | grep -A2 "Last State"

შენიშვნა: ძველი გაკვეთილები იყენებენ kubectl run --limits=… ფლაგს — ის kubectl 1.24-ში წაიშალა. მანიფესტები სწორი გზაა (რაც ამ გაკვეთილის მთავარი აზრია).

მოსალოდნელია: STATUS OOMKilled — ზუსტად ის ჩავარდნა, რომელსაც დაადიაგნოსტირებ, როცა vLLM მეხსიერების limit-ს გადააჭარბებს

kubectl delete pod oom-demo -n llm
მოგება შენ ახლა პრაქტიკაში ნახე ის ორი ჩავარდნის რეჟიმი, რომლებიც რეალურ vLLM ოპერაციებში დომინირებს: readiness-ის ჩავარდნა → ტრაფიკი იწრიტება და OOMKilled → limit-ები ზედმეტად მჭიდროა. ფაზა 4-ში ამ ბერკეტებს ნამდვილი engine-ისთვის დააყენებ.

წაშალე აპლიკაცია, namespace დაიტოვე

kubectl delete deployment hello -n llm
kubectl delete svc hello -n llm
kubectl delete configmap hello-config -n llm

შეამოწმე საკუთარი თავი

vLLM-ს მოდელის ჩატვირთვას 4 წუთი სჭირდება. რომელი probe აღკვეთს restart-ციკლებს ჩატვირთვისას?

startup probe თიშავს liveness შემოწმებას, სანამ თავად არ გაივლის. liveness-ის გაწელვა მუშაობს, მაგრამ სამუდამოდ ასუსტებს გაჭედვის აღმოჩენას; startup probe-ები მხოლოდ ჩატვირთვას ფარავენ.

Pod ითხოვს 8 CPU-ს node-ზე, რომელსაც 6 თავისუფალი აქვს. შედეგი?

Request-ები scheduling-ის კონტრაქტია. ვერცერთი node ვერ აკმაყოფილებს → Pending. kubectl describe pod → Events აჩვენებს "Insufficient cpu"-ს.

Readiness probe ორი რეპლიკიდან ერთზე ვარდება. რა მოუვა მას?

Readiness მხოლოდ ტრაფიკს აკონტროლებს. გადატვირთვები liveness-ის საქმეა; ჩანაცვლება — Deployment-ის საქმე, როცა Pod-ები კვდებიან.

პირველადი წყარო

Configure Liveness, Readiness and Startup Probes (K8s docs, 20 წთ). ზემოთ მოცემული probe-ების ცხრილი მას აჯამებს; დოკუმენტში კი ის edge case-ებია, რომლებიც ინტერვიუერებს უყვართ.
