Helm: შეფუთე სტეკი

შენი vLLM YAML მუშაობს, მაგრამ ეს copy-paste ინფრასტრუქტურაა. Helm მანიფესტებს აქცევს პარამეტრიზებულ, ვერსიონირებულ, დაინსტალირებად პაკეტად — ზუსტად ისე, როგორც ორივე საორიენტაციო რეპოზიტორია ვრცელდება, და ისე, როგორც შემდეგ გაკვეთილზე Prometheus-სა და Grafana-ს დააინსტალირებ.

ცოდნა: Helm ოთხ იდეაში

1. Chart = templates + values

chart (ჩარტი — Helm-ის პაკეტი) არის დირექტორია: Chart.yaml (სახელი, ვერსია), values.yaml (სამართავი პარამეტრები), templates/ (მანიფესტები {{ .Values.* }} ჩასასმელი ადგილებით). helm install ახდენს template-ების რენდერს values-ით და შედეგს იყენებს კლასტერზე. (დოკუმენტაცია)

2. Values შენი ინფრასტრუქტურის საჯარო API-ა

ნახე, როგორ ააშკარავებენ პროფესიონალები სამართავ პარამეტრებს — doronmak-ის values.yaml:

vllm:
  replicaCount: 1
  image: { repository: doronmak/vllm-cpu-image, tag: latest }
  model: { name: facebook/opt-125m, dtype: auto }
  resources: { limits: { memory: 10Gi, cpu: "4" } }
  env:
    VLLM_TARGET_DEVICE: "cpu"
    VLLM_CPU_KVCACHE_SPACE: "5"

და production-stack-ის modelSpec სია — თითო ჩანაწერი თითო მოდელზე, თითოეულს აქვს replicaCount, requestCPU/GPU, pvcStorage. იგივე იდეაა, ინდუსტრიული მასშტაბით. კარგი values.yaml არის სწორედ ის, რაც სადემონსტრაციო chart-ს საწარმოო chart-ისგან განასხვავებს.

3. Release-ები მდგომარეობის მქონე ინსტალაციებია

release (რელიზი — დაინსტალირებული ჩარტის ეგზემპლარი) = chart + values, დაინსტალირებული კონკრეტული სახელით. helm upgrade ადარებს განსხვავებებს და იყენებს მათ; helm rollback აბრუნებს უკან; helm history აჩვენებს ყველა რევიზიას. ვერსიონირებული ინფრასტრუქტურული ცვლილებები — ოქროს თემა გასაუბრებაზე.

4. რენდერი გამოყენებამდე — დებაგინგის სუპერძალა

helm template . ლოკალურად ახდენს რენდერს კლასტერზე შეხების გარეშე. chart გატყდა? ნახე, რას რეალურად აგენერირებს. ამას მუდმივად გამოიყენებ.

ლაბორატორია: გადაიყვანე შენი vLLM სტეკი chart-ში

დააგენერირე კარკასი და გაასუფთავე

cd ~/PycharmProjects/AI/llm-infra-playground
helm create charts/slm-stack
rm -rf charts/slm-stack/templates/*        # keep the scaffold structure, write our own templates
ls charts/slm-stack

მოსალოდნელია: Chart.yaml charts/ templates/ values.yaml

განსაზღვრე values

ჩაანაცვლე charts/slm-stack/values.yaml:

namespace: llm

vllm:
  replicaCount: 1
  image:
    repository: vllm-cpu
    tag: local
    pullPolicy: IfNotPresent
  model:
    name: Qwen/Qwen3.5-0.8B
    maxModelLen: 4096
    dtype: bfloat16
  kvCacheSpaceGiB: 4
  resources:
    requests: { cpu: "4", memory: 6Gi }
    limits:   { cpu: "6", memory: 8Gi }
  pvc:
    size: 10Gi
  service:
    port: 8000

გადაიყვანე Deployment template-ად

შექმენი charts/slm-stack/templates/vllm-deployment.yaml შენი k8s/vllm.yaml-ის კონვერტაციით. პატერნი ასეთია — ყოველი მყარად ჩაწერილი მნიშვნელობა ხდება მითითება:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm
  namespace: {{ .Values.namespace }}
  labels: { app: vllm }
spec:
  replicas: {{ .Values.vllm.replicaCount }}
  selector:
    matchLabels: { app: vllm }
  template:
    metadata:
      labels: { app: vllm }
    spec:
      containers:
      - name: vllm
        image: "{{ .Values.vllm.image.repository }}:{{ .Values.vllm.image.tag }}"
        imagePullPolicy: {{ .Values.vllm.image.pullPolicy }}
        args:
          - "--model"
          - {{ .Values.vllm.model.name | quote }}
          - "--max-model-len"
          - {{ .Values.vllm.model.maxModelLen | quote }}
          - "--dtype"
          - {{ .Values.vllm.model.dtype | quote }}
        env:
        - name: VLLM_CPU_KVCACHE_SPACE
          value: {{ .Values.vllm.kvCacheSpaceGiB | quote }}
        ports:
        - containerPort: 8000
        resources:
          {{- toYaml .Values.vllm.resources | nindent 10 }}
        startupProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 10
          failureThreshold: 60
        readinessProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 10
        livenessProbe:
          httpGet: { path: /health, port: 8000 }
          periodSeconds: 20
          failureThreshold: 3
        volumeMounts:
        - { name: hf-cache, mountPath: /root/.cache/huggingface }
        - { name: dshm, mountPath: /dev/shm }
      volumes:
      - name: hf-cache
        persistentVolumeClaim: { claimName: hf-cache }
      - name: dshm
        emptyDir: { medium: Memory, sizeLimit: 2Gi }

შემდეგ იმავე პრინციპით შექმენი templates/vllm-service.yaml და templates/vllm-pvc.yaml (პორტი და ზომა values-იდან). თუ დაეჭვდები, ჩახედე doronmak-ის template-ებს — ეს იგივე კონვერტაციაა.

ჯერ ლოკალური რენდერი, მერე ინსტალაცია

helm template charts/slm-stack | less     # inspect generated YAML — find your values in it

# replace the raw-YAML deploy with the chart-managed release:
kubectl delete -f k8s/vllm.yaml
helm install slm charts/slm-stack
helm list
kubectl get pods -n llm -w

მოსალოდნელია: release slm deployed; vllm Pod-ი READY 1/1 სწრაფად (PVC-მ შეინახა წონები? არა — ახალი PVC სახელი ხელახლა ჩამოტვირთვას ნიშნავს, თუ სახელები არ ემთხვევა. hf-cache დაარქვი? მაშინ კი: სწრაფად.)

იგრძენი ძალა: შეცვალე მნიშვნელობა, გააკეთე upgrade, დააბრუნე უკან

helm upgrade slm charts/slm-stack --set vllm.model.maxModelLen=2048
kubectl get pods -n llm -w              # rolling replacement
helm history slm
helm rollback slm 1
helm history slm

მოსალოდნელია: history აჩვენებს 3 რევიზიას; rollback-მა აღადგინა maxModelLen=4096

მოგება შენი სტეკი ახლა დაინსტალირებადი, ვერსიონირებული პაკეტია: helm install slm charts/slm-stack ყველაფერს ნულიდან აღადგენს. ეს ბრძანება README-ში პორტფოლიოს ენაზე ლაპარაკობს.

შეამოწმე საკუთარი თავი

helm upgrade-მა release გააფუჭა. ყველაზე სწრაფი უსაფრთხო აღდგენა?

rollback ხელახლა იყენებს შენახულ წინა რევიზიას — ატომურად და აუდიტირებადად. uninstall/ხელახალი ინსტალაცია კარგავს release-ის ისტორიას და შეიძლება PVC-ებიც წაშალოს; ხელით რედაქტირება კი chart-ისგან აცილებას (drift) იწვევს.

chart სადღაც არავალიდურ YAML-ს არენდერებს. საუკეთესო პირველი დიაგნოსტიკა?

გააკეთე რენდერი ლოკალურად და ნახე ზუსტად, რა დააგენერირა template-მა (ჩვეულებრივ ეს ჩაწევის (indentation) შეცდომაა — nindent). template ფაილები თავისთავად ვალიდური YAML არ არის, ამიტომ მათი პირდაპირ apply-ს აზრი არ აქვს.

პირველწყარო

Helm Chart Template Guide ("Getting Started"-იდან "Values Files" გვერდების ჩათვლით, ~25 წთ). შემდეგ თავიდან ბოლომდე წაიკითხე production-stack-ის values.yaml — ეს chart-ის API დიზაინის ნამდვილი მასტერკლასია.

გაკვეთილი 6: სრულფასოვანი production-stack — რამდენიმე engine რეპლიკა და მოთხოვნების router (მარშრუტიზატორი) service discovery-ით, სესიის მიბმით (session affinity) და საკუთარი მეტრიკებით.

template-ის შეცდომები ბუნდოვანია — გაუგზავნე ისინი შენს მასწავლებელს template ფაილთან ერთად.
