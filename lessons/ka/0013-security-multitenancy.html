SLM Production Stack · გაძლიერებული · გაკვეთილი 13 · ~2 სთ

უსაფრთხოება და მრავალმომხმარებლიანობა (Multi-Tenancy)

Inference-პლატფორმას ორი განძი აქვს: ძვირადღირებული გამოთვლითი რესურსი, რომლის "სესხებაც" ყველას სიამოვნებით უნდა, და prompt-ები, სავსე მომხმარებელთა მონაცემებით. დღეს: მრავალშრიანი დაცვა (defense in depth) ორივესთვის — ქსელი, იდენტობა, რესურსები და engine-ის საკუთარი შესასვლელი კარი.

ცოდნა

1. 4C მენტალური მოდელი

Cloud → Cluster → Container → Code: თითოეული შრე მხოლოდ იმდენად ძლიერია, რამდენადაც მის გარეთა შრეები. დღეს ვმუშაობთ Cluster-ზე (RBAC, NetworkPolicy, კვოტები) და Container/Code-ზე (image-ების ჰიგიენა, engine-ის ავთენტიფიკაცია). (დოკუმენტაცია)

2. NetworkPolicy: default-deny, შემდეგ დაუშვი შენ მიერ დახაზული გრაფი

ბრტყელი pod-ქსელი = ნებისმიერ კომპრომეტირებულ pod-ს შეუძლია curl-ით შენს engine-ებამდე მიწვდეს. NetworkPolicy (ქსელური პოლიტიკა) pod-ის დონის firewall-ია — მაგრამ მხოლოდ მაშინ, თუ CNI მას აღასრულებს (kind-ის ნაგულისხმევი kindnet ამას არ აკეთებს; სწორედ ამიტომ დააინსტალირა მე-11 ლაბორატორიამ Cilium). საპროდაქშენო პატერნი:

1. default-deny ingress in the namespace
2. allow: ui → router (port 8000)
3. allow: router → engines (port 8000)
4. allow: monitoring → engines (/metrics)
5. everything else stays dead — including engine → engine

შენი არქიტექტურული დიაგრამა მე-8 გაკვეთილიდან სწორედ ეს allowlist-ია. თუ დახაზვა შეგიძლია, აღსრულებაც შეგიძლია.

3. RBAC როგორც აუდიტირებადი ზედაპირი

მინიმალური პრივილეგიები განწყობა კი არა, query-ა:

kubectl auth can-i --list --as=system:serviceaccount:llm:vllm-router -n llm
kubectl auth can-i delete pods --as=system:serviceaccount:ui:default -n llm

სენიორის ჩვევა: ServiceAccount-ების აუდიტი კვარტალში ერთხელ; ყველაფერი, რასაც cluster-admin ან * verbs აქვს, ან თავს ამართლებს, ან კვდება. შენი router-ის 3-verb-იანი Role მე-6 გაკვეთილიდან სამაგალითო მოქალაქეა.

4. Tenancy = namespace + quota + limits + netpol, როგორც ერთი მთლიანობა

ობიექტირას აკონტროლებსLLM-პლატფორმის მაგალითი
ResourceQuotaNamespace-ის ჯამურ რესურსებსგუნდი A იღებს 8 GPU-ს, 200Gi RAM-ს — მკაცრი ლიმიტი
LimitRangePer-pod ნაგულისხმევებს/საზღვრებსულიმიტო pod-ები GPU-node-ებზე ვერ შეიპარებიან
NetworkPolicyვინ ვის ესაუბრებაTenant A ვერ წვდება tenant B-ს engine-ებს
PriorityClassვინ იმარჯვებს რესურსებზე კონკურენციისასProd > staging > ექსპერიმენტები (გაკვეთილი 11)

Namespace-ები რბილი tenancy-ა (ერთი API server, ერთი kernel). მტრულად განწყობილი tenant-ები ⇒ ცალკე კლასტერები ან მკაცრი sandbox-ები (Kata/gVisor) — იცოდე, სად გადის ეს ზღვარი.

5. Secret-ები და image-ები, გულახდილი ვერსია

K8s Secret-ები etcd-ში base64-ია — დაცვა მოდის RBAC + etcd encryption-at-rest + ხანმოკლე ტოკენებისგან. პროდაქშენი მათ secret-მენეჯერიდან იღებს (External Secrets Operator + Vault/ღრუბლის SM); git-ში ინახება მითითებები და არასდროს — მნიშვნელობები (ან sealed-secrets, თუ შიფროტექსტის შენახვა აუცილებელია). Image-ები: დააფიქსირე digest-ები, დაასკანერე (trivy), მინიმალური base image-ები, არავითარი :latest პროდაქშენში — შენი vllm-cpu:local იქნებოდა registry/vllm-cpu@sha256:….

6. Engine-ის საკუთარი კარი

vllm serve --api-key (Secret-იდან, env-ად ჩაწოდებული) ნიშნავს, რომ pod-ი, რომელიც netpol-ს გვერდს აუვლის, მოდელს მაინც ვერ მისცემს prompt-ს. მრავალშრიანი დაცვა: edge-ავთენტიფიკაცია (გაკვეთილი 12) + ქსელი (ეს გაკვეთილი) + engine-ის key.

ლაბორატორია (labs/lab-13): ჩაკეტე სტეკი

Default-deny — უყურე, როგორ ტყდება

გამოიყენე deny-all ingress llm-ზე. UI კვდება, scrape-ები კვდება, router→engine კვდება. იგრძენი აფეთქების რადიუსი — 30 წამი კონტროლირებადი წყვეტა მეტს გასწავლის, ვიდრე ერთსაათიანი კითხვა.

დაუშვი დახაზული გრაფი

დაამატე ოთხი allow-პოლიტიკა (ui→router, router→engines, monitoring→engines, edge→ui/router). ყველაფერი ბრუნდება. შემდეგ დაამტკიცე იზოლაცია: kubectl run attacker default namespace-ში → curl engine-ისკენ → timeout. გადაუღე screenshot ამ timeout-ს; ეს პორტფოლიოს არტეფაქტია.

დაუწესე tenant-ს კვოტა

ResourceQuota (cpu 12, mem 20Gi, pods 10) + LimitRange ნაგულისხმევები llm-ზე. სცადე engine-ების 5-მდე დასკეილება — უყურე, როგორ ეტყვის კვოტა თავაზიან უარს Events-ში.

Engine-ის API key

Secret → env → --api-key არგუმენტი შენს chart-ში (values-ით მართული). Curl key-ის გარეშე → 401; UI განახლებული key-ით → მუშაობს. გააკეთე RBAC-ის აუდიტი auth can-i --list-ით llm/ui-ის ყველა ServiceAccount-ისთვის.

შეამოწმე შენი თავი

სტანდარტულ kind-ზე (kindnet) გამოყენებული NetworkPolicy-ები…

პოლიტიკები უბრალოდ ობიექტებია; აღსრულება CNI-ის საქმეა. kindnet მათ არ ახორციელებს — უსაფრთხოების #1 ცრუ განცდა ლოკალურ კლასტერებში. Cilium/Calico კი აღასრულებენ.

რატომ არ არის base64 Secret-ში დაშიფვრა?

base64-ის დეკოდირება ნებისმიერს შეუძლია, ვისაც ბაიტები უჭირავს. ნამდვილი დაცვა: RBAC ობიექტზე, etcd encryption at rest, გარე secret-მენეჯერები.

ორი გადამხდელი კლიენტი მკაცრ იზოლაციას ითხოვს. საკმარისია namespace-ები?

Namespace-tenancy რბილია — ერთი API server, საზიარო kernel-ები, გაქცევის გზები არასწორი კონფიგურაციით. მკაცრი მოთხოვნები ⇒ კლასტერი თითო tenant-ზე ან hardware/VM sandbox-ები (გამოყოფილი node-ებიც კი control plane-ს იზიარებენ).
