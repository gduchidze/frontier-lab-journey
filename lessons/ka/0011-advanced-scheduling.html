გაძლიერებული Scheduling და GPU-ფლოტი

გაძლიერებული ტრეკი იწყება. Production-ში ის, თუ სად დაეშვება Pod, ღირებულებისა და საიმედოობის გადაწყვეტილებაა — მით უმეტეს, როცა node-ებზე $30k-იანი GPU-ებია. დღეს: როგორ წყვეტს scheduler-ი (განმთავსებელი) სინამდვილეში, და ყველა ბერკეტი, რომელიც მას გადაფარავს. შენს Mac-ზე GPU-ფლოტის სიმულაციას გააკეთებ.

გაძლიერებული ტრეკის ძირითადი წესები წინაპირობა: გაკვეთილები 1–10 დასრულებულია. მხოლოდ-GPU მექანიკა პატიოსნად ისწავლება: თეორია + სიმულაცია + სიმძლავრის მათემატიკა აქ; კონცეფციები 1:1 გადადის იმ დღეს, როცა რეალურ A100-ებს შეეხები. ლაბორატორია 11 შენს კლასტერს გაძლიერებული კონფიგურაციით აწყობს თავიდან (3 node, Cilium CNI, ingress-პორტები) — საკუთარი chart-იდან თავიდან აწყობა თავად პირველი სავარჯიშოა.

ცოდნა

1. Scheduler-ი ორფაზიანი კონვეიერია

ყოველი მომლოდინე pod-ისთვის: Filter (გამორიცხე node-ები, რომლებიც მას ვერ დაიტევენ: არასაკმარისი რესურსები, taint-ები, affinity-ის აცდენები, volume-ის ზონა) → Score (დაალაგე გადარჩენილები: least/most-allocated, გაფანტვა, image-ის ლოკალურობა) → მიბმა. ყველაფერი, რაც ქვემოთაა, შენ მიერ ამ ფაზებში შეზღუდვების ჩადებაა. (დოკუმენტაცია)

4. Bin-pack თუ გაფანტვა? დოლარების კითხვა

Web-დატვირთვები იფანტება (მდგრადობისთვის). GPU-ინფერენსი, როგორც წესი, bin-pack-დება (მჭიდროდ ილაგება): ჯერ node 1 ბოლომდე შეავსე და მერე გააღვიძე node 2, რომ cluster autoscaler-მა / Karpenter-მა ცარიელი $30/სთ-იანი node-ები ნულამდე დაიყვანოს. არასწორი default = ხუთი node, თითო 20%-ით დაკავებული = 5-ჯერ დიდი გადასახადი. Senior-ის პასუხი: bin-pack-ი ფლოტებისთვის თითო pool-ში, anti-affinity — მხოლოდ იმ ავარიის დომენებზე, რომელთა დაცვაშიც რეალურად იხდი.

თავიდან აწყობა: გაძლიერებული კლასტერი

kind delete cluster --name llm-stack → შექმენი k8s/kind-advanced.yaml-იდან (1 control-plane + 2 worker, default CNI გამორთული, პორტები 80/443/30080) → დააყენე Cilium CNI-ად → ხელახლა გაშალე მთელი შენი სტეკი chart-იდან. რეპროდუცირებადობის გამოცდა: რამდენ დროს გახარჯავს? ჩაინიშნე.

გააყალბე GPU-pool

დაადე worker2-ს label + taint (gpu=true:NoSchedule), შემდეგ გამოაცხადე ყალბი გაფართოებული რესურსი example.com/gpu: 2 node-ის status-ის PATCH-ით kubectl proxy-ის გავლით. Scheduler-ი მას ზუსტად ისე ეპყრობა, როგორც ნამდვილ GPU-ებს.

დაგეგმე მასზე

გაშალე „gpu" pod-ები, რომლებიც ითხოვენ example.com/gpu: 1-ს toleration + affinity-ით → ორივე worker2-ზე დაეშვება → მესამე რჩება Pending: Insufficient example.com/gpu. ზუსტად ის ავარია, რომელსაც ყველა GPU-კლასტერში გაარჩევ.

Preemption

PriorityClass-ები: prod-inference > batch-experiment; შეავსე „GPU" node batch-pod-ებით; გაშალე prod-pod → უყურე გამოსახლებას Events-ში.

გაფანტე engine-ები

დაამატე topologySpreadConstraints (ზონის key აქ = hostname) შენს vLLM chart-ის template-ს; 2 replica სხვადასხვა worker-ზე დაეშვება; გადაამოწმე -o wide-ით.

შეამოწმე შენი თავი

GPU-pod-ები ითხოვენ nvidia.com/gpu-ს, მაგრამ Pending-ში სხედან node-ზე, რომელსაც თავისუფალი GPU-ები აქვს. კლასიკური მიზეზი?

GPU-pool-ები თითქმის ყოველთვის taint-დადებულია; pod შესაბამისი toleration-ის გარეშე Score-ის ფაზამდე ვერ აღწევს — Filter-ი გამორიცხავს. გაფართოებული რესურსები ვერ ფრაგმენტირდება — ისინი მთელი რიცხვებია.

რატომ ვამჯობინებთ GPU-ინფერენსისთვის bin-pack-ს გაფანტვას?

მჭიდროდ დალაგებული = მთლიანად უქმე node-ები = autoscaler-ი მათ იბრუნებს = ფული. Pod-ები კონტეინერებს შორის წონებს არასოდეს იზიარებენ, TP კი ერთი pod-ის GPU-ების შიგნით ხდება.

MIG თუ time-slicing ორი გადამხდელი tenant-ისთვის ერთ A100-ზე?

Tenant-ებს იზოლაცია სჭირდებათ — MIG მეხსიერებას/SM-ებს აპარატურულად ყოფს. Time-slicing ყველაფერს აზიარებს: ერთი tenant-ის OOM ან პიკური დატვირთვა მეორეს ურტყამს.
