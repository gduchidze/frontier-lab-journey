SLM Production Stack · გაკვეთილი 7 · ფაზა 7 · ~2 სთ

Observability: Prometheus + Grafana

ყოველი სერიოზული ინფერენსის პლატფორმა სამ კითხვას მუდმივად პასუხობს: რამდენად სწრაფია (TTFT, latency), რამდენად დატვირთულია (running/pending მოთხოვნები, KV cache-ის გამოყენება), რამდენად ჯანმრთელია (up, შეცდომების სიხშირე). დღეს ააგებ სისტემას, რომელიც ამ კითხვებს პასუხობს.

ცოდნა: scrape მოდელი

1. Pull და არა push

Prometheus scrape-ავს (მონაცემებს თავად კრებს): ყოველ N წამში ის GET-ით უკავშირდება თითოეული სამიზნის /metrics-ს და ინახავს დროით მწკრივებს (time series). vLLM ამ endpoint-ს უკვე ააშკარავებს (მე-3 გაკვეთილში curl-ით მიმართე) — ინსტრუმენტირების ნულოვანი სამუშაო. router-იც ააშკარავებს საკუთარ სტატისტიკას თითო engine-ზე.

2. მეტრიკის ტიპები, რომლებსაც რეალურად წაიკითხავ

ტიპიფორმაvLLM მაგალითი
Gaugeმნიშვნელობა ახლავე, ზევით-ქვევითvllm:num_requests_running, vllm:kv_cache_usage_perc
Counterმხოლოდ იზრდება → იკითხება როგორც rate()vllm:prompt_tokens_total, vllm:request_success_total
Histogrambucket-ები → იკითხება როგორც histogram_quantile()vllm:time_to_first_token_seconds, vllm:e2e_request_latency_seconds

მეტრიკების სრული სია: vLLM production metrics დოკუმენტაცია · შემოკლებული ცხრილი observability-ის მასალაში.

3. PromQL — 3 პატერნი, რომელიც 90%-ს ფარავს

# current load per engine (gauge, as-is)
vllm:num_requests_running

# throughput: tokens/sec over 1m window (counter → rate)
rate(vllm:generation_tokens_total[1m])

# P90 TTFT over 5m (histogram → quantile)
histogram_quantile(0.9, rate(vllm:time_to_first_token_seconds_bucket[5m]))

4. ServiceMonitor: უთხარი Prometheus-ს, რა scrape-ოს — K8s-ის წესით

kube-prometheus-stack Helm chart-ი აინსტალირებს Prometheus-ს + Grafana-ს + Prometheus Operator-ს. Operator-ი უყურებს ServiceMonitor ობიექტებს — მანიფესტს, რომელიც ამბობს: "scrape-ე ამ label-ების შესაბამისი Service-ები ამ პორტზე". დეკლარაციული scrape კონფიგურაცია — იგივე ფილოსოფია, რაც ყველაფერ დანარჩენს, რაც უკვე ააგე.

ლაბორატორია: შუქი აინთე

დააინსტალირე kube-prometheus-stack

helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update
helm install kps prometheus-community/kube-prometheus-stack \
  --namespace monitoring --create-namespace \
  --set grafana.adminPassword=admin \
  --set prometheus.prometheusSpec.serviceMonitorSelectorNilUsesHelmValues=false
kubectl get pods -n monitoring -w     # wait until all Running (2-3 min)

მოსალოდნელია: prometheus, grafana, operator, kube-state-metrics, node-exporter Pod-ები Running

ის ბოლო --set flag-ი ნაგულისხმევად chart-ი მხოლოდ იმ ServiceMonitor-ებს scrape-ავს, რომლებსაც მისივე release label-ი აქვთ. სელექტორის nil-uses-false-ზე დაყენება = "scrape-ე ნებისმიერი ServiceMonitor". ეს გიშველის კლასიკური "ჩემი target არასოდეს ჩნდება" საათისგან.

მიმართე Prometheus-ი vLLM-ისკენ

შეინახე როგორც k8s/vllm-servicemonitor.yaml:

apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: vllm
  namespace: llm
  labels: { app: vllm }
spec:
  selector:
    matchLabels: { app: vllm }
  endpoints:
  - port: "8000"
    path: /metrics
    interval: 15s
პორტი დასახელებული უნდა იყოს ServiceMonitor-ის port: Service-ის პორტის სახელს გულისხმობს და არა ნომერს. vllm-ის Service template-ში პორტს დაუმატე name: http, აქ კი გამოიყენე port: http. ეს შეუსაბამობა ServiceMonitor-ის #1 ხაფანგია.
kubectl apply -f k8s/vllm-servicemonitor.yaml

გადაამოწმე target-ი, გაუშვი პირველი query-ები

kubectl port-forward -n monitoring svc/kps-kube-prometheus-stack-prometheus 9090:9090

გახსენი localhost:9090/targets → შენი vllm endpoint-ები უნდა აჩვენებდნენ UP-ს (მიეცი ~30 წამი). შემდეგ Graph ჩანართში გაუშვი ცოდნის სექციის სამივე PromQL პატერნი. ჯერ router-ის გავლით ტრაფიკი დააგენერირე, რომ გრაფიკები ამოძრავდეს:

for i in $(seq 1 10); do curl -s localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"Tell me a 50 word story."}],"max_tokens":80}' > /dev/null & done

Grafana: ააგე ინფერენსის dashboard-ი

kubectl port-forward -n monitoring svc/kps-grafana 3000:80

შედი localhost:3000-ზე (admin / admin). ორი გზაა — ორივე გააკეთე:

ა) დააიმპორტე საორიენტაციო dashboard-ი. production-stack-ს თან მოჰყვება: ჩამოტვირთე observability/vllm-dashboard.json რეპოზიტორიიდან → Grafana → Dashboards → Import → ატვირთვა. მყისიერი პანელები: TTFT-ის განაწილება, running/pending მოთხოვნები, ჯანმრთელი ინსტანსები.

ბ) ერთი პანელი შენით ააგე (სწორედ ეს არის უნარი): New dashboard → Add visualization → datasource Prometheus → query:

histogram_quantile(0.9, sum(rate(vllm:time_to_first_token_seconds_bucket[5m])) by (le))

დაარქვი "P90 TTFT", ერთეული — წამები, შეინახე dashboard-ი სახელით SLM Stack.

მოსალოდნელია: ცოცხალი TTFT მრუდი, რომელიც ტრაფიკის ციკლის გაშვებისას იჭიმება ზევით

მოგება production-stack-ის README-ს სქრინშოტი — ხელმისაწვდომი ინსტანსები, TTFT-ის განაწილება, pending მოთხოვნები — ახლა შენს კლასტერზე მუშაობს და შენს engine-ებს უყურებს. გადაუღე სქრინშოტი პორტფოლიოს README-სთვის.

დატვირთე-და-უყურე ვარჯიში (10 წთ)

გაყოფილი ეკრანი: მარცხნივ Grafana, მარჯვნივ ტერმინალი. გაუშვი 20 პარალელური მოთხოვნა. ხმამაღლა აღწერე, რასაც ხედავ: pending რიგი იზრდება → running პლატოზე გადის engine-ის სიმძლავრეზე → TTFT P90 იწევს → იცლება. ეს თხრობა ზუსტად ის არის, რასაც SRE-ს გასაუბრება dashboard-თან გთხოვს.

შეამოწმე საკუთარი თავი

რატომ უნდა შეიფუთოს counter-ები rate()-ში, რომ სასარგებლო იყოს?

counter-ი მთელი სიცოცხლის ჯამია — მისი დახრილობა (rate) არის აზრიანი სიგნალი. rate() ასევე კორექტულად უმკლავდება გადატვირთვებს (counter-ის განულებას).

ServiceMonitor-ი გამოყენებულია, target-ი Prometheus-ში არასოდეს ჩნდება. პირველი შემოწმება?

დასახელებული პორტის შეუსაბამობა (ან label-სელექტორის შეუსაბამობა) ჩუმ არაღმოჩენას იწვევს. Grafana ნაკადის ქვევითაა — target-ის აღმოჩენასთან კავშირი არ აქვს.

TTFT P90 იზრდება და ამავდროულად num_requests_waiting-იც — ეს ნიშნავს…

მოლოდინის რიგის ზრდა = დაშვების ბექლოგი; რიგში გატარებული დრო TTFT-ში ხვდება. ქსელური დაყოვნება მილიწამებია, ხოლო PagedAttention ფრაგმენტაციას ისედაც არ უშვებს.
