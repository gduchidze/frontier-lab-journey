SLM Production Stack · გაკვეთილი 3 · ფაზა 3 · ~2 სთ (ფონური build-ის ჩათვლით)

ძრავა: vLLM Docker-ში

ერთი სესია Kubernetes-ის გარეშე. თავად vLLM-ს გაიცნობ — რატომ არის ის ინდუსტრიის default, როგორ მუშაობს continuous batching (უწყვეტი დაჯგუფება) და PagedAttention, და Qwen-ს OpenAI-თან თავსებადი API-ის გავლით დაელაპარაკები საკუთარ მანქანაზე.

ცოდნა: რატომ გაიმარჯვა vLLM-მა

1. პრობლემა: GPU-ებს (და CPU-ებს) სძულთ წვრილმანი საუბრები

გულუბრყვილო serving ერთ მოთხოვნას ერთდროულად ამუშავებს; ტოკენებს შორის აპარატურა უსაქმოდ ზის. Continuous batching ამას ასწორებს: ყოველ decoding ნაბიჯზე scheduler-ს შეუძლია მიმდინარე batch-ში ახალი მოთხოვნების ჩამატება და დასრულებულების ამოგდება. batch-ის დაცლის ლოდინი აღარ არის. ეს არის ერთადერთი ყველაზე დიდი throughput-იდეა LLM serving-ში.

2. PagedAttention: ვირტუალური მეხსიერება KV cache-ისთვის

ყოველ დაგენერირებულ ტოკენს attention სჭირდება ყველა წინა ტოკენზე — ეს ინახება KV cache-ად, მეხსიერების მთავარ მომხმარებლად. vLLM-ის ფუძემდებლური იდეა (SOSP 2023 ნაშრომი): KV cache იმართოს ფიქსირებული ზომის ბლოკებით და მიმართვის ცხრილით, OS-ის ვირტუალური მეხსიერების გვერდების მსგავსად. შედეგი: თითქმის ნულოვანი ფრაგმენტაცია, საერთო პრეფიქსების მქონე მოთხოვნებს შორის გაზიარებული მეხსიერება, ბევრად დიდი batch-ები.

3. ერთი სერვერი, OpenAI-ის API

vllm serve MODEL უშვებს HTTP სერვერს, რომელიც ახორციელებს /v1/chat/completions, /v1/completions, /v1/models — OpenAI-თან თავსებად API-ს. ნებისმიერი OpenAI კლიენტი მუშაობს ერთი base URL-ის შეცვლით. სწორედ ამიტომ ერთვება ყველა UI, router და SDK უფასოდ — და ამიტომ არ სჭირდება ფაზა 8-ის ჩატის UI-ს არანაირი custom კოდი.

4. CPU backend: იგივე ძრავა, სხვა რკინა

vLLM-ის CPU backend (დოკუმენტაცია) იმავე scheduler-სა და API-ს უშვებს x86/ARM-ზე. დღეს ორ ბერკეტს აქვს მნიშვნელობა:

ბერკეტიმნიშვნელობაჩვენი მნიშვნელობა
VLLM_CPU_KVCACHE_SPACEKV cache-ისთვის დაჯავშნილი GiB4
--dtype bfloat16წონების სიზუსტე (M4 bf16-ს უჭერს მხარს)bfloat16
--max-model-lenზღუდავს კონტექსტის სიგრძეს → ზღუდავს KV მეხსიერებას4096
რეალობის შემოწმება CPU inference ~10–50-ჯერ ნელია GPU-ზე. 0.8B მოდელისთვის ეს მაინც რამდენიმე token/წმ-ია — საკმარისი ყველა production პატერნის სასწავლად. არქიტექტურა, რომელსაც ააგებ, GPU ვერსიის იდენტურია; მხოლოდ image და ერთი resource ხაზი განსხვავდება.

ლაბორატორია: build, serve, ჩატი

დაიწყე image-ის build ფონურად (ეს ᲞᲘᲠᲕᲔᲚᲘ გააკეთე)

ოფიციალური ARM CPU image არ არსებობს, ამიტომ მას ვაშენებთ vLLM-ის საკუთარი Dockerfile.cpu-დან. 30–60 წთ სჭირდება — გაუშვი და სანამ მუშაობს, კითხვა განაგრძე:

cd ~/PycharmProjects/AI
git clone https://github.com/vllm-project/vllm.git
cd vllm
docker build -f docker/Dockerfile.cpu -t vllm-cpu:local . > /tmp/vllm-build.log 2>&1 &
tail -f /tmp/vllm-build.log     # Ctrl+C anytime; build continues
რატომ build და არა pull? doronmak-ის რეპო საკუთარ doronmak/vllm-cpu-image-ს იმავე მიზეზით აქვეყნებს — ყველა პლატფორმისთვის ოფიციალური CPU image არ არსებობს. ნებისმიერი image-ის არქიტექტურა შეამოწმე docker manifest inspect IMAGE | grep architecture-ით. amd64 image შენს M4-ზე ემულაციით მუშაობს: მუშაობს, მაგრამ ~5-ჯერ ნელა.

სანამ build მიმდინარეობს: წაიკითხე ძრავის შესახებ

წაიკითხე vLLM-ის ორიგინალი საანონსო ბლოგი (10 წთ) — PagedAttention-ის ყველაზე ნათელი ახსნა ილუსტრაციებით. შემდეგ გადაათვალიერე vLLM reference ბარათი, რომელსაც კურსის დარჩენილ ნაწილში გამოიყენებ.

გაუშვი სერვერი

როცა build დასრულდება (docker images | grep vllm-cpu):

docker run -d --name vllm \
  -p 8000:8000 \
  -v hf-cache:/root/.cache/huggingface \
  -e VLLM_CPU_KVCACHE_SPACE=4 \
  vllm-cpu:local \
  --model Qwen/Qwen3.5-0.8B \
  --max-model-len 4096 \
  --dtype bfloat16

docker logs -f vllm      # watch model download + engine init; Ctrl+C when "Uvicorn running"

მოსალოდნელია (პირველ გაშვებაზე): წონების ჩამოტვირთვა (~2 GB), შემდეგ "Application startup complete"

თუ Qwen3.5-0.8B ჩავარდება unsupported architecture შეცდომით qwen3_5 2026 წლის მულტიმოდალური არქიტექტურაა; შენი აშენებული image vLLM main-ს მიჰყვება, ასე რომ, უნდა იმუშაოს. თუ არა: გადადი --model Qwen/Qwen3-0.6B-ზე (მხოლოდ ტექსტური, დიდი ხანია მხარდაჭერილი), ჩაინიშნე ეს სასწავლო ჩანაწერში და სამიზნე მოდელი მოგვიანებით სცადე git pull + rebuild-ის შემდეგ. ყველა სხვა გაკვეთილი ორივე მოდელით იდენტურად მუშაობს.

დაელაპარაკე — სამი endpoint

# 1. What models are loaded?
curl -s localhost:8000/v1/models | python3 -m json.tool

# 2. Chat (the endpoint everything else in this course uses)
curl -s localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen/Qwen3.5-0.8B",
    "messages": [{"role":"user","content":"In one sentence: what is PagedAttention?"}],
    "max_tokens": 60
  }' | python3 -m json.tool

# 3. Metrics — Prometheus format, phase 7 lives here
curl -s localhost:8000/metrics | grep -E "^vllm:" | head -20

მოსალოდნელია: JSON პასუხი Qwen-ისგან; მეტრიკების ხაზები, როგორიცაა vllm:num_requests_running

ნახე continuous batching საკუთარი თვალით

გაუშვი 5 პარალელური მოთხოვნა, შემდეგ შეამოწმე მიმდინარე მოთხოვნების gauge:

for i in 1 2 3 4 5; do
  curl -s localhost:8000/v1/chat/completions -H "Content-Type: application/json" \
    -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"Count to 30 slowly."}],"max_tokens":200}' > /dev/null &
done
sleep 3
curl -s localhost:8000/metrics | grep -E "num_requests_(running|waiting)"

მოსალოდნელია: num_requests_running > 1 — რამდენიმე მოთხოვნა ᲔᲠᲗ batch-ში, ერთდროულად

მოგება შენ ააგე inference ძრავის image, რომელსაც ყველა შემდეგი ფაზა დააპლოებს, დაელაპარაკე Qwen-ს ზუსტად იმ API-ით, რომელსაც შენი UI გამოიყენებს, და უყურე, როგორ მოემსახურა continuous batching 5 მოთხოვნას ერთდროულად. ძრავა შავი ყუთი აღარ არის.

გაჩერება (image + მოდელის cache დაიტოვე)

docker stop vllm && docker rm vllm
# model weights persist in the named volume:
docker volume ls | grep hf-cache

შეამოწმე საკუთარი თავი

მოთხოვნა B მოდის მაშინ, როცა მოთხოვნა A გენერაციის შუაშია. continuous batching ნიშნავს…

scheduler batch-ს ყოველ decoding ნაბიჯზე ხელახლა აწყობს. არც დაცლა, არც preemption ნორმალურ გზაზე.

რას აღმოფხვრის PagedAttention უპირველესად?

ბლოკებზე დაფუძნებული KV შენახვა მიმართვის ცხრილით = თითქმის ნულოვანი ფრაგმენტაცია და პრეფიქსების გაზიარება. გამოთვლის ღირებულება და წონები უცვლელია.

რატომ ეხმარება --max-model-len-ის შეზღუდვა მეხსიერებაში შეზღუდულ მანქანას?

KV cache იზრდება sequence-ის სიგრძესთან ერთად. სიგრძის შეზღუდვა ზღუდავს თითო მოთხოვნის ყველაზე ცუდ შემთხვევას და scheduler-ს მეტი მოთხოვნის უსაფრთხოდ მიღების საშუალებას აძლევს. წონები ხელუხლებელია.
