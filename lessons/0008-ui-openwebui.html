The UI: Open WebUI on the Stack

Everything so far spoke curl. Today the stack gets a face: Open WebUI — a full-featured chat frontend — deployed into the cluster, wired to the router, and exposed to your browser through the NodePort you mapped back in Lesson 1.

Knowledge: three ideas

1. The OpenAI API dividend pays out

Open WebUI speaks the OpenAI API. Point OPENAI_API_BASE_URL at any compatible server — vLLM, your router — and it lists models, streams chat, manages conversations. Zero custom code. This is why lesson 3 called the API "the dividend": every tool in the ecosystem plugs in for free. (vLLM's own docs feature this pairing: Open WebUI deployment guide.)

2. In-cluster DNS does the wiring

The UI reaches the router by Service DNS: http://vllm-router.llm.svc.cluster.local/v1. No IPs, no config discovery — the pattern from lesson 2, now earning its keep across namespaces.

3. NodePort: the third way to be reached

You know port-forward (dev tunnel) and ClusterIP (in-cluster only). NodePort opens a port (30000–32767) on every node. Your Lesson-1 kind config mapped node port 30080 → Mac port 30080 — planted exactly for today. Production would use an Ingress/LoadBalancer; NodePort is the honest local equivalent. (docs)

Lab: ship the face

Deploy Open WebUI

Save as k8s/webui.yaml:

apiVersion: v1
kind: Namespace
metadata:
  name: ui
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: webui-data
  namespace: ui
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 2Gi
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: open-webui
  namespace: ui
  labels: { app: open-webui }
spec:
  replicas: 1
  selector:
    matchLabels: { app: open-webui }
  template:
    metadata:
      labels: { app: open-webui }
    spec:
      containers:
      - name: open-webui
        image: ghcr.io/open-webui/open-webui:main
        env:
        - name: OPENAI_API_BASE_URL
          value: "http://vllm-router.llm.svc.cluster.local/v1"
        - name: OPENAI_API_KEY
          value: "not-needed-locally"
        - name: WEBUI_AUTH
          value: "False"
        - name: ENABLE_OLLAMA_API
          value: "False"
        ports:
        - containerPort: 8080
        resources:
          requests: { cpu: 250m, memory: 512Mi }
          limits:   { cpu: "1",  memory: 1Gi }
        readinessProbe:
          httpGet: { path: /health, port: 8080 }
          periodSeconds: 10
        volumeMounts:
        - name: data
          mountPath: /app/backend/data
      volumes:
      - name: data
        persistentVolumeClaim: { claimName: webui-data }
---
apiVersion: v1
kind: Service
metadata:
  name: open-webui
  namespace: ui
spec:
  type: NodePort
  selector: { app: open-webui }
  ports:
  - port: 8080
    targetPort: 8080
    nodePort: 30080
WEBUI_AUTH=False is local-only Fine on your laptop, disqualifying anywhere reachable by others. If you ever demo this off-machine, enable auth and put it behind TLS. Note it in your portfolio README — reviewers notice security awareness.
kubectl apply -f k8s/webui.yaml
kubectl get pods -n ui -w

expect: open-webui pod READY 1/1 in ~1 min (first image pull is ~1 GB, be patient)

Open it — no port-forward needed

Browser → http://localhost:30080. Trace why this works, hop by hop: browser → Mac:30080 → kind extraPortMapping → control-plane node:30080 → NodePort Service → UI Pod. Say it out loud; being able to trace a request path through K8s networking is an interview staple.

Resilience demo (5 min, do it — it's fun)

Mid-conversation, kill an engine:

kubectl delete pod -n llm $(kubectl get pods -n llm -l app=vllm -o name | head -1 | cut -d/ -f2)

Keep chatting. The router discovers the dead engine, routes to the survivor; the Deployment resurrects the victim. The user never knows. Every layer you built is now visibly doing its job.

Check yourself

How does Open WebUI find models to show in its picker?

Pure OpenAI-API client behavior — it asks the endpoint what models exist. Only the router touches the K8s API.

Why does localhost:30080 reach the UI without port-forward?

Two mechanisms chained: NodePort opens 30080 on the node; the Lesson-1 extraPortMappings forwards Mac:30080 → node:30080. Without the kind mapping, NodePort alone would only be reachable from the node containers.
