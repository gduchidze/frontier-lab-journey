vLLM Performance Engineering

Basics track treated the engine as a good black box. Seniors get paid to open it: quantization, batching knobs, chunked prefill, speculative decoding — and above all, a disciplined method for tuning instead of flag-flailing.

Knowledge

1. First principles: prefill is compute-bound, decode is bandwidth-bound

Prefill crunches the whole prompt in parallel — FLOPs-limited. Decode produces one token per step per sequence, re-reading weights + KV every step — memory-bandwidth-limited. Every optimization below attacks one side or their interference. Say which, and you sound senior; miss it, and knobs are astrology.

2. Quantization taxonomy

Decision shape: memory-starved → weight quant; batch/context-starved → KV quant; Hopper GPUs → FP8 by default; always re-run YOUR evals after — quantization quality loss is workload-dependent. (docs)

4. Chunked prefill: stop long prompts from starving decode

Naive scheduling runs a giant prefill as one step — every decoding user stalls (TPOT spike). Chunked prefill slices prompts into budget-sized chunks and mixes them WITH decode tokens each step: slightly slower prefill completion, dramatically smoother ITL. Default-on in modern vLLM (V1); knob is the token budget. This is the canonical "throughput vs tail latency" interview question in one feature. (optimization docs)

5. Speculative decoding: spend FLOPs to buy latency

Decode wastes parallel hardware on one token at a time. Fix: a cheap drafter (small model, n-gram lookup, or trained heads: EAGLE/MTP) proposes k tokens; the big model verifies all k in ONE parallel pass; accepted prefix keeps, mispredictions truncate — output provably identical in greedy mode. Wins when acceptance is high (code, templated text) and batch is small (latency-critical); loses at saturation (verify FLOPs compete with batch throughput). (docs)

6. The method (this is the actual lesson)

1. Fix a workload (input/output lens, concurrency) — bench is meaningless otherwise
2. Baseline: vllm bench serve → save JSON
3. ONE knob per run. Re-bench. Log: knob, TTFT P50/P90, TPOT, tok/s, KV usage
4. Accept/revert against YOUR SLO (not "bigger number")
5. Write the tuning report — the artifact that outlives the cluster

Lab (labs/lab-15): a real tuning study on CPU

CPU numbers differ from GPU; the METHOD is identical — that's what you're drilling.

Baseline

Single engine, fixed workload (128 in / 64 out, rate 0.5): bench, save JSON, record the table row.

Knob 1: max-num-seqs sweep

4 → 8 → 16. Watch TPOT degrade as batch grows while tok/s rises — the fundamental trade, measured by your own hands.

Knob 2: prefix caching

--enable-prefix-caching + a shared-prefix workload (same 400-token system prompt, different questions). Compare TTFT distributions on/off. Use the vllm-prefix-cache-bench skill for the harness.

Knob 3: max-model-len as capacity lever

4096 → 2048 with concurrency pushed high; watch admission improve (waiting queue drains faster) at the cost of context ceiling.

The report

docs/tuning-report.md: workload spec, table of runs, one chart (even ASCII), 3 conclusions, and a "what I'd try on H100s" section (FP8, kv fp8, spec decode — with predicted effects and why). The GPU section proves the theory transferred.

Check yourself

TPOT spikes whenever a long-prompt request arrives. Best-fit feature?

The spike is prefill monopolizing steps. Chunking bounds per-step prefill so decode tokens keep flowing. Spec decode and KV quant don't touch prefill interference.

70B model fits but batch stays tiny; queue grows. First lever?

Weights fit — the constraint is KV memory per sequence. fp8 KV ≈ doubles concurrent capacity. Raising max-num-seqs without memory does nothing (admission is KV-bound); 2-bit weights torch quality.

Speculative decoding helps least when…

At saturation the "idle parallel FLOPs" spec decode exploits don't exist — verification competes with batch throughput. Templated output (high acceptance) and sparse traffic are its best cases.
