Scale Out: Replicas and the Request Router

One engine can only serve so much. Today you run two, and learn why LLM serving needs a smarter load balancer than a plain Service — the request router at the heart of vLLM production-stack.

Memory check first Two vLLM replicas ≈ 12–16 Gi requested on a 16 Gi Mac. Before this lab: Docker Desktop → Settings → Resources → give Docker ≥ 12 GB. Also drop per-replica resources (step 1). If your machine swaps hard, run this lesson with facebook/opt-125m as the model — the routing lessons are identical.

Knowledge: why a plain Service isn't enough

1. LLM requests are not web requests

A web request costs milliseconds; an LLM request costs seconds-to-minutes of stateful compute with a growing KV cache. Random round-robin (what a Service does) ignores: engine load (queue depth), KV cache reuse (same user's conversation prefix already cached on engine A), and model placement (engine B serves a different model). Result: avoidable latency and evictions.

2. The router is an application-layer load balancer

production-stack's vllm-router sits in front of engines and adds what a Service can't:

CapabilityWhy it matters
K8s service discoveryWatches the API for engine Pods by label — replicas join/leave automatically
Routing algorithmsround-robin, session-ID affinity, prefix-aware (route same prefix → same engine → KV cache hit)
Per-engine metricsQPS, TTFT, pending requests per engine — feeds phase 7 dashboards
Model routingOne endpoint, many models: request's model field picks the backend

3. Session affinity = KV cache economics

A chat conversation resends the whole history each turn. If turn N lands on the engine that served turn N−1, the prefix is already in KV cache (with prefix caching enabled) → TTFT drops dramatically. Sticky routing isn't a nicety; it's a compute-cost lever. This is the "KV cache reuse" phrase in the production-stack README — and in ML infra job postings.

Lab: two engines, one router

Slim per-replica resources, scale to 2

Edit charts/slm-stack/values.yaml: replicaCount: 2, resources → requests: { cpu: "2", memory: 5Gi }, limits: { cpu: "4", memory: 6Gi }, kvCacheSpaceGiB: 2. One more change — two Pods can't share a ReadWriteOnce PVC on different nodes. Simplest local fix: in the deployment template, replace the PVC volume with a per-pod emptyDir (weights re-download once per pod; accept it locally) — or pin both replicas to one node with nodeSelector. Choose, then:

helm upgrade slm charts/slm-stack
kubectl get pods -n llm -o wide -w    # wait for both READY 1/1 (patience: CPU load × 2)

expect: vllm-xxxx and vllm-yyyy both Running READY 1/1

Watch a plain Service round-robin

kubectl port-forward -n llm svc/vllm 8000:8000 &
for i in 1 2 3 4 5 6; do
  curl -s localhost:8000/v1/chat/completions -H "Content-Type: application/json" \
    -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"hi"}],"max_tokens":5}' > /dev/null
done
kubectl logs -n llm -l app=vllm --tail=5 --prefix | grep "POST /v1"

expect: requests visible in BOTH pods' logs (note: port-forward pins to one pod — in-cluster traffic spreads; check both logs to compare)

Deploy the router

Save as k8s/router.yaml:

apiVersion: v1
kind: ServiceAccount
metadata:
  name: vllm-router
  namespace: llm
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: llm
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: vllm-router-pod-reader
  namespace: llm
subjects:
- kind: ServiceAccount
  name: vllm-router
  namespace: llm
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm-router
  namespace: llm
  labels: { app: vllm-router }
spec:
  replicas: 1
  selector:
    matchLabels: { app: vllm-router }
  template:
    metadata:
      labels: { app: vllm-router }
    spec:
      serviceAccountName: vllm-router
      containers:
      - name: router
        image: lmcache/lmstack-router:latest
        args:
          - "--host"
          - "0.0.0.0"
          - "--port"
          - "8000"
          - "--service-discovery"
          - "k8s"
          - "--k8s-namespace"
          - "llm"
          - "--k8s-label-selector"
          - "app=vllm"
          - "--routing-logic"
          - "roundrobin"
          - "--engine-scrape-interval"
          - "15"
          - "--request-stats-window"
          - "60"
        ports:
        - containerPort: 8000
        resources:
          requests: { cpu: 250m, memory: 256Mi }
          limits:   { cpu: "1",  memory: 512Mi }
---
apiVersion: v1
kind: Service
metadata:
  name: vllm-router
  namespace: llm
spec:
  selector: { app: vllm-router }
  ports:
  - port: 80
    targetPort: 8000
Two things to notice (1) The RBAC block: the router watches Pods via the K8s API, so it needs explicit permission — your first taste of cluster security. (2) Image tag: verify current image/args against the router README before applying — flags evolve. Ask your teacher to check if unsure.
kubectl apply -f k8s/router.yaml
kubectl logs -n llm -l app=vllm-router -f    # expect: discovers 2 engine pods

Route through it

kill %1 2>/dev/null   # stop old port-forward
kubectl port-forward -n llm svc/vllm-router 8080:80

Second terminal — same API, now behind the router:

curl -s localhost:8080/v1/models | python3 -m json.tool
for i in 1 2 3 4; do
  curl -s localhost:8080/v1/chat/completions -H "Content-Type: application/json" \
    -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"one word: ping"}],"max_tokens":5}' | python3 -c "import json,sys; print(json.load(sys.stdin)['choices'][0]['message']['content'])"

Switch to session-sticky routing

In k8s/router.yaml change --routing-logic to session and add --session-key / x-user-id args, then:

kubectl apply -f k8s/router.yaml
kubectl rollout status deployment/vllm-router -n llm
# same header → same engine every time:
for i in 1 2 3 4; do
  curl -s localhost:8080/v1/chat/completions -H "Content-Type: application/json" \
    -H "x-user-id: giorgi" \
    -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"ping"}],"max_tokens":5}' > /dev/null
done
kubectl logs -n llm -l app=vllm --tail=10 --prefix | grep "POST /v1"

expect: all 4 requests in ONE pod's log — affinity working

The win You now run the core production-stack architecture: N engines + discovering, affinity-aware router. From here on, everything (UI, monitoring, load tests) talks to vllm-router.llm.svc.cluster.local.

Check yourself

Why does session affinity reduce time-to-first-token in chat?

Chat resends history each turn; landing on the same engine means the shared prefix's KV cache can be reused instead of recomputed — that recompute IS most of TTFT.

Why did the router Deployment need a ServiceAccount + Role?

Service discovery = querying the K8s API for Pods matching a label. Default service accounts can't. The router never restarts engines — that's the Deployment controller's job.
