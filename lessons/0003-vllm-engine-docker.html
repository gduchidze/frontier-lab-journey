The Engine: vLLM in Docker

One session without Kubernetes. You meet vLLM itself — why it's the industry default, how continuous batching and PagedAttention work, and you chat with Qwen through the OpenAI-compatible API on your own machine.

Knowledge: why vLLM won

1. The problem: GPUs (and CPUs) hate small talk

Naive serving handles one request at a time; hardware sits idle between tokens. Continuous batching fixes this: at every decoding step, the scheduler can inject new requests into the running batch and evict finished ones. No waiting for the batch to drain. This is the single biggest throughput idea in LLM serving.

2. PagedAttention: virtual memory for the KV cache

Every generated token needs attention over all previous tokens — cached as the KV cache, the dominant memory consumer. vLLM's founding idea (SOSP 2023 paper): manage KV cache in fixed-size blocks with an indirection table, like OS virtual memory pages. Result: near-zero fragmentation, memory shared between requests with common prefixes, far bigger batches.

3. One server, OpenAI's API

vllm serve MODEL starts an HTTP server implementing /v1/chat/completions, /v1/completions, /v1/models — the OpenAI-compatible API. Any OpenAI client works by changing one base URL. This is why every UI, router, and SDK plugs in for free — and why phase 8's chat UI needs zero custom code.

4. CPU backend: same engine, different metal

vLLM's CPU backend (docs) runs the same scheduler and API on x86/ARM. Two knobs matter today:

KnobMeaningOur value
VLLM_CPU_KVCACHE_SPACEGiB reserved for KV cache4
--dtype bfloat16Weight precision (M4 supports bf16)bfloat16
--max-model-lenCap context length → caps KV memory4096
Reality check CPU inference is ~10–50× slower than GPU. For a 0.8B model that's still a few tokens/sec — enough to learn every production pattern. The architecture you build is identical to the GPU version; only the image and one resource line differ.

Lab: build, serve, chat

Start the image build in the background (do this FIRST)

No official ARM CPU image exists, so we build one from vLLM's own Dockerfile.cpu. Takes 30–60 min — start it, then read on while it runs:

cd ~/PycharmProjects/AI
git clone https://github.com/vllm-project/vllm.git
cd vllm
docker build -f docker/Dockerfile.cpu -t vllm-cpu:local . > /tmp/vllm-build.log 2>&1 &
tail -f /tmp/vllm-build.log     # Ctrl+C anytime; build continues
Why build and not pull? The doronmak repo pushes its own doronmak/vllm-cpu-image for the same reason — there's no official CPU image for every platform. Check any image's architecture with docker manifest inspect IMAGE | grep architecture. An amd64 image on your M4 runs under emulation: works, but ~5× slower.

While it builds: read the engine

Read the original vLLM announcement blog (10 min) — the clearest PagedAttention explanation with figures. Then skim the vLLM reference card you'll use for the rest of the course.

Run the server

When the build finishes (docker images | grep vllm-cpu):

docker run -d --name vllm \
  -p 8000:8000 \
  -v hf-cache:/root/.cache/huggingface \
  -e VLLM_CPU_KVCACHE_SPACE=4 \
  vllm-cpu:local \
  --model Qwen/Qwen3.5-0.8B \
  --max-model-len 4096 \
  --dtype bfloat16

docker logs -f vllm      # watch model download + engine init; Ctrl+C when "Uvicorn running"

expect (first run): weights download (~2 GB), then "Application startup complete"

If Qwen3.5-0.8B errors with unsupported architecture qwen3_5 is a 2026 multimodal arch; your built image tracks vLLM main, so it should work. If not: fall back to --model Qwen/Qwen3-0.6B (text-only, long-supported), note it in the learning record, and retry the target model after git pull + rebuild later. Every other lesson works identically with either model.

Talk to it — the three endpoints

# 1. What models are loaded?
curl -s localhost:8000/v1/models | python3 -m json.tool

# 2. Chat (the endpoint everything else in this course uses)
curl -s localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen/Qwen3.5-0.8B",
    "messages": [{"role":"user","content":"In one sentence: what is PagedAttention?"}],
    "max_tokens": 60
  }' | python3 -m json.tool

# 3. Metrics — Prometheus format, phase 7 lives here
curl -s localhost:8000/metrics | grep -E "^vllm:" | head -20

expect: JSON reply from Qwen; metrics lines like vllm:num_requests_running

See continuous batching with your own eyes

Fire 5 concurrent requests, then check the running-request gauge:

for i in 1 2 3 4 5; do
  curl -s localhost:8000/v1/chat/completions -H "Content-Type: application/json" \
    -d '{"model":"Qwen/Qwen3.5-0.8B","messages":[{"role":"user","content":"Count to 30 slowly."}],"max_tokens":200}' > /dev/null &
done
sleep 3
curl -s localhost:8000/metrics | grep -E "num_requests_(running|waiting)"

expect: num_requests_running > 1 — multiple requests in ONE batch, simultaneously

The win You built the inference engine image every later phase deploys, chatted with Qwen through the exact API your UI will use, and watched continuous batching serve 5 requests at once. The engine is no longer a black box.

Stop (keep image + model cache)

docker stop vllm && docker rm vllm
# model weights persist in the named volume:
docker volume ls | grep hf-cache

Check yourself

Request B arrives while request A is mid-generation. Continuous batching means…

The scheduler recomposes the batch at every decoding step. No draining, no preemption in the normal path.

What does PagedAttention primarily eliminate?

Block-based KV storage with an indirection table = near-zero fragmentation and prefix sharing. Compute cost and weights are unchanged.

Why does capping --max-model-len help a memory-tight box?

KV cache grows with sequence length. Capping length caps the per-request worst case, letting the scheduler admit more requests safely. Weights are untouched.
