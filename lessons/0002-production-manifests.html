Production-Shaped Manifests

Lesson 1 used generator commands. Real teams commit YAML to git. Today you write manifests by hand — with namespaces, resource limits, and health probes — the exact three knobs that make or break a vLLM deployment.

Prerequisite Cluster llm-stack from Lesson 1 running (kubectl get nodes → 2 Ready). If not: kind create cluster --config k8s/kind-config.yaml.

Knowledge: five ideas

1. Everything is a manifest

A manifest is YAML declaring desired state: apiVersion + kind + metadata + spec. kubectl apply -f sends it to the API server; controllers make it true. Git-committed manifests = your infrastructure has code review, history, rollback. This is why "GitOps" is a job requirement. Full annotated skeletons: manifest anatomy reference.

2. Namespaces partition the cluster

A Namespace is a named slice of the cluster. Our stack lives in llm; monitoring will live in monitoring. DNS inside the cluster: <service>.<namespace>.svc.cluster.local — memorize this, the router and UI lessons depend on it.

3. Requests and limits

Request = what the scheduler reserves for you ("I need 4 CPUs to be placed"). Limit = the ceiling you're killed for crossing (memory → OOMKilled). (docs) On a 16 GB Mac this is not academic: vLLM CPU will want ~6 Gi. Set requests too high → Pod stuck Pending forever. This is the #1 GPU-cluster skill too — GPUs are requested the same way (nvidia.com/gpu: 1).

4. The three probes

ProbeQuestionOn failurevLLM meaning
startupFinished booting?Keep waiting (up to threshold)Model still downloading/loading — can take minutes
readinessCan you take traffic now?Removed from Service endpointsEngine busy/overloaded — stop routing here
livenessAre you alive at all?Container restartedEngine hung — kill and reload

Without a startup probe, a slow-loading model gets liveness-killed in a loop: the classic CrashLoopBackOff-on-deploy. (docs)

5. ConfigMaps and Secrets

Config lives outside the image: ConfigMap for plain settings, Secret for credentials (your Hugging Face token in phase 4). Injected as env vars or files. Never bake tokens into images or YAML committed to git.

Lab: rebuild hello, production-shaped

Same nginx as lesson 1 — but now namespaced, resource-bounded, probed, and configured via ConfigMap. Then we break it on purpose.

Create the app manifests

Save as k8s/lab2.yaml (multiple docs separated by ---):

apiVersion: v1
kind: Namespace
metadata:
  name: llm
---
apiVersion: v1
kind: ConfigMap
metadata:
  name: hello-config
  namespace: llm
data:
  index.html: |
    <h1>Served by Kubernetes, configured by ConfigMap</h1>
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello
  namespace: llm
spec:
  replicas: 2
  selector:
    matchLabels: { app: hello }
  template:
    metadata:
      labels: { app: hello }
    spec:
      containers:
      - name: web
        image: nginx:alpine
        ports:
        - containerPort: 80
        resources:
          requests: { cpu: 100m, memory: 64Mi }
          limits:   { cpu: 500m, memory: 128Mi }
        startupProbe:
          httpGet: { path: /, port: 80 }
          periodSeconds: 2
          failureThreshold: 15
        readinessProbe:
          httpGet: { path: /, port: 80 }
          periodSeconds: 5
        livenessProbe:
          httpGet: { path: /, port: 80 }
          periodSeconds: 10
        volumeMounts:
        - name: content
          mountPath: /usr/share/nginx/html
      volumes:
      - name: content
        configMap: { name: hello-config }
---
apiVersion: v1
kind: Service
metadata:
  name: hello
  namespace: llm
spec:
  selector: { app: hello }
  ports:
  - port: 80
    targetPort: 80

Apply and verify

kubectl apply -f k8s/lab2.yaml
kubectl get pods -n llm -w      # wait for 2/2 READY 1/1, Ctrl+C
kubectl port-forward -n llm svc/hello 8080:80

Second terminal:

curl -s localhost:8080

expect: the ConfigMap HTML, not the default nginx page

Break the readiness probe — watch traffic drain

kubectl exec -n llm deploy/hello -- rm /usr/share/nginx/html/index.html 2>/dev/null; \
kubectl get pods -n llm -w

Wait — that deleted from ONE pod's mounted volume? No: ConfigMap mounts are read-only. Try it — the rm fails. Good. Break it differently — point the probe at a path that doesn't exist:

kubectl patch deployment hello -n llm --type=json -p='[
 {"op":"replace","path":"/spec/template/spec/containers/0/readinessProbe/httpGet/path","value":"/nope"}]'
kubectl get pods -n llm -w

expect: new pods appear but stay READY 0/1 — rollout stalls, OLD pods keep serving

This is the safety net in action: Kubernetes refuses to replace working Pods with broken ones. Check why it's not ready:

kubectl describe pod -n llm -l app=hello | grep -A3 "Readiness"
kubectl rollout undo deployment/hello -n llm
kubectl get pods -n llm

expect: after undo, back to 2 pods READY 1/1

Break memory — meet OOMKilled

Run a Pod that allocates more than its limit:

kubectl apply -f - <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: oom-demo
  namespace: llm
spec:
  restartPolicy: Never
  containers:
  - name: stress
    image: polinux/stress
    command: ["stress", "--vm", "1", "--vm-bytes", "128M", "--vm-hang", "0"]
    resources:
      limits:
        memory: 64Mi
EOF

sleep 10 && kubectl get pod oom-demo -n llm
kubectl describe pod oom-demo -n llm | grep -A2 "Last State"

Note: older tutorials use kubectl run --limits=… — that flag was removed in kubectl 1.24. Manifests are the way (which is this lesson's whole point).

expect: STATUS OOMKilled — the exact failure you'll diagnose when vLLM exceeds its memory limit

kubectl delete pod oom-demo -n llm
The win You have now seen, hands-on, the two failure modes that dominate real vLLM operations: readiness failure → traffic drained and OOMKilled → limits too tight. In phase 4 you'll set these knobs for the actual engine.

Clean up the app, keep the namespace

kubectl delete deployment hello -n llm
kubectl delete svc hello -n llm
kubectl delete configmap hello-config -n llm

Check yourself

vLLM takes 4 minutes to load a model. Which probe prevents restart-loops during load?

The startup probe disables liveness checking until it passes. Stretching liveness works but weakens hang-detection forever; startup probes only cover boot.

A Pod requests 8 CPUs on a node with 6 free. Result?

Requests are a scheduling contract. No node can satisfy it → Pending. kubectl describe pod → Events shows "Insufficient cpu".

Readiness probe fails on 1 of 2 replicas. What happens to it?

Readiness only controls traffic. Restarts are liveness's job; replacement is the Deployment's job when Pods die.
